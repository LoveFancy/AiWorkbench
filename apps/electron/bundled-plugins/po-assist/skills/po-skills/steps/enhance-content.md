# 独立工具：enhance-content（图片增强）

**触发词：** `enhance-content` 或"内容增强"或"图片分析"

> 正常流程中，enhance-content 在 doc-convert 完成后**自动执行**，无需单独触发。
> 仅当需要对已有 `[PROD_ORI]` 文件单独重新执行图片增强时，才显式触发本工具。

## 职责边界

**AI 主导图片理解，脚本只负责确定性文件操作。**

AI 负责：

1. 读取 `references/image-classify-prompt.md`。
2. 读取完整 `[PROD_ORI]` Markdown，识别所有图片引用。
3. 在当前模型支持多模态且图片可读取时，使用多模态能力读取图片本身；运行环境提供 `Z.ai Built-in Tool: analyze_image` 时，必须优先调用 `Z.ai Built-in Tool: analyze_image`。
4. 只基于图片可见内容进行分类、命名和内容提取。
5. 构造 `--rename`、`--keep`、`--describe` 参数调用脚本。

脚本负责：

1. 图片物理重命名。
2. Markdown 图片链接同步更新。
3. 图片引用的全局搜索定位。
4. 基于 Markdown 语法结构判定插入位置（独立图片 / 表格内图片 / 行内图片）。
5. 图片内容提取说明块插入或更新。

脚本不负责：

- 不调用模型 API。
- 不判断图片内容。
- 不判断图片语义类型。
- 不校验 `--describe` 文本是否真实。

## 图片路径约束（强制）

更新 `[PROD_ORI]` 图片链接时，必须使用相对于当前 Markdown 文件所在目录的相对路径。图片在同级 `images/` 目录下时写成 `./images/<文件名>`；禁止使用绝对路径、项目根路径、`file://` 路径或 Windows 盘符路径。跨目录引用时必须使用相对当前文档目录的路径，例如 `../REFERENCES/<文档名>/images/<文件名>`；如无法稳定计算，先复制到当前文档同级 `images/` 后使用 `./images/<文件名>`。

## 核心约束：只基于图片可见内容

图片内容提取必须遵守以下硬规则：

- 必须优先调用 `Z.ai Built-in Tool: analyze_image` 或当前环境可用的等价多模态图片理解工具读取图片本身；不得只基于 Markdown 上下文、alt、文件名或附近文字生成图片说明。
- 只基于图片可见内容，不做任何推断。
- 禁止推断图片中没有体现的信息。
- 上下文只用于定位图片所属章节和识别图片引用，不用于补全图片内容。
- 文件名、章节标题、附近文字不能作为图片内容提取的事实来源。
- 看不清、被遮挡、被截断、分辨率不足或图片中未出现的信息，必须写“图片中未体现”或不写。
- 不允许根据常见产品经验补充交互、跳转、保存、提交、审批、权限、校验等规则。
- 不允许使用“可能”“应该”“通常”“用于”等推测性表达，除非图片中有明确文字、连线、状态、按钮或页面关系支撑。
- 识别目标是辅助大模型理解需求文档，不是为了前端实现或视觉还原；优先提取内容、结构、字段、流程、状态、条件、交互线索和业务逻辑。
- 颜色、字体、圆角、阴影、视觉风格、背景色等视觉样式默认不写；只有当视觉信息直接影响内容理解或状态判断时才写，例如红色错误态、灰色禁用态、选中高亮、流程连线颜色代表分支。

示例：

- 可以写：图片中可见按钮“查看”“编辑”。
- 不可以写：点击“查看”后进入详情页，除非图片中明确展示了跳转箭头、详情页标题或相关说明。

## AI 执行流程

### 阶段 A：读取规则和文档

用 `read` 工具依次读取：

1. `references/image-classify-prompt.md`
2. 输入的 `[PROD_ORI]` Markdown 文件完整内容

扫描 Markdown 中所有图片，记录：

- 图片路径
- alt 文本
- 所在章节
- 是否在表格内
- 附近文本
- 本地图片绝对路径（仅用于读取图片，不写入文档）

### 阶段 B：多模态能力分流

#### B1. 当前模型支持多模态且图片可读取

必须读取图片本身后，才能生成图片内容提取。

执行要求：

- 对每张图片读取图片本身；运行环境显示 `Z.ai Built-in Tool: analyze_image` 可用时，必须优先调用 `Z.ai Built-in Tool: analyze_image`，并以该工具返回的图片理解结果作为生成 `--describe` 的依据。
- 识别图片时围绕文档理解组织信息，重点写内容、结构、字段、流程、状态、条件、交互线索和业务逻辑。
- 避免把说明写成视觉稿审美点评或前端还原说明；颜色、字体、圆角、阴影、视觉风格、背景色等只在影响内容理解或状态判断时保留。
- 按图片类型差异化提取可见内容。
- 可以结合上下文判断重命名标题，但 `--describe` 文本只能来自图片中可见、可读、可辨认的信息。
- 如果图片看不清或无法可靠识别，说明中只写可辨认内容，并明确未体现或无法确认的信息。

#### B2. 当前模型不支持多模态或图片无法读取

降级规则：

- 不生成图片内容提取。
- 可以保留现有基于上下文的图片重命名能力，但必须标记为非图片内容提取。
- 不允许把上下文推断结果插入为“图片内容提取”。
- 默认不传 `--describe`，避免污染原文。

### 阶段 C：按图片类型提取

#### 设计稿 / 原型图

重点提取图片中可见的：

- 页面标题或模块标题
- 页面结构：筛选区、列表区、详情区、操作区、弹窗区等
- 字段、表单项、表格列名
- 按钮、链接、菜单、图标操作、批量操作、行内操作
- 状态文案、提示、错误信息、空状态
- 明确展示的交互线索：弹窗、页面连线、步骤标注、禁用态、展开收起状态

禁止提取：

- 图片中没有出现的点击后结果
- 图片中没有出现的保存、提交、审批、权限、校验规则
- 根据页面名称推断出的业务目标
- 根据按钮名称推断出的完整流程

#### 流程图 / 时序图 / 状态图 / 架构图

重点提取图片中可见的：

- 参与方、系统、角色、泳道
- 节点名称
- 连线方向和步骤顺序
- 分支条件
- 状态名称和状态流转关系
- 异常路径、回退路径、终止节点
- 输入、输出、消息、接口或事件名称

禁止补全图中没有标注的异常处理、系统内部规则、权限、数据一致性、性能要求或缺失节点。

#### 数据图 / 表格截图 / 报表

重点提取图片中可见的：

- 报表标题或图表标题
- 指标名称
- 维度、筛选条件、时间范围
- 字段列名
- 数值、单位、趋势文字
- 图例和分组

禁止推断图片中没有写明的数据口径、业务原因或趋势结论。

#### 其他图片

只描述可见对象、可读文字和图片本身直接呈现的信息。禁止推断用途或业务含义。

### 阶段 D：构造命令并执行脚本

AI 已完成图片分类、命名和可见内容提取后，直接构造参数调用脚本。**无需用 write 工具修改 `[PROD_ORI]`，脚本会自动更新图片链接并插入说明块。**

`--rename` / `--keep` 规则：

- 可以稳定命名：传 `--rename OLD NEW`
- 无法稳定命名：传 `--keep PATH`

`--describe` 规则：

- 仅当已读取图片本身并生成可见内容提取时传入。
- `PATH` 必须使用最终 Markdown 中的图片路径。
- 如果图片被重命名，`PATH` 使用重命名后的路径。
- `TEXT` 只能包含图片中可见、可读、可辨认的信息。
- 同一路径图片在 Markdown 中出现多次时，AI 只需传入一次 `--describe`；脚本会全局搜索所有出现位置并分别插入或更新说明块。

示例：

```bash
python run.py enhance-content \
    --input "<PROD_ORI路径>" \
    --rename "./images/image-001.png" "./images/[原型图]客户列表页面-01.png" \
    --describe "./images/[原型图]客户列表页面-01.png" "图片中可见页面标题为“客户列表”，包含筛选区、列表区和分页区；筛选项可见“客户名称”“客户状态”；列表操作中可见“查看”“编辑”按钮。"
```

脚本插入结果示例：

```markdown
![客户列表页面](./images/[原型图]客户列表页面-01.png)

<!-- image-desc:start ./images/[原型图]客户列表页面-01.png -->
> 图片内容提取：图片中可见页面标题为“客户列表”，包含筛选区、列表区和分页区；筛选项可见“客户名称”“客户状态”；列表操作中可见“查看”“编辑”按钮。
<!-- image-desc:end -->
```

## 分批调用规则

当 rename 总条数 > 20 时强制分批：

- 每批不超过 15 条 `--rename`
- 每批使用同一个 `--input`
- `--describe` 应与对应最终图片路径在同一批或后续批次传入
- 脚本具备幂等性，已完成的说明块会更新而不是重复插入

## 脚本输出

脚本输出统计信息：

```text
OUTPUT_FILE=<路径>
RENAMED=<已重命名数量>
KEPT=<保留数量>
DESCRIBED=<有匹配并处理的 describe 路径数量>
FAILED=<失败数量>
LINKS_UPDATED=<图片链接替换次数>
DESCRIPTIONS_UPDATED=<说明块插入或更新次数>
```

## 降级规则

- 文档没有图片：直接跳过，不报错。
- 无法读取图片：不传 `--describe`，不插入图片内容提取。
- 无法判断某张图的命名：`--keep`，保留原文件名和原链接。
- 某张图重命名失败：不中断整个步骤，失败计入统计。
- `--describe` 路径未在 Markdown 中找到：不中断整个步骤，失败计入统计。

完成后输出：

```text
✅ enhance-content 完成！
文件：<路径>

处理结果：
- 已重命名：Y 张
- 保留原名：Z 张
- 已插入 / 更新图片内容提取：D 处
- 处理失败：N 项
```
