# brainstorming：产品阶段头脑风暴与需求澄清

**触发词：** `brainstorming`、"头脑风暴"、"需求头脑风暴"、"需求澄清"、"需求梳理"、"产品梳理"、"先梳理"、"先澄清"

**职责：** 在产品设计阶段把模糊想法澄清成可进入 PRD 的结构化输入。只做产品阶段澄清，不引入 Visual Companion，不做工程方案，不生成实现计划。

边界：`需求澄清` 属于 PRD 形成前的产品讨论，进入 brainstorming；`需求评审` 属于 PRD 形成后的质量检查，进入 req-review。

> 参考 superpowers brainstorming 的核心思路：先理解、再澄清、再收敛。
> 本文件不是照搬其工程化流程，只保留产品阶段需要的轻量约束。

---

## 输入

```text
/brainstorming <产品想法 / 业务问题 / 需求背景> [--save] [--continue-prd]
```

默认只在对话中完成头脑风暴，不写文件，不生成 PRD。

---

## 执行方式

你需要像产品搭档一样和用户讨论，而不是机械填表。

brainstorming 是 AI 对话流程，不是脚本命令。不要调用 `run.py brainstorming`，因为 `run.py` 没有这个子命令；如需保存纪要，只在完成澄清后按本文档规则写入 `[BRAINSTORM]需求澄清纪要.md`。

1. 提问前必须先理解项目 / 文档上下文。先阅读用户提供的文档、片段、链接摘要或项目背景；如果用户给了本地文件路径，先读取相关内容再提问。
   - 识别到 REQID 或 `newreq/<REQID>` 路径时，必须先理解整个需求空间，而不是只看用户当前提到的单个文件。
   - 需求空间上下文至少包括：`newreq/<REQID>/req.md`、`newreq/<REQID>/PRODUCT_DESIGN/` 下的 PRD / `[PROD_ORI]` / brainstorming 纪要，以及 `newreq/<REQID>/REFERENCES/` 下的参考文档和摘要。
   - 参考资料较多时，先用文件名、标题、摘要、一级/二级标题建立索引，再有针对性读取和当前问题相关的内容；不要盲目全文塞入上下文。
2. 先用自己的判断理解用户的原始想法和上下文，识别最影响 PRD 质量的缺口。
3. 首次提问前，先输出一段简短的上下文理解，说明你已经理解的业务对象、目标、已知信息和最大不确定点。
4. 如果上下文文档已经明确回答了某个问题，先直接把文档结论列出来，再决定是否还需要追问；不要对已回答内容重复发问。
5. 必要时提问，默认一次只问 1 个问题，先问当前最影响方向判断的问题。
6. 每个问题优先提供 2-4 个选项。必须优先使用宿主环境提供的按钮 / 快捷回复 / 可点击选项控件承载完整选项文本，让用户直接点击选择；只有在确认当前工具环境不支持按钮或快捷回复控件时，才退回到文本选项。文本退回格式使用 A/B/C/D，并明确提示用户可以直接回复选项字母或补充“其他”。
7. 问题之间有依赖关系时必须按顺序提问，不能把后续依赖前置答案的问题一次性抛给用户。
8. 只有用户明确说“可以一次多问 / 快速梳理 / 批量问”时，才允许每轮最多问 3 个问题。
9. 用户回答后，先用一句话确认你吸收了什么，再继续问下一个最关键问题。
10. 收敛判断：当业务对象、目标用户、核心场景、本期范围和暂不处理项都已明确，且继续追问只能得到重复信息或边缘细节时，停止追问并进入方案收敛；如果关键维度仍缺失、且这些缺口会改变方案方向，继续问下一题。
11. 通常 1-2 轮即可收敛；如果用户要求先给方案，就直接给方案，把不确定内容放入待确认问题。
12. 根据需求特点给出 1-3 个产品方案。可以按 MVP / 完整闭环 / 增强版组织，也可以按互斥路径组织，不强制套模板。
13. 给出推荐方案和理由，说明主要风险或待确认点。

提问示例：

```text
先确认第一个关键问题：这个平台主要面向谁？

A. B2C 自营平台：平台自己卖商品给消费者，类似京东自营
B. C2C / 多商家平台：商家入驻售卖，类似淘宝
C. B2B 采购平台：企业客户采购、合同、账期、审批更重要
D. 其他：你可以直接描述

如果当前界面没有显示可点击按钮，你可以直接回复 A/B/C/D，也可以补充自己的描述。
```

优先澄清这些产品问题，但不要逐项机械询问：为什么做、给谁用、解决什么问题、本期范围、核心流程、关键规则、成功标准、依赖系统。

## 输出结构

输出结论时，读取 `references/brainstorming-template.md` 作为格式参考。

纪要重点记录：

- 原始想法
- 你问了什么
- 用户怎么回答
- 基于回答形成了什么结论
- 方案方向和推荐方向
- 仍需确认的问题
- 给后续 `prd-write` 使用的“PRD 写作参考”

不需要强制输出固定字段表。信息不明确时写“待确认”，后续 `prd-write` 不得自行编造。

## 可选保存

`[BRAINSTORM]需求澄清纪要.md` 不是必须生成。

生成规则：

| 场景 | 是否生成文件 |
|------|--------------|
| 用户只想快速讨论 | 不生成 |
| 用户说"保存一下 / 沉淀成文档 / 后面继续写 PRD" | 生成 |
| `prd-write` 自动串联 brainstorming | 默认不生成 |
| `prd-write` 自动串联且用户要求保留过程记录 | 生成 |

保存路径：

```text
{DESIGN_DIR}/[BRAINSTORM]需求澄清纪要.md
```

REQID 规则：

- 用户输入中已有 REQID：直接使用
- 独立 brainstorming 且用户要求保存但未提供 REQID：先执行 `newreq --init-only` 获取正式需求空间
- `prd-write` 自动串联：复用 `prd-write` 阶段 A 确认或生成的 REQID
- 不保存文件：不生成 REQID

---

## 后续动作

独立 brainstorming 完成后：

- 默认询问用户是否继续进入 `/prd-write`
- 用户确认后，将本轮头脑风暴纪要和“PRD 写作参考”作为主输入继续执行 `prd-write`
- 用户未确认时，不自动生成 PRD

如果输入中包含 `--continue-prd`、"继续写 PRD"、"再生成 PRD"，在输出头脑风暴纪要后继续 `prd-write`。
