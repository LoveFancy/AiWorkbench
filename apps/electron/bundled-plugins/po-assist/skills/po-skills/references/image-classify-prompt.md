# 图片可见内容提取提示词

你正在处理需求文档中的图片。你的任务是：在当前模型支持多模态且图片可读取时，基于图片本身的可见内容进行分类、命名和内容提取，并生成可传给 `enhance-content` 脚本的处理结果。

识别图片的目标是辅助大模型理解需求文档，不是为了前端实现或视觉还原。输出应更多关注内容、结构、字段、流程、状态、条件、交互线索和业务逻辑；颜色、字体、圆角、阴影、视觉风格、背景色等视觉样式默认不写，只有当视觉信息直接影响内容理解或状态判断时才写。

## 任务目标

对每一张图片输出结构化判断：

1. 判断图片类型：
   - `设计稿`
   - `流程图`
   - `数据图`
   - `其他`
   - `无法判断`
2. 生成图片功能标题 `title`
3. 生成最终文件名 `new_filename`
4. 给出处理决策 `decision`
5. 生成图片可见内容提取 `description_for_insert`
6. 标明不可见或无法确认的信息 `not_visible_or_unclear`

## 输入信息

你会拿到以下信息：

- 图片当前文件名
- Markdown 中的图片路径
- 图片 `alt`
- 所在章节标题
- 图片附近文本上下文
- 图片本身

上下文只用于定位图片所在位置、辅助识别当前图片与文档关系，不得用于补全图片内容。

## 核心硬约束

- 运行环境提供 `Z.ai Built-in Tool: analyze_image` 时，必须优先调用 `Z.ai Built-in Tool: analyze_image` 读取图片本身；没有该工具时，使用当前环境可用的等价多模态图片理解能力。
- 必须读取图片本身后，才能生成 `description_for_insert`。
- 不得只基于 Markdown 上下文、alt、文件名或附近文字生成图片说明。
- 只基于图片可见内容，不做任何推断。
- 禁止推断图片中没有体现的信息。
- 不能从 Markdown 上下文推断图片内容。
- 上下文只用于定位，不用于补充事实。
- 文件名、章节标题、附近文字不能作为图片内容提取的事实来源。
- 只能输出图片中明确可见、可读、可辨认的信息。
- 看不清、被遮挡、被截断、分辨率不足或图片中未出现的信息，必须放入 `not_visible_or_unclear`，或不写。
- 不允许根据常见产品经验补充隐含逻辑。
- 不允许使用“可能”“应该”“通常”“用于”等推测性表达，除非图片中有明确文字、连线、状态、按钮或页面关系支撑。
- 识别目标是辅助大模型理解需求文档，不是为了前端实现或视觉还原；优先提取内容、结构、字段、流程、状态、条件、交互线索和业务逻辑。
- 颜色、字体、圆角、阴影、视觉风格、背景色等视觉样式默认不写；只有当视觉信息直接影响内容理解或状态判断时才写，例如红色错误态、灰色禁用态、选中高亮、流程连线颜色代表分支。

示例：

- 可以写：图片中可见按钮“查看”“编辑”。
- 不可以写：点击“查看”后进入详情页，除非图片中明确展示了跳转箭头、详情页标题或相关说明。

## 执行顺序（强制）

1. 扫描 `[PROD_ORI]` 文档，列出所有图片路径和所在位置。
2. 检查当前模型是否支持读取图片本身。
3. 支持多模态且图片可读取时，逐张读取图片本身；运行环境提供 `Z.ai Built-in Tool: analyze_image` 时，必须优先调用该工具。
4. 按图片类型差异化提取可见内容。
5. 生成图片命名和说明结果。
6. 如果当前模型不支持多模态，或图片无法读取，不生成 `description_for_insert`。

## 图片类型与提取重点

### 1. 设计稿

适用对象：

- 页面设计稿
- 产品原型图
- 系统截图
- 表单、列表、详情页、弹窗、看板、移动端页面

重点提取图片中可见的：

- 页面标题或模块标题
- 页面结构：筛选区、列表区、详情区、操作区、弹窗区等
- 字段、表单项、表格列名
- 按钮、链接、菜单、图标操作、批量操作、行内操作
- 状态文案、提示、错误信息、空状态
- 明确展示的交互线索：弹窗、页面连线、步骤标注、禁用态、展开收起状态
- 图片中明确呈现的业务逻辑：条件、分支、步骤顺序、前后页面关系、字段之间的可见约束

禁止提取：

- 图片中没有出现的点击后结果
- 图片中没有出现的保存、提交、审批、权限、校验规则
- 根据页面名称推断出的业务目标
- 根据按钮名称推断出的完整流程
- 与需求理解无关的颜色、字体、圆角、阴影、视觉风格、背景色等视觉样式

### 2. 流程图

适用对象：

- 流程图
- 时序图
- 状态流转图
- 泳道图
- 架构图
- 调用链路图

重点提取图片中可见的：

- 参与方、系统、角色、泳道
- 节点名称
- 连线方向和步骤顺序
- 分支条件
- 状态名称和状态流转关系
- 异常路径、回退路径、终止节点
- 输入、输出、消息、接口或事件名称
- 明确展示的处理逻辑、判断条件、前后依赖关系

禁止补全图中没有标注的异常处理、系统内部规则、权限、数据一致性、性能要求或缺失节点。

### 3. 数据图

适用对象：

- 数据看板
- 报表截图
- 统计图
- 表格截图
- 监控面板

重点提取图片中可见的：

- 报表标题或图表标题
- 指标名称
- 维度、筛选条件、时间范围
- 字段列名
- 数值、单位、趋势文字
- 图例和分组

禁止推断图片中没有写明的数据口径、业务原因或趋势结论。

### 4. 其他

只描述可见对象、可读文字和图片本身直接呈现的信息。禁止推断用途或业务含义。

## 命名规则

`new_filename` 使用以下格式：

```text
[图片类型]功能描述-01.扩展名
```

兼容现有图片前缀：

- `设计稿` 使用 `[原型图]`
- `流程图` 使用 `[流程图]`
- `数据图` 使用 `[其他]`
- `其他` 使用 `[其他]`

示例：

- `[原型图]客户列表页面-01.png`
- `[流程图]审批状态流转-01.png`
- `[其他]客户数据报表-01.jpg`

如果无法稳定命名：

- `decision` 使用 `keep`
- `title` 使用空字符串
- `new_filename` 使用空字符串
- 不要强行生成“图片1”“截图2”等无语义名称

## description_for_insert 规则

`description_for_insert` 是将插入到 Markdown 图片位置的文本，必须满足：

- 只汇总图片中可见、可读、可辨认的信息。
- 不引入图片中不存在的新事实。
- 不复述 Markdown 上下文中的推断。
- 不写产品经验、业务目的或隐含流程。
- 信息不足时明确写“图片中未体现”或“无法可靠辨认”。

推荐写法：

```text
图片中可见页面标题为“客户列表”，包含筛选区、列表区和分页区；筛选项可见“客户名称”“客户状态”；列表操作中可见“查看”“编辑”按钮。图片中未体现点击按钮后的跳转页面。
```

脚本会自动加上 `> 图片内容提取：` 前缀；你传给 `--describe` 的文本可以不带此前缀。

## 输出格式

不要输出 JSON，不要输出代码块，不要输出过程解释。

请只输出 Markdown 的“图片分析结果”表格，每行对应一张图片，至少包含这些列：

- `原图片地址`
- `图片类型`
- `图片标题`
- `可见内容提取`
- `未体现或无法确认`
- `新文件名`
- `处理决策`
- `describe_path`
- `describe_text`
- `所在章节`
- `判定依据`

约束：

- `处理决策` 只允许填写 `renamed` 或 `keep`。
- `describe_path` 必须使用最终 Markdown 中的图片路径；如果图片被重命名，使用新路径。
- `describe_text` 只在已经读取图片本身并生成可见内容提取时填写。
- 当前模型不支持多模态或图片无法读取时，`describe_text` 留空。
- `判定依据` 只能说明图片中的可见元素，不得引用上下文推断图片内容。
- 这张表格用于直接构造 Python 脚本参数，不要先展示给用户。
