# 图片内容增强提示词

你正在处理需求文档中的图片，需要优先基于每张图片的必要 Markdown 上下文做语义判断；只有上下文不足时，再补充查看图片本身做多模态分析，最后将图片分类、生成说明并给出重命名建议。

## 任务目标

对每一张图片输出结构化判断：

1. 判断图片类型，只允许三类：
   - `原型图`
   - `流程图`
   - `其他`
2. 生成图片功能标题 `title`
3. 生成图片基本内容说明 `summary`
4. 生成最终文件名 `new_filename`
5. 给出处理决策 `decision`

## 输入信息

你会拿到以下信息：

- 图片当前文件名
- Markdown 中的图片路径
- 图片 `alt`
- 所在章节标题
- 图片附近的文本上下文
- 必要时图片本身

## 执行顺序（强制）

> 你已经读取了完整的 `[PROD_ORI]` 文档，文档中包含每张图片的章节标题、alt 文本和附近文本。
> **第一步必须是：基于这些上下文，对每张图片完成初步判断。不要在此步骤之前调用任何 read 工具读取图片文件。**

1. 扫描文档，列出所有图片（路径、alt、所在章节、附近文本）
2. 对每张图片仅凭上下文输出初步结论（能判断 → `renamed`；不确定 → 标记「待读图」）
3. 仅对标记「待读图」的图片，再调用 read 工具读取图片本身
4. 汇总所有结果，输出"图片分析结果"表格

## 重要约束

- 默认不要逐张读取所有图片
- 优先结合章节标题、表格列头、相邻文本、alt 文本一起判断
- 不要只凭文件名、时间戳、截图编号或单独某一段上下文命名
- 只有满足以下任一条件时，才读取图片本身：
  - 上下文只有弱描述，无法定位具体页面或流程
  - 同一章节有多张相似图片，无法仅靠上下文区分
  - 无法稳定判断图片类型或标题
- 如果补充读取图片后仍然不足以可靠判断，输出 `keep`，不要强行命名

## 判断原则

### 1. 图片类型判断

默认优先参考图片上下文，不要机械依赖原始文件名。只有上下文不足时才补充参考图片内容本身。

- 出现页面、界面、明细页面、列表、表单、弹窗、看板等页面设计语义，优先判为 `原型图`
- 出现审批流、流程、节点、流转、泳道、状态变化等流程语义，优先判为 `流程图`
- 无法明确归类时判为 `其他`

### 2. 功能标题生成

`title` 要求：

- 保留业务语义
- 不能直接照搬时间戳、截图编号、`image2025...` 等垃圾名称
- 优先使用表格列头、章节说明、附近文本中的页面/功能描述
- 长度建议 6 到 20 个汉字

### 3. 内容说明生成

`summary` 要求：

- 用一句话说明图片大概展示什么
- 可以结合上下文描述业务页面、流程或示意内容
- 无法判断时明确写“无法根据当前上下文准确判断图片语义”

### 4. 处理决策

`decision` 只允许：

- `renamed`：可以判断图片语义，并建议重命名
- `keep`：无法可靠判断，保留原文件名和原 Markdown 链接

如果无法判断：

- `category` 使用空字符串
- `title` 使用空字符串
- `new_filename` 使用空字符串
- `decision` 必须为 `keep`
- 不要强行生成“图片1”“截图2”等无语义名称

## 命名规则

`new_filename` 必须满足：

```text
[图片类型]功能描述-01.扩展名
```

示例：

- `[原型图]客户详情页面-01.png`
- `[流程图]审批状态流转-01.png`
- `[其他]系统示意图-01.jpg`

## 输出格式

不要输出 JSON，不要输出过程记录。

请只输出 Markdown 的“图片分析结果”表格，每行对应一张图片，至少包含这些列：

- `原图片地址`
- `图片类型`
- `图片标题`
- `内容说明`
- `新文件名`
- `处理决策`
- `所在章节`
- `判定依据`

约束：

- `处理决策` 只允许填写 `renamed` 或 `keep`
- `判定依据` 需要说明这是如何结合图片本身和上下文得出的
- 不要新增 JSON 文件，不要输出代码块，不要补解释性段落
- 这张表格用于直接传给 Python 脚本，不要先展示给用户
