# prd-write：从零合成首版草稿 PRD

**触发词：** `prd-write` 或"编写PRD"或"写PRD"或"创建PRD"

**角色设定：** 你是一位经验丰富的产品经理，负责根据用户输入和参考资料，为本次产品或功能编写详细、清晰、可落地的产品需求文档。该文档将作为正式需求说明，帮助业务、产品、设计、研发、测试等相关人员达成一致，并指导后续开发和验收工作。

**职责：** 从用户自由文字描述和可选的关联文档中，合成首版草稿 PRD。

写作时要站在产品负责人的角度，主动补齐背景、目标、用户、范围、流程、规则、验收标准和待确认问题；没有依据的内容不得编造，必须标记为 `<!-- 待补充 -->`。

"首版草稿"指：
- 输出文件结构与最终 PRD 完全一致
- 内容允许存在 `<!-- 待补充 -->`、待确认项
- 目标是帮助用户快速形成可审查、可迭代的第一版

---

## 阶段 A：输入解析与需求空间创建

### Step 1：解析用户输入

| 类型 | 识别模式 | 用途 |
|------|----------|------|
| Wiki URL | `http://wiki...pageId=...` | 阶段 B 用 doc-convert 转换 |
| EIP 文档 | `eip.htsc.com.cn/htscPortalDocs/` | 暂不支持自动下载；登记为待补充并提示用户手工下载 |
| LinkApp 短链 | `linkapp.htsc.com.cn/S/` | 暂不支持自动下载；登记为待补充并提示用户手工下载 |
| 本地文档 | `.docx` / `.pdf` / `.doc` | 阶段 B 调用 `run.py doc-to-md --file`，使用 markitdown 转换 |
| REQID | 大写字母字段 + `-` + 数字，正则 `^[A-Z]+-\d+$`，如 `TAILOR-124` | 需求空间命名 |
| 自然语言描述 | 剩余所有文字 | PRD 主素材 |

### Step 2：先创建或复用正式需求空间

先创建或复用正式需求空间，再转换和理解可访问的关联文档。云文档 URL 暂不支持自动下载，不要先转换 Wiki / 本地文档后再决定需求目录。

```
├─ 输入中已含 REQID → 执行 `newreq --reqid <REQID>` 创建或复用正式需求空间
└─ 无 REQID → 执行 `newreq --mock` 生成 mock 需求编号并创建正式需求空间
```

无明确需求编号时，不追问，直接执行 `newreq --mock`。mock 编号是临时正式工作空间编号，后续如用户提供真实需求编号，再按迁移/重命名流程处理。

### Step 3：读取目录上下文

```bash
python ${CLAUDE_PLUGIN_ROOT}/skills/po-skills/run.py newreq --reqid "{REQID}"
python ${CLAUDE_PLUGIN_ROOT}/skills/po-skills/run.py newreq --title "{标题}" --mock
```

从 stdout 读取 `REQID`、`REQ_ROOT`、`DESIGN_DIR`、`REFERENCES_DIR`、`REFERENCE_IMAGES_DIR`、`IMAGES_DIR`、`NEXT_STEP`。后续路径一律使用这些变量，不自行 `mkdir -p`。

### Step 4：关联文档确认

若未发现任何 Wiki URL、EIP / LinkApp URL 或本地文档 → 询问："是否有需要关联的参考文档（Wiki 链接、EIP/LinkApp 云文档或本地 docx/pdf 文件）？没有请回复'无'。"

## 阶段 A.5：需求成熟度判断与内部 brainstorming 串联

> 仅在用户输入不足以直接生成可用 PRD 时执行。该阶段只做产品阶段需求澄清，不做工程方案，不生成实现计划，不引入 Visual Companion。
> 此阶段发生在 `prd-write` 内部，且必须复用阶段 A 已创建或复用的 `{REQID}`、`{DESIGN_DIR}` 和 `{REFERENCES_DIR}`；不要把 `brainstorming` 当成 `newreq` 的前置步骤。

### Step 1：判断是否需要执行内部 brainstorming

满足以下任一条件时，在 `prd-write` 内部执行 `steps/brainstorming.md`：

- 用户明确说"先头脑风暴"、"先梳理"、"先澄清"、"brainstorm"、"需求澄清"、"帮我梳理"
- 纯自然语言输入少于 120 个中文字符，且没有 Wiki URL、EIP / LinkApp URL、本地文档作为主参考
- 在以下 4 个关键维度中缺少 2 项及以上：目标用户、业务目标、范围边界、核心流程
- 输入不包含结构化章节标题，例如"背景"、"目标"、"范围"、"流程"、"验收标准"
- 用户输入包含想法型表达，例如"想做一个"、"考虑做"、"有没有必要做"、"看看能不能做"

### Step 2：判断是否必须跳过 brainstorming

满足以下任一条件时，跳过 brainstorming，直接进入阶段 B/C：

- 用户明确说"直接写 PRD"、"不用澄清"、"不要追问"、"先出草稿"
- 输入已经覆盖背景、目标、用户、范围、流程、约束等主要 PRD 信息
- 主输入是 Wiki URL、EIP / LinkApp URL 或本地文档，用户要求基于资料生成 PRD
- 用户正在基于已有 `[PROD_ORI]` 或 `[PROD_FORMAT]` 文件继续处理

冲突裁决：如果同一句话同时包含"先梳理/需求澄清"和"写 PRD"，且未出现"直接写 PRD/不用澄清"，在 `prd-write` 内部执行 brainstorming，得到头脑风暴纪要后继续生成 PRD。

### Step 3：执行 brainstorming 子流程

需要 brainstorming 时：

1. 静默加载 `${CLAUDE_PLUGIN_ROOT}/skills/po-skills/steps/brainstorming.md`
2. 按产品澄清流程进行 2-3 轮内的收敛
3. 输出头脑风暴纪要，重点记录问题、用户答复、形成的结论、待确认问题和“PRD 写作参考”
4. 默认不生成 `[BRAINSTORM]需求澄清纪要.md`
5. 若用户明确要求"保存 / 沉淀成文档 / 保留过程记录"，写入 `{DESIGN_DIR}/[BRAINSTORM]需求澄清纪要.md`
6. 继续进入阶段 B/C

头脑风暴纪要是 `brainstorming -> prd-write` 的输入接口，不要求固定字段表。`prd-write` 应优先读取其中的“PRD 写作参考”、问答结论和待确认问题；待确认内容在 PRD 中标记为 `<!-- 待补充 -->`，不得自行编造。

## 阶段 B：关联文档转换

> 无需关联文档时跳过。
>
> **强制规则：**
> - 只要输入中出现 1 个及以上关联文档，阶段 C 生成 PRD 时就必须在 `1.2 关联需求` 表中逐条登记。
> - 若输入中出现多个关联文档，禁止只做摘要描述，必须一条文档一行，至少包含：关联类型、标题、ID/来源、关联说明/用途、当前状态。
> - 文档尚未下载完成时也要先登记，`当前状态` 填 `待补充`；下载并纳入分析后更新为 `已纳入`。

对每个文档：

**Wiki URL：**

Wiki URL 在 `prd-write` 中属于关联文档，不是主需求文档。必须转换到 `{REFERENCES_DIR}`，不得使用 `--reqid` 输出到 `{DESIGN_DIR}`；`{DESIGN_DIR}` 只保存最终 PRD、主需求文档、Story 文档和主文档图片。

```bash
python ${CLAUDE_PLUGIN_ROOT}/skills/po-skills/run.py doc-convert --url "<URL>" --output-dir "{REFERENCES_DIR}"
```

**EIP / LinkApp 云文档：**

暂不支持自动下载。不要派发 `cloud-doc-downloader`，不要同步执行 `steps/doc-browser-download.md`，不要尝试通过 chrome-devtools 点击下载。

处理规则：

1. 在 `1.2 关联需求` 表中登记该 URL，`关联类型` 填 `云文档`，`关联说明` 标明 `待补充`。
2. 输出稳定提示：

```text
CLOUD_DOC_MANUAL_DOWNLOAD_REQUIRED：暂不支持自动下载 EIP/LinkApp 云文档。
请先在浏览器中手工下载原始 .doc/.docx/.pdf 文件，然后把本地文件路径发给我；我会继续执行 doc-to-md 转换并纳入 PRD。
```

3. 继续基于已有自然语言、Wiki 或本地资料生成 PRD V0；不要因为云文档未下载而阻断首版草稿。
4. 用户后续提供手工下载后的本地文件路径时，按“本地文档”流程处理，并把 `1.2 关联需求` 中对应记录更新为 `已纳入`。

**本地文档：**
```bash
python ${CLAUDE_PLUGIN_ROOT}/skills/po-skills/run.py doc-to-md --file "<路径>" --output-dir "{REFERENCES_DIR}"
```

转换后如有图片引用，执行 enhance-content。

**图片路径约束（强制）：** 首版 PRD、补充后的 PRD 以及后续引用的图片，必须使用相对于当前 Markdown 文件所在目录的相对路径。禁止使用绝对路径、项目根路径、`file://` 路径或 Windows 盘符路径。引用 `{DESIGN_DIR}/images/` 下图片时使用 `./images/<文件名>`；引用 `{REFERENCES_DIR}/images/` 下图片时，必须从 PRD 文件所在目录计算路径，例如 `../references/images/<文件名>`。如果无法稳定计算跨目录相对路径，应先复制图片到 `{DESIGN_DIR}/images/`，再使用 `./images/<文件名>`。

## 阶段 C：生成首版草稿 PRD

> 执行过程中最多向用户输出 4 句进展提示，用产品语言，不暴露内部步骤。

1. 输出 `"正在分析你的需求描述…"`
2. `read` 模板：`${CLAUDE_PLUGIN_ROOT}/skills/po-skills/references/prd-template.md`
3. 输出 `"已确定文档结构，正在填充 PRD 内容…"`
4. 写作前先在内部完成“四问分析”，再生成 PRD。分析内容要融入正文，不要单独输出长篇推理：
   - 我们正在解决什么问题？
   - 我们是在为谁解决问题？
   - 我们如何衡量成功？
   - 我们的限制条件和假设是什么？
5. 用户输入为主素材，按模板结构生成完整 PRD；如阶段 A.5 已输出头脑风暴纪要，必须优先消费其中的“PRD 写作参考”、问答结论和待确认问题，再结合用户原始输入和 references 生成 PRD；用户没说的不编，缺什么标 `<!-- 待补充 -->`
   - 输出必须是格式良好的 Markdown 文档，标题层级清晰，段落短，表格完整。
   - 用通俗易懂的语言写，按小学毕业生能读懂的水平表达；句子要短，少用专业术语。
   - 必须使用专业术语时，第一次出现要用一句简单的话解释。
   - 如果用户提供了本地文件、Wiki、云文档或参考资料，必须先仔细阅读/转换/摘要后再写，不得只根据文件名猜测。
   - 验收用例统一输出到“整体验收用例”章节
   - 不要在每个功能点下面单独生成验收用例小节
   - 模板章节可以按需裁剪。简单 Story 只要用文字、表格和验收用例能说明白，就不要强制补流程图、时序图、架构图或占位原型图。
   - 流程图仅在存在多步骤流转、审批/工作流、跨页面跳转、状态变化或异常分支时添加；时序图仅在存在多系统调用、异步回调、消息流转或接口交互顺序时添加；架构图仅在跨系统协作复杂且文字难以说明时添加。
   - 不要输出空图、占位图、`path/to/mockup.png`、无实际信息的 Mermaid。若某节不适用，删除该节或用一两句话说明“不涉及复杂流程/交互”。
   - 头脑风暴纪要中标记为"待确认"的内容，PRD 对应位置写 `<!-- 待补充 -->`，不得自行编造
   - 只要本轮输入中含关联文档，必须在 `1.2 关联需求` 表中逐条落表
   - 当输入中有多个关联文档时，`1.2 关联需求` 必须一条文档一行，不得省略、不得合并成一句话
   - 存量文档 + 新变更诉求场景：不要把存量文档里的老逻辑完整复述成新 PRD 正文。存量文档只作为背景、术语、现状和差异判断依据；新 PRD 正文重点写新增/变更内容、触发条件、影响范围、改前改后差异、兼容规则和验收标准。
   - 如果必须引用老逻辑，只保留理解本次变更所必需的现状摘要，并在对应章节标明“现状/本次变更/影响说明”，避免把历史功能说明改写成整篇新需求。
   - 图片路径必须相对于当前 Markdown 文件所在目录；不得照抄关联文档中的图片路径。PRD 同级图片使用 `./images/<文件名>`；引用 references 图片时使用 `../references/images/<文件名>`，或先复制到 PRD 同级 `images/` 后改写为 `./images/<文件名>`。
6. 输出 `"正在保存 PRD 文档…"`
7. 一次 `write` 写入：`{DESIGN_DIR}/[PROD_FORMAT]{标题}.md`（标题不超过 20 字）
8. 完成输出：`首版草稿 PRD 已生成：<路径>。下一步建议：/req-review 进行质量审查。需要吗？`

## 阶段 D：补充文档（可选）

用户补充新文档时：
1. 处理新文档（→ references/）
2. 重新读取所有 references + 现有 PRD
3. **先更新 `1.2 关联需求` 表**：每个新补充的文档必须新增一行；如果原表中已有该文档，则更新其 `当前状态`、`关联说明/用途`、`ID/来源`
4. 再列出受影响的章节，**征得用户确认后**用 Edit 工具做定向替换

补充文档场景下的硬约束：
- 不允许只更新正文、不更新 `1.2 关联需求` 表
- 用户后续对话中追加的 Wiki / 云文档 / 本地文档，也按首次输入同样规则处理
- 若文档已进入 `references/` 但暂未消费到正文，仍需登记到 `1.2 关联需求` 表，`关联说明` 标记为 `待补充`

## 阶段 E：后续引导

PRD 完成后主动引导：`/req-review {DESIGN_DIR}/[PROD_FORMAT]{标题}.md`
