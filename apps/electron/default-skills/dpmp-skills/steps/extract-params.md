# 参数提取指导 - 为 LLM 提供参数提取指导

**触发词：** `提取参数`、`参数指导`、`extract-params`、`参数提取`

**职责：** 为 LLM 提供从自然语言中提取结构化参数的指导，不直接解析自然语言，而是提供模板和指导让 LLM 来理解用户意图。

## 使用场景

当用户用自然语言描述需求时，AI 使用此工具获取参数提取指导，然后根据指导从用户输入中提取结构化参数。

## AI 执行流程

### Phase A: 调用参数提取指导

```bash
python3 ${CLAUDE_PLUGIN_ROOT}/skills/dpmp-skills/run.py extract-params \
    --text "用户输入的自然语言描述"
```

### Phase B: 分析指导信息

工具返回的指导信息包含：

1. **用户输入**：原始的自然语言描述
2. **任务说明**：参数提取的任务描述
3. **提取指导**：分步指导 LLM 如何提取参数
4. **建议的模板**：根据输入内容推荐的操作模板
5. **模板信息**：包含实体类型、操作类型、必需字段、可选字段
6. **字段描述**：所有字段的中文描述和说明

### Phase C: 根据指导提取参数

LLM 根据指导信息执行：
1. 判断用户意图：创建、更新、查询还是状态更新？
2. 判断操作对象：REQ（需求）还是 STORY（任务）？
3. 根据意图和对象选择合适的模板
4. 从用户输入中提取对应的参数
5. 标记哪些参数是缺失的（需要询问用户）

### Phase D: 向用户展示提取结果

使用工具提供的格式化方法展示提取结果：

```
📋 参数确认：

✅ 已提取的参数：
  • 产品空间key: PRODU
  • 名称: 客户管理系统
  • 优先级: 高(一般)
  • 描述: 需要支持客户标签管理功能

❓ 需要确认的参数：
  • 需求提出人：
  • 报告人：
  • 经办人：

请确认以上参数是否正确，或补充缺失的信息。
```

## 支持的模板类型

### 1. 创建类模板
- **create_req**: 创建 REQ（需求）
- **create_story**: 创建 STORY（任务）

### 2. 更新类模板
- **update_req**: 更新 REQ
- **update_story**: 更新 STORY
- **update_status**: 更新状态（REQ 或 STORY）

### 3. 查询类模板
- **query_req**: 查询 REQ
- **query_story**: 查询 STORY

## 环境变量配置

所有配置通过 `check-config` 动态检测，不要手动读取 `.env`。参考文件：`.env.example`

## 错误处理

1. **输入为空**：提示用户提供自然语言描述
2. **无法识别意图**：返回通用的提取指导，让 LLM 自行判断
3. **网络错误**：返回本地缓存的模板信息
4. **配置错误**：检查环境变量是否正确配置

## 最佳实践

1. **先提取后确认**：先提取参数，再向用户确认，避免错误
2. **标记缺失字段**：明确告诉用户哪些信息需要补充
3. **使用字段描述**：使用中文描述让用户更容易理解
4. **多轮对话**：对于复杂需求，支持多轮对话补充信息
5. **保存上下文**：在对话中保存已确认的参数，避免重复询问

## 与 CLI 命令的集成

参数提取完成后，AI 可以生成对应的 CLI 命令：

```bash
# 根据提取的参数生成创建 REQ 命令
python3 ${CLAUDE_PLUGIN_ROOT}/skills/dpmp-skills/run.py create-req \
    --product-key "PRODU" \
    --name "客户管理系统" \
    --priority "高(一般)" \
    --desc "需要支持客户标签管理功能"
```

## 注意事项

1. **不直接解析**：此工具不直接解析自然语言，而是提供指导
2. **LLM 负责提取**：实际的参数提取由 LLM 完成
3. **模板是建议**：建议的模板仅供参考，LLM 可以根据实际情况调整
4. **字段映射**：注意 API 字段名与 CLI 参数名的映射关系
5. **认证一致**：使用与其他 DPMP 工具相同的认证方式
