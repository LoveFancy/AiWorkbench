# 自动切换模型可配置化需求

编辑时间：2026年6月15日

---

## 一、背景

Trae 客户端具备**自动切换模型**能力：当用户发起的模型调用返回错误（如超时、服务不可用、并发限制等），客户端会自动从模型列表中选取下一个可用模型重试，保证对话连续性。

当前自动切换范围为**首页/设置页配置的全部模型列表**，包含两大类模型：

| 类别 | 示例 | 部署方式 |
|------|------|----------|
| 本地/私有化模型 | `local-deepseek-v32`、`local-qwen36-27b` | 华泰内部部署 |
| SaaS 云端模型 | `saas-doubao-15-pro-32k`、`saas-deepseek-v32` | 第三方云服务 |

---

## 二、问题分析

### 2.1 核心风险：数据泄露

当本地模型报错后，自动切换机制可能在**用户无感知的情况下**将请求路由到 SaaS 模型。由于 SaaS 模型的数据会经过第三方云服务，存在公司机密数据泄露的风险。

### 2.2 痛点

- **不可控**：用户无法指定哪些模型参与自动切换，只能被动接受全量模型列表
- **无感知**：切换过程静默，用户可能未意识到当前对话已路由到 SaaS 模型
- **缺乏粒度**：无法按场景（如敏感项目 vs 日常使用）区分切换策略

### 2.3 典型场景

```
用户选择 local-qwen36-27b 发起对话
  → 模型返回 503 错误
  → 自动切换到下一个模型：saas-doubao-15-pro-32k  ← 风险！
  → 请求携带公司内部代码/数据发送至第三方
```

---

## 三、需求目标

参考 Trae 的 "Auto Mode" 设计，在模型列表顶部提供开关，实现：

1. **Auto Mode 开关**：一键开启/关闭自动切换，开关 OFF 时等价于固定模型
2. **自定义候选模型**：Auto Mode 下仅使用用户指定的模型进行切换，而非全量列表
3. **安全可控**：默认排除 SaaS 模型，候选池由用户明确指定
4. **未配置不可开启**：候选池为空时，开关置灰，禁止开启 Auto Mode
5. **用户可感知**：切换时给出明确提示，让用户知道当前使用的模型已变更

---

## 四、详细需求

### 4.1 Auto Mode 开关

在模型列表最上方，提供 **Auto Mode** 开关，交互参考 Trae：

```
┌──────────────────────────────────────────────────┐
│  模型列表                                         │
│  ┌──────────────────────────────────────────────┐ │
│  │ ⚠ 建议优先使用本地（local）模型进行对话。        │ │
│  │ 自动模式下若包含 SaaS 模型，模型报错时可能        │ │
│  │ 自动切换至云端服务，存在数据泄露风险。            │ │
│  └──────────────────────────────────────────────┘ │
│                                                   │
│  Auto Mode                              [开关]    │
│  模型异常时自动切换候选池中的下一个可用模型        │
│  ─────────────────────────────────────────────    │
│  ● local-qwen36-27b         私有化                │
│    local-deepseek-v32       私有化     Auto       │
│    local-qwen3-235b-moe     私有化     Auto       │
│    saas-doubao-15-pro-32k   云端 (SaaS)          │
│    ...                                           │
│                                                   │
│  [管理候选模型]                                    │
└──────────────────────────────────────────────────┘
```

**开关行为：**

| 开关状态 | 候选池状态 | 效果 |
|----------|-----------|------|
| **OFF** | 任意 | 固定模型模式，当前模型失败直接报错，不切换 |
| **ON** | 已配置（≥1 个模型） | 自动切换模式，按候选池顺序切换 |
| **置灰（不可开启）** | 未配置（0 个模型） | 开关显示为禁用态，hover 提示"请先配置候选模型" |

**关键规则：**
- 默认状态：**OFF**（首次使用不开自动切换）
- 候选池为空时，开关置灰，点击无效
- 用户必须先配置候选模型，才能开启 Auto Mode
- 开关状态即时生效，无需额外点击保存
- 开关在 ON 状态下，若用户清空了候选池，开关自动回拨到 OFF 并置灰

#### 4.1.1 对话框模型选择器联动

Auto Mode 开关影响对话框顶部模型选择器的展示：

| 操作 | 模型选择器展示 | 说明 |
|------|---------------|------|
| **打开 Auto Mode** | 显示 **"Auto"**，不展示具体模型名 | 表示当前由自动切换机制决定实际使用的模型 |
| **关闭 Auto Mode** | 回退到上次手动选择的模型 | 记住关闭前用户最后一次手动选的模型 |
| **关闭 Auto Mode（无历史选择）** | 显示未选择模型的初始状态（placeholder） | 如用户从未手动选过模型就开了 Auto Mode |

```
Auto Mode ON：

  ┌──────────────────────────────────────┐
  │  模型：Auto                    ▼     │
  │  自动切换已启用                        │
  └──────────────────────────────────────┘

Auto Mode OFF（有历史选择）：

  ┌──────────────────────────────────────┐
  │  模型：local-qwen36-27b        ▼     │
  └──────────────────────────────────────┘

Auto Mode OFF（无历史选择）：

  ┌──────────────────────────────────────┐
  │  模型：请选择模型               ▼     │
  └──────────────────────────────────────┘
```

- 历史选择指用户**手动在模型下拉中选择的模型**，不包括 Auto Mode 自动切换到的模型
- 对话框模型选择器在 Auto Mode ON 时仍然可下拉，用户可以选择具体模型（此时模型选择不影响自动切换行为，切换仍按候选池执行），并更新历史选择记录

#### 4.1.2 本地模型优先提醒

在模型列表页面固定展示安全提示条（始终可见，不可关闭），引导用户优先选择本地模型：

```
⚠ 建议优先使用本地（local）模型进行对话。
自动模式下若包含 SaaS 模型，模型报错时可能自动切换至云端服务，存在数据泄露风险。
```

- **展示位置**：模型列表顶部，Auto Mode 开关上方
- **始终可见**：不随开关状态变化而隐藏，无论 Auto Mode ON/OFF 都展示
- **目的**：在用户选择模型时持续提醒数据安全风险

#### 4.1.3 存储

```json
{
  "autoModeEnabled": false,
  "autoSwitchCandidateModels": ["local-deepseek-v32", "local-qwen36-27b"]
}
```

### 4.3 切换行为

#### 4.3.1 整体流程

```
模型调用失败
  │
  ├─ Auto Mode = OFF → 不切换，直接返回错误给用户
  │
  └─ Auto Mode = ON → 从候选池中按序选取下一个模型重试
       │
       ├─ 候选池中有可用模型 → 切换并通知用户
       └─ 候选池已耗尽 → 停止切换，提示所有候选模型均不可用
```

#### 4.3.2 切换范围

```
Auto Mode OFF：
  仅使用当前选中模型 → 失败直接报错

Auto Mode ON：
  全部模型列表 → 过滤：仅候选池中的模型 → 按序切换
```

#### 4.3.3 切换顺序

候选池中模型的切换顺序沿用当前模型列表的排序，而非候选池中勾选的顺序。即：在模型列表的原顺序中，跳过不在候选池内的模型。

#### 4.3.4 边界情况

| 场景 | 行为 |
|------|------|
| Auto Mode OFF，模型报错 | 不切换，直接报错，提示"模型 xxx 响应异常" |
| Auto Mode ON，候选池仅 1 个模型 | 该模型 2 次机会（首次 + 重试 1 次），均失败后停止 |
| Auto Mode ON，候选池中所有模型均失败 | 停止切换，提示"所有候选模型均不可用" |
| 当前模型本身不在候选池中 | 允许手动使用，但自动切换时跳过该模型 |
| 候选池中某模型已被下线/删除 | 切换时跳过该过期 ID（不在 availableModels 中），继续尝试下一个 |

#### 4.3.5 切换后 Agent 调用模型的行为

```
用户发送消息
  │
  ├─ 使用当前激活模型发起调用
  │     │
  │     ├─ 成功 → Agent 工具调用继续使用该模型（同一轮内不换模型）
  │     │
  │     └─ 失败 → 触发自动切换
  │           ├─ 找到候选模型 → 用新模型重试本轮调用
  │           └─ Agent 后续工具调用均使用新模型
  │
  └─ 下一轮用户消息 → 继续使用上次成功的模型（不再回退到原始模型）
       （若该模型也失败，继续按候选池顺序切换）
```

| 轮次 | 描述 |
|------|------|
| **同一轮内** | 切换后模型对该轮内所有后续 Agent 工具调用生效（Agent 一次性完成本轮任务） |
| **跨轮行为** | 切换后模型成为新的"激活模型"，下一轮用户消息默认使用该模型，不再回退到切换前的模型。避免反复撞击已故障的模型 |
| **Auto Mode 关闭时** | 激活模型被清空，回到手动选择的模型（见 4.1.1 联动规则） |

示例：

```
候选池：[local-qwen36-27b, local-deepseek-v32, local-qwen3-235b]

第 1 轮：使用 local-qwen36-27b → 报错 503
         → 自动切换到 local-deepseek-v32 → 成功
         → Agent 后续工具调用全部走 local-deepseek-v32

第 2 轮：默认使用 local-deepseek-v32（上次成功的模型）
         → 不再回退到 local-qwen36-27b

第 N 轮：local-deepseek-v32 也报错
         → 自动切换到 local-qwen3-235b
         → 之后默认使用 local-qwen3-235b
```

### 4.4 切换机制实现设计

本节描述 Auto Mode 下模型切换在代码层面的实现方式。

#### 4.4.1 改造点：从同模型重试到跨模型切换

**现状（agent-orchestrator.ts 第 1703 行起）**：目前 Agent 的自动重试是**同模型重试**——`for` 循环中每次用同一个模型 ID 重新发起 SDK 调用，仅依赖指数退避延迟。

```
现状逻辑：
  for (attempt = 1; attempt <= 25; attempt++)
    → 等待退避延迟
    → 同一个模型 ID query(sdk)       ← 始终同一模型
    → 成功 → break
    → 可重试错误 → continue（同模型再来一次）
```

**改造方向**：在重试循环中加入**模型切换逻辑**——当重试次数超过当前模型的容忍上限后，不再用同一模型硬怼，而是切换到候选池中的下一个模型重新发起调用。

```
改造后逻辑（Auto Mode ON）：
  for (attempt = 1; attempt <= 25; attempt++)
    → 等待退避延迟（同模型指数退避，切模型时重置）
    → 检查：累计等待 > 5 分钟？ → 停止
    → 当前模型 ID query(sdk)
    → 成功 → break（记录 activeModelId）
    → 不可重试错误 → 停止，提示用户
    → 可重试错误
          ├─ 同模型重试次数 < 1 → continue（同模型再试 1 次）
          └─ 同模型重试次数 >= 1（已试 2 次仍失败）
              ├─ 候选池中还有下一个模型 → 切换模型，重置计数器，continue
              └─ 候选池已耗尽 → 停止，提示"所有候选模型均不可用"
```

#### 4.4.2 触发切换的错误类型

沿用现有的错误分类体系（agent-orchestrator.ts），以下错误触发自动切换（同模型重试 1 次，仍失败则切模型）：

| 错误来源 | 错误码/模式 | 所属模块 |
|----------|-----------|---------|
| **SDK typed_error** | `rate_limited`、`provider_error`（overloaded）、`service_error`、`service_unavailable`、`network_error` | `AUTO_RETRYABLE_ERROR_CODES` |
| **HTTP 状态码** | 429（限流）、5xx（含 502 Bad Gateway、529 Overloaded） | `isAutoRetryableCatchError()` |
| **瞬时网络** | TCP 连接重置（ECONNRESET）、超时（ETIMEDOUT）、管道断裂（EPIPE）、DNS 抖动、`fetch failed`、stream prematurely close 等 | `error-patterns.ts` 的 `isTransientNetworkError()` |
| **上下文管理** | `context_management` 错误模式 | `isAutoRetryableCatchError()` |

以下错误**不触发切换**，直接报错停止：

- `api_key_invalid`（401）— 认证问题，切任何模型都没用
- `prompt_too_long` — 内容问题，非模型问题
- `thinking_signature_incompatible` — 先走签名回填恢复，不直接切换
- `session_not_found` — 先清除 SDK session 并回填上下文恢复

#### 4.4.3 重试策略与总次数上限

每个模型 2 次机会（首次调用 + 重试 1 次），仍失败则切换到下一个候选模型：

| 参数 | 建议值 | 说明 |
|------|--------|------|
| `MAX_SAME_MODEL_RETRIES` | 1 | 同一模型失败后重试 1 次（共 2 次机会），仍失败则切换 |
| `MAX_AUTO_RETRIES` | 25 | 全局总重试次数上限（复用现有值） |
| `MAX_AUTO_RETRY_WAIT_MS` | 300000（5 分钟） | 累计等待时间预算（复用现有值） |
| `RETRY_DELAY_BASE_MS` | 1000 | 同模型退避基础延迟 |
| `RETRY_DELAY_MAX_MS` | 15000 | 同模型退避上限 |

**总重试次数计算**（取以下两者中的较小值）：

```
总次数 = min(candidatePoolSize × 2, MAX_AUTO_RETRIES)
```

即：每个模型 2 次机会（首次 + 重试 1 次），总次数 = 候选池大小 × 2，受全局 25 次上限约束。

**候选池全部不可用时的停止条件**（满足任一即停止）：

| 条件 | 描述 |
|------|------|
| 候选池中所有模型均已尝试 | 每个模型都失败，无更多候选 |
| 全局重试达 25 次 | 达到 `MAX_AUTO_RETRIES` 硬上限 |
| 累计等待超过 5 分钟 | 达到 `MAX_AUTO_RETRY_WAIT_MS` 时长上限 |

**停止后的处理**：触发 `retry_failed` 事件通知前端，显示"所有候选模型均不可用，请检查模型状态或调整候选池配置"。

**具体示例**：

```
候选池 3 个模型 [A, B, C]：
  A: 首次失败 → 重试 1 次 → 仍失败 → 切到 B
  B: 首次失败 → 重试 1 次 → 仍失败 → 切到 C
  C: 首次失败 → 重试 1 次 → 仍失败 → 候选池耗尽 → 停止（共 6 次）
  耗时估算：(1+1+1)s × 3 ≈ 3s（未达 5 分钟上限）

候选池 10 个模型：
  每个模型 2 次，共 20 次 → 候选池耗尽 → 停止
  耗时估算：约 10s+（切模型退避延迟复位）

候选池 13 个模型：
  前 12 个模型各 2 次 = 24 次
  第 13 个模型首次失败后达到 25 次全局上限 → 停止（共 25 次）
```

注：切换模型后的退避延迟复位，因为新模型是独立的后端，不需要继续退避。但全局 attempt 计数和累计等待时间仍受 25 次 / 5 分钟上限约束。

#### 4.4.4 候选模型选取逻辑

在 `agent-orchestrator.ts` 的 `queryAgent` 方法中，新增 `selectNextCandidateModel()` 函数：

```
输入：
  - currentModelId: 当前使用的模型 ID
  - candidatePool: 候选池模型 ID 列表（按模型列表原始顺序）
  - excludeModelIds: 本轮已尝试过且失败的模型 ID 集合
  - availableModels: 当前系统实际可用的模型 ID 集合（来自模型列表接口）

输出：下一个候选模型 ID，或 null（候选池耗尽）

逻辑：
  1. 在 candidatePool 中从 currentModelId 的下一个位置开始遍历
  2. 跳过 excludeModelIds 中的模型
  3. 跳过不在 availableModels 中的模型（处理模型下线/被删除的兜底）
  4. 取第一个未排除且在线的模型
  5. 若已遍历完候选池 → 返回 null
```

注：候选池存储的是模型 ID 字符串。当后端下线某个模型后，该 ID 不再出现在可用模型列表中，切换时应跳过该过期 ID，避免切换到不存在的模型导致立即失败。

示例：

```
候选池排序：[A, B, C, D]
当前模型 = A，排除列表 = {A, B}（A 失败，B 也失败）
→ 从 A 的下一个开始遍历 → B(排除) → C(可用) → 返回 C
```

#### 4.4.5 跨轮模型记忆

Auto Mode 下，每次成功的模型会被记录到 `sessionMeta.activeModelId`，下一轮用户消息使用该模型发起调用：

```
首次开启 Auto Mode / 开启后发消息：
  if (autoModeEnabled):
    modelId = sessionMeta.activeModelId              // 有切换记忆取记忆
           ?? (userSelectedModelId 在候选池中 ? userSelectedModelId : null)  // 用户当前手动选的模型若在候选池中则优先
           ?? candidatePool[0]                       // 兜底取候选池第一个
  else:
    modelId = userSelectedModelId

切换过程：
  queryAgent(sessionId, modelId, ...)
    → on success: sessionMeta.activeModelId = modelId  // 成功时更新记忆
    → on switch to newModel: sessionMeta.activeModelId = newModel  // 切换后更新记忆
```

#### 4.4.6 涉及改动的文件

| 文件 | 改动 |
|------|------|
| `agent-orchestrator.ts` | 新增候选模型选取逻辑；在 retry 循环中加入切换判定；跨轮模型记忆 |
| `error-patterns.ts` | 无需改动（复现有错误分类） |

## 五、非功能需求

### 5.1 向后兼容

- 升级到新版本后，Auto Mode 默认 **OFF**，不启用自动切换
- 若用户之前依赖全部模型自动切换，升级后需**手动配置候选模型并开启 Auto Mode** 才会恢复自动切换能力

### 5.2 性能

- 候选池过滤逻辑在切换时执行，不增加正常调用路径的延迟
- 配置读取在启动时加载到内存，切换过程无需磁盘 IO
