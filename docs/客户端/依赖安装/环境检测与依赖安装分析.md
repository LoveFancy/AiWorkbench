# 环境检测与依赖安装分析及一键安装方案

## 一、当前实现逻辑分析

### 1.1 整体架构

环境检测和依赖安装涉及两套并行的 UI 流程，共用底层的检测和下载模块：

```
┌─────────────────────────────────────────────────────────────┐
│                      UI 层 (Renderer)                        │
├────────────────────────────┬────────────────────────────────┤
│  AboutSettings.tsx         │  EnvironmentCheckPanel.tsx      │
│  (设置→关于/更新)           │  (Windows 环境检测弹窗)          │
│  action: openExternal      │  action: download (一键下载)     │
├────────────────────────────┴────────────────────────────────┤
│                      IPC 通道                                │
├─────────────────────────────────────────────────────────────┤
│  environment:check         │  installer:download/launch/...  │
├────────────────────────────┴────────────────────────────────┤
│                      Main 进程                               │
├─────────────────────────────────────────────────────────────┤
│  environment-checker.ts    │  installer-manifest.ts          │
│    → node-detector.ts      │    → 远程 API 获取清单           │
│    → git-detector.ts       │    → 内置 fallback              │
│                             │  installer-downloader.ts        │
│  runtime-init.ts           │    → HTTP 流式下载 + sha256 校验  │
│    → git-bash-detector.ts  │    → shell.openPath() 拉起安装    │
│    → wsl-detector.ts       │                                 │
│    → node-detector.ts      │                                 │
└────────────────────────────┴─────────────────────────────────┘
```

### 1.2 环境检测流程

#### 流程 A：设置页（AboutSettings.tsx）

**入口**：`设置 → 关于/更新 → 环境检测`

**检测逻辑**（[environment-checker.ts](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/environment-checker.ts)）：
1. 并行调用 `detectNodeRuntime()` 和 `detectGitRuntime()`
2. 版本判断：Node.js 最低 18 LTS，推荐 22 LTS；Git >= 2.0
3. 返回 `EnvironmentCheckResult`，包含 `installed`、`version`、`meetsMinimum`、`meetsRecommended`、`downloadUrl`

**下载链接**（硬编码在 `getDownloadUrls()` 中）：

| 平台    | Node.js                                   | Git                              |
|---------|-------------------------------------------|----------------------------------|
| macOS   | `https://nodejs.org/dist/v22.13.1/node-v22.13.1.pkg` | `https://git-scm.com/download/mac` |
| Windows | `https://htpan.htsc.com.cn/l/vF2xEX`      | `https://htpan.htsc.com.cn/l/iFghB9` |
| Linux   | `https://nodejs.org/en/download/`         | `https://git-scm.com/download/linux` |

**问题**：使用 `openExternal` 打开浏览器，Windows 下是 htpan 内网短链接，普通用户在公网无法访问或访问到错误页面。

#### 流程 B：Windows 环境检测弹窗（EnvironmentCheckPanel.tsx）

**入口**：聊天错误卡片"打开环境检测" / Onboarding Step 2

**检测逻辑**（[runtime-init.ts](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/runtime-init.ts)）：
1. 检测 Node.js：`detectNodeRuntime()`（[node-detector.ts](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/node-detector.ts)）
   - 策略：`where node` 查找 PATH → 注册表路径 → 常见安装目录（Scoop/Chocolatey/官方默认）→ 执行 `node --version` 获取版本
2. 检测 Git Bash：`detectGitBash()`（[git-bash-detector.ts](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/git-bash-detector.ts)）
   - 策略：常见安装路径（Program Files/Scoop）→ 注册表 → `where bash`
   - 验证：执行 `bash --version` 解析 GNU bash 版本
3. 检测 WSL：`detectWsl()`（[wsl-detector.ts](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/wsl-detector.ts)）
4. 推荐策略：Git Bash > WSL 2 > WSL 1

**UI 行为**：
- Shell 环境缺失 → 显示 `type: 'download', installerId: 'git-for-windows'` 按钮
- Node.js 缺失 → 显示 `type: 'download', installerId: 'nodejs'` 按钮

### 1.3 依赖安装下载流程

#### 安装包清单（[installer-manifest.ts](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/installer-manifest.ts)）

**清单获取优先级**：
1. 远程 API：`https://api.proma.cool/api/v1/installers/manifest`（缓存 5 分钟）
2. 远程失败 → 降级到内置 fallback

**本地覆盖机制**：`applyManagedInstallerOverrides()` 强制将 Windows x64 的安装包替换为 htpan 内网链接：

```typescript
// 内置 fallback 中 Windows x64 的配置
{
  id: 'git-for-windows',
  platform: 'win32', arch: 'x64', version: '2.54.0',
  downloadUrl: 'https://htpan.htsc.com.cn/l/iFghB9',   // ← 内网 CDN
  fallbackUrl: '',                                        // ← 无公网回退！
  sha256: '',
  sizeBytes: 76000000,
  filename: 'Git-2.54.0-64-bit.exe',
},
{
  id: 'nodejs',
  platform: 'win32', arch: 'x64', version: '24.15.0',
  downloadUrl: 'https://htpan.htsc.com.cn/l/vF2xEX',   // ← 内网 CDN
  fallbackUrl: '',                                        // ← 无公网回退！
  sha256: '',
  sizeBytes: 32000000,
  filename: 'node-v24.15.0-x64.msi',
},
```

**arm64 有公网 fallback，x64 没有**：
```typescript
// arm64 有 fallback
{ id: 'git-for-windows', arch: 'arm64', fallbackUrl: 'https://github.com/.../Git-2.47.1-arm64.exe' }
{ id: 'nodejs', arch: 'arm64', fallbackUrl: 'https://nodejs.org/dist/v22.13.1/node-v22.13.1-arm64.msi' }
```

#### 下载器（[installer-downloader.ts](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/installer-downloader.ts)）

1. 下载到 `%TEMP%/proma-installers/` 目录
2. 流式下载，250ms 推送一次进度（速度/百分比）
3. 下载完后 sha256 校验
4. 主 URL 失败自动尝试 fallbackUrl
5. 支持取消

#### 拉起安装（[installer-downloader.ts#L257](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/installer-downloader.ts#L257)）

```typescript
export async function launchInstaller(filePath: string): Promise<void> {
  const errorMsg = await shell.openPath(filePath)
  if (errorMsg) throw new Error(`无法拉起安装程序：${errorMsg}`)
}
```

使用 `shell.openPath()` 直接打开安装程序，用户仍需手动走完安装向导。

## 二、核心问题分析

### 问题 1：Windows x64 下载源为内网地址，普通用户无法下载

| 问题                   | 详情                                                                 |
|------------------------|----------------------------------------------------------------------|
| **根因**               | `MANAGED_WINDOWS_X64_INSTALLERS` 和内置 fallback 的 x64 条目 `fallbackUrl` 为空 |
| **影响范围**           | 所有非 HTSC 内网的 Windows x64 用户                                  |
| **涉及文件**           | [installer-manifest.ts](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/installer-manifest.ts) |

### 问题 2：设置页使用 openExternal，完全不具备一键安装能力

AboutSettings.tsx 的下载按钮用的是 `action: 'openExternal'`，只是打开浏览器跳转到下载页面，用户需要：
1. 在浏览器中找到正确的下载按钮
2. 手动下载
3. 手动运行安装程序
4. 手动选择安装选项

### 问题 3：安装过程无自动化，用户需手动操作

下载完成后仅通过 `shell.openPath()` 拉起安装程序，用户仍需：
- Git for Windows：选择安装路径、选择组件、选择 PATH 环境（默认编辑器、CRLF 处理等选项对普通用户不友好）
- Node.js：选择安装路径、是否添加到 PATH

### 问题 4：Shell 环境检测与安装存在割裂

- 环境检测提示需要"Git for Windows"来获取 Git Bash
- 但安装包下载的是 `git-for-windows` installerId
- 安装完 Git for Windows 后，Git 和 Git Bash 都有了，但 Node.js 还需要单独操作
- 用户可能需要安装两个东西，没有一键全装的入口

## 二·附、不安装各依赖的具体影响

### Git Bash（Windows 上必需）

**影响级别：阻断性（Agent 完全不可用）**

WorkMate 在 Windows 上的 Agent 核心依赖一个 Unix-like Shell 来执行 AI 生成的 Bash 命令（如 `ls`、`grep`、`git`、文件操作等）。代码中有两处硬阻断：

#### 阻断点 1：Preflight 检查（[agent-orchestrator.ts#L1000-L1026](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/agent-orchestrator.ts#L1000)）

```typescript
// 在启动 Agent 前执行飞前检查
if (shellStatus && !shellStatus.gitBash?.available && !shellStatus.wsl?.available) {
  reportPreflightError({
    code: 'windows_shell_missing',
    title: 'Windows 环境未就绪',
    message: '需要 Git Bash 或 WSL 才能运行 Agent。...',
    ...
  })
  return  // ← 直接 return，Agent 不会启动
}
```

**后果**：发送任何 Agent 消息都会直接报错，Agent 完全无法运行。UI 会显示错误卡片，用户可以点"打开环境检测"跳转到检测弹窗。

#### 阻断点 2：SDK Shell 配置（[agent-orchestrator.ts#L617-L633](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/agent-orchestrator.ts#L617)）

```typescript
// 配置 SDK 使用的 Shell 环境
if (shellStatus.gitBash?.available && shellStatus.gitBash.path) {
  sdkEnv.CLAUDE_CODE_SHELL = shellStatus.gitBash.path  // 如 C:\Program Files\Git\bin\bash.exe
} else if (shellStatus.wsl?.available) {
  sdkEnv.CLAUDE_CODE_SHELL = 'wsl'
}
```

`CLAUDE_CODE_SHELL` 环境变量告诉 Codex Agent SDK 用哪个 Shell 来执行工具调用中的 Bash 命令。Windows 的 cmd.exe / PowerShell 与 Agent SDK 的 Bash 工具不兼容。没有这个变量，SDK 子进程无法执行任何 Bash 命令。

#### 附加影响：插件 Hook 系统（[run-hook.cmd](file:///d:/AiWorkbench-workmate/apps/electron/bundled-plugins/superpowers/hooks/run-hook.cmd)）

```batch
REM 插件 hook 脚本需要 bash.exe 来执行
if exist "C:\Program Files\Git\bin\bash.exe" (
    "C:\Program Files\Git\bin\bash.exe" "%HOOK_DIR%%~1" ...
    exit /b %ERRORLEVEL%
)
REM 找不到 bash → 静默退出，hook 不生效
exit /b 0
```

插件（如 superpowers）的 SessionStart 等生命周期 hook 脚本都是 bash 脚本，通过 `run-hook.cmd` 调用。没有 Git Bash 时 hook 静默失败，不影响核心功能但插件的部分能力会缺失。

#### 小结

| 功能               | 无 Git Bash 时的表现          |
|--------------------|------------------------------|
| Agent 对话         | **完全不可用，发送即报错**    |
| Bash 命令执行      | 无法执行                      |
| 插件 hook 脚本     | 静默失效                      |
| 文件浏览/编辑/设置 | 正常                          |
| 聊天 UI            | 正常（但发送消息会报错）      |

> **备选方案**：WSL 也可作为替代 Shell。如果用户已安装 WSL（Windows Subsystem for Linux），Agent 可以使用 WSL 的 bash，不需要额外安装 Git for Windows。

### Node.js（Windows/macOS 上推荐但非必需）

**影响级别：部分功能受限（Agent 核心可用）**

Node.js 在 WorkMate 中的定位是"软依赖"。从代码可以看出：

1. **Agent SDK 不依赖系统 Node.js**：[agent-orchestrator.ts](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/agent-orchestrator.ts) 中 Agent 运行时使用的是打包在 Electron 内的 Codex Agent SDK 二进制文件（`Codex.exe`），不调用系统的 `node` 命令。

2. **UI 中标注为"可选"**：[EnvironmentCheckPanel.tsx#L97-L102](file:///d:/AiWorkbench-workmate/apps/electron/src/renderer/components/environment/EnvironmentCheckPanel.tsx#L97)：
   ```
   未检测到 Node.js。如果不使用基于 npx 的 MCP 服务器，可以忽略此项。
   ```

3. **状态为 warning 而非 error**：`isNodeJsOkAtom` 仅影响提示，不阻断 Agent 启动。

#### 没有 Node.js 时具体受影响的功能

| 功能                       | 无 Node.js 时的表现                 |
|----------------------------|-------------------------------------|
| Agent 基础对话             | 正常                                |
| Bash 命令执行              | 正常（通过 Git Bash/WSL）           |
| 文件读写/搜索              | 正常                                |
| MCP 服务器（npx 方式启动） | **无法启动**                        |
| Git 操作                   | 正常（依赖 Git，不依赖 Node.js）    |
| 插件系统                   | 正常                                |

MCP（Model Context Protocol）服务器是 Agent 扩展能力的关键机制。WorkMate 内置了 Memos、Nano Banana、Web Search 等 MCP 服务器（见 [agent-orchestrator.ts#L654-L778](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/agent-orchestrator.ts#L654)）。部分 MCP 通过 `npx` 启动子进程，没有 Node.js 时这些服务器会启动失败。

**典型影响场景**：
- 用户配置了 `npx @anthropic/mcp-server-xxx` 的 MCP 服务 → 报错
- 用户想要 Agent 执行 `node script.js` 或 `npm install` → 报错
- Agent 想要用 `npx` 调用某个工具 → 失败

### Git（各平台都需要）

Git 在 WorkMate 中有两层含义：

1. **Git 命令行工具**（`git.exe`）：被 [environment-checker.ts](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/environment-checker.ts) 检测，用于版本控制操作
2. **Git Bash**（仅 Windows）：即上面说的 Shell 环境，来自 Git for Windows 安装包

**没有 Git 命令行工具的影响**：

| 功能               | 无 Git 时的表现         |
|--------------------|------------------------|
| Agent 对话         | 正常（只要有 Shell）    |
| Git 版本控制操作   | 无法执行               |
| 代码 diff 展示     | 可能不可用             |
| 代码变更追踪       | 不可用                 |

不过 WorkMate 的核心价值是 Agent 对话，Git 操作的缺失不会阻断核心流程。而且 Windows 用户如果安装了 Git for Windows（为了拿到 Git Bash），Git 命令行工具也会一并安装。

---

### 依赖关系图

```
Git for Windows（Windows 平台）
├── git.exe ─────────────→ Git 版本控制操作
└── bash.exe ────────────→ Agent 核心（必需）✅
    └── CLAUDE_CODE_SHELL → Codex Agent SDK 的 Bash 子进程

Node.js（各平台推荐）
└── node.exe / npx ─────→ MCP 服务器启动（可选）⚠️
    └── 用户配置的 npx MCP 服务
    └── Agent 执行 node 脚本 / npm 命令
```

> **结论**：Windows 用户**必须**安装 Git for Windows（或 WSL），否则 Agent 模式完全无法使用。Node.js 是推荐安装，仅在使用 npx MCP 服务器时需要。

### Chat 模式 vs Agent 模式：依赖需求对比

Chat 模式和 Agent 模式的底层实现完全不同，对运行环境的依赖也完全不同。

#### Chat 模式的实现方式（[chat-service.ts](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/chat-service.ts)）

Chat 模式**不经过 Codex Agent SDK**，而是直接调用 AI Provider 的 HTTP API：

```
用户输入 → chat-service.ts
  ↓
@proma/core Provider 适配器（OpenAI / Anthropic / Google / ...）
  ↓
HTTP SSE 流式响应
  ↓
webContents.send() → React UI
```

Chat 模式的特点：
- **无 Bash 工具**：Chat 工具的 function calling 是纯 API 层面的调用（如联网搜索调用搜索 API），不执行任何 shell 命令
- **无 Preflight 检查**：`chat-service.ts` 中没有任何 shell 环境检测逻辑
- **无 `CLAUDE_CODE_SHELL`**：不需要配置 shell 环境变量
- **无文件系统操作**：Chat 模式不会主动修改用户本地文件

#### Agent 模式的实现方式（[agent-orchestrator.ts](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/agent-orchestrator.ts)）

Agent 模式使用 **Codex Agent SDK**（打包在 Electron 内的 `Codex.exe` 二进制文件），SDK 内部子进程需要用 Bash 执行工具命令：

```
用户输入 → agent-orchestrator.ts
  ↓
Codex Agent SDK (Codex.exe) → 子进程执行 Bash 命令
  ↓                               ↑
  │                     CLI: bash.exe / wsl
  ↓
SDKMessage 流 → convertSDKMessage() → webContents.send() → React UI
```

#### 依赖对比表

| 依赖项            | Chat 模式 | Agent 模式 | 说明                           |
|-------------------|-----------|------------|--------------------------------|
| Git Bash           | **不需要**  | **必须**     | Agent SDK 需要 Bash 执行命令    |
| WSL                | **不需要**  | 可替代      | 可替代 Git Bash                 |
| Git 命令行         | **不需要**  | 推荐        | Agent 做版本控制时需要           |
| Node.js            | **不需要**  | 推荐        | Agent 使用 npx MCP 时需要        |
| 网络 / API Key     | 必须       | 必须        | 两者都需要                      |

#### 什么时候只需要 Chat 模式就够了

如果用户只需要以下功能，**不安装 Git Bash 和 Node.js 完全不碍事**：
- AI 问答、翻译、润色
- 粘贴代码让 AI 分析、解释
- 上传附件做总结、对比
- 联网搜索
- 多模型对比

#### 什么时候必须装 Git Bash

用户想用以下功能时就必须装了：
- 让 AI 自动修改本地项目文件
- 让 AI 执行命令（如 `npm install`、`git commit`、运行测试）
- 使用 MCP 服务器或 Skills
- 多步骤编程任务的自动化推进

> **核心结论**：Chat 模式 = 轻量级，零环境依赖（只需要网络和 API Key）；Agent 模式 = 重量级，Windows 上需要 Git Bash（或 WSL）。

## 三、打包内置方案分析 — 能否把 Git/Node 直接打进安装包？

用户期望的理想体验：安装 WorkMate 后直接就能用 Agent 模式，不需要额外装任何东西。下面从技术可行性和工程成本两个维度分析。

### 3.1 方案 A：完全打包进安装包（PortableGit + Node.js Portable）

**原理**：用 electron-builder 的 `extraResources` 将预下载的 Git Portable 和 Node.js Portable 的完整目录打包进 NSIS 安装包，安装时解压到应用目录或 `AppData` 下，运行时自动检测并使用。

#### 可行性

| 层面           | 分析 |
|----------------|------|
| electron-builder | `extraResources` 在现有配置中已用于打包 Skills、插件、教程等，完全支持打包任意目录 |
| Git Portable    | 官方 [git-for-windows/git](https://github.com/git-for-windows/git/releases) 每版都提供 `PortableGit-{version}-64-bit.7z.exe`，解压后是一个完整可独立运行的 MinGW64/MSYS2 环境（含 bash.exe + 全套 Unix 工具），**不需安装，不需管理员权限** |
| Node.js Portable | 官方 [nodejs.org/dist](https://nodejs.org/dist/) 每版提供 `node-v{version}-win-x64.zip`，解压即用，无需安装 |
| 运行时配置       | `CLAUDE_CODE_SHELL` 环境变量直接指向 `resources/portable-git/bin/bash.exe` 即可（[agent-orchestrator.ts#L624](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/agent-orchestrator.ts#L624)） |

#### 体积成本

| 组件               | 压缩包   | 解压后   | 说明 |
|--------------------|----------|----------|------|
| PortableGit        | ~40 MB   | ~280 MB  | 含 bash + git + MinGW64 全套 |
| Node.js Portable   | ~25 MB   | ~100 MB  | 含 node.exe + npm + npx |
| **合计**            | **~65 MB** | **~380 MB** | |

当前 WorkMate Windows 安装包约 **250-300 MB**（Electron + Chromium + SDK binary + 前端资源 + Skills + 插件）。加入 PortableGit + Node.js 后，NSIS 安装包将膨胀到约 **400-450 MB**（压缩后）。

#### Git Portable 是否可以精简？

可以。Agent 实际只需要以下内容：

```
PortableGit/
├── bin/
│   └── bash.exe           ← 核心
├── usr/bin/                ← Unix 工具集
│   ├── grep.exe, sed.exe, awk.exe, find.exe, sort.exe, uniq.exe, ...
│   ├── ls.exe, cat.exe, rm.exe, cp.exe, mv.exe, mkdir.exe, ...
│   ├── git.exe            ← 版本控制（Agent 大概率用到）
│   └── ssh.exe, curl.exe  ← 网络工具
├── etc/                    ← 配置文件（bash 需要）
└── usr/lib/                ← DLL 依赖
```

精简后可降至约 **100-150 MB**（解压后），但需要精确梳理 DLL 依赖链，维护成本高。直接打包完整 PortableGit 虽然大但无需维护。

#### 方案 A 结论

| 优点                                          | 缺点                                              |
|-----------------------------------------------|---------------------------------------------------|
| 用户零操作，装完即用                          | 安装包膨胀 ~150 MB（压缩后）                      |
| 不受网络环境影响（离线可用）                  | 下载安装包耗时更长                                |
| 与 electron-builder 集成简单                  | CI 构建需先下载 PortableGit（~40MB）+ Node.js（~25MB） |
| 版本可控，不依赖用户系统的 Git/Node 版本      | 更新 Git/Node 时需发新版                          |

### 3.2 方案 B：首次使用自动下载（Download-on-First-Use）

**原理**：安装包不变，当用户首次进入 Agent 模式且检测不到 Git Bash 时，自动弹出"正在准备运行环境..."，下载 PortableGit + Node.js Portable 到 `~/.proma/runtime/` 目录并解压，配置环境后自动重试。用户体验接近"一键安装"。

#### 流程

```
用户点击切换到 Agent 模式
  ↓
Preflight 检查 → Shell 缺失
  ↓
┌──────────────────────────────────────┐
│  ⚠ Agent 模式需要 Git Bash 环境       │
│                                      │
│  正在为您自动准备运行环境...          │
│  [████████░░░░░░] 67%  8.2 MB/s     │
│                                      │
│  下载 PortableGit...                 │
│  [████████████░░] 82%  10.1 MB/s    │
│                                      │
│  解压并配置环境...                    │
└──────────────────────────────────────┘
  ↓
下载完成 → 解压到 ~/.proma/runtime/portable-git/
  → 检测到 bash.exe → 配置 CLAUDE_CODE_SHELL
  → 自动重试 Agent 启动
```

#### 下载源

| 组件         | 下载地址                                                                                     | 大小   |
|--------------|----------------------------------------------------------------------------------------------|--------|
| PortableGit  | `https://github.com/git-for-windows/git/releases/download/v2.54.0.windows.1/PortableGit-2.54.0-64-bit.7z.exe` | ~41 MB |
| Node.js 便携版 | `https://nodejs.org/dist/v24.15.0/node-v24.15.0-win-x64.zip`                               | ~28 MB |

> 也可使用现有 htpan CDN 作为内网优先下载源（速度快），GitHub/nodejs.org 作为公网 fallback。

#### 实现要点

1. **复用现有下载器**：[installer-downloader.ts](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/installer-downloader.ts) 已有流式下载 + 进度推送 + sha256 校验能力
2. **解压 PortableGit**：`PortableGit-*.7z.exe` 是 7z 自解压包，用 `child_process.spawn` 执行 `/VERYSILENT /DIR="target"` 即可静默解压
3. **解压 Node.js**：用 Node.js 内置的 `zlib` + `tar` 或 `adm-zip` 解压
4. **刷新检测**：解压完成后更新 `detectGitBash()` 的搜索路径，加入 `~/.proma/runtime/portable-git/bin/bash.exe`，自动重新检测

#### 方案 B 结论

| 优点                                          | 缺点                                              |
|-----------------------------------------------|---------------------------------------------------|
| 安装包体积不增加                              | 首次使用需等待下载（3-10 分钟，视网络）           |
| 不需要发新版就能更新 Git/Node                  | 离线环境无法使用 Agent（除非已缓存）              |
| 实现复杂度中等，大部分基础设施已有             | 下载失败需要友好降级提示                          |
| 不打扰纯 Chat 用户                            |                                                   |

### 3.3 方案对比与推荐

| 维度           | 方案 A：完全打包内置 | 方案 B：首次使用自动下载 | 当前方案：手动下载 |
|----------------|---------------------|--------------------------|-------------------|
| 安装包大小     | +~65 MB（压缩）     | 不变                     | 不变              |
| 用户操作       | **零操作**          | **零操作**（等待下载）   | 多步操作          |
| 离线可用       | 是                  | 否（除非已缓存）         | 否                |
| 网络依赖       | 仅安装时            | 首次 Agent 使用时        | 下载时            |
| CI 构建影响    | 需预下载 + ~2 分钟  | 无                       | 无                |
| 工程复杂度     | 低（extraResources）| 中（实现下载+解压+配置） | 低（已有）        |
| 版本更新       | 需发新版            | 更新 manifest 即可       | 更新 manifest 即可 |
| Git/Node 更新  | 依赖 WorkMate 发版  | 独立于 WorkMate 发版     | 独立于 WorkMate 发版 |

#### 推荐策略：方案 B（首次使用自动下载）为基础，逐步演进

**理由**：
1. **不牺牲安装包体积**：纯 Chat 用户不应该为 Agent 功能下载额外的 65 MB
2. **维护独立**：Git/Node 漏洞修复不需要 WorkMate 发新版
3. **现有基础设施充足**：`installer-downloader.ts` 的下载能力可直接复用，只需加解压逻辑
4. **用户体验接近零操作**：用户只需点"确认"（或可做成完全自动），等待下载完成后自动进入 Agent

**进阶：方案 A+B 混合**

可以做一个"完整版安装包"（含 PortableGit + Node.js）和"标准版安装包"（不含），官网提供两个下载选项：
- 标准版：安装包更小，首次 Agent 时自动下载 → 适合网络好的用户
- 完整版：装完即用 → 适合企业内网/离线环境

#### 改造点 2：Git for Windows 静默安装（关键突破）

Git for Windows 支持静默安装参数，可以直接跳过安装向导：

```powershell
# 静默安装 Git for Windows（/VERYSILENT 完全无界面，/NORESTART 不重启）
Git-2.54.0-64-bit.exe /VERYSILENT /NORESTART /SUPPRESSMSGBOXES `
  /DIR="C:\Program Files\Git" `
  /COMPONENTS="icons,ext\reg\shellhere,assoc,assoc_sh" `
  /NOICONS
```

或在 `installer-downloader.ts` 中通过 `child_process.exec` 执行静默安装：

```typescript
// 新增：静默安装 Git for Windows
export async function silentInstallGit(installerPath: string): Promise<void> {
  const args = [
    '/VERYSILENT',
    '/NORESTART',
    '/SUPPRESSMSGBOXES',
    '/DIR=C:\\Program Files\\Git',
  ]
  return new Promise((resolve, reject) => {
    const proc = spawn(installerPath, args, { stdio: 'ignore' })
    proc.on('close', (code) => {
      if (code === 0) resolve()
      else reject(new Error(`Git 安装退出码: ${code}`))
    })
    proc.on('error', reject)
  })
}
```

**关键考虑**：安装目录固定为 `C:\Program Files\Git`，安装后需要更新当前进程的 PATH 或通知用户重启应用，因为 Git for Windows 默认会将 `C:\Program Files\Git\bin` 添加到系统 PATH（安装后立即生效需要新开 cmd）。

```typescript
/**
 * 一键安装编排器
 *
 * 负责：
 * 1. 检测缺失的环境项
 * 2. 按顺序下载并静默安装
 * 3. 安装后验证
 */

interface InstallTask {
  id: string           // 'git-for-windows' | 'nodejs'
  name: string         // 显示名称
  arch: 'x64' | 'arm64'
}

interface InstallTaskStatus {
  task: InstallTask
  status: 'pending' | 'downloading' | 'installing' | 'done' | 'failed'
  error?: string
}

export async function oneClickInstall(
  tasks: InstallTask[],
  sender: BrowserWindow,
  onProgress: (statuses: InstallTaskStatus[]) => void
): Promise<InstallTaskStatus[]> {
  const manifest = await fetchInstallerManifest()
  const results: InstallTaskStatus[] = []

  for (const task of tasks) {
    const source = findInstallerSource(manifest, task.id, task.arch)
    if (!source) {
      results.push({ task, status: 'failed', error: '未找到安装包' })
      continue
    }

    // 下载
    updateStatus(results, task, 'downloading')
    onProgress([...results])
    const key = `${task.id}:${task.arch}`
    const { filePath } = await downloadInstaller(source, key, sender)

    // 静默安装
    updateStatus(results, task, 'installing')
    onProgress([...results])
    if (task.id === 'git-for-windows') {
      await silentInstallGit(filePath)
    } else if (task.id === 'nodejs') {
      await silentInstallNode(filePath)
    }

    updateStatus(results, task, 'done')
    onProgress([...results])
  }

  return results
}
```

#### 改造点 6：安装后 PATH 刷新

Git/Node.js 安装后，新的 PATH 不会立即对当前 Electron 进程生效。需要：

1. **读取注册表**获取系统 PATH 更新
2. **更新 `process.env.PATH`** 以包含新安装的路径
3. **重新运行环境检测**验证安装成功

```typescript
export function refreshPathAfterInstall(): void {
  // 读取系统 PATH 注册表（HKLM + HKCU）
  const systemPath = readRegistryValue(
    'HKLM\\SYSTEM\\CurrentControlSet\\Control\\Session Manager\\Environment', 'Path'
  )
  const userPath = readRegistryValue('HKCU\\Environment', 'Path')
  
  const merged = [
    process.env.PATH,
    'C:\\Program Files\\Git\\bin',
    'C:\\Program Files\\Git\\usr\\bin',
    'C:\\Program Files\\nodejs',
    systemPath,
    userPath,
  ].filter(Boolean).join(';')
  
  process.env.PATH = merged
}
```
