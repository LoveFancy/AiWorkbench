## 一、问题分析

### 1.1 现状

当前每次关闭 WorkMate 后重新打开，都需要重新登录。经排查，存在两个问题：

**问题 1：启动时未恢复持久化的登录状态**

`authStateAtom` 初始值为 `{ isLoggedIn: false }`，App 启动时从未调用主进程的 `getAuthInfo()` / `getAuthState()` 去检查磁盘上 `auth.json` 中是否已有未过期的 Token。即使上次登录的 Token 仍然有效，渲染进程也始终认为用户未登录。

```
启动流程（现状）：
  App 挂载
    → authStateAtom 初始值 { isLoggedIn: false }
    → 不检查磁盘 auth.json
    → UI 始终显示未登录 ❌
```

**问题 2：长期 Token 有效期过长**

- 长期 Token 有效期：365 天
- 强制重登周期：180 天

对于企业安全策略而言周期过长，需要收紧。

### 1.2 目标

| 优化点 | 现状 | 目标 |
|--------|------|------|
| 启动时自动恢复登录 | 不检查，始终未登录 | 检查 `auth.json`，Token 有效则自动恢复登录状态 |
| 长期 Token 有效期 | 365 天 | 30 天 |
| 强制重登周期 | 180 天 | 30 天 |

## 二、方案设计

### 2.1 启动时自动恢复登录

#### 整体流程

```
启动流程（优化后）：
  App 挂载
    → IPC: auth:check-session
      → 主进程 getAuthInfo()
        → 读取 auth.json
        → Token 存在且未过期 → 返回 { isLoggedIn: true, jobId }
        → Token 不存在或已过期 → 返回 { isLoggedIn: false }
    → 设置 authStateAtom
    → UI 根据实际状态展示 ✅
```

#### 改动点

共涉及 3 个文件，每个文件改动量极小：

| 文件 | 改动 | 说明 |
|------|------|------|
| `auth/ipc-handlers.ts` | 新增 `CHECK_SESSION` channel | 调用 `getAuthInfo()` 返回登录状态 |
| `auth/preload-bridge.ts` | 新增 `checkSession` API | 暴露给渲染进程 |
| `renderer/App.tsx` | 初始化时调用 `checkSession` | 启动时恢复登录状态 |

#### 2.1.1 `auth/ipc-handlers.ts` — 新增 IPC channel

```typescript
// 在 AUTH_IPC_CHANNELS 中新增
export const AUTH_IPC_CHANNELS = {
  GET_AUTH_STATE: 'auth:get-state',
  CHECK_SESSION: 'auth:check-session',   // ← 新增
  LOGIN: 'auth:login',
  LOGOUT: 'auth:logout',
} as const

// 在 registerAuthIpcHandlers() 中新增 handler
ipcMain.handle(AUTH_IPC_CHANNELS.CHECK_SESSION, () => {
  const info = getAuthInfo()
  if (!info) {
    return { isLoggedIn: false }
  }
  return {
    isLoggedIn: true,
    jobId: info.jobId,
    needsReauth: info.needsReauth,
  }
})
```

> **注意**：`CHECK_SESSION` 与现有 `GET_AUTH_STATE` 的区别：
> - `GET_AUTH_STATE`：调用 `isLoggedIn()`，会解密 Token 并校验 JWT 格式，较重
> - `CHECK_SESSION`：调用 `getAuthInfo()`，返回更完整的信息（含 `needsReauth`），供启动时使用
>
> 两个 channel 语义不同，保留 `GET_AUTH_STATE` 供登录成功后刷新状态使用。

#### 2.1.2 `auth/preload-bridge.ts` — 新增 preload API

```typescript
// AuthElectronAPI 接口中新增
export interface AuthElectronAPI {
  getAuthState: () => Promise<{ isLoggedIn: boolean; jobId?: string }>
  checkSession: () => Promise<{ isLoggedIn: boolean; jobId?: string; needsReauth?: boolean }>  // ← 新增
  login: (username: string, password: string, days?: number) => Promise<{ ... }>
  logout: () => Promise<{ success: boolean }>
}

// createAuthPreloadApi() 中新增
checkSession: () => ipcRenderer.invoke(AUTH_IPC_CHANNELS.CHECK_SESSION),
```

#### 2.1.3 `renderer/App.tsx` — 启动时恢复登录

在 `App` 组件的初始化 `useEffect` 中，增加一步：调用 `checkSession` 恢复登录状态。

```typescript
// App.tsx 初始化 useEffect 中新增
React.useEffect(() => {
  const initialize = async () => {
    try {
      // 1. 恢复登录状态（新增）
      const session = await window.electronAPI.auth.checkSession()
      if (session.isLoggedIn) {
        store.set(authStateAtom, {
          isLoggedIn: true,
          jobId: session.jobId,
        })
      }

      // 2. 检查是否需要显示 Onboarding（原有逻辑）
      const settings = await window.electronAPI.getSettings()
      if (!settings.onboardingCompleted) {
        setShowOnboarding(true)
      }
    } catch (error) {
      console.error('[App] 初始化失败:', error)
    } finally {
      setIsLoading(false)
    }
  }
  initialize()
}, [])
```

> **注意**：`checkSession` 放在 `getSettings` 之前，确保登录状态尽早恢复，后续 `PlatformChannelSync` 等组件挂载时能立即感知到已登录状态。

### 2.2 Token 有效期调整为 30 天

改动仅涉及 `auth/auth-service.ts` 一个文件，两处常量修改：

#### 2.2.1 长期 Token 有效期

```typescript
// auth-service.ts

// 修改前
export async function loginWithEipGateway(
  username: string, password: string, longTermDays: number = 365,
): Promise<LoginResult> {

// 修改后
export async function loginWithEipGateway(
  username: string, password: string, longTermDays: number = 30,
): Promise<LoginResult> {
```

#### 2.2.2 强制重登周期

```typescript
// auth-service.ts

// 修改前
const FORCED_REAUTH_DAYS = 180

// 修改后
const FORCED_REAUTH_DAYS = 30
```

> **说明**：长期 Token 有效期和强制重登周期统一为 30 天，意味着：
> - 登录后获取的长期 Token 自身有效期为 30 天
> - 自 Token 签发日起 30 天后，`needsReauth` 返回 `true`，UI 可据此提示用户重新登录
> - 两者一致，不会出现"Token 还有效但被强制要求重登"的情况

## 四、测试要点

### 4.1 启动自动恢复登录

| 场景 | 预期 |
|------|------|
| 上次登录后关闭，Token 未过期，重新打开 | 自动恢复登录状态，显示已登录 |
| 上次登录后关闭，Token 已过期，重新打开 | 显示未登录状态 |
| 从未登录过（无 auth.json），打开 | 显示未登录状态 |
| auth.json 文件损坏 | 显示未登录状态，不崩溃 |

### 4.2 Token 有效期

| 场景 | 预期 |
|------|------|
| 登录成功 | `expiresAt` = 当前时间 + 30 天 |
| 登录后 29 天 | `isLoggedIn()` 返回 true，`needsReauth()` 返回 false |
| 登录后 31 天 | `isLoggedIn()` 返回 false（Token 自身过期），`needsReauth()` 返回 true |

### 4.3 兼容性

| 场景 | 预期 |
|------|------|
| 旧版本登录的 auth.json（365 天有效期） | 如果未过期，仍可正常恢复登录；过期后按正常流程处理 |
| 旧版本 auth.json 无 `createdAt` 字段 | `getAuthInfo()` 已兼容，回退使用 `lastLoginAt` |
