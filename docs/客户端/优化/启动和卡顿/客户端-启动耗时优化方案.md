# 客户端启动耗时优化方案

> 数据来源：`docs/客户端/优化/启动耗时.txt`
> 测量环境：Windows，受管终端（htsc.com.cn 域），打包正式版 WorkMate v0.12.37
> 现象：从双击 EXE 到首屏可见约 **83 秒**

---

## 一、总体结论

从进程启动到"窗口已显示（首屏可见）"耗时约 **83 秒**，严重异常。

根因不是单一环节慢，而是三类问题叠加：

1. **bootstrap 被完整执行了 3 次**（单实例锁未生效 / 进程重复拉起）
2. **单次运行时探测就要约 12 秒**，且全程串行 + 同步阻塞
3. **所有初始化都堵在主窗口创建之前**，用户长时间看到"双击没反应"

辅以日志同步写盘进一步放大每步开销。

计时口径说明：日志中的 `+X.XXs` 以主进程模块顶层的 `STARTUP_TIME` 为基准（`apps/electron/src/main/index.ts:5`），同一进程内必然单调递增。

---

## 二、问题清单与证据

### 问题 1（最致命）：bootstrap 被完整执行了 3 次

**证据：**

- `[运行时初始化] 完成` 出现 **3 次**：`11930ms` / `11459ms` / `11944ms`
- `app.whenReady() 回调触发` 出现 **3 次**
- `单实例锁已获取` 出现 **3 次**（`+1.59s` / `+21.85s` / `+21.78s`）
- 计时 `+23.99s` 之后又跳回 `+21.85s` —— 同一进程计时不可能回退，说明这是**另一个进程**的计时

**矛盾点：**
按代码逻辑，拿不到单实例锁的进程应在 `app.quit()` 处退出（`apps/electron/src/main/index.ts:38-43`），**不会**打印 `单实例锁已获取`。但日志里三个进程都打印了该行，说明**单实例锁实际未生效**。

**疑似成因：**
- 旧版本进程残留 / 卡死，未释放锁也未正常被新实例唤起
- 自更新（updater）或安装器 stub 触发了重复 relaunch
- 多个用户态进程并发启动

**影响：** 仅重复执行就额外浪费约 `12s × 2 ≈ 24s`。

#### 成因推断：用户反复双击 + 前序进程已退出（叠加）

一个直觉假设是"用户因为打不开而不断双击"。但**单纯反复双击解释不了这份日志**，必须叠加"前一个进程已经死了"才成立。

单实例锁逻辑（`apps/electron/src/main/index.ts:37-49`）：
- 拿到锁 → 打印 `单实例锁已获取`，执行完整 bootstrap
- 拿不到锁 → 打印 warn `已有 Proma 进程持有单实例锁` 后 `app.quit()`，并触发主实例的 `second-instance`

若进程 A 仍存活并持锁，用户再双击出来的 B / C 应**拿不到锁**而直接退出，日志里应出现 warn + `second-instance`。

但实际日志中：
- `单实例锁已获取` 出现 **3 次**
- 拿不到锁的 warn **一次都没有**
- `second-instance` **没有**

说明三个进程**全部成功拿到了锁**。在锁正常工作的前提下，这只可能意味着：**每次新进程启动时，前一个持锁进程已经不在了（锁已释放）。**

时间线佐证：三个进程启动时刻约为 `01:24:56` / `01:25:08` / `01:26:00`，间隔几十秒，符合"人等不到反应、隔一会儿再点"的节奏，而非毫秒级连点或程序自动 relaunch。

**最可能的真实场景：**
1. 用户双击 → A 启动，卡在约 12s 的运行时探测中，窗口迟迟不出现；
2. 用户以为没启动 → A 崩溃 / 被杀软拦截结束 / 被用户在任务管理器结束，锁随之释放；
3. 用户再双击 → B 拿到锁，重复同样的卡顿……如此三次。

**结论：** 反复双击是**诱因**，但前提是**前序进程已退出释放了锁**。若前序进程一直存活，反复双击只会被锁挡住，不会重复执行 bootstrap。

#### 验证手段

- 启动期间观察任务管理器，确认是否存在多个 `WorkMate.exe`，以及它们是否会消失（异常退出）；
- 检查崩溃日志 / 杀软拦截记录，确认前序进程是否被异常结束；
- 在单实例锁失败分支补一条更醒目的日志，确认 `second-instance` 是否真的触发。

> 注：无论触发方式如何，根因仍是首屏太慢（首屏 ~83s / 单次探测 ~12s）。落地方案 1「窗口优先、探测后置」将首屏降到秒级后，用户不会再误判"没启动"，反复双击与重复启动大概率自然消失。

---

### 问题 2：单次 `initializeRuntime` 约 12 秒，全程串行 + 同步阻塞

**证据：**

- 单次 `[运行时初始化] 完成 (11930ms)`
- 相邻探测步骤几乎固定相差 **~1.5s**，连"新增 0 个路径""PATH 加载完成"这类基本不干活的步骤也耗时 1.5s
- `Bun 检测` 未找到却仍耗时

**代码定位：**
- 探测顺序串行 `await`：`loadShellEnv → Node → Bun → Git → Git Bash`（`apps/electron/src/main/lib/runtime-init.ts:38-110`）
- Windows PATH 加载使用同步阻塞调用 `execSync('reg query ...')`（`apps/electron/src/main/lib/windows-env.ts:32-40`）

**根因判断：**
每步都 spawn 子进程或查询注册表。在受管终端上，**每次进程 spawn / 注册表查询都会被 EDR / 杀软拦截扫描**，产生约 1.5s 的固定单步开销。这解释了为何"不干活"的步骤也慢。

---

### 问题 3：所有初始化都堵在主窗口创建之前

**证据：**

- `准备创建主窗口` 发生在 `+64s`
- `窗口已显示（首屏可见）` 在 `+83s`

运行时探测、Skills/Plugins 初始化、IPC 注册等**全部位于关键路径上**，窗口在最后才创建。用户在 80 多秒内的直观体验是"双击没反应"。

---

### 问题 4：日志同步写盘放大开销

**代码定位：**
`apps/electron/src/main/lib/file-logger.ts:83-99` —— 每条日志都同步执行 `existsSync` / `statSync` + `appendFileSync`，运行在主线程。

**影响：** 受管机器上文件写入同样被杀软 hook，每条日志的写入开销被进一步放大。

---

### 问题 5：日志时间戳是 UTC，而非本地时区（北京时间）

**证据：**

- 日志行形如 `2026-06-22T05:40:39.515Z`，末尾 `Z` 表示 UTC（零时区）
- 与日志开头 `2026-06-22T01:24:56.202Z` 一致，相比北京时间（UTC+8）整整慢 8 小时

**代码定位：**
`apps/electron/src/main/lib/file-logger.ts:64` —— `formatLogLine` 使用 `new Date().toISOString()` 生成时间戳。`toISOString()` **始终输出 UTC 时间**并带 `Z` 后缀，与本地时区无关。

**影响：** 排查问题时需要手动 +8 小时换算，容易与其他系统日志、用户反馈时间对不上，干扰故障定位。

---

## 三、优化方案（按性价比排序）

### 方案 1（首选）：窗口优先，探测后置

把 `initializeRuntime` 从启动关键路径移出：先创建并显示主窗口，待 `ready-to-show` 后再异步执行运行时探测；渲染层用 loading / 降级态等待结果，探测完成后通过 IPC 推送。

**当前阻塞点：** bootstrap 中 `await initializeRuntime()`（`apps/electron/src/main/index.ts:539`）在 `createWindow()`（`apps/electron/src/main/index.ts:590`）**之前**，窗口被排在全部探测之后，直接导致 `准备创建主窗口` 发生在 +64s。

**后置的探测项**（均来自 `initializeRuntime`，`apps/electron/src/main/lib/runtime-init.ts:38-110`，单次合计约 12s）：

| 探测项 | 代码 | 说明 / 为何可后置 |
| --- | --- | --- |
| Shell 环境加载 / Windows PATH | `loadShellEnv`（Win 下为 `loadWindowsEnv`，`windows-env.ts`） | 从注册表合并 PATH，仅影响后续子进程能否找到工具，首屏不需要 |
| Node.js 检测 | `detectNodeRuntime`（`node-detector`） | 只有真正运行 Agent / 执行命令时才需要，首屏不需要 |
| Bun 检测 | `detectBunRuntime`（`bun-finder`） | 非必须运行时，日志显示未找到，更应延后 |
| Git 检测 | `detectGitRuntime`（`git-detector`） | 仓库相关操作才用到，首屏不需要 |
| Git Bash 检测（Windows） | `detectGitBash`（`git-bash-detector`） | Shell 环境探测，首屏不需要 |

**注意：** 仅后置"运行时探测"，IPC handlers 注册（`registerIpcHandlers`，`apps/electron/src/main/index.ts:558`）等需要在渲染层调用前就绪的逻辑仍应前置，避免窗口加载后调用 IPC 失败。

- **收益：** 首屏从约 83s 降到几秒级

### 方案 2：修复重复启动

阅读当前单实例锁实现（`apps/electron/src/main/index.ts:41-52`）后，发现其设计依赖一条脆弱的链路，且存在几处具体缺陷。

**当前实现：**
```js
if (!app.requestSingleInstanceLock()) {
  console.warn('已有 Proma 进程持有单实例锁，本次启动将退出。')
  app.quit()          // 没拿到锁 → 退出
} else {
  registerProtocolsAndHandlers()  // 拿到锁 → 注册 second-instance 监听
}
```
设计意图：副本进程退出，由主实例的 `second-instance` 事件调 `showAndFocusMainWindow()`（`apps/electron/src/main/index.ts:303-319`）把已有窗口唤到前台。

**问题 A：`second-instance` 唤起依赖"主窗口已存在"，但主实例正卡在探测里**
`mainWindow` 要等 bootstrap 跑到 `createWindow()`（`apps/electron/src/main/index.ts:590`）才被赋值，即 `await initializeRuntime()`（约 12s）之后。在这段窗口期内主实例**持有锁但无窗口**：副本被锁挡住退出并触发 `second-instance`，但 `showAndFocusMainWindow()` 里 `mainWindow` 仍为 `null`，会走 `createWindow()`，与主流程稍后的 `createWindow()` 存在竞态/重复创建风险，用户仍感觉"双击没反应"。注释亦自承："如果窗口未出现，可能旧进程已卡死"。

**问题 B：锁失败用 `app.quit()` 而非 `app.exit(0)`，副本进程可能无法秒退**
`app.quit()` 是优雅退出，会触发 `before-quit`（`apps/electron/src/main/index.ts:733`）。该钩子在 `isWorkmateShutdownComplete` 为 `false` 时 `event.preventDefault()` 并调 `shutdownWorkmateServices()`，而副本进程**从未初始化过**这些服务，可能被钩子拖住、延迟甚至卡住退出，加剧资源竞争。应改用 `app.exit(0)` 立即终止。

**问题 C：日志证据表明锁"没挡住"是因为前序进程已死**
日志中 3 次 `单实例锁已获取`、0 次 warn、0 次 `second-instance`，说明每个进程都拿到了锁——前序进程在下次启动前已退出（崩溃 / 被杀软结束 / 被用户结束）。锁机制本身没坏，但"主实例卡死 → 退出 → 锁释放 → 新进程又卡死"形成恶性循环。

**修复措施：**
1. **治本**：落地方案 1（窗口优先），让窗口秒级显示，消除 ~12s 无窗口期 → `second-instance` 可正常唤起，用户也不会误判"没启动"。
2. **加固锁失败分支**：`app.quit()` → `app.exit(0)`，确保副本进程秒退、不被 `before-quit` 钩子拖住。
3. **加固 `second-instance`**：当 `mainWindow` 尚未创建时，记录"启动中"状态并合并请求（如设标志位 / 暂存待唤起），避免与主流程 `createWindow()` 竞态重复创建。
4. **可观测**：在锁失败分支与 `second-instance` 各补一条醒目日志，便于确认实际走了哪条路径。

- **收益：** 消除约 24s 的重复开销，并修复"卡死期间二次启动唤不起窗口"的体验问题

### 方案 3：探测并行化

当前 `initializeRuntime`（`apps/electron/src/main/lib/runtime-init.ts:38-110`）是 5 项**全串行 `await`**：`loadShellEnv → Node → Bun → Git → Git Bash`。但并不能简单把 5 项全塞进一个 `Promise.all`——存在一处**顺序依赖**必须保留。

#### 关键依赖：PATH 加载必须先于探测器

`loadShellEnv()`（Windows 为 `loadWindowsEnv()`）会**修改 `process.env.PATH`**（`windows-env.ts:120-145` 的 `mergeRegistryPath`）。而 Node / Bun / Git / Git Bash 探测器都依赖 PATH 查找可执行文件，`git-bash-detector.ts:24` 注释明确："在调用时读取 `process.env`，确保 `loadWindowsEnv()` 已执行后路径完整。"

因此正确结构是**两阶段**：

- **阶段 1（串行）**：先 `await loadShellEnv()`，确保 PATH 合并完成。
- **阶段 2（并行）**：Node / Bun / Git / Git Bash 之间互不依赖，用 `Promise.all` 并发。

> 佐证：`environment-checker.ts:86` 已有先例——`Promise.all([detectNodeRuntime(), detectGitRuntime()])`，证明这些探测器并行安全。

#### 改造示意

改造前（串行，约 12s = 各项之和）：
```js
const shellEnvResult = await loadShellEnv()      // 阶段1
const nodeStatus = await detectNodeRuntime()      // ┐
const bunStatus  = await detectBunRuntime()       // │ 串行，逐项等待
const gitStatus  = await detectGitRuntime()       // │
const gitBash    = await detectGitBash()          // ┘
```

改造后（PATH 先行 + 探测并行，约等于"最慢一项"）：
```js
// 阶段1：PATH 必须先加载（探测器依赖它）
const shellEnvResult = await loadShellEnv()

// 阶段2：四项探测互不依赖，并行执行
const [nodeStatus, bunStatus, gitStatus, gitBashStatus] = await Promise.all([
  options.skipNodeDetection ? Promise.resolve(SKIPPED_NODE) : detectNodeRuntime(),
  options.skipBunDetection  ? Promise.resolve(SKIPPED_BUN)  : detectBunRuntime(),
  options.skipGitDetection  ? Promise.resolve(SKIPPED_GIT)  : detectGitRuntime(),
  (process.platform === 'win32' && !options.skipShellDetection)
    ? detectGitBash()
    : Promise.resolve(undefined),
])
```

#### 注意事项

1. **错误隔离**：用 `Promise.allSettled` 或给每个探测包一层 try/catch，避免任一项 reject 导致整体失败（现状各探测器内部已自行兜底返回 `available:false`，但并行后仍建议显式隔离）。
2. **保留各项的 `skipXxx` 选项**：并行写法里用 `Promise.resolve(默认值)` 占位，保持与现有 `RuntimeInitOptions` 行为一致。
3. **不要把 `loadShellEnv` 并入 `Promise.all`**：否则探测器可能在 PATH 合并完成前读取到不完整的 PATH，导致漏检（尤其 Windows 从注册表加载 PATH 的场景）。

- **收益：** 单次约 12s 降到约"最慢一项"的耗时（阶段 1 + 阶段 2 中最慢者）

### 方案 4：缓存探测结果

把 node / git / bun 的路径与版本写入磁盘缓存，下次启动**先读缓存、后台静默校验**，避免每次冷探测。

#### 缓存结果在哪里被应用

| 消费点 | 代码 | 用途 | 风险等级 |
| --- | --- | --- | --- |
| Agent preflight | `apps/electron/src/main/lib/agent-orchestrator.ts:244` | Windows 下校验 `gitBash.available`，不可用直接报错阻止运行 | 高 |
| 注入 Shell 路径 | `apps/electron/src/main/lib/orchestrator/sdk-env.ts:79-84` | 把 `gitBash.path` 写入 `CLAUDE_CODE_SHELL` 传给 SDK 子进程 | 最高 |
| UI 展示 | AboutSettings / EnvironmentCheckPanel（IPC `GET_RUNTIME_STATUS`） | 展示环境状态 | 低 |

即缓存主要在**运行 Agent 时**和**展示环境状态时**被应用。

#### 风险：缓存陈旧（用户卸载 / 移动了 git / node）

- **最危险**：`sdk-env.ts:84` 将缓存里的 `gitBash.path` 直接当 `CLAUDE_CODE_SHELL` 注入。用户卸载 / 移动 Git 后路径已失效 → SDK 子进程指向不存在的 bash → Agent 执行失败且报错隐晦。
- node 同理：缓存标记 available，实际已卸载 → 依赖功能失败。
- 反向：缓存标记"无 git"、用户后来装了 → preflight 误报"环境未就绪"挡住用户（可由手动刷新补救）。

> 注意：现状探测是"找路径 + 实跑取版本"两步（如 `node-detector.ts:150-181`），已验证"可执行"。**纯读缓存会丢掉这层保证**，这是缓存方案必须正视的核心风险。

#### 失效兜底设计（缓存不能成为唯一真相）

1. **启动后必做后台重探校正**：缓存只作"乐观快速值"用于秒级展示；窗口显示后后台静默跑一次真实探测，结果有差异则更新缓存 + 通过 IPC 推送刷新 UI。
2. **使用前轻量校验（关键）**：在 `sdk-env.ts:79-84` 注入 `CLAUDE_CODE_SHELL` 前、以及 orchestrator preflight 时，对缓存 `path` 做一次廉价 `existsSync`；不存在则视为不可用并触发按需重探，**绝不把失效路径喂给子进程**。
3. **缓存带环境指纹 + TTL**：记录 PATH 哈希、各工具 path/version、OS、appVersion。下次启动指纹不变才信任缓存（仍后台校验）；指纹变化（PATH 变动往往意味着装/卸载工具）则不信任、走完整探测。
4. **保留手动刷新**：`reinitRuntime`（`runtime:reinit`）作为用户主动纠正入口。

**原则：** 缓存用于"加速首屏可用感"，但**运行 Agent 的关键路径必须有 `existsSync` 级别的兜底**，否则卸载 git 后会出现"以为能跑、结果失败"的隐蔽 bug。

- **收益：** 二次及以后启动几乎不受探测开销影响

### 方案 5：PATH 加载改异步

将 `execSync('reg query')` 改为异步 `execFile`，或直接使用 Node 的注册表读取方式，避免阻塞主线程（`apps/electron/src/main/lib/windows-env.ts:32-40`）。

### 方案 6：日志异步化

改为内存缓冲 + 异步批量刷盘，去掉每行的 `existsSync` / `statSync`（`apps/electron/src/main/lib/file-logger.ts:83-99`）。

### 方案 7：日志时间戳改为本地时区

将 `formatLogLine` 中的 `new Date().toISOString()` 改为输出本地时区时间（`apps/electron/src/main/lib/file-logger.ts:64`）。建议：

- 使用本地时间格式（如 `YYYY-MM-DD HH:mm:ss.SSS`），并保留时区偏移信息（如 `+08:00`）便于跨时区核对；
- 或使用 `Intl.DateTimeFormat` 指定时区，避免依赖运行机器的系统时区设置不确定性。

- **收益：** 日志时间与本地时间一致，排查无需手动 +8 小时换算

## 四、建议落地顺序

0. **方案 6 已实现 → 先做对照实验（前置 Gate）**：日志异步化已落地（`file-logger.ts`）。出包在受管机实测首屏耗时，**用数据确认日志是否为头号瓶颈**，据此校正下方优先级。

## 四-B、v0.12.51 实测分析与修复（2026-06-26）

> 测量环境：Windows，受管终端（htsc.com.cn 域），打包正式版 WorkMate v0.12.51
> 首屏可见耗时：**79 秒**

### 版本演进

| 版本 | 关键变化 | 首屏耗时 |
|------|----------|----------|
| v0.12.37 | 同步日志写盘 + 探测阻塞窗口创建 | 83s |
| v0.12.39 | **日志异步化**（方案 6 落地） | 快（推测 ~15s） |
| v0.12.51 | 窗口优先 + 后台探测 + 缓存预热 + `BACKGROUND_INIT_FALLBACK_MS=3s` | 79s |

### 问题定位：并发 I/O 触发 AV 串行化

v0.12.51 中方案 1/2/3/4 的部分优化已落地（窗口提前创建、单实例生效、缓存预热、探测并行化），但首屏仅从 83s 降到 79s，几乎无改善。根因是**主进程的后台探测与渲染进程的页面加载并发执行，在受管机上触发了 AV/EDR 的资源竞争串行化**。

#### 时间线证据

```
主进程                                    渲染进程
+6.45s  loadFile 发出 ────────────────────→ Chromium 开始加载
+6.57s  bootstrap 核心初始化完成
        安排 setTimeout(scheduleBackgroundInit, 3000)
                                          
        █ 主进程事件循环被 AV 间歇性阻塞
        █ 3s 定时器实际 36s 后才触发
        █ 15s Bridge 超时实际 32s 后才触发
                                          
+38.45s Bridge 超时兜底触发（预期 15s，实际延迟 17s）
+42.59s 后台初始化开始（预期 3s，实际延迟 33s）
        initializeRuntime 开始
          execSync('reg query') ────────→ ← 争夺文件 I/O / CPU
          spawn Node/Bun/Git 检测 ──────→ ← 争夺文件 I/O / CPU
                                          
+54.80s initializeRuntime 完成（12.2s）
+54.85s 全部后台初始化完成
                                          
                                          █ 24s：渲染进程恢复加载
                                          
+79.19s  dom-ready → ready-to-show → 首屏可见
+79.19s  preload 执行（<1ms）
+79.19s  React 渲染（<10ms）
```

#### 两个关键定时器异常延迟

| 定时器 | 设定值 | 预期触发 | 实际触发 | 延迟 |
|--------|--------|----------|----------|------|
| `BACKGROUND_INIT_FALLBACK_MS` | 3s | +9.6s | +42.6s | +33.0s |
| `DEFAULT_FALLBACK_DELAY_MS` (Bridge) | 15s | +21.6s | +38.5s | +16.9s |

两个独立的 `setTimeout` 都大幅延迟且延迟量不同，说明主进程事件循环在 +6.57s~+42.6s 期间被**间歇性阻塞**——正是 AV 扫描 asar 归档时占用内核态、不归还 CPU 给 Node.js 事件循环的典型特征。

#### 根因链

1. `BACKGROUND_INIT_FALLBACK_MS = 3000` 在渲染进程加载期间过早启动后台探测
2. 后台探测的 `execSync('reg query')` 和子进程 spawn 与 Chromium 的 asar 读取并发执行
3. AV/EDR 发现同一可执行文件的多进程同时做 I/O → 串行化扫描
4. 渲染进程加载从正常的 ~7s 恶化到 72s

#### 对比 v0.12.39 为何快

v0.12.39 中 `BACKGROUND_INIT_FALLBACK_MS` 大概率不存在或值很大，后台探测只在 `ready-to-show` 之后才触发，渲染进程加载期间主进程完全空闲，无 I/O 竞争。

### 已实施的修复（v0.12.51+）

**修复 1：删除 `BACKGROUND_INIT_FALLBACK_MS` 兜底定时器**

代码位置：`apps/electron/src/main/index.ts`

改动：
- 删除 `const BACKGROUND_INIT_FALLBACK_MS = 3000`
- 删除 `setTimeout(scheduleBackgroundInit, BACKGROUND_INIT_FALLBACK_MS)`
- 后台初始化现在仅由 `createWindow` 的 `ready-to-show` 回调触发

**修复 2：`createQuickTaskWindow` 移出关键路径**

`createQuickTaskWindow` 在 bootstrap 关键路径中会创建第二个 `BrowserWindow` 并执行 `loadFile`，与主窗口的渲染进程并发从 asar 加载，加剧 AV 串行化。现已移至 `runBackgroundInit`（`ready-to-show` 之后执行），保留在后台初始化中的调用。

**修复 3：拉长 Bridge 启动超时兜底**

`startup-bridge-scheduler.ts` 中 `DEFAULT_FALLBACK_DELAY_MS` 从 15s 拉长到 120s，避免 Bridge 在受管机上过早启动加剧资源竞争。

**修复 4：`seedDefaultPlugins` 加 `safeRun` 包裹**

与其他 seed 操作保持一致，防止单点异常崩溃 bootstrap。

**修复 5：补空窗日志**

在 `bootstrap()` 返回后追加一条日志 `bootstrap 完成，等待渲染进程加载（主进程空闲，无后台任务）`，明确标记主进程空闲期，便于排查。

### 修复后的启动流程

```
+0.02s   模块加载完成
+6.11s   app.whenReady() 回调触发
+6.45s   loadFile 发出（仅主窗口，无其他渲染进程，无后台探测）
         ↓ 主进程事件循环空闲，无 setTimeout，无 execSync
+~13s    ready-to-show → 首屏可见
         → scheduleBackgroundInit()
             → createQuickTaskWindow()（此时主窗口已加载完，无 asar 竞争）
             → initializeRuntime()（12s，但首屏已可见）
             → Bridge 启动
+~25s    全部后台初始化完成
```

### 预期收益

| 指标 | 修复前 | 修复后（预期） |
|------|--------|----------------|
| 首屏可见 | 79s | ~15s |
| 渲染进程加载 | 72s（竞争） | ~7s（无竞争） |
| 后台探测完成 | +54.8s | +25s |

## 五、运行时探测的触发入口与现状

运行时探测（`initializeRuntime` 全套 5 项，或 `checkEnvironment` 的 Node+Git 子集）除启动外，还可由以下入口触发。落地方案 1/4 时需一并核对这些入口，避免缓存被绕过或重复探测。

| 入口 | 是否可达 | 打开是否自动探测 | 调用 | 探测范围 |
| --- | --- | --- | --- | --- |
| 启动 bootstrap | 必经 | 是 | `initializeRuntime` | 全套 5 项 |
| Onboarding Step2（仅 Windows，首次启动） | 可达 | 是（`autoDetectOnMount`） | `reinitRuntime` | 全套 5 项（清缓存重跑） |
| Agent 报错 →「打开环境检测」弹窗（仅 Windows 缺 Git Bash） | 条件触发 | 是（`autoDetectOnMount={open}`） | `reinitRuntime` | 全套 5 项（清缓存重跑） |
| ~~设置 - 环境检测弹窗~~ | **不可达（已无入口）** | — | — | — |
| 关于页 - 环境检测卡 | 可达 | 否，需手点「重新检查」 | `checkEnvironment` | 仅 Node + Git（并行） |
| 关于页 - Shell 环境卡 | 可达 | 否，需手点「重新检测」 | `reinitRuntime` | 全套 5 项（清缓存重跑） |

**关键现状与隐患：**

1. **`EnvironmentCheckDialog` 已无设置入口**：它仍在 `App.tsx:207-209` 全局挂载，但唯一打开方式是 Agent 报错卡片的 recovery action（`SDKMessageRenderer.tsx:1196`，仅 Windows 缺 Git Bash 时出现，来源 `agent-orchestrator.ts:257`）。它已从"设置入口"变为"运行报错急救入口"。`EnvironmentCheckPanel.tsx:5` 注释中"设置里的 EnvironmentCheckDialog 复用"一句**已过时**，与现状不符。

2. **首次启动 Windows 会重复探测**：bootstrap 已探一次（写入缓存），进入 Onboarding Step2 时 `autoDetectOnMount` 又调 `reinitRuntime` **清缓存重跑**，完全不复用。配合方案 1/4 时，应改为默认读缓存（`getRuntimeStatus`），仅用户手点刷新才 `reinitRuntime`。

3. **关于页打开不自动探测**：环境检测卡挂载时只读 `settings.lastEnvironmentCheck`（旧值，`AboutSettings.tsx:283-291`），Shell 卡只读内存缓存 `getRuntimeStatus`（`AboutSettings.tsx:397-401`）；二者都需用户手点按钮才实跑。

4. **两套探测结果互不相通**：关于页用 `checkEnvironment` → 落 `settings.lastEnvironmentCheck`；其余走 `runtimeStatusCache`（内存）。即便在关于页刷新了 Node/Git 状态，Agent 实际读取的缓存仍是旧的。这也是方案 4「统一到同一套带缓存探测」的动机之一。

5. **缓存无自动失效**：当前 `runtimeStatusCache` 是内存变量（`runtime-init.ts:20`），无 TTL / 无 `existsSync` 复检 / 无 PATH 监听。用户中途卸载 git 后缓存保持陈旧，直到重启或手动刷新——印证方案 4 的兜底设计必要性。

## 六、实现待补设计（开放问题与动作项）

> 本节回应方案评审中发现的实现空白。**日志异步化已落地（方案 6 已实现，见 `file-logger.ts`：内存缓冲 + `setImmediate` 批量刷盘 + `flushFileLoggerSync` 退出兜底）**，下列其余项需在动手前确认。

### 6.1 方案 1（窗口优先）待补管线

**(a) 运行时状态推送通道（当前是 pull，需补 push）**
现状只有 pull 式 `GET_RUNTIME_STATUS`（`ipc.ts:767`）。窗口先出、探测后台跑完后，需要主动通知渲染层刷新，否则 UI 停在 loading。动作项：
- 主进程新增事件 `runtime:status-updated`，在后台 `initializeRuntime()` 完成时 `webContents.send`；
- preload 增加订阅桥（参照现有 `electronAPI` 模式）；
- renderer 订阅后更新 `runtimeStatusAtom`，loading 态切到结果态。

**(b) 探测未完成时的 null 消费策略（关键）**
后台探测期间，`agent-orchestrator.ts:244`、`sdk-env.ts:79` 读到的 `getRuntimeStatus()` 为 `null`。必须区分"**探测中**"与"**探测完成但不可用**"，否则会误报"环境未就绪"。动作项：
- `initializeRuntime()` 启动时保存其 in-flight Promise（如 `runtimeReadyPromise`）；
- preflight / sdk-env 在状态为 `null` 时 **`await` 该 Promise（带超时）** 而非直接判定缺失；
- 探测完成后正常读缓存。即"首屏不阻塞，但真正跑 Agent 时按需等待探测结果"。

**(c) `ANTHROPIC_*` 环境变量清理时序**
现状：模块顶层先清一次（`index.ts:100-107`，注释要求"必须在 `loadShellEnv` 之前"），因为 `loadShellEnv` 可能从 shell 配置重新注入 `ANTHROPIC_*`。探测后置后，`loadShellEnv` 改为窗口显示后的后台执行，故：
- 顶层首次清理保留（进程一启动即清）；
- **后台 `loadShellEnv` 完成后需再清一次**，确保被 shell 重新注入的变量被移除；
- 兜底：`sdk-env.ts:98-103` 在构造 SDK env 时已对未显式管理的 `ANTHROPIC_*` 置空，作为最后一道防线（保留）。

**(d) 启动耗时上报口径**
现状 `app_startup` 上报的是"bootstrap 全部完成"的时长（`index.ts:565`，`initWorkmateServices(Date.now()-STARTUP_TIME)`）。窗口优先后"启动完成"定义改变，需拆分指标避免前后不可比：
- `app_first_paint`：进程启动 → 窗口可见（`ready-to-show`，`index.ts:409`）——新的主指标；
- `runtime_ready`：进程启动 → 后台探测完成——独立指标；
- 旧 `app_startup` 字段语义需在上报侧标注口径变更（版本分界）。

### 6.2 方案 2（单实例锁）：先验证后修复

**Gate（必须先做）**：修复建立在"前序进程已死、锁已释放"的推断上。动手前先用 §二·问题1 的验证手段确认真实成因：
- 若确为"卡死→退出→重启"：方案 1 + `app.exit(0)` 有效；
- 若为 **auto-updater relaunch / 安装器 stub 重复拉起**：`app.exit(0)` 无效，需改更新器/安装器逻辑（另案）。

**待验证点**：副本进程改 `app.exit(0)` 后，主实例 `second-instance` 是否仍可靠触发。Electron 在副本调用 `requestSingleInstanceLock()` 时即向主实例发消息，理论上先于退出，但需在受管机实测确认（补一条 `second-instance` 命中日志即可观测）。

### 6.3 方案 4（缓存）落地细节

**(a) 存储位置与 schema**
- 位置：`app.getPath('userData')/runtime-cache.json`；
- 结构（含版本字段，便于未来迁移）：
```jsonc
{
  "schemaVersion": 1,
  "fingerprint": "<hash>",      // 见 (b)
  "probedAt": 1782100000000,
  "appVersion": "0.12.37",
  "platform": "win32",
  "node": { "available": true, "path": "...", "version": "24.16.0" },
  "git":  { "available": true, "path": "...", "version": "2.54.0" },
  "bun":  { "available": false },
  "shell":{ "gitBash": { "available": true, "path": "..." } }
}
```

**(b) 指纹必须取"注册表源 PATH（合并前）"**
`process.env.PATH` 会被 `loadShellEnv`/`loadWindowsEnv` 运行时改写（`windows-env.ts` 的 `mergeRegistryPath`），若对改写后的 PATH 取哈希会**自我污染**（每次启动指纹都变）。正确做法：指纹 = hash(注册表 HKLM+HKCU 原始 PATH 字符串) + OS + appVersion。指纹一致才信任缓存（仍后台校验），变化则走完整探测。

**(c) 统一两套探测（明确动作项）**
现状 `checkEnvironment`（落 `settings.lastEnvironmentCheck`）与 `runtimeStatusCache`（内存）互不相通（见 §五·4）。动作项：
- 让 `checkEnvironment` 与关于页改为读写**同一份统一缓存**（即上面的 `runtime-cache.json`）；
- 关于页「重新检查」、Onboarding、环境弹窗统一走带缓存的探测入口；
- 废弃 `settings.lastEnvironmentCheck` 独立字段（或令其只读映射到统一缓存）。

**(d) 首次启动 Windows 重复探测（从 §五·2 落为动作项）**
Onboarding Step2 的 `autoDetectOnMount` 当前调 `reinitRuntime`（清缓存重跑）。改为：
- 默认调 `getRuntimeStatus()` 复用 bootstrap/后台已探好的缓存；
- 仅当用户**手点刷新**时才 `reinitRuntime`。
- 同样适用于 Agent 报错弹窗入口。

### 6.4 横切项

**(a) 验收标准 / 量化目标**（实测后据真实数据微调）

| 阶段 | 指标 | 目标 |
| --- | --- | --- |
| 方案 6（已实现）实验 | 首屏耗时（受管机） | 验证日志是否为头号瓶颈，决定后续优先级 |
| 方案 1 | `app_first_paint` | < 3s（首屏可见） |
| 方案 3 | 单次 `initializeRuntime` | 从 ~12s 降到约"最慢一项" |
| 方案 4 | 二次启动 `runtime_ready` | 命中缓存时 < 1s |

**(b) macOS 覆盖**：文档现以 Windows 视角为主。macOS 上 `loadShellEnv()` 会 `execSync` 运行登录 shell 提取环境变量，可能是该平台的主导成本。方案 1（窗口优先）、方案 3（并行）、方案 4（缓存）对 macOS 同样适用，且 macOS 的 shell 加载更值得后置/缓存。需补一轮 macOS 实测。

**(c) 回归面评估**：列出隐式依赖"窗口显示时 runtime 已就绪"的功能，逐一确认在后台探测模式下的行为：
- Windows Agent preflight（`agent-orchestrator.ts:244`）—— 由 6.1(b) 的 await 策略覆盖；
- SDK Shell 注入（`sdk-env.ts:79`）—— 同上；
- 关于页 / Onboarding 的环境展示 —— 由 6.1(a) 推送刷新覆盖；
- 任何在 bootstrap 早期同步读 `getRuntimeStatus()` 的调用点（需 grep 复核）。
