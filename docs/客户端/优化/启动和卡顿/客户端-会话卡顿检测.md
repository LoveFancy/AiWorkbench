# 客户端会话卡顿检测（发送消息阻塞主线程）

> 数据来源：`docs/客户端/优化/会话卡顿情况.txt`
> 现象：发送消息时整个应用无响应（窗口卡死），疑似主进程（主线程）被同步文件 I/O 阻塞
> 背景：此前通过"异步写日志"缓解过一次卡顿，后续改动引入回归，发送消息再次卡顿

---

## 一、总体结论

发送一条消息的核心编排（`AgentOrchestrator.sendMessage`，`apps/electron/src/main/lib/agent-orchestrator.ts`）在 **Electron 主线程上执行了大量同步文件读写**，足以阻塞事件循环导致整窗无响应。

之前的"日志异步化"只解决了 `file-logger`，但**会话索引、会话 JSONL、插件/连接器配置的持久化与扫描仍是全同步**，因此发送消息卡顿回归。

代码顶部已埋下 Event Loop 健康检查（`agent-orchestrator.ts:88-97`，`setImmediate` 回调延迟 >50ms 告警），印证团队已怀疑主线程拥堵——而下述同步 I/O 正是拥堵来源。

计时口径：受管终端（EDR/杀软 hook）下，**每个 syscall（stat/read/write/copy/rename）都被拦截扫描**，单步开销被放大，因此"次数多"与"单次重"都会累加成卡顿。

> 路径约定：配置根目录 `~/.workmate/`；会话历史目录 `~/.workmate/agent-sessions/`；会话工作目录 `~/.workmate/agent-workspaces/{slug}/`；连接器目录 `{工作区}/connectors/`。

## 四、关键热点详解

### 热点 1（最致命）：每条消息全量扫描并读取所有 Skill/Plugin 内容

调用链：`getAgentPluginPaths(workspaceSlug)`（`orchestrator/sdk-path.ts:82-86`）→ `buildPluginRuntimePaths`（`plugin-registry-service.ts:763-773`）→ `listInstalledPlugins`（`plugin-registry-service.ts:626-654`）。

`listInstalledPlugins` 对**所有内置 + 用户插件**逐个调 `pluginInfoFromPath`（`plugin-registry-service.ts:529-562`），其中：

- `readManifest` → `readFileSync` 读 `plugin.json`（`plugin-registry-service.ts:163-183`）
- `discoverSkills` → `discoverSkillDirs`（`plugin-registry-service.ts:246-286`）：`readdirSync` 列 skill 目录，并对**每个 SKILL.md** 调 `readDescriptionFromMarkdown` 做 `readFileSync` **读全文**（`plugin-registry-service.ts:185-218`）
- `discoverMarkdownCapabilities('commands' / 'agents')`：`collectMarkdownFiles` **递归** `readdirSync` 整个目录树，并对**每个 .md** `readFileSync` 读全文（`plugin-registry-service.ts:220-233 / 291-311`）
- `discoverMcp` → `readFileSync` 读 `.mcp.json`（`plugin-registry-service.ts:313-322`）

**问题本质：**

1. **每发一条消息**就把所有插件的 plugin.json、每个 SKILL.md 全文、commands/agents 下每个 .md 全文、.mcp.json 全部同步读一遍；插件/skill 越多，`readFileSync` 次数越多（可达几十上百次）。
2. **读出的内容几乎全被浪费**：`buildPluginRuntimePaths`（`plugin-registry-service.ts:766-771`）最终只用 `plugin.enabled` 与 issues 过滤，输出的仅是**插件路径字符串**；所有 description/markdown 解析结果在此路径上未被使用——纯为拿路径而全量读盘。
3. 对照：`buildDynamicContext`（`agent-prompt-builder.ts:631`）注释明确 "Skills 已通过 SDK plugin 机制自动发现，无需手动注入"（`agent-prompt-builder.ts:671-672`），prompt 构建本身不读 skill 内容；真正读内容的是这条 `buildPluginRuntimePaths` 路径。

### 热点 2：会话索引 `agent-sessions.json` 的"原子写"被频繁触发

`updateAgentSessionMeta` → `writeIndex` → `writeJsonFileAtomic`（`safe-file.ts:14-33`）。一次"原子写"内部是 **4 个同步 syscall**：

1. `existsSync(filePath)`
2. `copyFileSync(filePath, bakPath)` —— **整份索引文件同步拷贝**（.bak 备份）
3. `writeFileSync(tmpPath, JSON.stringify(index))` —— **整份索引序列化 + 同步写**
4. `renameSync(tmpPath, filePath)`

一条消息会多次触发（#1 / #4 / #15 / #17 / #19）。会话越多、索引越大，单次 `copyFileSync` + `writeFileSync` 越慢。

> 这极可能就是"后来改动引入回归"的根源：崩溃安全机制（temp + rename + .bak 备份）把原本一次 `writeFileSync` 放大成 4 次同步 I/O + 一次全文件 copy。

### 热点 3：完成时全量回读会话 JSONL

每个回合结束用 `getAgentSessionMessages`（`agent-session-manager.ts:183-203`）把**整个会话历史**同步 `readFileSync` 全量读出再逐行 `JSON.parse`（调用点 #20）。会话越长，单次同步读 + 解析越久。

### 热点 4：连接器逐文件读 + connectors.json 重复读

- `connectors.json` 被读两次（#10、#13），#13 处（`agent-orchestrator.ts:1006`）未复用 #10（`agent-orchestrator.ts:597`）的预读结果，是可省的重复 I/O。
- `buildMcpServers` 对每个 enabled 连接器逐个 `readFileSync` 读 `mcp.json`（`mcp-builder.ts:70-73`）；`readSkillDirsFromConnectorJson` 对每个 CLI 连接器逐个读 `connector.json`（`agent-workspace-manager.ts:969-982`）。次数随连接器数线性增长。

## 五、卡顿主因排序与优化建议

| 优先级 | 热点 | 问题 | 建议 |
|---|---|---|---|
| 最高 | 热点 1（插件全量扫描读内容） | 每条消息读所有 SKILL.md / markdown 全文，次数随数量爆炸，且内容被丢弃 | `buildPluginRuntimePaths` 只需"启用插件路径"，应避免全量 `listInstalledPlugins`；改为只 `readdirSync` 列目录 + 读 `plugin.json` 的 enabled 状态，**跳过 skill/command/agent 内容读取**；或对枚举结果做进程级缓存 + 目录 mtime/指纹失效 |
| 高 | 热点 2（索引原子写） | 每次 4 syscall + 全文件拷贝，一条消息触发多次 | `writeJsonFileAtomic` 改异步（`fs.promises`）；合并/防抖 `updateAgentSessionMeta` 调用，减少触发次数 |
| 高 | 热点 3（JSONL 全量回读） | 每回合同步读 + 解析整文件 | 完成回执改用内存中已累积的消息，避免回读磁盘 |
| 中 | 热点 4（连接器逐文件读 + 重复读） | 次数随连接器数增长，含 1 次重复读 | #13 复用 #10 预读的 `connectorsConfig`；确认 `syncDefaultConnectorsToWorkspace` 是否每条消息都写 connectors.json |
| 中 | `appendSDKMessages` 同步追加 | 同步 appendFileSync | 改异步追加或内存缓冲批量刷盘 |
