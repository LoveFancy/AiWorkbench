**关键特性**：
- 单一全局 `keydown` listener（capture 阶段），通过 Map 分发
- 支持 `exclusive` 模式（最内层组件优先捕获）
- 支持用户自定义覆盖 + 冲突检测
- 输入法组合过程自动忽略 (`e.isComposing`)

**Electron 应用菜单层**（`main/menu.ts`）：

编辑菜单注册了以下 `role` 项，隐式绑定了系统级加速键：

| role | 隐式加速键 | 影响 |
|------|-----------|------|
| `cut` | Ctrl+X / Cmd+X | Chromium 剪切编辑命令 |
| `copy` | Ctrl+C / Cmd+C | Chromium 复制编辑命令 |
| `paste` | Ctrl+V / Cmd+V | Chromium 粘贴编辑命令 |
| `delete` | 仅 macOS Backspace | Chromium 删除编辑命令 |
| `selectAll` | Ctrl+A / Cmd+A | Chromium 全选编辑命令 |

与自定义 `accelerator + click` 菜单项（如 `Cmd+W → menu:close-tab`）不同，`role` 菜单项不会拦截 DOM 事件——键盘事件**仍然传播到渲染进程的 JS 监听器**。但 Chromium 同时会执行对应的编辑命令（如 selectAll 会选中聚焦元素内的所有文本）。在 JS handler 中调用 `e.preventDefault()` 可阻止该 Chromium 编辑命令的默认行为。

## 2. 方案设计

### 2.1 方案选型

**方案 A：直接在 FileBrowser 内监听 keydown 事件（推荐）**

在 FileBrowser 组件容器上挂载 `keydown` 事件监听，仅在文件树获得焦点/有选中项时响应。不走全局快捷键注册表，避免与已有 app 级快捷键冲突。

- 优势：局部作用域，不影响其他组件；实现简单直接；Delete/F2 等无修饰键按键不会和全局冲突
- 劣势：需要自行管理焦点状态

**方案 B：注册到全局快捷键系统**

在 `shortcut-defaults.ts` 中新增 `file-delete`、`file-copy` 等定义，通过 `useShortcut` 在 FileBrowser 中注册。

- 优势：统一管理，可在设置面板自定义
- 劣势：文件操作快捷键（Delete、Ctrl+C/V/X）在非文件树上下文中也会触发，需要额外判断焦点；Ctrl+C/V 等与系统剪贴板行为冲突；Delete 为无修饰键，全局注册会干扰输入框

**推荐方案 A**。文件树快捷键本质上是**局部作用域**的操作（仅在文件树有选中项且文件树区域获得焦点时才应响应），不适合注册为全局快捷键。参考 VS Code、Finder/Explorer 的行为，均是在文件树组件内部局部监听。

### 2.2 焦点管理策略

文件操作快捷键（尤其是 Delete、Ctrl+C/V/X）与系统/编辑器行为重叠，必须精确控制作用域：

```
焦点判定规则：
1. FileBrowser 容器 div 设置 tabIndex={0}，使其可聚焦
2. 用户点击文件行时，容器自动获得焦点
3. keydown 监听挂在容器上，仅在容器处于 focus 状态时生效
4. 点击容器外部区域（输入框、对话区等）时，容器失焦，快捷键自动失效
5. 重命名编辑状态下，快捷键监听需跳过（避免 Delete 键删文件而非删字符）
```

### 2.3 快捷键定义

| 快捷键 | 操作 | 触发条件 | 说明 |
|--------|------|---------|------|
| `Delete` / `Backspace` | 删除选中文件 | 有选中项，非重命名态 | 弹出确认对话框，复用现有 `handleRequestDelete` |
| `Ctrl+C` / `Cmd+C` | 复制选中文件路径 | 有选中项 | 将选中文件路径存入内部剪贴板（应用内状态），标记为"复制"模式 |
| `Ctrl+X` / `Cmd+X` | 剪切选中文件 | 有选中项 | 将选中文件路径存入内部剪贴板，标记为"剪切"模式；视觉上给被剪切项加淡化效果 |
| `Ctrl+V` / `Cmd+V` | 粘贴文件 | 内部剪贴板有内容 | 复制/移动文件到当前选中目录或根目录 |
| `F2` | 重命名 | 单选一个文件 | 复用现有 `handleStartRename` |
| `Ctrl+A` / `Cmd+A` | 全选 | 文件树聚焦 | 选中当前 FileBrowser 根级列表项；不含已展开子目录 |
| `Escape` | 取消选中/取消剪贴板 | 任意状态 | 清空选中 + 清空内部剪贴板 |
| `↑` / `↓` | 上下移动选中 | 文件树聚焦 | 键盘导航，移动焦点到上/下一个可见行 |
| `Enter` | 打开/展开 | 单选一个文件 | 文件→预览；文件夹→展开/收起 |

### 2.4 文件剪贴板设计

由于 Web 端无法直接操作系统文件级剪贴板（浏览器安全限制），采用**应用内虚拟剪贴板**方案：

```typescript
// 文件剪贴板状态（可用 Jotai atom 管理，跨 FileBrowser 实例共享）
interface FileClipboard {
  /** 剪贴板中的文件路径列表 */
  paths: string[]
  /** 操作模式：copy 复制 / cut 剪切 */
  mode: 'copy' | 'cut'
  /** 来源根路径（用于判断是否跨区域粘贴） */
  sourceRoot: string
}

// atoms/file-clipboard-atoms.ts
export const fileClipboardAtom = atom<FileClipboard | null>(null)
```

**粘贴行为**：
- **确定目标目录**：若选中项是文件夹则为该文件夹，若选中项是文件则为其父目录，无选中则为当前根目录
- **复制模式**：调用主进程 `copyFile(sourcePath, targetDir)` 逐个复制
- **剪切模式**：调用现有 `moveFile(sourcePath, targetDir)` 逐个移动；完成后清空剪贴板
- **同名冲突**：复制时自动添加 ` (1)` 后缀；移动时如目标同名则提示用户

### 3.2 FileBrowser keydown 监听核心代码结构

**架构难点**：FileBrowser 的 `entries` 状态仅包含根目录第一层项，但用户可选中任意深度的文件（通过展开子目录并点击）。`selectedPaths` 是 `Set<string>`，只存路径，不含 FileEntry 元数据（`name`、`isDirectory`）。因此键盘操作不能依赖 `entries.find()`，需要额外的元数据追踪机制。

**解决方案**：

1. **`entryMetaMapRef`**：维护 `path → { name, isDirectory }` 的可见节点元数据表。每个 FileTreeItem 渲染/更新时注册自身元信息，键盘 Delete/F2/Enter/粘贴目标判断都从该 Map 查询，避免依赖根级 `entries` 或“最近点击项”。
2. **点击聚焦容器**：FileBrowser 容器设置 `tabIndex={0}` 后，还需要在文件行点击/选中时主动调用 `containerRef.current?.focus()`，否则焦点可能仍停留在 body 或其他控件，导致容器 `onKeyDown` 不触发。
3. **`keyboardToggleSignal`**：状态信号 `{ path, ts }`，用于从 FileBrowser 父组件通知 FileTreeItem 子组件展开/收起。FileTreeItem 监听此信号，匹配自身 path 时执行 toggleDir。
4. **直接路径操作**：`setRenamingPath(path)` 无需 FileEntry；`selectAll` 选中根级 `entries` 所有项。

```tsx
// FileBrowser.tsx 核心改动

export function FileBrowser({ rootPath, ... }: FileBrowserProps) {
  const containerRef = React.useRef<HTMLDivElement>(null)
  const [fileClipboard, setFileClipboard] = useAtom(fileClipboardAtom)

  // ===== 可见节点元信息追踪 =====
  // FileBrowser 的 entries 只包含根级项；递归子项在 FileTreeItem 内部维护。
  // 因此由每个 FileTreeItem 渲染时把自身元信息注册到 Map，供键盘操作查找。
  const entryMetaMapRef = React.useRef(new Map<string, {
    name: string; isDirectory: boolean
  }>())

  const registerEntryMeta = React.useCallback((entry: FileEntry): void => {
    entryMetaMapRef.current.set(entry.path, {
      name: entry.name,
      isDirectory: entry.isDirectory,
    })
  }, [])

  // 修改现有 handleSelect：点击文件行时主动聚焦 FileBrowser 容器
  const handleSelect = React.useCallback((entry: FileEntry, event: React.MouseEvent) => {
    containerRef.current?.focus()
    const isMulti = event.metaKey || event.ctrlKey
    // ... 原有 selectedPaths 更新逻辑不变 ...
  }, [onSelectedDirectoryChange])

  // ===== 键盘展开/收起信号 =====
  // Enter 键按下时设置此信号，FileTreeItem 监听后执行 toggleDir
  const [keyboardToggleSignal, setKeyboardToggleSignal] = React.useState<{
    path: string; ts: number
  } | null>(null)

  // ===== 键盘快捷键处理 =====
  const handleKeyDown = React.useCallback((e: React.KeyboardEvent) => {
    // 重命名编辑状态下不拦截
    if (renamingPath) return
    // 忽略输入法组合
    if (e.nativeEvent.isComposing) return

    const isMod = navigator.userAgent.includes('Mac') ? e.metaKey : e.ctrlKey

    // Delete / Backspace → 删除选中
    if ((e.key === 'Delete' || e.key === 'Backspace') && !isMod && selectedPaths.size > 0) {
      e.preventDefault()
      e.stopPropagation()
      // 键盘删除：用 entryMetaMapRef 获取确认对话框所需信息
      // handleRequestDelete 接受 FileEntry 用于对话框文案；handleDelete 使用 selectedPaths 批量删除
      const firstPath = [...selectedPaths][0]
      const meta = entryMetaMapRef.current.get(firstPath)
      handleRequestDelete({
        path: firstPath,
        name: meta?.name ?? firstPath.split(/[/\\]/).pop() ?? '',
        isDirectory: meta?.isDirectory ?? false,
      } as FileEntry)
      return
    }

    // Ctrl+C / Cmd+C → 复制
    if (isMod && e.key === 'c' && selectedPaths.size > 0) {
      e.preventDefault()
      e.stopPropagation()
      setFileClipboard({ paths: [...selectedPaths], mode: 'copy', sourceRoot: rootPath })
      return
    }

    // Ctrl+X / Cmd+X → 剪切
    if (isMod && e.key === 'x' && selectedPaths.size > 0) {
      e.preventDefault()
      e.stopPropagation()
      setFileClipboard({ paths: [...selectedPaths], mode: 'cut', sourceRoot: rootPath })
      return
    }

    // Ctrl+V / Cmd+V → 粘贴
    if (isMod && e.key === 'v' && fileClipboard) {
      e.preventDefault()
      e.stopPropagation()
      handlePaste(fileClipboard)
      return
    }

    // F2 → 重命名（直接设置 renamingPath，无需 FileEntry 查找）
    if (e.key === 'F2' && selectedPaths.size === 1) {
      e.preventDefault()
      setRenamingPath([...selectedPaths][0])
      return
    }

    // Ctrl+A / Cmd+A → 全选（选中根级 entries 所有项）
    // 必须 preventDefault 阻止 Chromium selectAll 文本编辑命令
    if (isMod && e.key === 'a') {
      e.preventDefault()
      e.stopPropagation()
      setSelectedPaths(new Set(entries.map(item => item.path)))
      onSelectedDirectoryChange?.(null)
      return
    }

    // Escape → 清除选中 + 清除剪贴板
    if (e.key === 'Escape') {
      e.preventDefault()
      e.stopPropagation()
      clearSelection()
      setFileClipboard(null)
      return
    }

    // Enter → 打开/展开
    if (e.key === 'Enter' && selectedPaths.size === 1) {
      e.preventDefault()
      const path = [...selectedPaths][0]
      const meta = entryMetaMapRef.current.get(path)
      if (meta?.isDirectory) {
        // 目录：通过信号通知 FileTreeItem 展开/收起
        setKeyboardToggleSignal({ path, ts: Date.now() })
      } else {
        // 文件：打开预览
        onFilePreview?.(path)
      }
      return
    }
  }, [renamingPath, selectedPaths, fileClipboard, entries, rootPath, ...])

  return (
    <div
      ref={containerRef}
      tabIndex={0}
      className={cn('flex flex-col outline-none', ...)}
      onKeyDown={handleKeyDown}
      onClickCapture={handleRootClickCapture}
    >
      {/* 文件树：将 keyboardToggleSignal 传递给 FileTreeItem */}
      {entries.map((entry) => (
        <FileTreeItem
          key={entry.path}
          entry={entry}
          // ... 其他现有 props ...
          registerEntryMeta={registerEntryMeta}
          keyboardToggleSignal={keyboardToggleSignal}
        />
      ))}
    </div>
  )
}
```

> **注意**：`selectAll` 仅选中根级 `entries`，不递归选中已展开子目录的项。这与 macOS Finder 的行为一致（Finder 中 Cmd+A 仅选中当前视图层级的项目）。如需选中所有可见项（含已展开子目录），需要 FileTreeItem 向父组件上报已展开的子项路径，实现复杂度较高，建议 P2 阶段再考虑。

> **FileTreeItem 侧改动**：新增 `registerEntryMeta` 和 `keyboardToggleSignal` props。FileTreeItem 在 `useEffect` 中调用 `registerEntryMeta(entry)` 注册自身元信息，并监听 `keyboardToggleSignal`：当 `signal.path === entry.path` 且 `signal.ts` 变化时调用已有的 `toggleDir()`。递归渲染 children 时也要继续透传这两个 props，确保任意深度的可见子项都能被键盘操作识别。

### 3.3 粘贴操作处理

```typescript
const handlePaste = React.useCallback(async (clipboard: FileClipboard) => {
  // 确定目标目录：优先用 entryMetaMapRef 判断选中项是否为文件夹
  let targetDir = rootPath
  if (selectedPaths.size === 1) {
    const selectedPath = [...selectedPaths][0]
    const meta = entryMetaMapRef.current.get(selectedPath)
    if (meta?.isDirectory) {
      targetDir = selectedPath
    } else {
      // 选中的是文件，粘贴到其父目录
      const sep = selectedPath.includes('\\') ? '\\' : '/'
      const lastSep = selectedPath.lastIndexOf(sep)
      if (lastSep > 0) targetDir = selectedPath.substring(0, lastSep)
    }
  }

  try {
    for (const sourcePath of clipboard.paths) {
      if (clipboard.mode === 'copy') {
        await window.electronAPI.copyFile(sourcePath, targetDir)
      } else {
        await window.electronAPI.moveFile(sourcePath, targetDir)
      }
    }
    // 剪切完成后清空剪贴板
    if (clipboard.mode === 'cut') {
      setFileClipboard(null)
    }
    await loadRoot()
  } catch (err) {
    console.error('[FileBrowser] 粘贴失败:', err)
  }
}, [rootPath, selectedPaths, loadRoot, setFileClipboard])
```

```typescript
  // 安全校验：sourcePath 和 targetDir 必须在允许的工作区范围内
  // ...（复用现有的路径校验逻辑）

  // 防止复制目录到自身或子目录，避免递归复制/异常
  // normalizeFsPath / isSameOrChildPath 需要在 main/ipc.ts 就近实现，不能直接复用 renderer/FileBrowser.tsx 中的导出。
  if (normalizeFsPath(sourcePath) === normalizeFsPath(targetDir) || isSameOrChildPath(sourcePath, targetDir)) {
    throw new Error('不能复制到自身或子目录')
  }

  const baseName = path.basename(sourcePath)
  let destPath = path.join(targetDir, baseName)

  // 同名冲突处理
  if (existsSync(destPath) && destPath !== sourcePath) {
    const ext = path.extname(baseName)
    const nameWithoutExt = ext ? baseName.slice(0, -ext.length) : baseName
    let counter = 1
    while (existsSync(destPath)) {
      destPath = path.join(targetDir, `${nameWithoutExt} (${counter})${ext}`)
      counter++
    }
  }

  const stat = statSync(sourcePath)
  if (stat.isDirectory()) {
    cpSync(sourcePath, destPath, { recursive: true })
  } else {
    copyFileSync(sourcePath, destPath)
  }

  return destPath
})
```

## 4. 注意事项

### 4.1 快捷键冲突防护

- **与渲染进程全局快捷键系统的隔离**：全局快捷键系统在 `window` 上以 capture 阶段监听（`shortcut-registry.ts`），FileBrowser 在容器 div 上以 bubble 阶段监听（`onKeyDown`）。`Ctrl+C/V/X/A` 未注册在 `DEFAULT_SHORTCUTS` 中，全局 `dispatchShortcut` 不会匹配，事件正常传播到 bubble 阶段供 FileBrowser 处理。**不需要使用 `onKeyDownCapture`**，bubble 阶段即可。

- **与 Electron 应用菜单 role 的交互**：`main/menu.ts` 编辑菜单注册了 `role: 'copy'/'paste'/'cut'/'selectAll'/'delete'`，这些 role 会触发 Chromium 的文本编辑命令。关键行为差异：
  - `role` 菜单项（如 `copy`）：键盘事件**仍然传播到渲染进程**，Chromium 同时执行文本编辑命令。`e.preventDefault()` 可阻止。
  - `accelerator + click` 菜单项（如 `Cmd+W`）：键盘事件**被主进程拦截**，不传播到渲染进程。

  因此 FileBrowser 的 `onKeyDown` handler 可以正常接收 `Ctrl+C/V/X/A` 事件。**必须对所有处理的按键调用 `e.preventDefault()`**，以阻止 Chromium 的文本编辑命令（尤其是 `Ctrl+A` 的 selectAll 会导致文件树 div 内文本被全选，产生蓝色覆盖层与文件级全选 UI 叠加）。

- **与 Chromium 内置文本操作的关系**：本应用基于 Electron，渲染进程运行在 Chromium WebContents 中。`Ctrl+C/V/X` 在 Chromium 中有内置的文本复制/粘贴/剪切行为（无需 JS 注册即可生效）。FileBrowser 拦截这些按键后会覆盖该内置行为，但文件树中文件名为普通 `<span>` 渲染，用户几乎不会框选文件名复制；重命名态（input 元素）已排除拦截，因此影响可忽略。

- **重命名态保护**：当 `renamingPath` 非空时，所有快捷键处理必须跳过，让键盘输入正常进入 input。

- **输入框焦点保护**：FileBrowser 的 `tabIndex={0}` 确保只有显式点击文件树区域才会获得焦点，不会与消息输入框、搜索框等冲突。

- **事件传播完整链路**：

  ```
  OS 键盘事件
    → Electron 主进程：menu accelerator 匹配
      → role 菜单项：不拦截 DOM 事件，Chromium 编辑命令与 JS 并行（e.preventDefault() 可阻止）
      → click 菜单项：拦截 DOM 事件，JS 收不到（如 Cmd+W）
    → Chromium 渲染进程 DOM 事件
      → window capture 阶段：shortcut-registry dispatchShortcut（仅匹配 parsedCache 中的快捷键）
      → 容器 div bubble 阶段：FileBrowser onKeyDown（本方案新增）
  ```

### 4.2 跨 FileBrowser 实例的复制粘贴

SidePanel 中有两个 FileBrowser 实例（会话文件 + 工作区文件）。使用 Jotai atom 管理 `fileClipboard` 状态，可天然支持跨实例的复制→粘贴（如从会话文件复制到工作区文件）。

### 4.3 嵌入模式下的焦点行为

FileBrowser 有 `embedded` 模式（由 SidePanel 嵌入），此时不使用内部 ScrollArea。焦点管理需确保在嵌入模式下，滚动容器不会抢夺 FileBrowser 的焦点。

### 4.4 Backspace 键的平台差异

- **macOS**：删除键为 `Backspace`（标记为 ⌫），无独立 `Delete` 键（需 Fn+Backspace）
- **Windows**：`Delete` 键独立存在，`Backspace` 通常用于"返回上级"

建议两个键都监听为删除操作，与 Finder/Explorer 的行为对齐。
