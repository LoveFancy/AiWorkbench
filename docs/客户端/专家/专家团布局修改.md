# 专家团布局修改方案

> 编辑时间：2026年6月15日

***

## 一、现状分析

### 1.1 当前架构问题

| 问题         | 现状                     | 影响                                    |
| ---------- | ---------------------- | ------------------------------------- |
| 专家团藏在设置里   | 用户需进入 设置 → 专家团 才能看到    | 发现性极差，新用户不知道专家团存在                     |
| 设置页与使用场景割裂 | 在 Agent 模式工作时要回设置页召唤专家 | 操作路径长：Agent 会话 → 设置 → 找专家 → 召唤 → 回到会话 |
| 无快速访问机制    | 39 个专家团全部平铺，无收藏/常用标记   | 用户高频使用 3-5 个专家，每次都要从头翻                |
| Chat 模式也显示 | 设置页的专家团在 Chat 模式下也能看到  | Chat 模式下专家团无意义（专家团仅用于 Agent 模式）       |
| 无关注/收藏能力   | WorkBuddy 也无此功能        | 用户无法快速筛选常用专家                          |

### 1.2 WorkBuddy 的专家团布局（参考）

WorkBuddy 将专家团放在**侧边栏**中，作为独立入口：

```
WorkBuddy 侧边栏布局：
├── 🔍 搜索
├── 📁 文件浏览器
├── 🤖 Agent 会话列表
├── ⚡ Automation（自动任务）
├── 👥 专家团 ← 侧边栏直接可见
│   ├── ai-engineer
│   ├── backend-architect
│   ├── data-analysis
│   └── ...
└── ⚙️ 设置
```

优点：专家团与会话在同一视觉平面，召唤路径短。

### 1.3 业界参考

| 产品                     | 设计特点                                          | 可借鉴点                 |
| ---------------------- | --------------------------------------------- | -------------------- |
| **Cursor 2.0/3.0**     | Agent-centric sidebar，agents/plans/runs 为一级对象 | 侧边栏按功能分区，Agent 是核心入口 |
| **VS Code Extensions** | 侧边栏独立视图，支持搜索/分类/已安装筛选                         | 搜索+筛选+列表的标准模式        |
| **GitHub Copilot**     | Agent picker 下拉菜单 + 快速切换                      | 快捷选择器减少操作步数          |
| **Linear**             | Project favorites with star toggle            | 星标/关注机制简单直观          |
| **Figma**              | 团队库 + 收藏夹双层结构                                 | 收藏夹与全部列表分离           |

## 二、整体布局方案

### 2.1 核心原则

1. **Agent 模式专属**：专家团仅在 Agent 模式下可见，Chat 模式隐藏
2. **侧边栏就近访问**：从设置页迁出，嵌入左侧边栏，紧贴工作流
3. **关注优先**：关注的专家团置顶，减少查找成本
4. **渐进式展现**：侧边栏紧凑展示 → 展开完整面板 → 详情弹窗，三层递进

### 2.2 目标布局

```
LeftSidebar (Agent 模式)
├── 🔍 搜索
├── 🤖 Agent 会话列表（日期分组）
│   ├── 今天
│   ├── 昨天
│   └── 更早
├── ⏰ 自动任务                       ← 现有，位置不变
│
├── 👥 专家团                         ← 新增，Agent 模式专属
│   ├── ⭐ 已关注 (3)    ──→ 点击打开"已关注"全页面
│   ├── 🕐 最近使用 (5)  ──→ 点击打开"最近使用"全页面
│   └── 📋 全部专家团 (39) ──→ 点击打开"全部专家团"全页面
│
├── ───────── 分隔线 ─────────
│
└── 📌 置顶会话
```

> 三个子入口均为**导航链接**（非展开/折叠），点击后切换到对应的全屏视图页面。页面设计统一，仅数据范围不同。

### 2.3 Chat 模式下的布局

Chat 模式下专家团区域**完全不渲染**：

```
LeftSidebar (Chat 模式)
├── 🔍 搜索
├── 💬 Chat 对话列表
│   ├── 今天
│   ├── 昨天
│   └── 更早
├── ───────── 分隔线 ─────────
├── 📌 置顶对话
└── （无专家团区域）
```

## 三、侧边栏专家团区域设计

### 3.1 区域组件：`ExpertSidebarSection`

位置：在 `LeftSidebar` 中，`AutomationSidebarEntry` 之后，置顶区域之前。

**设计原则**：三个子入口均为**导航链接**，点击后切换主视图到对应的全屏页面，而非在该区域展开下拉列表。这与"自动任务"的交互模式一致。

```
┌─────────────────────────────────────┐
│ 👥 专家团                           │  ← 标题行
├─────────────────────────────────────┤
│ ⭐ 已关注 · 3                       │  ← 导航链接（点击打开全页面）
├─────────────────────────────────────┤
│ 🕐 最近使用 · 5                     │  ← 导航链接（点击打开全页面）
├─────────────────────────────────────┤
│ 📋 全部专家团 · 39                  │  ← 导航链接（点击打开全页面）
└─────────────────────────────────────┘
```

**每条导航链接的行为：**
- 点击"已关注" → 切换到专家团视图，筛选标签默认选中"收藏"
- 点击"最近使用" → 切换到专家团视图，筛选标签默认选中"最近"
- 点击"全部专家团" → 切换到专家团视图，筛选标签默认选中"全部"

**计数实时更新机制：**

三个 Badge 的数字来源不同，通过 `useMemo` 从对应 atom 派生：

| 计数 | 数据源 | 更新触发 |
|------|--------|---------|
| 已关注数 | `followedExpertGroupsAtom` (localStorage) | 星标切换立即更新（atom 响应式） |
| 最近使用数 | `recentExpertGroupsAtom` (localStorage) | 召唤/切换会话时更新 |
| 全部数量 | `agentExpertGroupsAtom` (IPC) | 插件安装/卸载/启用切换后 `loadGroups()` 刷新 |

```typescript
function ExpertSidebarSection(): React.ReactElement {
  const allGroups = useAtomValue(agentExpertGroupsAtom)
  const followed = useAtomValue(followedExpertGroupsAtom)
  const recent = useAtomValue(recentExpertGroupsAtom)

  const followedCount = useMemo(
    () => allGroups.filter(g => followed[g.id]).length,
    [allGroups, followed]
  )
  const recentCount = useMemo(
    () => allGroups.filter(g => recent[g.id] && !followed[g.id]).length,
    [allGroups, recent, followed]
  )
  const totalCount = allGroups.length

  // ...
}
```

**交互细节：**
- 当前激活的视图对应的侧边栏条目高亮（与"自动任务"一致）
- 右侧数字为计数 Badge（关注数、最近数、总数）
- 每个条目都是标准的侧边栏导航项，与 `AutomationSidebarEntry` 样式统一

### 3.2 与现有导航模式的关系

侧边栏现有导航模式：

| 导航项 | 计数字段 | 激活判断 | 点击行为 |
|--------|---------|---------|---------|
| Agent 会话列表 | 无 | activeTabId 匹配 | 切换标签页 |
| 自动任务 | automationCount | activeView === 'automations' | 切换 activeView |
| **已关注** | followedCount | activeView === 'expert-followed' | 切换 activeView |
| **最近使用** | recentCount | activeView === 'expert-recent' | 切换 activeView |
| **全部专家团** | totalCount | activeView === 'expert-all' | 切换 activeView |

需要扩展 `ActiveView` 类型：

```typescript
// active-view.ts
export type ActiveView =
  | 'conversations'
  | 'automations'
  | 'expert-all'      // 全部专家团
  | 'expert-followed'  // 已关注
  | 'expert-recent'    // 最近使用
```

### 3.3 状态指示

状态指示应用于全页面视图中的卡片上（非侧边栏）：

| 状态             | 指示方式 | 说明         |
| -------------- | ---- | ---------- |
| available      | 绿色圆点 | 可正常召唤      |
| skill\_missing | 橙色圆点 | 缺少依赖 Skill |
| mcp\_missing   | 橙色圆点 | 缺少 MCP 配置  |
| disabled       | 灰色圆点 | 已禁用        |

## 四、关注（Follow/Favorite）机制

### 4.1 设计思路

借鉴 **GitHub Stars** + **Linear Favorites** 模式：

- 每个专家团有一个星标按钮（☆/⭐）
- 关注的专家团置顶显示
- 关注状态跨会话/工作区持久化
- 支持在侧边栏和管理面板两处切换关注状态（双向同步）

### 4.2 存储方案

```typescript
// 关注列表存储在本地，不在插件系统内（用户偏好而非项目配置）
interface FollowedExpertsState {
  // key: expertGroupId（如 "backend-architect"）
  // value: 关注时间戳
  followed: Record<string, number>
}

// 存储位置：~/.proma/user-preferences.json 或 electron-store
// 与工作区无关，跟随用户账号
```

使用 `electron-store` 或 `atomWithStorage`（localStorage 兜底）：

```typescript
const followedExpertGroupsAtom = atomWithStorage<Record<string, number>>(
  'followed-expert-groups',
  {}
)
```

### 4.3 UI 交互

| 操作   | 位置             | 行为                 |
| ---- | -------------- | ------------------ |
| 点击 ☆ | 侧边栏条目 / 管理面板卡片 | 切换为 ⭐，加入置顶区        |
| 点击 ⭐ | 侧边栏条目 / 管理面板卡片 | 切换为 ☆，从置顶区移除       |
| 关注   | 侧边栏 hover 时显示  | 即时响应，toast 提示"已关注" |
| 取消关注 | 侧边栏 hover 时显示  | 即时响应，无 toast（减少打扰） |

### 4.4 关注列表排序

已关注专家团按**关注时间倒序**排列（最近关注的在前），方便用户快速找到刚关注的专家。

## 五、代码组织：`experts/` 目录

### 5.1 设计原则

1. **集中管理**：所有专家相关代码收敛到 `experts/` 目录，核心代码只引用此目录
2. **单一职责**：每个组件只做一件事，通过组合构建页面
3. **零重复**：三个页面（全部/已关注/最近使用）共享同一套组件，差异仅通过 props 和条件渲染控制
4. **最小入口**：`experts/index.ts` 统一导出，外部只通过此文件的导出消费

## 六、组件树与页面拼接

### 6.1 核心思路：一个页面、三种入口

三个侧边栏入口映射到同一个 `ExpertPageView` 组件，仅通过 props 区分：

```typescript
// 侧边栏点击后，MainArea 根据 activeView 渲染：
switch (activeView) {
  case 'expert-all':      return <ExpertPageView showFeaturedScenes initialFilter="all" />
  case 'expert-followed': return <ExpertPageView initialFilter="followed" />
  case 'expert-recent':   return <ExpertPageView initialFilter="recent" />
}
```

**`ExpertPageView` Props**：

```typescript
interface ExpertPageViewProps {
  /** 是否显示精选场景区域（仅"全部专家团"入口为 true） */
  showFeaturedScenes?: boolean
  /** 初始筛选标签 */
  initialFilter: 'all' | 'followed' | 'recent'
}
```

### 6.2 组件树

```
ExpertPageView                              ← 顶层容器（读写 atom，管理 filter state）
│
├── ExpertPageHeader                        ← 标题行 "👥 专家团" + ExpertImportDropdown
│
├── [if showFeaturedScenes]                 ← ⚠️ 仅"全部专家团"显示
│   └── ExpertFeaturedScenes
│       └── ExpertSceneCard × N             ← 场景卡片（内容创作、办公场景、法律咨询…）
│
├── ExpertSearchBar                         ← 搜索框，共享
│
├── ExpertFilterPills                       ← 筛选标签（收藏/最近/可用/不可用），共享
│
├── ExpertSortDropdown                      ← 排序下拉，共享
│
├── ExpertCardGrid                          ← 卡片网格容器，共享
│   ├── ExpertCard × N                      ← 核心卡片
│   │   ├── ExpertStatusBadge               ← 状态标记
│   │   └── [内部星标交互]                   ← 读写 followedExpertGroupsAtom
│   └── [if 列表为空] ExpertEmptyState       ← 空态（类型由 filter 决定）
│
└── [if 弹窗打开] ExpertDetailDialog        ← 详情弹窗，共享
```

### 6.3 数据流

```
┌──────────────────────────────────────────────────────────┐
│  ExpertPageView                                          │
│                                                          │
│  // 1. 首次渲染触发数据加载                                │
│  useEffect(() => { loadGroups() }, [])                   │
│                                                          │
│  // 2. 读取 atom                                        │
│  const allGroups   = useAtomValue(agentExpertGroupsAtom) │
│  const loadGroups  = useSetAtom(loadAgentExpertGroupsAtom)
│  const followed    = useAtomValue(followedExpertGroupsAtom)│
│  const recent      = useAtomValue(recentExpertGroupsAtom) │
│                                                          │
│  // 3. 局部状态：筛选/搜索/排序                            │
│  const [filter, setFilter] = useState(initialFilter)     │
│  const [searchQuery, setSearchQuery] = useState('')      │
│  const [sortBy, setSortBy] = useState('smart')           │
│                                                          │
│  // 4. 数据管道：全部 → 筛选 → 搜索 → 排序（useMemo）      │
│  const filtered = useMemo(() => {                        │
│    let result = filterByTag(allGroups, filter,           │
│                              followed, recent)           │
│    result = searchByName(result, searchQuery)            │
│    result = sortExpertGroups(result, followed,           │
│                               recent, sortBy)            │
│    return result                                         │
│  }, [allGroups, filter, searchQuery, sortBy,             │
│      followed, recent])                                 │
│                                                          │
│  // 5. 渲染                                              │
│  return (                                                │
│    <div>                                                  │
│      {showFeaturedScenes && <ExpertFeaturedScenes       │
│        allGroups={allGroups}                            │
```

```typescript
// experts/utils/sort.ts
export function sortExpertGroups(
  groups: AgentExpertGroupInfo[],
  followed: Record<string, number>,
  recent: Record<string, number>,
  sortBy: 'smart' | 'name' | 'recent'
): AgentExpertGroupInfo[] {
  return [...groups].sort((a, b) => {
    if (sortBy === 'name') return a.name.localeCompare(b.name, 'zh')
    if (sortBy === 'recent') return (recent[b.id] ?? 0) - (recent[a.id] ?? 0)
    // smart: 关注优先 → 最近关注 → 最近使用 → 名称
    const aF = followed[a.id] ?? 0, bF = followed[b.id] ?? 0
    if (aF && !bF) return -1
    if (!aF && bF) return 1
    if (aF && bF) return bF - aF
    const aR = recent[a.id] ?? 0, bR = recent[b.id] ?? 0
    if (aR !== bR) return bR - aR
    return a.name.localeCompare(b.name, 'zh')
  })
}
```

```typescript
// experts/utils/filter.ts
export function filterByTag(
  groups: AgentExpertGroupInfo[],
  tag: 'all' | 'followed' | 'recent' | 'available' | 'unavailable',
  followed: Record<string, number>,
  recent: Record<string, number>
): AgentExpertGroupInfo[] {
  switch (tag) {
    case 'all': return groups
    case 'followed': return groups.filter(g => followed[g.id])
    case 'recent':
      const r = groups.filter(g => recent[g.id] && !followed[g.id])
      return r.sort((a, b) => (recent[b.id] ?? 0) - (recent[a.id] ?? 0)).slice(0, 8)
    case 'available': return groups.filter(g => g.status === 'available')
    case 'unavailable': return groups.filter(g => g.status !== 'available')
  }
}

export function searchByName(
  groups: AgentExpertGroupInfo[],
  query: string
): AgentExpertGroupInfo[] {
  if (!query.trim()) return groups
  const q = query.trim().toLowerCase()
  return groups.filter(g =>
    getExpertGroupSearchTerms(g).some(t => t.toLowerCase().includes(q))
  )
}

/** 按 tags 匹配场景下的专家数量 */
export function countByScene(
  groups: AgentExpertGroupInfo[],
  sceneTags: string[]
): number {
  return groups.filter(g =>
    g.tags?.some(t => sceneTags.some(st => t.includes(st)))
  ).length
}
```
