# 客户端：专家模式（Expert Group）实现方案

编辑时间：2026年6月12日

***

## 一、概述

专家模式允许用户在 Agent 会话中**绑定一个专家团**（Expert Group），获得专属的**主角色 System Prompt**、**SubAgent 协作能力**、**Skills/MCP/内置工具推荐**，让 AI 以特定领域的专家角色完成复杂任务。

### 核心概念

| 概念                    | 说明                                            |
| --------------------- | --------------------------------------------- |
| **专家团（Expert Group）** | 一个预配置的专家协作方案，包含主角色、SubAgent、Skills、MCP 等      |
| **主角色（Main Role）**    | 专家团的主 AI 角色，拥有专属的 system prompt               |
| **SubAgent**          | 主角色可按需调度的子代理，每个有独立的 prompt + 工具列表             |
| **插件（Plugin）**        | 专家团的载体，内置插件（builtin）随应用发布，用户插件（user）通过 zip 安装 |

### 整体流程

```
┌──────────────────────────────────────────────────────────────────────┐
│  1. 插件系统发现 expert-groups/*.json → AgentExpertGroupInfo         │
│  2. 用户在 UI 中"召唤"专家团 → createAgentSession(..., expertGroupId)│
│  3. Agent 编排时 resolveExpertGroupRuntime() → ExpertGroupRuntime    │
│  4. Runtime 注入 system prompt + SubAgent + MCP + Skills             │
│  5. Agent 以专家角色运行，按需调度 SubAgent                           │
└──────────────────────────────────────────────────────────────────────┘
```

### 3.2 专家团信息（AgentExpertGroupInfo）

清单信息 + 运行时元数据：

```typescript
interface AgentExpertGroupInfo extends AgentExpertGroupManifest {
  sourcePluginId: string        // 来源插件 ID
  sourceLabel: string           // 来源显示名
  sourcePluginVersion: string   // 来源插件版本
  sourcePluginKind: 'builtin' | 'user'  // 内置/用户插件
  sourcePluginPath: string      // 插件目录路径
  filePath: string              // 配置文件完整路径
  enabled: boolean              // 是否启用
  status: AgentExpertGroupStatus  // 可用状态
  issues: Array<{              // 配置问题列表
    level: 'warning' | 'error'
    message: string
  }>
}
```

### 3.3 专家团状态（AgentExpertGroupStatus）

| 状态                   | 含义             |
| -------------------- | -------------- |
| `available`          | 可用，可召唤         |
| `plugin_disabled`    | 所属插件已禁用        |
| `plugin_uninstalled` | 来源插件已卸载        |
| `invalid_manifest`   | 配置文件错误         |
| `missing_subagent`   | 缺少 SubAgent 定义 |
| `missing_skill`      | 缺少 Skill 定义    |
| `mcp_conflict`       | MCP 连接器冲突      |

### 3.4 专家团运行时（ExpertGroupRuntime）

在 Agent 编排时从 AgentExpertGroupInfo 解析得到：

```typescript
interface ExpertGroupRuntime {
  group: AgentExpertGroupInfo          // 原始专家团信息
  mainPrompt: string                   // 主角色 System Prompt
  agents: Record<string, AgentDefinition>  // SubAgent 定义 Map
  pluginPaths: Array<{ type: 'local'; path: string }>  // 插件路径（SDK 注入用）
  mcpServers: Record<string, McpServerEntry>  // MCP Server 配置
  promptHints: string[]                // 调度提示
  allowedTools?: string[]              // restrict 模式的白名单
  disallowedTools?: string[]           // 禁用的工具列表
}
```

### 4.1 发现流程

```
discoverExpertGroups(pluginPath, pluginId, sourceLabel, enabled, manifest)
  │
  ├─ 1. 读取 manifest.expertGroup / manifest.expertGroups
  │     （package.json 中声明的专家团 ID）
  │
  ├─ 2. 扫描 expert-groups/ 目录下的 *.json 文件
  │
  ├─ 3. 取声明的第一个 ID 或目录下第一个 JSON 文件作为 groupId
  │     每个插件只允许声明一个专家团
  │
  ├─ 4. 验证 expert-groups/{groupId}.json 文件存在且为合法 JSON
  │
  └─ 5. 返回 AgentPluginCapability[]（type: 'expert-group'）
```

### 4.2 插件清单声明方式

```json

  "name": "architecture-decision-team",
  "version": "1.0.0",
  "expertGroup": "architecture-decision-team"
}
```

或兼容旧格式：

```json
{
  "expertGroups": ["architecture-decision-team"]
}
```

### 4.3 重复 ID 检查

安装用户插件时，会检查是否与已安装的其他插件的专家团 ID 冲突（[plugin-registry-service.ts#L698](file:///d:/AiWorkbench/apps/electron/src/main/lib/plugin-registry-service.ts#L698)）。

## 五、专家团管理器

代码：[agent-expert-group-manager.ts](file:///d:/AiWorkbench/apps/electron/src/main/lib/agent-expert-group-manager.ts)

### 5.1 列出所有专家团

```typescript
listAgentExpertGroups(paths?): AgentExpertGroupInfo[]
```

**执行流程**：

1. 遍历所有已安装插件（`listInstalledPlugins()`）
2. 筛选 capability.type === 'expert-group' 的能力
3. 对每个专家团：
   - `readExpertManifest()` — 读取并校验 `expert-groups/{id}.json`
   - `normalizeExpertManifest()` — 解析 JSON，校验必填字段（name, mainRole.name, mainRole.prompt）
   - `validateExpertReferences()` — 验证引用的 SubAgent、Skills、MCP 是否存在
4. 按 `name` 字母序排序返回

### 5.2 获取单个专家团

```typescript
getAgentExpertGroup(input: { expertGroupId: string; expertPluginId?: string }): AgentExpertGroupInfo | undefined
```

按 `id + pluginId` 精确匹配。

### 5.3 解析专家团运行时

```typescript
resolveExpertGroupRuntime(input: ResolveExpertGroupRuntimeInput): ExpertGroupRuntime | null
```

这是**将静态清单转换为可注入 Agent 的运行时结构**的核心方法：

1. 调用 `getAgentExpertGroup()` 获取清单
2. 检查 `group.status === 'available'`
3. 遍历 `group.subagents[]`，读取 `agents/{agentName}.md`：
   - 解析 frontmatter（description, tools, maxTurns）
   - 正文作为 prompt
   - 组装为 `AgentDefinition`
4. 读取 `.mcp.json`，提取 `group.mcpServers[]` 中声明的 MCP Server 配置
5. 构建 `pluginPaths`（SDK 注入用）
6. 构建 `promptHints`（调度提示）
7. 处理 `toolsPolicy`（restrict 模式的白名单）

### 5.4 SubAgent 定义解析

SubAgent 文件格式（`agents/{agentName}.md`）：

```markdown
---
description: 代码审查专家，负责审查代码质量
tools:
  - Read
  - Grep
  - Glob
maxTurns: 5
---

你是代码审查专家。你的职责是...
```

通过 `parseFrontmatter()` 分离元数据和正文。

### 5.5 引用完整性校验

`validateExpertReferences()` 检查：

| 引用类型         | 校验方式                                                | 缺失状态               |
| ------------ | --------------------------------------------------- | ------------------ |
| SubAgent     | `agents/{name}.md` 存在                               | `missing_subagent` |
| Skill        | `skills/{name}/SKILL.md` 存在于插件或默认 skills 目录         | `missing_skill`    |
| MCP Server   | 名称在 `.mcp.json` 的 mcpServers 中                      | 仅 warning          |
| builtinTools | 名称在 `SUPPORTED_BUILTIN_TOOLS` 集合中（当前仅 `web-search`） | 仅 warning          |

### 7.3 SubAgent 注入

`buildAgentsForSession()` 将 `expertRuntime.agents` 中的 SubAgent 定义注入到 SDK query options：

```typescript
if (expertRuntime?.agents) {
  for (const [name, def] of Object.entries(expertRuntime.agents)) {
    agents[name] = def
  }
}
```

### 7.4 插件路径注入

`expertRuntime.pluginPaths` 作为 SDK local plugin 路径注入，使 SubAgent 可以访问插件内的资源。

### 7.5 MCP Server 注入

`expertRuntime.mcpServers` 中的 MCP 配置合并到 SDK mcpServers 配置。

### 7.6 工具策略

- `restrict` 模式：`expertRuntime.allowedTools` 作为 SDK allowedTools 传递
- `disallowedTools`：合并到 SDK disallowedTools

### 7.7 主角色 Prompt

`expertRuntime.mainPrompt` 作为主 system prompt 注入（替代默认 prompt），同时保留基础指令（语言规则、当前时间、工作区状态等）。

## 十一、完整交互流程

### 11.1 召唤专家团

```
用户点击"召唤专家"按钮
  │
  ├─ ExpertSummonButton
  │   ├─ 首次打开且无数据 → loadAgentExpertGroupsAtom()
  │   └─ 打开 ExpertGroupPicker 弹框
  │
  ├─ ExpertGroupPicker
  │   ├─ 展示可召唤/不可召唤分区
  │   ├─ 支持搜索过滤
  │   ├─ 点击卡片 → ExpertGroupDetailDialog（查看详情）
  │   └─ 点击"召唤"→ handleSummon(group)
  │
  ├─ handleSummon()
  │   ├─ 显示 ExpertSummoningOverlay（全屏遮罩 + 加载动画）
  │   └─ 调用 createExpertSessionAtom(group)
  │       ├─ 创建新会话（标题 = "{group.name} · 新任务"）
  │       ├─ 继承当前会话的 channelId / workspaceId
  │       ├─ 绑定 expertGroupId + expertPluginId
  │       ├─ 持久化 introduction 为首条 assistant 消息
  │       └─ openSession(newSessionId) → 跳转到新会话
  │
  └─ ExpertSummoningOverlay 消失
      └─ 用户进入专家会话，开始对话
```

### 11.2 专家会话运行

```
用户发送消息
  │
  ├─ agent-orchestrator.ts
  │   ├─ 读取 sessionMeta.expertGroupId
  │   ├─ resolveExpertGroupRuntime({ expertGroupId, expertPluginId })
  │   │   ├─ getAgentExpertGroup() → 获取清单
  │   │   ├─ 校验 status === 'available'
  │   │   ├─ 读取 SubAgent 定义
  │   │   ├─ 读取 MCP 配置
  │   │   └─ 构建 promptHints
  │   │
  │   ├─ buildSystemPrompt({ expertRuntime })
  │   │   └─ buildExpertModeSummary() → "## 专家团模式" 章节
  │   │
  │   ├─ buildAgentsForSession({ expertRuntime })
  │   │   └─ 将 SubAgent 注册到 SDK agents
  │   │
  │   └─ SDK query({ systemPrompt, agents, mcpServers, ... })
  │
  └─ Agent 以专家团主角色身份运行
      ├─ 按专家的 system prompt 执行任务
      ├─ 按需调度 SubAgent（Task 工具）
      ├─ 使用注入的 Skills / MCP / builtinTools
      └─ 回复用户
```

### 11.3 查看/管理专家团（设置页）

```
用户打开设置页 → 专家团标签页
  │
  ├─ ExpertGroupSettings
  │   ├─ loadAgentExpertGroupsAtom() 加载列表
  │   ├─ 展示搜索框
  │   ├─ 展示"全部专家团"分区

  │   │   └─ installAgentPluginZip() → 校验是否含 expert-group
  │   └─ "刷新"按钮 → 重新加载
  │
  └─ ExpertGroupDetailDialog
      ├─ 能力介绍 / 专家成员 / 依赖能力 / 示例问题 / 配置问题
      └─ "召唤"按钮 / "关闭"按钮
```
