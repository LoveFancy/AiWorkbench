## 一、设计结论

直接在现有 `apps/electron/src/main/lib/updater/` 模块中实现 WorkMate Server 升级链路，替换原有的 `electron-updater`（GitHub Release）方式。

保留现有 IPC、preload、renderer atom、设置页入口和弹窗入口不变：

```text
main/index.ts
  → initAutoUpdater(mainWindow)
main/ipc.ts
  → 保留现有 updater IPC 注册
preload/index.ts
  → window.electronAPI.updater
renderer/atoms/updater.ts
  → updateStatusAtom
UpdateDialog.tsx / AboutSettings.tsx
  → 继续消费 updater 状态
```

升级链路：

```text
应用启动 / 定时 / 手动触发
  ↓
调用 WorkMate Server 检测接口
  ↓
有更新 → 自动下载安装包（支持进度展示）
  ↓
下载完成 → 校验 sha256 / fileSize / 版本
  ↓
弹窗提示安装 → 用户确认 → 启动安装器 → 退出应用
```

职责划分：

| 层 | 职责 |
|------|------|
| 检测 API | 调用 `/workmate/upgrade/check`，解析响应，做端侧版本方向校验。 |
| 下载器 | 下载到临时文件、进度统计、backpressure 处理、sha256/fileSize 校验。 |
| 安装器 | 安装前校验、版本提取、`shell.openPath` 启动安装器并退出应用。 |
| manifest | 持久化已下载安装包信息，启动时恢复以便重新提示安装。 |
| 主入口 | 统一状态管理、定时检测调度、对外 API（`checkForUpdates`、`quitAndInstall` 等）。 |

## 二、配置设计

### 2.1 配置字段

登录和升级检测共用同一个 EIP Lite 地址。配置在客户端 `settings.json` 中：

```json
{
  "eipGatewayBase": "http://eiplite.htsc.com.cn",
  "allowedUpgradeDownloadHosts": ["eiplite.htsc.com.cn"]
}
```

字段说明：

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `eipGatewayBase` | string | `http://eiplite.htsc.com.cn` | EIP Lite 地址，登录和升级检测共用。开发环境可配置为 `http://localhost:6173`。 |
| `allowedUpgradeDownloadHosts` | string[] | 空 | 可选下载域名白名单。为空时只校验 hostname 与 `eipGatewayBase` 一致，不要求端口一致。 |

### 2.2 配置读取

`getEipGatewayBase()` 已有实现（`apps/electron/src/auth/auth-service.ts` 第 27 行），登录已使用该函数。升级检测直接复用，无需新建。

```ts
// 已有实现，直接 import
import { getEipGatewayBase } from '../auth/auth-service'
// 返回如 "http://eiplite.htsc.com.cn/gateway" 或 settings.eipGatewayBase 配置值
```

## 四、状态类型

状态按"渲染端需要什么"定义，不暴露内部字段（如 `sha256`、`fileSize` 保留在下载器和 manifest 内部）。

```ts
export interface DownloadProgress {
  transferred: number
  total: number          // -1 表示未知大小
  bytesPerSecond: number
  percent: number
}

/** 渲染端可见的更新状态 */
export type UpdateStatus =
  | { status: 'idle' }
  | { status: 'checking' }
  | { status: 'not-available'; hint?: string }
  | {
      status: 'available'
      version: string
      releaseNotes?: string
      forceUpdate?: boolean
      releaseType?: 'UPGRADE' | 'ROLLBACK'
      hint?: string
    }
  | {
      status: 'downloading'
      version: string
      releaseNotes?: string
      progress: DownloadProgress
      forceUpdate?: boolean
      releaseType?: 'UPGRADE' | 'ROLLBACK'
    }
  | {
      status: 'downloaded'
      version: string
      releaseNotes?: string
      forceUpdate?: boolean
      releaseType?: 'UPGRADE' | 'ROLLBACK'
      hint?: string
    }
  | { status: 'error'; error: string }
```

字段归属说明：

| 字段 | 在 UpdateStatus | 说明 |
|------|:---:|------|
| `version` | ✅ | `available`/`downloading`/`downloaded` 展示目标版本 |
| `releaseNotes` | ✅ | 直接展示服务端返回的字符串，不调 GitHub API |
| `forceUpdate` | ✅ | 决定弹窗是否可关闭 |
| `releaseType` | ✅ | `ROLLBACK` 时按钮文案改为"回退安装" |
| `hint` | ✅ | 服务端提示文案；`not-available` 时也可带 hint（如 minVersion 场景） |
| `progress` | ✅ | `downloading` 时展示进度 |
| `sha256` | ❌ | 下载器和 manifest 内部使用，渲染端无需感知 |
| `fileSize` | ❌ | 同上 |
| `fileName` | ❌ | 内部用；渲染端不展示文件名（展示 `version` 即可） |
| `packageType` | ❌ | 内部用；安装器根据平台已有默认扩展名 |
| `downloadUrl` | ❌ | `available` 时自动下载，渲染端不需要手动触发 |
| `installerPath` | ❌ | IPC `quitAndInstall` 内部使用，不传给渲染端 |

状态流转规则：

- `not-available`：手动检查后无更新时使用，可附带 `hint`（如 minVersion 过低提示）；自动静默检查无更新时回到 `idle`。
- `downloading` / `downloaded`：跳过后续自动检查，避免并发下载。

## 五、检测接口

### 5.1 请求

```http
GET {eipGatewayBase}/workmate/upgrade/check?currentVersion=X&platform=Y&arch=Z
Cookie: EIPGW-TOKEN=<jwt>
```

`eipGatewayBase` 通过已有的 `getEipGatewayBase()`（`auth/auth-service.ts`）获取，检测路径直接拼接 `/workmate/upgrade/check`。请求经过 EIP 网关，`hteip-client.ts` 自动注入 `Cookie: EIPGW-TOKEN=<jwt>`，无需额外处理认证。

请求参数：

| 参数 | 类型 | 说明 |
|------|------|------|
| `currentVersion` | string | 当前客户端版本号，来自 `app.getVersion()`。 |
| `platform` | string | `win32` / `darwin` / `linux`，来自 `process.platform`。 |
| `arch` | string | `x64` / `arm64` 等，来自 `process.arch`。 |

> **注意**：当前服务端 `UpgradeCheckRequest` 仅有 `currentVersion` 和 `platform` 两个字段，缺少 `arch`。需要服务端新增此参数，详见 §11。

### 5.2 响应

服务端响应包装在 `ApiResponse<T>` 结构内，客户端使用 `httpGet<{ code: number; data: UpgradeCheckData }>()` 后，需先判断 `code === 0` 再取 `data` 字段。

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "hasUpdate": true,
    "forceUpdate": false,
    "releaseType": "UPGRADE",
    "latestVersion": "1.2.0",
    "downloadUrl": "http://eiplite.htsc.com.cn/workmate/releases/1.2.0/WorkMate-1.2.0-win32-x64.exe",
    "releaseNotes": "修复了若干已知问题，优化了性能",
    "minVersion": "1.0.0",
    "hint": "有新版本可用，建议更新",
    "publishedAt": "2026-05-20T10:00:00Z",
    "sha256": "...",
    "fileSize": 123456789,
    "fileName": "WorkMate-1.2.0-win32-x64.exe",
    "packageType": "exe"
  },
  "timestamp": 1717300000000
}
```

`data` 内字段说明：

| 字段 | 必填 | 说明 |
|------|------|------|
| `hasUpdate` | 是 | 是否有可用更新。 |
| `forceUpdate` | 是 | 是否强制升级。当前服务端硬编码为 `false`（见 §11）。 |
| `releaseType` | 有更新时必填 | `UPGRADE` 或 `ROLLBACK`。 |
| `latestVersion` | 有更新时必填 | 目标版本。 |
| `downloadUrl` | 有更新时必填 | 安装包下载地址。当前取自 `UpgradeStrategy` 表冗余字段而非 `UpgradeRelease` 表（见 §11 修复方案）。 |
| `releaseNotes` | 否 | 发布说明。服务端返回纯文本或 Markdown 字符串，客户端直接展示，不调 GitHub API。 |
| `minVersion` | 否 | 最低允许运行版本。特殊场景见 §5.3。 |
| `hint` | 否 | 提示文案，如回退提示。 |
| `publishedAt` | 否 | 发布时间。当前服务端未返回。 |
| `sha256` | 建议必填 | 安装包 SHA-256，用于完整性校验。当前服务端未返回（数据已有，在 `UpgradeRelease` 表中）。 |
| `fileSize` | 建议必填 | 安装包大小（byte），用于下载完成校验和 UI 展示。当前服务端未返回。 |
| `fileName` | 建议必填 | 安装包文件名，用于保存和展示。当前服务端未返回。 |
| `packageType` | 建议必填 | 安装包类型。当前服务端未返回。 |

> **注意**：当前服务端 `UpgradeCheckResponse` 仅有 8 个字段（`hasUpdate`/`forceUpdate`/`releaseType`/`latestVersion`/`downloadUrl`/`releaseNotes`/`minVersion`/`hint`），缺 `sha256`、`fileSize`、`fileName`、`packageType`、`publishedAt` 和 `arch` 参数。这些数据在 `UpgradeRelease` 表中已有存储，需要服务端补齐。详见 §11。

### 5.3 端侧二次校验

收到 `hasUpdate=true` 后，按以下顺序校验：

```text
1. downloadUrl 为空或无效：
   服务端在 currentVersion < minVersion 时会返回 downloadUrl=null
   （hint 如 "当前版本过低，请先升级到 1.0.0"）。
   → setStatus('not-available', { hint })，不进入下载流程。

2. releaseType = UPGRADE:
  latestVersion > currentVersion 才有效，否则忽略更新

3. releaseType = ROLLBACK:
  latestVersion < currentVersion 才有效，否则忽略更新

4. minVersion 兜底（仅在 downloadUrl 有效时生效）：
   如果 minVersion 存在且 currentVersion < minVersion：
   客户端将 forceUpdate 视为 true
```

版本比较必须使用 SemVer 解析，不能用字符串大小比较。

## 六、启动与定时检测

### 6.1 初始化流程

```text
auto-updater.init(mainWindow)
  ↓
读取 eipGatewayBase 配置
  ↓
恢复 manifest.json（getConfigDir()/upgrade/manifest.json）
  ↓
如果 manifest 存在且校验通过：
  setStatus(downloaded)，等待渲染端弹窗提示安装
  ↓
启动随机首检定时器
```

### 6.2 随机检测策略

```text
首次检测延迟:
  delay = random(10_000, 3_600_000)    // 10秒到60分钟

周期检测间隔:
  delay = random(6 * 3600_000, 8 * 3600_000)   // 6到8小时
```

使用递归 `setTimeout`，每次检测完成后重新计算下一次间隔。

### 6.3 行为矩阵

| 场景 | 行为 |
|------|------|
| 正常启动 | 10秒~60分钟随机首检 |
| 周期检测 | 每次6~8小时重新随机 |
| 登录成功 | 10秒后触发一次检查，不影响周期计时器 |
| 用户手动检查 | 立即检查，不影响周期计时器 |
| 登出 | 不主动触发检查；已有定时检查发现无 token 时跳过 |
| 状态为 downloading/downloaded | 跳过自动检查，避免并发下载 |

### 6.4 并发保护

`checkForUpdates()` 加锁，同一时间只允许一个检查流程。手动检查时若已处于 `checking` 状态则忽略。

## 七、下载与持久化

### 7.1 下载流程

```text
检测到有效更新
  ↓
校验 downloadUrl 安全边界（见 §7.2）
  ↓
setStatus(downloading) 并推送进度
  ↓
下载到临时文件：<fileName>.tmp
  ↓
下载完成后校验 fileSize 和 sha256（如果服务端提供）
  ↓
rename 为正式安装包文件
  ↓
写入 manifest.json
  ↓
setStatus(downloaded)，渲染端弹窗提示安装
```

下载实现要求：

1. 使用 `.tmp` 临时文件，成功后再 `rename`。
2. 失败时删除 `.tmp`，不保留半成品。
3. 写入文件流时处理 backpressure：`write()` 返回 `false` 时等待 `drain`。
4. 支持通过 `content-length` 计算下载进度；缺失时 `total=-1`，UI 展示"未知大小"。
5. 一期不要求 Range 断点续传。
6. 下载失败后自动重试 1 次（间隔 5 秒）。两次都失败则进入 `error` 状态。

### 7.2 下载 URL 安全校验

下载前校验 URL：

```text
1. 仅允许 https；开发环境允许 http://localhost 或 http://127.0.0.1。
2. 如果 allowedUpgradeDownloadHosts 非空，downloadUrl.hostname 必须在白名单内。
3. 如果 allowedUpgradeDownloadHosts 为空，downloadUrl.hostname 必须与 eipGatewayBase.hostname 一致。
4. 禁止 file://、ftp:// 等非 HTTP(S) 协议。
```

> 注意：hostname 比对不要求端口一致，因为下载走 Nginx（如 18080 端口），检测走 Spring Boot（如 6173 端口）。

### 7.3 manifest 持久化

保存位置：

```text
getConfigDir()/upgrade/manifest.json
```

`getConfigDir()` 已有实现（`apps/electron/src/main/lib/config-paths.ts`），正式版返回 `~/.proma`，`PROMA_DEV=1` 时返回 `~/.proma-dev`。manifest 与 `settings.json`、`auth.json` 放在同一目录下。

示例：

```json
{
  "version": "1.2.0",
  "releaseType": "UPGRADE",
  "forceUpdate": false,
  "releaseNotes": "修复了若干已知问题，优化了性能",
  "downloadedAt": 1717300000000,
  "sha256": "...",
  "fileSize": 123456789,
  "fileName": "WorkMate-1.2.0-win32-x64.exe",
  "packageType": "exe"
}
```

> `installerPath` 不写死在 manifest 中，由代码按 `join(getConfigDir(), 'upgrade', fileName)` 动态计算，避免路径迁移问题。

启动恢复流程：

```text
读取 manifest
  ↓
按 join(getConfigDir(), 'upgrade', fileName) 计算文件路径
  ↓
检查文件是否存在
  ↓
校验 fileSize / sha256（如果 manifest 中有）
  ↓
校验安装包版本 == manifest.version
  ↓
校验版本方向仍符合当前 app.getVersion()
  ↓
通过：setStatus(downloaded)
失败：删除 manifest 和安装包，重新走检测
```

## 八、安装前校验与安装

### 8.1 安装前校验

`installerPath` 由 `join(getConfigDir(), 'upgrade', fileName)` 动态计算，校验项：

```text
1. 当前状态必须是 downloaded（manifest 已校验通过）
2. installerPath 文件必须存在
3. fileSize 与 manifest 一致（如果提供）
4. sha256 与 manifest 一致（如果提供）
5. 安装包版本 == latestVersion
6. UPGRADE 时安装包版本 > 当前 app.getVersion()
7. ROLLBACK 时安装包版本 < 当前 app.getVersion()
```

任一失败：进入 `error` 状态，删除异常安装包和 manifest，不执行安装。

### 8.2 安装包版本读取

| 平台 | 一期方案 | 二期增强 |
|------|----------|----------|
| Windows | 文件名提取版本 + sha256 校验 | 读取 EXE PE VersionInfo |
| macOS | 文件名提取版本 + sha256 校验 | 解析包内 `.app/Contents/Info.plist` |
| Linux | 文件名提取版本 + sha256 校验 | 按 AppImage/deb/rpm 格式读取元数据 |

### 8.3 执行安装

```text
shell.openPath(installerPath)
  ↓
成功：标记应用正在退出，移除 window close 拦截，app.quit()
失败：setStatus(error)
```

保留现有 `quitAndInstall()` 函数和 `updater:quit-and-install` IPC 通道名。UI 文案从"立即重启更新"改为"立即安装"。

## 九、IPC、preload 与渲染端

### 9.1 IPC

继续复用现有 IPC 通道，不新增：

```text
updater:check

```

`updater-ipc.ts` 基本不改，只继续调用 `auto-updater.ts` 导出的函数。

### 9.2 preload

`window.electronAPI.updater` 保持不变，仅更新 TypeScript 类型以包含新字段。

### 9.3 UpdateDialog 弹窗行为

| 场景 | 行为 |
|------|------|
| downloading | 展示下载进度；`total <= 0` 时展示"未知大小" |
| downloaded (非强制) | [下次安装] [立即安装]，允许关闭 |
| downloaded (强制) | 只显示 [立即安装]，阻止关闭弹窗（`onOpenChange` 中拒绝关闭） |
| 回退 (ROLLBACK) | 弹窗标题和按钮文案改为"回退安装"，提示用户目标版本低于当前版本 |

`releaseNotes` 渲染：不再调用 `getReleaseByTag()` + `ReleaseNotesViewer`（那是 GitHub Release 专用）。改为直接从 `status.releaseNotes` 取值，用简单 `<pre>` 或 Markdown 组件渲染纯文本/Markdown 字符串。

### 9.4 About 页

- 保留 "检查更新" 按钮，点击调用 `checkForUpdates`。
- 检查中显示 loading；无更新显示 "已是最新版本"（`not-available` 时展示 `hint` 中的说明）。
- 有更新时显示目标版本和 `releaseNotes`，`releaseNotes` 同样直接渲染字符串，不调 GitHub API。
- 下载中展示进度条。

## 十一、服务端待配合事项

基于 2026-06-07 的接口匹配度分析，以下改动才能使客户端完成完整的检测→下载→校验链路。

### 11.1 `UpgradeCheckRequest` 增加 `arch` 参数

**文件**：`src/main/java/com/workmate/server/dto/request/UpgradeCheckRequest.java`

当前只有 `currentVersion` 和 `platform`，缺少 `arch`。导致同一平台 `x64`/`arm64` 的安装包无法区分。

新增字段：

```java
@NotBlank(message = "arch 不能为空")
@Pattern(regexp = "x64|arm64", message = "arch 必须为 x64/arm64")
private String arch;
```

### 11.2 `UpgradeCheckResponse` 补齐文件元数据

**文件**：`src/main/java/com/workmate/server/dto/response/UpgradeCheckResponse.java`

当前仅有 8 个字段，缺 5 个文件元数据字段。这些数据在 `UpgradeRelease` 表中已有存储。

新增字段：

```java
private String sha256;       // UpgradeRelease.sha256
private Long fileSize;       // UpgradeRelease.fileSize
private String fileName;     // UpgradeRelease.fileName
private String packageType;  // UpgradeRelease.packageType
private String publishedAt;  // UpgradeRelease.publishedAt 的 ISO 字符串
```

### 11.3 `UpgradeStrategy` 增加 `forceUpdate` 和 `arch`

**文件**：`src/main/java/com/workmate/server/entity/UpgradeStrategy.java`

当前策略表没有强制升级标记和架构维度。

新增字段：

```java
private Boolean forceUpdate; // 是否强制升级，默认 false
private String arch;         // x64 / arm64，与 platform 组合定位安装包
```

同时需要更新：
- `UpgradeStrategyMapper.xml`：resultMap 和查询列增加 `forceUpdate`/`arch`
- `findFirstByPlatformAndStatus` 查询需增加 `arch` 条件，改为 `findFirstByPlatformAndArchAndStatus`
- 或保持现有查询只按 platform 过滤，在 `checkUpgrade()` 中单独校验 `arch` 匹配

### 11.4 `checkUpgrade()` 改为从 `UpgradeRelease` 表获取文件信息

**文件**：`src/main/java/com/workmate/server/service/UpgradeService.java`

当前 `checkUpgrade()` 中 `downloadUrl` 取自 `activeStrategy.getDownloadUrl()`（策略表冗余字段），其他文件元数据（`sha256`、`fileSize`、`fileName`、`packageType`）完全没有返回。

修改方案 —— 在 `checkUpgrade()` 返回有更新结果前，通过 `targetVersion + platform + arch` 从 `UpgradeRelease` 表查询对应的安装包记录：

```java
// 在 hasUpdate=true 的返回分支前新增：
UpgradeRelease release = releaseMapper.findFirstByPlatformAndVersionAndArch(
    platform, targetVersion, request.getArch())
    .orElse(null);

return UpgradeCheckResponse.builder()
    .hasUpdate(true)
    .forceUpdate(activeStrategy.getForceUpdate() != null && activeStrategy.getForceUpdate())
    .releaseType(ReleaseType.valueOf(releaseType))
    .latestVersion(targetVersion)
    .downloadUrl(release != null ? release.getDownloadUrl() : null)
    .releaseNotes(activeStrategy.getReleaseNotes())
    .minVersion(activeStrategy.getMinVersion())
    .hint("ROLLBACK".equals(releaseType) ? "当前版本将回退到 " + targetVersion : null)
    .sha256(release != null ? release.getSha256() : null)
    .fileSize(release != null ? release.getFileSize() : null)
    .fileName(release != null ? release.getFileName() : null)
    .packageType(release != null ? release.getPackageType() : null)
    .publishedAt(release != null ? release.getPublishedAt().toString() : null)
    .build();
```

需要新增对应的 Mapper 方法：

**文件**：`src/main/java/com/workmate/server/mapper/UpgradeReleaseMapper.java`

```java
Optional<UpgradeRelease> findFirstByPlatformAndVersionAndArch(
    @Param("platform") String platform,
    @Param("version") String version,
    @Param("arch") String arch);
```

**文件**：`src/main/resources/mapper/UpgradeReleaseMapper.xml`

```xml
<select id="findFirstByPlatformAndVersionAndArch" resultMap="BaseResultMap">
    SELECT * FROM upgrade_release
    WHERE platform = #{platform} AND version = #{version} AND arch = #{arch}
    LIMIT 1
</select>
```

### 11.5 `forceUpdate` 逻辑修复

**文件**：`src/main/java/com/workmate/server/service/UpgradeService.java`

当前第 131 行 `.forceUpdate(false)` 是硬编码。改为从策略表读取：

```java
// 旧代码：
.hasUpdate(true).forceUpdate(false)

// 新代码：
.hasUpdate(true)
.forceUpdate(activeStrategy.getForceUpdate() != null && activeStrategy.getForceUpdate())
```

同时 `minVersion` 场景（第 107-113 行）中 `.forceUpdate(false)` 保持不变，因为该场景目标不是强制升到 latestVersion，而是提示用户先满足最低版本要求。

## 十二、推荐实施顺序

服务端 §11 的改动（`arch`/`sha256`/`fileSize`/`fileName`/`packageType`/`forceUpdate`）必须在客户端下载阶段之前完成，否则客户端无法按架构匹配安装包、无法做完整性校验。

```text

  - UpgradeCheckRequest 增加 arch 参数
  - UpgradeCheckResponse 增加 sha256 / fileSize / fileName / packageType / publishedAt
  - UpgradeStrategy 增加 forceUpdate / arch 字段
  - checkUpgrade() 改为从 UpgradeRelease 表获取文件元数据
  - 新增 UpgradeReleaseMapper 按 platform + version + arch 查询方法

阶段 1：客户端基础骨架
  - 创建 workmate-version.ts（SemVer 比较）
  - 创建 workmate-api.ts（检测接口 + 二次校验）
  - 重写 auto-updater.ts（移除 electron-updater，接入 WorkMate 检测）
  - 更新 updater-types.ts（扩展状态类型）
  - 渲染端类型更新（preload/index.ts、atoms/updater.ts）

阶段 2：下载与持久化
  - 创建 workmate-downloader.ts（依赖阶段 0 的 sha256/fileSize/arch）
  - 创建 workmate-manifest.ts
  - 下载进度 UI
  - 启动恢复 manifest

阶段 3：安装与 UI
  - 创建 workmate-installer.ts

  - 下载重试策略完善
```
