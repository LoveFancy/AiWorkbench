# 专家团召唤时版本升级方案

> 编辑时间：2026年6月22日
> 修订记录：2026-06-22 第二版——抽出共享召唤编排、修正召唤 guard、确定来源判定、补并发锁

---

## 一、背景与目标

### 1.1 现状

专家团服务端化后，`/workmate/expert-groups` 列表项中已带有 `version` 字段（见 [`ServerExpertGroupSummary`](file:///d:/AiWorkbench/packages/shared/src/types/agent.ts)），但目前**没有任何版本升级管理**：

- 列表合并时，[`mergeExpertGroups`](file:///d:/AiWorkbench/apps/electron/src/renderer/atoms/agent-atoms.ts#L249) 用 `server.version !== local.sourcePluginVersion`（字符串不等）把本地已安装的专家团标记为 `remote_update_available`，但**点击召唤时不会真正去下载新版**。
- 召唤逻辑 `handleSummon` 在 [`ExpertSummonButton.tsx`](file:///d:/AiWorkbench/apps/electron/src/renderer/experts/picker/ExpertSummonButton.tsx#L69) 和 [`ExpertPageView.tsx`](file:///d:/AiWorkbench/apps/electron/src/renderer/experts/views/ExpertPageView.tsx#L100) **两处几乎完全重复**，只处理两种情况：`remote_not_downloaded` → 下载后召唤；`available` → 直接召唤。
- 关键缺陷：两处都有 `if (group.status !== 'available') return`，使得 `remote_update_available` 状态的专家团**点击召唤直接被拦截、什么都不做**。
- 客户端尚未对接单个专家团详情接口 `/workmate/expert-groups/group-detail/:id`（[`expert-remote-service.ts`](file:///d:/AiWorkbench/apps/electron/src/main/lib/expert-remote-service.ts) 只有 list / featured-scenes / categories）。

### 1.2 目标

用户点击「召唤」按钮时：

1. 调用 `/workmate/expert-groups/group-detail/:id` 检查该专家团的最新版本号；
2. 若服务端版本**高于**本地已安装版本，自动调用 `/workmate/expert-groups/:id/download` 下载并覆盖安装最新专家团；
3. 用最新版本创建专家会话并开启窗口。

### 1.3 设计约束（已与需求方确认）

| 约束 | 说明 |
|---|---|
| **检查时机** | 每次召唤都检查（实时调 group-detail），不依赖列表刷新时机 |
| **超时** | group-detail 检查超时尽可能短（**1 秒**），防止影响召唤体验 |
| **失败降级** | 检查或下载失败（网络异常/离线/超时）时，**静默降级为本地现有版本召唤，绝不阻断** |
| **无检查态 overlay** | 检查更新阶段**不显示任何 overlay**，用户点击后对"检查/下载"过程无感知；现有 `ExpertSummoningOverlay`（召唤遮罩）保持原状，仅在创建会话期间展示，不新增"检查更新/下载最新版"文案态 |

## 二、整体流程

```
用户点击「召唤」(summon(group))   ← experts/ 下唯一召唤入口
  │
  ├── 远程未下载 (status === 'remote_not_downloaded')
  │     → 走现有「下载后召唤」流程（不变）
  │
  └── 本地已安装 (status === 'available' 或 'remote_update_available')   ← guard 需放行后者
        │
        ├─ 1. 仅当来源为远程下载(user:remote/*)时，静默调用
        │       ensureExpertGroupLatest(id, localVersion)
        │       （主进程编排，1s 超时，无 overlay，per-id 互斥）
        │       ├─ detail.version > localVersion → 后台下载安装(overwrite:true)，返回 { updated:true, plugin }
        │       ├─ detail.version ≤ localVersion → 返回 { updated:false }
        │       └─ 超时 / 请求失败 / 下载失败 → 吞掉异常，返回 { updated:false }（降级本地版）
        │
        ├─ 2. 若 updated=true → 用 IPC 返回的 plugin 解析出最新 group（无需二次 listAgentExpertGroups）
        │
        └─ 3. createExpertSession(最新 group) → openSession('agent', ...)（开窗）
```

核心原则：**版本检查/下载是"尽力而为"的增量，任何异常都不影响"用本地版本召唤"这条主路径。**

## 三、架构设计（方案 A：主进程编排 + 渲染层共享召唤入口）

两个落点：

1. **主进程**收敛"拉详情 → 比较版本 → 下载覆盖安装"的升级编排（`ensureExpertGroupLatest`），渲染层只调一次。
2. **渲染层**把当前重复在两个组件里的召唤逻辑抽到 `experts/` 下的**单一共享入口**（`useSummonExpert` hook），版本升级步骤只在这一个地方实现，满足开闭原则与"尽量在 experts 目录修改"的诉求。

### 3.1 主进程：新增详情拉取（1s 超时、不走缓存）

文件：[`expert-remote-service.ts`](file:///d:/AiWorkbench/apps/electron/src/main/lib/expert-remote-service.ts)

`httpGet` 已原生支持 `timeoutMs`（见 [`HttpRequestOptions`](file:///d:/AiWorkbench/apps/electron/src/shared/hteip-client.ts#L49)），超时返回 `{ ok:false, error:'请求超时' }` **不抛异常**，因此 1s 超时直接可用，无需自行包 `AbortController`。

注意：**版本检查必须取实时版本，不复用列表/详情的磁盘缓存**（list 接口有 30 分钟缓存，会漏判新版本）。

```typescript
/**
 * 拉取单个专家团详情（用于召唤时的实时版本检查）。
 *
 * 1s 超时；失败不抛异常（httpGet 已归一化），返回 null 交由上层降级。
 * 不读写磁盘缓存——版本检查必须命中服务端最新值。
 */
export async function fetchServerExpertGroupDetail(
  id: string,
): Promise<import('@proma/shared').ServerExpertGroupSummary | null> {
  const path = `/workmate/expert-groups/group-detail/${encodeURIComponent(id)}`
  const res = await httpGet<{ code: number; data: import('@proma/shared').ServerExpertGroupSummary }>(
    path,
    { timeoutMs: 1000 },
  )
  if (!res.ok || !res.data || res.data.code !== 0) {
    console.warn('[expert-remote] 获取专家团详情失败: id=%s status=%d', id, res.status)
    return null
  }
  return res.data.data
}
```

### 3.2 主进程：下载支持覆盖安装

文件：[`expert-download-service.ts`](file:///d:/AiWorkbench/apps/electron/src/main/lib/expert-download-service.ts)

现有 [`downloadAndInstallRemoteExpert`](file:///d:/AiWorkbench/apps/electron/src/main/lib/expert-download-service.ts#L74) 调用 `installUserPluginZip(tempPath, { marketplaceId: 'remote' })`，**未传 `overwrite`，默认 `false`**。升级场景下目标目录 `user-plugins/remote/<slug>` 已存在，`copyPluginAtomically` 会失败。

改动：让该函数接受 `overwrite` 选项并透传（首次下载传 `false`，升级传 `true`）：

```typescript
export async function downloadAndInstallRemoteExpert(
  groupId: string,
  options: { overwrite?: boolean } = {},
): Promise<AgentPluginInfo> {
  // ...下载到 tempPath 不变...
  const plugin = installUserPluginZip(tempPath, {
    marketplaceId: 'remote',
    overwrite: options.overwrite ?? false,
  })
  // ...清理临时文件、广播完成不变...
  return plugin
}
```

### 3.3 主进程：升级编排接口（含 per-id 并发锁）

新建 `apps/electron/src/main/lib/expert-upgrade-service.ts`（与下载服务解耦，职责单一）：

```typescript
import type { AgentPluginInfo } from '@proma/shared'
import { compareVersion } from './updater/workmate-version'
import { fetchServerExpertGroupDetail } from './expert-remote-service'
import { downloadAndInstallRemoteExpert } from './expert-download-service'

export interface EnsureExpertGroupLatestResult {
  updated: boolean
  plugin?: AgentPluginInfo
}

/** per-id 互斥：同一专家团的并发升级请求复用同一个 Promise，避免重复下载 + overwrite 写竞争 */
const inFlight = new Map<string, Promise<EnsureExpertGroupLatestResult>>()

/**
 * 召唤前确保专家团为最新版（尽力而为）。
 *
 * @param id           专家团 ID
 * @param localVersion 本地已安装版本
 * @returns updated 是否实际下载了新版；任何异常均降级为 { updated:false }
 */
export function ensureExpertGroupLatest(
  id: string,
  localVersion: string,
): Promise<EnsureExpertGroupLatestResult> {
  const existing = inFlight.get(id)
  if (existing) return existing

  const task = (async (): Promise<EnsureExpertGroupLatestResult> => {
    try {
      const detail = await fetchServerExpertGroupDetail(id)        // 1s 超时
      if (!detail) return { updated: false }                       // 检查失败 → 降级
      if (compareVersion(detail.version, localVersion) <= 0) {
        return { updated: false }                                  // 无更新
      }
      const plugin = await downloadAndInstallRemoteExpert(id, { overwrite: true })
      return { updated: true, plugin }
    } catch (err) {
      console.warn('[expert-upgrade] 召唤前版本检查/升级失败，降级本地版: id=%s', id, err)
      return { updated: false }                                    // 下载失败 → 降级
    } finally {
      inFlight.delete(id)
    }
  })()

  inFlight.set(id, task)
  return task
}
```

版本比较复用 [`compareVersion`](file:///d:/AiWorkbench/apps/electron/src/main/lib/updater/workmate-version.ts#L20)（`detail.version > localVersion` 才升级），不用脆弱的字符串 `!==`。

### 3.4 IPC + preload 桥接

- `apps/electron/src/main/ipc.ts`：新增通道 `EXPERT_IPC_CHANNELS.ENSURE_LATEST`，handler 调用 `ensureExpertGroupLatest(id, localVersion)`。
- `apps/electron/src/preload/index.ts`：新增方法签名与实现：

```typescript
/** 召唤前确保专家团为最新版（静默，失败降级本地版） */

```

### 3.5 渲染层：抽出共享召唤入口（消除重复）

**问题**：`handleSummon` 当前在 [`ExpertSummonButton.tsx`](file:///d:/AiWorkbench/apps/electron/src/renderer/experts/picker/ExpertSummonButton.tsx#L69) 与 [`ExpertPageView.tsx`](file:///d:/AiWorkbench/apps/electron/src/renderer/experts/views/ExpertPageView.tsx#L100) 重复实现。若按"两处各加一遍版本检查"会持续违反开闭原则、逻辑易漂移。

**做法**：新建 `apps/electron/src/renderer/experts/hooks/useSummonExpert.ts`，作为召唤的**唯一编排出口**。两个组件删除各自的 `handleSummon`，改为调用该 hook。版本升级逻辑只存在于此处。

> 选 hook 而非 atom：召唤需要 `useOpenSession()`（hook）与 `toast`，无法在纯 atom 内调用；`summoningGroup` 遮罩状态也由 hook 暴露，组件自行决定是否渲染 `ExpertSummoningOverlay`。

```typescript
// experts/hooks/useSummonExpert.ts
import * as React from 'react'
import { useSetAtom } from 'jotai'
import { toast } from 'sonner'
import type { AgentExpertGroupInfo } from '@proma/shared'
import { createExpertSessionAtom, loadAgentExpertGroupsAtom } from '@/atoms/agent-atoms'
import { recordRecentExpertGroupAtom } from '@/experts/atoms/expert-follow'
import { useOpenSession } from '@/hooks/useOpenSession'

/** 远程下载来源判定：仅 user:remote/* 才参与召唤时自动升级 */
function isRemoteSourced(group: AgentExpertGroupInfo): boolean {
  return group.sourcePluginId.startsWith('user:remote/')
}

export function useSummonExpert() {
  const createExpertSession = useSetAtom(createExpertSessionAtom)
  const loadGroups = useSetAtom(loadAgentExpertGroupsAtom)
  const recordRecent = useSetAtom(recordRecentExpertGroupAtom)
  const openSession = useOpenSession()
  const [summoningGroup, setSummoningGroup] = React.useState<AgentExpertGroupInfo | null>(null)

  const summon = React.useCallback(async (group: AgentExpertGroupInfo): Promise<void> => {
    // ── 分支一：远程未下载 → 下载后召唤（保持现有行为）
    if (group.sourcePluginKind === 'remote' && group.status !== 'available') {
      if (group.status === 'remote_downloading') { toast('正在下载中，请稍候'); return }
      setSummoningGroup(group)
      try {
        const installed = await window.electronAPI.downloadRemoteExpert(group.id)
        await loadGroups()
        toast.success(`已下载 ${group.name}，正在召唤...`)
        const session = await createExpertSession({
          ...group, status: 'available', sourcePluginKind: 'user',
          sourcePluginId: installed.name, sourcePluginVersion: installed.version,
        })
        recordRecent(group.id)
        openSession('agent', session.id, session.title)
      } catch (err) {
        console.error('[专家团] 下载远程专家团失败:', err)
        toast.error(`下载 ${group.name} 失败`)
      } finally {
        setSummoningGroup(null)
      }
      return
    }

    // ── 分支二：本地已安装（available / remote_update_available）→ 先升级再召唤
    //    注意：必须放行 remote_update_available，不能再用 `status !== 'available'` 直接 return
    if (group.status !== 'available' && group.status !== 'remote_update_available') return

    setSummoningGroup(group)
    try {
      let target = group
      if (isRemoteSourced(group)) {
        const { updated, plugin } = await window.electronAPI.ensureExpertGroupLatest(
          group.id, group.sourcePluginVersion,
        )
        if (updated && plugin) {
          await loadGroups()
          // 用 IPC 返回的 plugin 直接构造最新召唤目标，避免二次 listAgentExpertGroups
          target = { ...group, status: 'available', sourcePluginVersion: plugin.version }
        }
      }
      const session = await createExpertSession(target)
      recordRecent(target.id)
      openSession('agent', session.id, session.title)
      toast.success(`已召唤${target.name}`)
    } catch (error) {
      console.error('[专家团] 召唤失败:', error)
      toast.error('召唤专家团失败')
    } finally {
      setSummoningGroup(null)
    }
  }, [createExpertSession, loadGroups, recordRecent, openSession])

  return { summon, summoningGroup }
}
```

> 说明：`createExpertSession` 内部按 `expertGroupId + expertPluginId` 从主进程重新解析 runtime（见 [`getAgentExpertGroup`](file:///d:/AiWorkbench/apps/electron/src/main/lib/agent-expert-group-manager.ts#L257)），`loadGroups()` 后本地已是新版，因此 `target` 主要用于 toast 名称与 recent 记录；最终 runtime 仍取本地最新版。

两个组件改为：

```typescript
const { summon, summoningGroup } = useSummonExpert()
// ExpertSummonButton：onSummon={(g) => void summon(g)} + 渲染 ExpertSummoningOverlay(summoningGroup)
// ExpertPageView：onSummon={(g) => void summon(g)}
```

## 四、UI 行为

| 阶段 | UI 表现 |
|---|---|
| 点击召唤 → 版本检查 | **无 overlay、无 toast、无感知**（1s 内完成或超时） |
| 检查发现新版 → 后台下载 | 无独立检查态 overlay；下载在编排接口内串行完成后才创建会话 |
| 创建会话 | 沿用现有 `ExpertSummoningOverlay`（召唤遮罩），文案不变 |
| 召唤成功 | `toast.success(已召唤{name})` |
| 检查/下载失败 | 静默降级为本地版召唤，**不弹错误**（仅 console.warn）；只有创建会话本身失败才 `toast.error` |

> 注：现有 `EXPERT_IPC_CHANNELS.DOWNLOAD_PROGRESS` 进度广播保留，但召唤路径不订阅、不展示进度条。

## 五、关键约束与边界

1. **召唤 guard 必须放行 `remote_update_available`**（本方案最易遗漏点）：现有 `if (group.status !== 'available') return` 会拦掉"可更新"的本地专家团，导致升级逻辑根本不触发。共享 hook 中改为 `status !== 'available' && status !== 'remote_update_available'` 才放行。

2. **来源判定用插件 ID 前缀**：远程下载安装后 `sourcePluginKind === 'user'`、`sourcePluginId` 形如 `user:remote/<slug>`（见 [`listInstalledPlugins`](file:///d:/AiWorkbench/apps/electron/src/main/lib/plugin-registry-service.ts#L498)）。`sourcePluginKind === 'remote'` 仅表示"服务端合成、尚未下载"项。故自动升级判定为 `sourcePluginId.startsWith('user:remote/')`，不要用 `sourcePluginKind`。

3. **内置 / 用户上传专家团不参与召唤时自动升级**：[`assertNoDuplicateExpertGroups`](file:///d:/AiWorkbench/apps/electron/src/main/lib/plugin-registry-service.ts#L707) 会拒绝"另一个插件已声明相同专家团 ID"。若专家团本地来源是 builtin 或用户上传，再向 `user:remote/` 安装同 ID 会触发重复校验报错。因此这两类来源跳过版本检查、直接用本地版召唤（仍不阻断）。这是明确的 UX 取舍。

4. **下载必须覆盖安装**：升级时目标目录已存在，必须 `overwrite: true`，否则 `copyPluginAtomically` 失败。

5. **并发/重入保护**：`ensureExpertGroupLatest` 按 `groupId` 做 in-flight 去重（§3.3），快速双击或跨组件并发召唤同一专家团时复用同一 Promise，避免并发下载 + `overwrite` 写竞争损坏 `user-plugins/remote/<slug>`。

6. **超时范围**：1s 超时**仅作用于 group-detail 检查**；真正的 zip 下载阶段不强加 1s（下载本就耗时，且仅在已确认有更新后才发生）。下载失败同样降级本地版。

7. **版本比较语义**：SemVer 比较，`detail.version > localVersion` 才升级；相等或更低不动本地版本。

8. **版本检查不走缓存**：`fetchServerExpertGroupDetail` 不读写磁盘缓存，确保命中服务端实时 `version`。

## 八、测试要点

- `compareVersion` 在 `version > local` / `=` / `<` / 非法版本号下的升级决策正确。
- `fetchServerExpertGroupDetail` 在 1s 内无响应时返回 `null`（依赖 httpGet 超时归一化）。
