# 专家团不可用状态与技能检测方案

> 编辑时间：2026年6月15日

---

## 一、概述

专家团在加载过程中会经过多层校验，任何校验失败都会导致专家团标记为"不可用"状态。本文档详细说明各不可用状态的触发条件、检测逻辑和用户可见信息。

---

## 二、状态枚举

```typescript
// packages/shared/src/types/agent.ts

export type AgentExpertGroupStatus =
  | 'available'           // 可正常召唤
  | 'plugin_disabled'     // 所属插件已禁用
  | 'plugin_uninstalled'  // 来源插件已卸载
  | 'invalid_manifest'    // expert-groups/*.json 配置错误
  | 'missing_subagent'    // 缺少 SubAgent 定义文件
  | 'missing_skill'       // 缺少依赖的 Skill
  | 'mcp_conflict'        // MCP 连接器冲突
```

---

## 三、状态判定流程

```
┌─────────────────────────────────────────────────────┐
│  statusFor(enabled, issues)                          │
│                                                     │
│  1. if (!enabled) → 'plugin_disabled'               │
│                                                     │
│  2. 遍历 issues，查找第一个带 status 字段的 issue      │
│     → 返回该 status 值                               │
│                                                     │
│  3. if (issues 中有 level === 'error')               │
│     → 'invalid_manifest'                            │
│                                                     │
│  4. 否则 → 'available'                              │
└─────────────────────────────────────────────────────┘
```

```typescript
// agent-expert-group-manager.ts

function statusFor(enabled: boolean, issues: ExpertIssue[]): AgentExpertGroupStatus {
  if (!enabled) return 'plugin_disabled'
  const statusIssue = issues.find((issue) => issue.status)
  if (statusIssue?.status) return statusIssue.status
  if (issues.some((issue) => issue.level === 'error')) return 'invalid_manifest'
  return 'available'
}
```

## 四、各状态详解

### 4.1 `available` — 可正常召唤

**条件**：插件已启用 + 所有校验通过。

**UI 表现**：绿色 Badge "可用"，召唤按钮可点击。

### 4.2 `plugin_disabled` — 插件已禁用

**条件**：`plugin.enabled === false`。

**触发场景**：用户在设置 → 插件管理中禁用了该插件。

**UI 表现**：红色 Badge "插件已禁用"，召唤按钮禁用。

### 4.3 `plugin_uninstalled` — 来源已卸载

**条件**：插件已从系统中移除，但专家团信息仍缓存在某处。

**触发场景**：用户卸载了插件，但会话历史中仍引用该专家团。

**UI 表现**：红色 Badge "来源已卸载"。

### 4.4 `invalid_manifest` — 配置错误

**条件**：`expert-groups/{id}.json` 文件解析失败或缺少必要字段。

**触发场景**：
- JSON 格式错误
- 缺少 `id`、`name`、`mainRole` 等必填字段
- `mainRole.prompt` 为空

**UI 表现**：红色 Badge "配置错误"，详情弹窗中显示具体错误信息。

### 4.5 `missing_subagent` — 缺少子专家

**条件**：`manifest.subagents` 中声明的 Agent 名称在 `agents/{name}.md` 中不存在。

**检测逻辑**：

```typescript
// agent-expert-group-manager.ts — validateExpertReferences()

for (const agentName of manifest.subagents ?? []) {
  if (!existsSync(join(pluginPath, 'agents', `${agentName}.md`))) {
    issues.push({
      level: 'error',
      status: 'missing_subagent',
      message: `缺少 SubAgent: ${agentName}`,
    })
  }
}
```

**触发场景**：
- WorkBuddy 转换时 SubAgent prompt 未正确生成到 `agents/` 目录
- 手动编辑 expert-groups/*.json 时声明了不存在的 subagent
- 插件打包时遗漏了 agents 目录

**UI 表现**：红色 Badge "缺少子专家"，详情弹窗中列出缺失的 SubAgent 名称。

### 4.6 `missing_skill` — 缺少技能

**条件**：`manifest.skills` 中声明的 Skill 在以下两个位置均不存在：
1. 插件目录 `skills/{skillName}/SKILL.md`
2. 全局 `default-skills/{skillName}/SKILL.md`

**检测逻辑**：

```typescript
// agent-expert-group-manager.ts

function hasSkill(pluginPath: string, defaultSkillsDir: string, skillName: string): boolean {
  return existsSync(join(pluginPath, 'skills', skillName, 'SKILL.md')) ||
    existsSync(join(defaultSkillsDir, skillName, 'SKILL.md'))
}

for (const skillName of manifest.skills ?? []) {
  if (!hasSkill(pluginPath, defaultSkillsDir, skillName)) {
    issues.push({
      level: 'error',
      status: 'missing_skill',
      message: `缺少 Skill: ${skillName}`,
    })
  }
}
```

**什么是 Skill**：

Skill 是 LLM 执行特定任务时加载的提示词/规则文件，格式为 `skills/{name}/SKILL.md`。专家团在 `skills` 字段中声明依赖的 Skill 名称列表，运行时由 Orchestrator 注入对应 Skill 的提示词。

**Skill 的查找路径**：

```
1. 插件目录
   {pluginPath}/skills/{skillName}/SKILL.md
   ↓ 不存在则尝试
2. 全局默认 Skills 目录
   {appData}/default-skills/{skillName}/SKILL.md
```

**触发场景**：
- WorkBuddy 转换时引用了 WorkMate 中不存在的 Skill（如 `ima-skills`、`web-research`）
- 插件依赖的 Skill 被删除或重命名
- 插件打包时遗漏了 skills 目录

**UI 表现**：红色 Badge "缺少技能"，详情弹窗中列出缺失的 Skill 名称。

**内置专家团的情况**：

内置专家团 `architecture-decision-team` 的 `skills: []`，不依赖任何外部 Skill，因此不会触发 `missing_skill`。其能力通过 `builtinTools: ["web-search"]` 声明，由系统内置工具提供。

### 4.7 `mcp_conflict` — 连接器冲突

**条件**：专家团声明的 MCP Server 与其他已安装插件存在名称冲突。

**触发场景**：多个插件声明了同名的 MCP Server 但配置不同。

**UI 表现**：红色 Badge "连接器冲突"。

## 五、不可用状态的 UI 展示

### 5.1 卡片状态

```typescript
// experts/card/ExpertStatusBadge.tsx

const STATUS_LABELS: Record<AgentExpertGroupStatus, string> = {
  available: '可用',
  plugin_disabled: '插件已禁用',
  plugin_uninstalled: '来源已卸载',
  invalid_manifest: '配置错误',
  missing_subagent: '缺少子专家',
  missing_skill: '缺少技能',
  mcp_conflict: '连接器冲突',
}
```

- `available`：绿色 Badge
- 其他状态：红色 Badge（`variant="destructive"`）

### 5.2 详情弹窗

不可用专家团的详情弹窗中会显示 `issues` 列表：

```typescript
// experts/detail/ExpertDetailDialog.tsx

{group.issues.length > 0 && (
  <section>
    <h4 className="text-sm font-medium text-destructive">配置问题</h4>
    <div className="mt-2 space-y-2">
      {group.issues.map((issue, index) => (
        <div key={`${issue.message}-${index}`}
             className="rounded-md bg-destructive/10 px-3 py-2 text-sm text-destructive">
          {issue.message}
        </div>
      ))}
    </div>
  </section>
)}
```

### 5.3 召唤按钮

不可用专家团的召唤按钮始终禁用（`disabled={group.status !== 'available'}`）。

### 5.4 筛选

用户可通过筛选标签切换"可用"/"不可用"视图：

```typescript
// experts/shared/ExpertFilterPills.tsx

{ tag: 'available', label: '可用', icon: CheckCircle },
{ tag: 'unavailable', label: '不可用', icon: AlertTriangle },
```

## 六、WorkBuddy 转换时的 Skill 处理

### 6.1 转换 Skill 的映射规则

WorkBuddy 专家的 `skills` 字段指向 `.codebuddy-plugin/skills/{name}/` 目录。转换 Skill 会：

1. 读取 WorkBuddy 源目录的 `skills/` 列表
2. 检查 WorkMate 的 `default-skills/` 中是否存在同名 Skill
3. 如果存在 → 在 `expert-groups/{id}.json` 的 `skills` 字段中声明
4. 如果不存在 → 尝试将 Skill 内容融入 `mainRole.prompt`（由 LLM 判断是否相关），并在 issues 中记录警告

### 6.2 常见缺失 Skill 场景

| WorkBuddy Skill | WorkMate 对应 | 处理方式 |
|-----------------|--------------|---------|
| `ima-skills` | 无 | 融入 prompt 或标记为缺失 |
| `web-research` | 无（使用 builtinTools: web-search） | 转换为 builtinTools 声明 |
| `code-review` | 可能有 | 检查 default-skills/ |
| `pdf` | 可能有 | 检查 default-skills/ |

## 七、数据流总结

```
插件安装/启用
    │
    ▼
listAgentExpertGroups()
    │
    ├── readExpertManifest(filePath)
    │       ↓ 解析 expert-groups/{id}.json
    │       ↓ 校验必填字段
    │
    ├── validateExpertReferences(pluginPath, manifest)
    │       ↓ 检查 agents/{name}.md 是否存在
    │       ↓ 检查 skills/{name}/SKILL.md 是否存在
    │       ↓ 检查 .mcp.json 中的 MCP Server
    │       ↓ 检查 builtinTools 是否受支持
    │
    └── statusFor(enabled, allIssues)
            ↓ 综合判定最终状态
            ↓
        AgentExpertGroupInfo.status
```
