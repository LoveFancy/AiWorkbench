# 服务端：专家团管理后台方案

编辑时间：2026年6月16日

> **关联文档**：[专家团对workmate接口](./专家团对workmate接口.md)
>
> 本方案补充专家团管理后台功能：上传 ZIP 批量导入、元数据管理、精选场景配置。管理台通过 Spring Boot 上传 ZIP 并落盘到 `/app/workmate/plugins/`，客户端下载由 Spring Boot 实时打包返回 zip 流。

---

## 一、整体架构

```
管理台前端（上传 zip）
     ↓ POST multipart/form-data
Spring Boot（ConsoleExpertGroupController）
     ↓ 调用
ExpertGroupService
     ├── 解压 zip → 临时目录
     ├── 逐目录解析 JSON → 提取元数据
     ├── 移动文件 → /app/workmate/plugins/{id}/
     └── 写入 DB → expert_groups 表
```

职责划分：

| 模块 | 职责 |
|------|------|
| 管理台前端 | 上传 ZIP、展示列表、编辑元数据、管理精选场景 |
| Spring Boot | 接收上传、解压校验、文件落盘、元数据写入、客户端列表/详情/下载 API |
| MySQL | expert_groups（专家团元数据）、featured_scenes（精选场景） |
| 客户端 | 调用列表/精选场景接口，按需下载 zip（服务端实时打包返回） |

## 二、数据库设计

### 2.1 `expert_groups` 专家团元数据表

```sql
-- V11__expert_groups.sql
CREATE TABLE `expert_groups` (
    `id`           VARCHAR(64)   NOT NULL COMMENT '专家团 ID（来自目录名/JSON）',
    `name`         VARCHAR(256)  NOT NULL COMMENT '展示名称',
    `description`  TEXT          NULL     COMMENT '简要描述',
    `version`      VARCHAR(32)   NOT NULL COMMENT '版本号',
    `author`       VARCHAR(128)  NULL     COMMENT '作者/团队',
    `category`     VARCHAR(64)   NULL     COMMENT '分类标签',
    `tags`         JSON          NULL     COMMENT '标签列表 JSON 数组',
    `icon_url`     VARCHAR(512)  NULL     COMMENT '图标 URL',
    `is_active`    BOOLEAN       NOT NULL DEFAULT TRUE COMMENT '是否启用',
    `file_path`    VARCHAR(512)  NOT NULL COMMENT '插件文件目录（相对 plugins 根路径）',
    `file_size`    BIGINT        NULL     COMMENT '目录总大小（字节）',
    `meta_json`    JSON          NULL     COMMENT '完整 JSON 元数据（expert-groups/{id}.json 原文，含 mainRole.prompt/skills/subagents 等）',
    `created_at`   DATETIME(3)   NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    `updated_at`   DATETIME(3)   NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
    PRIMARY KEY (`id`),
    INDEX `idx_category` (`category`),
    INDEX `idx_is_active` (`is_active`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='专家团元数据表';

CREATE TABLE `featured_scenes` (
    `id`               VARCHAR(64)   NOT NULL COMMENT '场景标识，如 "content"',
    `name`             VARCHAR(128)  NOT NULL COMMENT '场景名称，如 "内容创作"',
    `icon_url`         VARCHAR(512)  NULL     COMMENT '场景图标',
    `sort_order`       INT           NOT NULL DEFAULT 0 COMMENT '排序权重',
    `expert_group_ids` JSON          NOT NULL COMMENT '该场景下的专家团 ID 列表',
    `created_at`       DATETIME(3)   NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    `updated_at`       DATETIME(3)   NOT NULL DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3),
    PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='精选场景分类表';
```

### 2.2 字段映射（JSON → DB）

每个专家团目录下有**两个 JSON 文件**，分别提供不同维度的元数据：

#### 文件 1：`.claude-plugin/plugin.json`（插件清单）

```json
{
  "name": "深网网",
  "version": "1.0.0",
  "description": "精通ML模型开发部署优化的全栈AI工程师",
  "author": { "name": "WorkBuddy Import" },
  "license": "MIT",
  "keywords": ["expert-group", "AI工程", "模型部署", "ML流水线"],
  "expertGroup": "ai-engineer"
}
```

| 字段 | 说明 |
|------|------|
| `name` | 插件展示名称 |
| `version` | 版本号 |
| `description` | 简要描述 |
| `author.name` | 作者名 |
| `keywords` | 关键字数组（不含 `expert-group` 这个固定标签） |
| `expertGroup` | 指向的专家团 ID（**必须**与目录名一致） |

#### 文件 2：`expert-groups/{id}.json`（专家团定义）

```json
{
  "id": "ai-engineer",
  "name": "深网网",
  "description": "精通ML模型开发部署优化的全栈AI工程师",
  "introduction": "我们想在公司产品中集成AI功能...",
  "mainRole": {
    "name": "AI工程师",
    "prompt": "# AI Engineer Agent\n\nYou are an **AI Engineer**..."
  },
  "subagents": [],
  "subagentLabels": {},
  "builtinTools": ["web-search"],
  "skills": ["agent-browser-core", "agent-team-orchestration", ...],
  "mcpServers": [],
  "tags": ["AI工程", "模型部署", "ML流水线"],
  "samplePrompts": ["我们想在公司产品中集成AI功能..."],
  "toolsPolicy": { "mode": "inherit", "allowedTools": [], "disallowedTools": [] }
}
```

| 字段 | 说明 |
|------|------|
| `id` | 专家团 ID（**必须**与目录名一致） |
| `name` | 展示名称 |
| `description` | 简要描述（与 plugin.json 可能相同） |
| `introduction` | 引导语/开场白 |
| `mainRole` | 核心角色定义（含 `name` 和 `prompt`） |
| `subagents` | 子 Agent 名称列表 |
| `subagentLabels` | 子 Agent 中文标签映射 |
| `builtinTools` | 内置工具列表 |
| `skills` | 依赖的 Skill 名称列表 |
| `mcpServers` | MCP 服务器配置 |
| `tags` | 标签数组 |
| `samplePrompts` | 示例提示词 |
| `toolsPolicy` | 工具策略配置 |

#### 提取到 DB 列的映射

| DB 列 | 取值来源 | 优先级 | 必填 |
|--------|----------|--------|------|
| `id` | `plugin.json` → `expertGroup`，或 `expert-groups/{id}.json` → `id`，两者必须一致 | — | 是 |
| `name` | `expert-groups/{id}.json` → `name`，回退 `plugin.json` → `name` | expert-groups 优先 | 是 |
| `version` | `plugin.json` → `version` | — | 是 |
| `description` | `expert-groups/{id}.json` → `description`，回退 `plugin.json` → `description` | expert-groups 优先 | 否 |
| `author` | `plugin.json` → `author.name` | — | 否 |
| `category` | `plugin.json` → `keywords[1]`（跳过首项 `expert-group` 后取第一个关键字） | — | 否 |
| `tags` | `expert-groups/{id}.json` → `tags`，回退 `plugin.json` → `keywords`（去掉 `expert-group`） | expert-groups 优先 | 否 |
| `icon_url` | 暂不从 JSON 提取（后续可约定目录下的 icon.png） | — | 否 |
| `meta_json` | `expert-groups/{id}.json` 完整原文，回退 `plugin.json` 完整原文 | expert-groups 优先 | 否 |

#### 两种文件情况处理

| 情况 | 处理方式 |
|------|---------|
| 两个文件**都存在** | 交叉校验 `expertGroup`/`id` 与目录名一致，不一致则失败；优先使用 `expert-groups/{id}.json` 字段 |
| **仅有** `expert-groups/{id}.json` | 从该文件提取全部字段，其中 `version` 取不到则记录为 `"0.0.0"` |
| **仅有** `plugin.json` | 从该文件提取基础字段（name/version/description/author/category），`meta_json` 存 plugin.json 原文；**允许导入但标记为"仅清单"** |
| 两个文件**都不存在** | 记录失败，跳过该目录 |

## 三、上传 ZIP 包处理流程

### 3.1 ZIP 包结构约定

```
upload.zip
├── ai-engineer/                 ← 完整结构：plugin.json + expert-groups 都有
│   ├── .claude-plugin/
│   │   └── plugin.json          ← 插件清单（必读：name/version/author/keywords/expertGroup）
│   ├── expert-groups/
│   │   └── ai-engineer.json     ← 专家团定义（必读：mainRole.prompt/skills/tags 等）
│   ├── agents/                  ← SubAgent 定义（如有）
│   │   └── reviewer.md
│   ├── skills/                  ← 依赖的 Skills（如有）
│   │   └── agent-browser-core/
│   │       └── SKILL.md
│   └── .mcp.json                ← MCP 配置（如有）
├── ai-content-creator-team/     ← 仅有 plugin.json（无 expert-groups/ 目录）
│   ├── .claude-plugin/
│   │   └── plugin.json
│   ├── agents/
│   │   └── ...
│   └── skills/
│       └── ...
├── content-creator/             ← 完整结构
│   ├── .claude-plugin/
│   │   └── plugin.json
│   ├── expert-groups/
│   │   └── content-creator.json
│   └── skills/
│       └── ...
└── (跳过 __MACOSX/  .DS_Store 等非目录项)
```

约束：
- 每个顶层目录对应一个专家团，目录名即为专家团 ID
- 至少需要存在 `.claude-plugin/plugin.json` 或 `expert-groups/{目录名}.json` 之一
- `plugin.json` 的 `expertGroup` 字段必须与目录名一致（如果存在）
- `expert-groups/{id}.json` 的 `id` 字段必须与目录名一致（如果存在）
- 两个 JSON 同时存在时，`version` 以 `plugin.json` 为准

### 3.2 上传处理步骤

```
1. 接收 MultipartFile (zip)
   ├── 校验：非空、contentType 为 zip、大小 ≤ 50MB
   └── 保存到 /app/workmate/upload-tmp/{uuid}.zip

2. 解压到 /app/workmate/upload-tmp/{uuid}/
   └── ⚠️ 安全校验：每个条目都校验 canonicalPath 在 targetDir 内，
       防止 Zip Slip（../../../etc/passwd 路径穿越攻击）

3. 扫描顶层目录
   └── 过滤：只保留目录、跳过隐藏文件（.DS_Store）、跳过 __MACOSX

4. 对每个专家团目录处理：
   ├── a. 校验目录名：只允许 [a-zA-Z0-9_-]+，拒绝含特殊字符的目录名
   ├── b. 读取 .claude-plugin/plugin.json（如果存在）
   │     └── 解析 JSON → 提取 name/version/description/author/keywords/expertGroup
   │     └── 校验 expertGroup == 目录名（不一致 → 失败跳过）
   ├── c. 读取 expert-groups/{目录名}.json（如果存在）
   │     └── 解析 JSON → 提取 id/name/description/tags/skills/mainRole 等
   │     └── 校验 id == 目录名（不一致 → 失败跳过）
   ├── d. 合并字段（按优先级映射）
   │     ├── id：两个文件交叉校验一致
   │     ├── name：expert-groups 优先，plugin.json 回退
   │     ├── version：plugin.json 优先
   │     ├── description：expert-groups 优先，plugin.json 回退
   │     ├── author：plugin.json → author.name
   │     ├── category：plugin.json → keywords 第二个元素
   │     ├── tags：expert-groups 优先，plugin.json → keywords 回退
   │     └── meta_json：expert-groups 优先存完整原文
   ├── e. 两个 JSON 都不存在 → 记录失败，跳过
   ├── f. 重复处理（根据 mode 参数）：
   │     ├── mode=cover（默认）：加锁 → 删旧文件 → 写新文件 → 更新 DB
   │     └── mode=skip：若 DB 已存在该 ID → 跳过
   ├── g. 移动文件到 /app/workmate/plugins/{目录名}/
   │     └── 先删旧目录，再 move（ATOMIC_MOVE，失败时回退到 REPLACE_EXISTING）
   ├── h. 递归计算目录总大小，写入 file_size
   └── i. 写入/更新 DB 记录（事务）

5. 清理临时文件（finally 块确保一定执行）

6. 返回详细结果
```

### 3.3 异常处理策略

| 场景 | 处理方式 |
|------|---------|
| 某个专家团两个 JSON 都不存在 | 跳过该目录，结果记为 FAILED："缺少 plugin.json 和 expert-groups/{id}.json" |
| 仅有 plugin.json，无 expert-groups | 允许导入，从 plugin.json 提取基础字段，meta_json 存 plugin.json 原文 |
| 仅有 expert-groups，无 plugin.json | 允许导入，version 记为 "0.0.0" |
| JSON 格式错误 | 跳过该目录，结果记为 FAILED，注明具体错误 |
| plugin.json 的 expertGroup ≠ 目录名 | 跳过，结果记为 FAILED："expertGroup 字段与目录名不一致" |
| expert-groups/{id}.json 的 id ≠ 目录名 | 跳过，结果记为 FAILED："id 字段与目录名不一致" |
| 两个 JSON 的 name 不一致 | 不阻塞，以 expert-groups 为准（仅日志 warn） |
| 文件移动失败（磁盘满等） | 该条回滚 DB，继续处理其他 |
| DB 写入失败 | 回滚该条 DB + 删除已移动的文件 |
| 全部失败 | 返回空 success，所有结果标记 FAILED |

**不做整体回滚**：部分成功部分失败是合理的，避免因一个坏数据导致整包导入失败。

### 3.4 并发控制

- 使用 `ConcurrentHashMap<String, ReentrantLock>` 按 `expertGroupId` 粒度加锁
- 同一 ID 的两次上传串行执行，避免文件覆盖竞争

## 四、API 设计

### 4.1 管理后台接口（/workmate/console）

| # | 方法 | 路径 | 说明 |
|---|------|------|------|
| 1 | POST | `/workmate/console/expert-groups/upload` | 上传 ZIP 批量导入 |
| 2 | GET | `/workmate/console/expert-groups` | 分页列表（管理用） |
| 3 | GET | `/workmate/console/expert-groups/:id` | 详情（含 metaJson） |
| 4 | PUT | `/workmate/console/expert-groups/:id` | 更新元数据 |
| 5 | DELETE | `/workmate/console/expert-groups/:id` | 删除（DB + 文件） |
| 6 | POST | `/workmate/console/expert-groups/:id/toggle` | 启用/禁用 |
| 7 | GET | `/workmate/console/featured-scenes` | 精选场景列表 |
| 8 | POST | `/workmate/console/featured-scenes` | 新增场景 |
| 9 | PUT | `/workmate/console/featured-scenes/:id` | 更新场景基础信息 |
| 10 | DELETE | `/workmate/console/featured-scenes/:id` | 删除场景 |
| 11 | POST | `/workmate/console/featured-scenes/:id/expert-groups` | 向场景添加专家团 |

| 14 | POST | `/workmate/console/expert-groups/import` | 一键导入（从导出 ZIP 恢复，覆盖模式） |

```
POST /workmate/console/featured-scenes/:id/expert-groups
Content-Type: application/json

Body:
{
  "expertGroupId": "code-expert"   // 要添加的专家团 ID（必填）
}

服务端处理：
  1. 校验 featured_scene 存在
  2. 校验 expertGgroups 表中存在该 ID 且 is_active=true
     → 不存在或已禁用 → 返回 400 "专家团不存在或已禁用"
  3. 检查该 expertGroupId 是否已在 expert_group_ids 数组中
     → 已存在 → 返回 400 "该专家团已在此场景中"
  4. 将 expertGroupId 追加到 expert_group_ids JSON 数组末尾
  5. 更新 featured_scenes.updated_at

响应：
{ "code": 0, "message": "已添加专家团 code-expert 到场景 content" }
```

#### 接口 12：从场景移除专家团

```
DELETE /workmate/console/featured-scenes/:id/expert-groups/:expertGroupId

服务端处理：
  1. 校验 featured_scene 存在
  2. 检查该 expertGroupId 是否在 expert_group_ids 数组中
     → 不存在 → 返回 400 "该专家团不在此场景中"
  3. 从 expert_group_ids JSON 数组中移除该 ID
  4. 更新 featured_scenes.updated_at

响应：
{ "code": 0, "message": "已从场景 content 中移除专家团 code-expert" }
```

```
GET /workmate/console/expert-groups/export

响应：ZIP 文件下载流
  Content-Type: application/zip
  Content-Disposition: attachment; filename="export-expert-groups-{yyyyMMddHHmmss}.zip"

ZIP 包内部结构：
export-expert-groups-{timestamp}.zip
├── data.json          ← { exportTime, expertGroups: [...], featuredScenes: [...] }
│                         expertGroups 含 id/name/version/description/author/category/tags/isActive/metaJson 等全字段

└── plugins/           ← 所有专家团文件目录
    ├── ai-engineer/
    │   ├── .claude-plugin/plugin.json
    │   ├── expert-groups/ai-engineer.json
    │   └── ...
    └── content-creator/
        └── ...
```

#### 接口 14：一键导入

```
POST /workmate/console/expert-groups/import
Content-Type: multipart/form-data

参数：
  file: MultipartFile    (必填)  导出 ZIP 文件

服务端处理：
  1. 解压 ZIP，读取 data.json
     → 不存在 data.json → 返回 400 "ZIP 包缺少 data.json"
  2. 专家团数据：逐条 upsert（insert or update by id）
  3. 精选场景数据：逐条 upsert
  4. 插件文件：覆盖 /app/workmate/plugins/{id}/（先删旧再复制新）
  5. 重新计算所有专家团存储大小并更新 DB

冲突策略：完全以导入数据为准（DB upsert + 文件覆盖），不做合并/保留

响应：
{
  "code": 0,
  "message": "导入完成：专家团 3 个，场景 2 个",
  "data": { "groupCount": 3, "sceneCount": 2 }
}
```

### 4.2 客户端接口（/workmate）

| # | 路径 | 说明 | 实现要点 |
|---|------|------|---------|
| 1 | `GET /workmate/expert-groups` | 全量列表 | 查 `expert_groups WHERE is_active=TRUE` |
| 2 | `GET /workmate/expert-groups/:id` | 详情 | 查单条记录 |
| 3 | `GET /workmate/expert-groups/featured-scenes` | 精选场景 | 查 `featured_scenes` 表 |
| 4 | `GET /workmate/expert-groups/:id/download` | 下载插件 zip | 实时打包 `/app/workmate/plugins/{id}/` 返回 zip 流 |

下载接口实现要点：
- Content-Type: `application/zip`
- Content-Disposition: `attachment; filename="{id}-{version}.zip"`
- 不缓存 zip（每次实时打包），保证内容最新
- 设置 Content-Length（先打包到临时文件获取大小，或使用 chunked 传输）

## 八、查漏补缺清单

### 8.1 安全

| 风险 | 措施 |
|------|------|
| Zip Slip 路径穿越 | 解压时对每个 ZipEntry 做 `canonicalPath.startsWith(targetDir)` 校验，拒绝含 `../` 的路径 |
| 恶意大文件 | 上传接口限制 zip 最大 50MB（`spring.servlet.multipart.max-file-size` 已有 500MB 兜底） |
| JSON 注入/反序列化攻击 | 使用 Jackson 严格解析，禁用 default typing，白名单字段映射 |
| 未授权访问管理接口 | 管理台接口走已有 `AdminWhitelistInterceptor` 鉴权 |
| 文件覆盖竞争条件 | 按 `expertGroupId` 粒度加 JVM 锁，先删后写 |
| 目录名注入 | 限制目录名正则 `[a-zA-Z0-9_-]+`，拒绝特殊字符 |

### 8.2 数据一致性

| 场景 | 处理 |
|------|------|
| DB 写入成功但文件移动失败 | 回滚 DB 事务 |
| 文件移动成功但 DB 写入失败 | 删除已移动的文件目录，回滚 DB |
| 部分专家团失败 | 不做整体回滚，逐个处理，返回详细成功/失败列表 |
| 同一 ID 并发上传 | JVM 锁按 ID 串行化 |
| 删除时 featured_scenes 仍有引用 | 自动从 `expert_group_ids` JSON 数组中移除该 ID |
| 一键导出 → 一键导入往返 | `@JsonRawValue` 导出时 tags/metaJson/expertGroupIds 写入原生 JSON 数组/对象，导入时 `normalizeRawJson` 转回字符串再反序列化 |
| `ATOMIC_MOVE` 跨分区 | 捕获 `AtomicMoveNotSupportedException`，回退到 `REPLACE_EXISTING` |
| `importAll` 事务 | 加 `@Transactional`，DB upsert 失败时回滚 |

### 8.3 功能边界与待确认

| 问题 | 建议 |
|------|------|
| 上传后默认启用 | 默认 `is_active=true`，上传后即对客户端可见 |
| 删除模式 | 物理删除（删文件 + 删 DB），不做软删除。因为重新上传即可恢复 |
| 是否需要版本历史/回滚 | 初版不做。后续可加 `expert_group_versions` 表记录变更历史 |
| 下载 zip 缓存 | 每次实时打包，不缓存。避免文件变更后客户端拿到旧版本 |
| 临时文件清理 | 加定时任务（已有的 `PartitionMaintenanceService` 或新增 `@Scheduled`），清理 24 小时前的 `/app/workmate/upload-tmp/*` |
| 上传 zip 内混合合法/非法目录 | 合法的正常处理，非法的跳过并记录原因，不阻塞其他 |
| 仅有 plugin.json 无 expert-groups | 允许导入，但 DB 中缺少 `mainRole.prompt`/`skills` 等完整定义，客户端下载 zip 后可能无法完整运行 |
| `tags`/`metaJson`/`expertGroupIds` 序列化 | MySQL JSON 列存字符串，Jackson 默认二次转义。Entity 对应字段加 `@JsonRawValue`，序列化时原样输出 |
| `updateMeta` 中 tags 类型 | 前端传 JSON 数组时 Spring 反序列化为 List，需 instanceof 判断并转 JSON 字符串 |

### 8.4 前端管理页面功能点

## 九、总结

| 模块 | 核心要点 |
|------|---------|
| 数据库 | 2 张表：`expert_groups`（元数据）+ `featured_scenes`（场景） |
| 上传 | ZIP → 解压 → 逐目录校验 JSON → 移动文件 → 写入 DB，支持 cover/skip 模式 |
| 管理 CRUD | 14 个接口覆盖上传/列表/详情/编辑/删除/启禁用 + 场景管理 + 一键导出导入 |
| 客户端 API | 4 个接口，与已有文档对齐，从 DB 读取 + 实时打包下载 |
| 安全 | Zip Slip 防护、目录名校验、鉴权复用、并发锁 |
| 一致性 | 先文件后 DB，失败回滚；部分失败不整体回滚；导出导入往返序列化兼容 |
| 序列化 | `@JsonRawValue` 原样输出 MySQL JSON 列，避免二次转义 |
