# 专家团服务端化方案：列表按需获取 + 按需下载 + 召唤

> 编辑时间：2026年6月16日

---

## 一、背景与目标

### 1.1 现状

当前专家团完全内置在安装包中（`apps/electron/bundled-plugins/`），通过本地插件发现机制（扫描 `expert-groups/*.json`）加载。目前仅有 1 个内置专家团（architecture-decision-team），但如果将 39 个专家团全部内置，安装包会急剧膨胀。

### 1.2 目标

1. **专家团列表服务端化**：从后端动态拉取可用的专家团列表（含元数据），前端不再从本地插件扫描中获取列表。
2. **按需下载**：用户"召唤"某个专家团时，先检查本地是否已有该插件；若没有，先下载插件 zip 包并安装到本地 `~/.proma/plugins/`，再执行召唤。
3. **精选场景服务端化**：精选场景分类从后端接口获取，支持动态更新。
4. **保持兼容**：本地已安装的专家团（内置+用户自行导入）继续可用，与服务端列表合并展示。

### 1.3 核心思路

```
┌──────────────────────────────────────────────────────────────┐
│                        服务端 (Proma Backend)                   │
│                                                              │
│  GET /workmate/expert-groups              → 专家团列表（含元数据）     │
│  GET /workmate/expert-groups/:id          → 单个专家团详情           │
│  GET /workmate/expert-groups/featured-scenes → 精选场景分类          │
│  GET /workmate/expert-groups/:id/download → 下载插件包（zip 文件流）  │
└──────────────────────────────────────────────────────────────┘
        │                              │
        ▼                              ▼
┌──────────────────────────────────────────────────────────────┐
│                    客户端 (Electron Renderer)                   │
│                                                              │
│  列表页 ← serverGroups + localGroups → 合并 → 筛选/搜索/展示     │
│  召唤按钮 → 检查本地是否存在 → 不存在则下载 → 安装 → 创建会话      │
│  精选场景 ← 服务端接口动态获取                                  │
└──────────────────────────────────────────────────────────────┘
```

## 二、服务端专家团列表字段定义

### 2.1 列表项类型：`ServerExpertGroupSummary`

服务端返回的列表需要足够的前端展示和筛选信息，但**不包含**完整的 `mainRole.prompt`、SubAgent 定义等大段内容（这些只在下载后的 `expert-groups/{id}.json` 中才需要）。

```typescript
/** 服务端专家团列表项 —— 前端筛选/展示用的摘要信息 */
export interface ServerExpertGroupSummary {
  /** 专家团唯一标识（对应插件 expertGroup/expertGroups 字段） */
  id: string

  /** 显示名称 */
  name: string

  /** 简短描述（卡片展示） */
  description: string

  /** 唤起后展示给用户的自我介绍文本 */
  introduction: string

  /** 主角色名称（卡片/搜索用，不含 prompt） */
  mainRoleName: string

  /** 专家类型：agent=单专家，team=多专家团队 */
  expertType: 'agent' | 'team'

  /** SubAgent 个数（前端展示"X 位专家"用，不含具体定义） */
  subagentCount: number

  /** SubAgent 调用名 → 中文显示名映射（详情弹窗展示用） */
  subagentLabels: Record<string, string>

  /** 标签列表（精选场景匹配 + 筛选标签匹配） */
  tags: string[]

  /** 示例问题（详情弹窗展示用） */
  samplePrompts: string[]

  /** 依赖的内置工具（如 web-search） */
  builtinTools: string[]

  /** 依赖的 Skill 名称列表（用于"不可用"状态判定） */
  skills: string[]

  /** 依赖的 MCP Server 名称列表 */
  mcpServers: string[]

  /** 插件版本号（用于版本更新检查） */
  version: string

  /** 下载 URL（相对路径或完整 URL，前端拼接 baseUrl） */
  downloadUrl: string

  /** 下载包大小（字节，前端展示"XX MB"） */
  downloadSize: number

  /** 服务端排序权重（数字越大越靠前，用于"精品推荐"排序） */
  sortWeight: number

  /** 上架时间 ISO 字符串（用于"最新上架"排序） */
  publishedAt: string

  /** 更新时间 ISO 字符串 */
  updatedAt: string
}
```

## 三、服务端精选场景接口

### 3.1 现有实现（硬编码）

当前精选场景在 `ExpertFeaturedScenes.tsx` 中硬编码了 8 个分类及模糊匹配标签：

```typescript
// 现状 — 硬编码在客户端，客户端用 filterTags 做模糊匹配
const SCENE_CATEGORIES = [
  { id: 'content', name: '内容创作', filterTags: ['写作', '文案', '翻译'] },
  { id: 'office', name: '办公场景', filterTags: ['文档', '邮件', '日程'] },
  // ...
]
```

**问题**：
1. 新增场景需要发版更新客户端
2. 模糊匹配不精确 — "数据分析"可能匹配到不相关的专家团
3. 场景与专家团的归属关系应由服务端统一管理，而非前端猜测

### 3.2 接口定义

**改为由服务端直接指定每个场景下有哪些专家团**，前端不再做 `filterByScene` 模糊匹配：

```typescript
/** 精选场景分类 */
export interface FeaturedScene {
  /** 场景唯一标识 */
  id: string

  /** 场景显示名称 */
  name: string

  /** 该场景下的专家团 id 列表 */
  expertGroupIds: string[]

  /** 场景图标 URL（可选，前端可兜底用默认图标） */
  iconUrl?: string

  /** 排序权重（数字越大越靠前） */
  sortOrder: number
}

/** GET /workmate/expert-groups/featured-scenes 响应 */
export interface FeaturedScenesResponse {
  scenes: FeaturedScene[]
}
```

### 3.3 前端筛选逻辑变更

**旧逻辑**（客户端模糊匹配 — 删除）：

```typescript
// filter.ts — filterByScene / countByScene 不再需要
export function filterByScene(groups, sceneTags) {
  // 用 tags 模糊匹配，兜底搜索 name/description/mainRole.name
}
```

**新逻辑**（直接 ID 筛选）：

```typescript
// ExpertPageView.tsx — 场景点击处理
const handleSceneClick = (sceneId: string | null, expertGroupIds: string[] | null) => {
  if (!expertGroupIds) {
    setSceneFilter(null)  // 清除场景筛选
  } else {
    setSceneFilter(new Set(expertGroupIds))  // 用 Set 做 O(1) 查重
    setActiveSceneId(sceneId)
  }
}

// 数据管道中直接用 Set.has 筛选
const displayGroups = React.useMemo(() => {
  let result = allGroups
  result = filterByTag(result, filterTag, followed, recent)
  if (sceneFilter) {
    result = result.filter(g => sceneFilter.has(g.id))
  }
  result = searchByName(result, query)
  return result
}, [allGroups, filterTag, query, followed, recent, sceneFilter])
```

### 3.4 场景计数

场景卡片上的数字（如"3 位专家"）应取 `expertGroupIds` 与 `allGroups` 的交集数，避免服务端下架某个专家团后计数偏大：

```typescript
// ExpertFeaturedScenes.tsx
{SCENE_CATEGORIES.map((scene) => {
  const count = allGroupIds.size > 0
    ? scene.expertGroupIds.filter(id => allGroupIds.has(id)).length
    : scene.expertGroupIds.length  // 初始加载时服务端数据为准
  // ...渲染卡片
})}
```

### 3.5 前端组件适配

`ExpertFeaturedScenes` 回调签名变更：

```typescript
// 旧
onSceneClick: (sceneId: string | null, tags: string[] | null) => void

// 新
onSceneClick: (sceneId: string | null, expertGroupIds: string[] | null) => void
```

## 四、服务端筛选字段对照

当前前端支持的筛选能力需要服务端列表字段一一支撑：

| 前端筛选 | 依赖字段 | 说明 |
|---|---|---|
| 搜索（名称/描述/标签/角色） | `name`、`description`、`tags`、`mainRoleName`、`subagentLabels` | 全在前端做，服务端暂不需要搜索接口 |
| 收藏 | 前端 localStorage | 不需要服务端支持 |
| 最近使用 | 前端 localStorage | 不需要服务端支持 |
| 专家/专家团 | `expertType` | `'agent'` / `'team'` |
| 可用/不可用 | `skills`、`mcpServers` | 前端根据本地已安装的 skills/默认 skills 判定 |
| 精选场景 | `id` ↔ `expertGroupIds` | 前端用 `Set.has(g.id)` 直接筛选，不再模糊匹配 |
| 智能排序 | `sortWeight`、`publishedAt`、`downloadSize` | 前端组合排序 |
| 名称排序 | `name` | 前端本地排序 |

> **说明**：搜索和筛选都在客户端执行（与现有逻辑一致），服务端仅负责返回全量列表。若未来专家团数量很大（>200），可后续增加服务端分页/搜索参数。

## 五、按需下载 + 召唤流程

### 5.1 状态扩展

当前 `AgentExpertGroupStatus` 的枚举值：

```typescript
type AgentExpertGroupStatus =
  | 'available'
  | 'plugin_disabled'
  | 'plugin_uninstalled'
  | 'invalid_manifest'
  | 'missing_subagent'
  | 'missing_skill'
  | 'mcp_conflict'
```

**新增服务端专家团专属状态**：

```typescript
type AgentExpertGroupStatus =
  // ... 现有状态 ...
  | 'remote_not_downloaded'   // 服务端专家团，尚未下载到本地
  | 'remote_downloading'      // 正在下载中
  | 'remote_download_failed'  // 下载失败
  | 'remote_update_available' // 服务端有更新版本
```

### 5.2 召唤流程

```
用户点击"召唤"
    │
    ├─ 1. 检查 sourcePluginKind
    │
    ├── sourcePluginKind === 'builtin' || 'user'
    │   → 直接走现有召唤流程（专家团已存在于本地）
    │
    └── sourcePluginKind === 'remote'（新增）
         │
         ├─ 2. 检查本地是否已安装该 expertGroupId
         │     ├─ 已安装 → 走现有召唤流程
         │     └─ 未安装 ↓
         │
         ├─ 3. 发起下载
         │     ├─ 状态更新为 'remote_downloading'
         │     ├─ 显示下载进度（底部进度条或卡片内进度）
         │     ├─ 下载成功后调用 installUserPluginZip()
         │     └─ 下载失败 → 'remote_download_failed' + toast error
         │
         ├─ 4. 重新加载专家团列表（listAgentExpertGroups 会返回
         │    新安装的插件中的 expert-group）
         │
         └─ 5. 执行召唤（createExpertSession）
```

### 5.4 下载流程（主进程）

```typescript
/**
 * 下载远程专家团插件包并安装到本地
 *
 * 流程：
 * 1. 通过 hteip-client 拼接完整下载 URL（自动注入 EIPGW-TOKEN Cookie）
 * 2. 流式下载到临时目录（fetch + ReadableStream）
 * 3. 调用 installUserPluginZip() 安装到 user-plugins/remote/
 * 4. 清理临时文件，返回安装后的 AgentPluginInfo
 */
async function downloadAndInstallRemoteExpert(
  groupId: string,
): Promise<AgentPluginInfo> {
  const downloadPath = `/workmate/expert-groups/${groupId}/download`
  const downloadUrl = `${resolveApiBase()}${downloadPath}`  // hteip-client 域名拼接
  const tempPath = `${app.getPath('temp')}\\proma-expert-${groupId}-${Date.now()}.zip`

  // 1. 流式下载（fetch + Cookie 注入，与 updater 下载逻辑一致）
  await downloadFile(downloadUrl, tempPath, (downloaded, total) => {
    broadcastProgress({ groupId, status: 'downloading', progress, ... })
  })

  // 2. 安装插件（marketplaceId: 'remote' 安装到 user-plugins/remote/）
  const plugin = installUserPluginZip(tempPath, { marketplaceId: 'remote' })

  // 3. 清理临时文件
  try { unlinkSync(tempPath) } catch { /* ignore */ }

  return plugin
}
```

**HTTP 客户端选型**：查询接口走 `hteip-client` 的 `httpGet()`（自动注入 Cookie、超时控制、错误归一化），下载走 `fetch()` + `getToken()` 手动注入 Cookie（与 `workmate-downloader.ts` 升级包下载逻辑一致）。不再使用原生 `node:http`。

#### 5.4.1 落盘路径

下载的插件包安装到 `user-plugins/remote/` 目录下，与用户自行上传的 `user-plugins/local/` 隔离：

```
{configDir}/user-plugins/
├── local/                         # 用户自行上传的 zip 插件
│   └── {pluginName}/
│       ├── .claude-plugin/plugin.json
│       ├── expert-groups/{id}.json
│       ├── agents/
│       └── skills/
│
└── remote/                        # 从服务端下载的专家团插件 ← 新增
    └── {pluginName}/
        ├── .claude-plugin/plugin.json
        ├── expert-groups/{id}.json
        ├── agents/
        └── skills/
```

对应的 `pluginId` 格式：`user:remote/{pluginName}`，与现有 `user:{marketplaceId}/{pluginName}` 格式一致。

为此需要改造 `downloadAndInstallRemoteExpert`，在调用 `installUserPluginZip` 时传入 `marketplaceId: 'remote'`，将现有安装逻辑中的硬编码 `marketplaceId = 'local'` 改为可覆盖参数。

**`installUserPluginZip` 签名扩展**：

```typescript
// plugin-registry-service.ts
interface InstallUserPluginZipOptions extends PluginRegistryPaths {
  overwrite?: boolean
  tempRoot?: string
  /** 插件市场 ID，默认 'local'；远程下载的专家团传入 'remote' */
  marketplaceId?: string
}
```

## 六、本地专家团 + 服务端专家团合并

### 6.1 合并策略

```
                  ┌─────────────────────┐
                  │ agentExpertGroupsAtom │
                  │   (合并后的全量列表)    │
                  └──────────┬──────────┘
                             │
              ┌──────────────┴──────────────┐
              │                             │
    ┌─────────┴─────────┐       ┌──────────┴──────────┐
    │ localExpertGroups │       │ serverExpertGroups  │
    │ (listAgentExpert  │       │ (fetchServerExpert  │
    │  Groups IPC)      │       │  GroupsList IPC)    │
    │ sourcePluginKind: │       │ sourcePluginKind:   │
    │  'builtin'|'user' │       │  'remote'           │
    └───────────────────┘       └─────────────────────┘
```

合并规则：
1. 以 `id` 为唯一键合并
2. **本地优先**：如果 `localExpertGroups` 中已存在同 `id` 的专家团（用户已下载安装），使用本地版本（拥有完整的 `mainRole.prompt`、`subagents` 等），仅从服务端数据补充 `sortWeight`、`publishedAt` 等元数据
3. **服务端补充**：仅存在于服务端列表中的条目，作为 `sourcePluginKind: 'remote'` 展示，`sourcePluginPath`、`filePath` 等为空
4. **本地独有**：用户自行导入或内置的专家团若服务端列表中不存在，仍保留展示

#### 6.1.1 关于合并键 `id`

本地专家团和服务端专家团**都有 `id` 字段**，来源一致：

| 来源 | `id` 来源 | 示例 |
|---|---|---|
| 本地 `listAgentExpertGroups()` | `expert-groups/{id}.json` 中的 `id` 字段，缺失时回退为文件名 | `architecture-decision-team` |
| 服务端 `ServerExpertGroupSummary` | 同上 — 服务端打包 zip 时 `expert-groups/{id}.json` 中的 `id` | `architecture-decision-team` |

所以当用户从服务端下载 `architecture-decision-team` 后，本地 `listAgentExpertGroups()` 会扫描 `user-plugins/remote/architecture-decision-team/expert-groups/` 得到相同的 `id`，合并时自动匹配。

**为什么不用 `name` 合并**：

| 字段 | 性质 | 示例 |
|---|---|---|
| `id` | 内部唯一标识，不可变 | `architecture-decision-team` |
| `name` | 展示名，可修改/可能重复 | `架构决策专家团` |

`name` 来自插件的显示名，两个不同的插件可能有相同的显示名，且服务端修改展示名后会导致匹配失败。`id` 是稳定键。

```typescript
function mergeExpertGroups(
  localGroups: AgentExpertGroupInfo[],
  serverSummaries: ServerExpertGroupSummary[],
): AgentExpertGroupInfo[] {
  const result: AgentExpertGroupInfo[] = []
  const localIds = new Set(localGroups.map((g) => g.id))

  // 1. 本地已有的：保留本地完整信息，补充服务端元数据
  for (const local of localGroups) {
    const server = serverSummaries.find((s) => s.id === local.id)
    result.push({
      ...local,
      // 如果服务端版本更新，标记可更新
      status: server && server.version !== local.sourcePluginVersion
        ? 'remote_update_available'
        : local.status,
    })
  }

  // 2. 仅服务端有的：构建 Remote 条目
  for (const server of serverSummaries) {
    if (localIds.has(server.id)) continue
    result.push({
      id: server.id,
      name: server.name,
      description: server.description,
      introduction: server.introduction,
      mainRole: { name: server.mainRoleName, prompt: '' },
      subagents: undefined,
      subagentLabels: server.subagentLabels,
      builtinTools: server.builtinTools,
      skills: server.skills,
      mcpServers: server.mcpServers,
      tags: server.tags,
      samplePrompts: server.samplePrompts,
      expertType: server.expertType,
      sourcePluginId: '',  // 下载后动态赋值
      sourceLabel: server.name,
      sourcePluginVersion: server.version,
      sourcePluginKind: 'remote',
      sourcePluginPath: '',  // 下载后动态赋值
      filePath: '',
      enabled: true,
      status: 'remote_not_downloaded',
      issues: [],
    })
  }

  return result
}
```

## 七、缓存策略

### 7.1 服务端列表缓存

| 位置 | 存储内容 | TTL | 说明 |
|---|---|---|---|
| 主进程内存 | `serverExpertGroupsCache` | 进程生命周期 | 避免重复请求 |
| 磁盘 | `~/.proma/expert-groups-cache.json` | 30 分钟 | 离线可用（不刷新时展示上次缓存） |
| 渲染进程 | `serverExpertGroupsAtom` | 页面生命周期 | 响应式 UI 绑定 |

**刷新触发**：
- 应用启动时（后台静默刷新）
- 用户进入专家团页面时（若缓存过期）
- 用户下拉刷新（手动触发）
- 安装/卸载插件后

### 7.2 精选场景缓存

| 位置 | 存储内容 | TTL | 说明 |
|---|---|---|---|
| 磁盘 | `~/.proma/featured-scenes-cache.json` | 1 小时 | 精选场景变化频率低 |

### 7.3 API 加载失败降级策略

| 场景 | 行为 |
|---|---|
| 首次启动，无缓存，API 不可达 | 专家团列表为空（仅展示已安装本地插件），精选场景不展示，页面顶部展示"网络不可用，仅显示已安装的专家团" |
| 有磁盘缓存，API 不可达 | 使用缓存数据，toast 提示"无法获取最新数据，显示的是上次缓存" |
| API 超时（>10s） | 同上，使用缓存降级，后台静默重试一次 |
| API 返回非 200 | 同上，使用缓存降级 |
| API 正常返回 | 覆盖缓存，清除重试标记 |

### 10.4 下载专家团插件包

```
GET /workmate/expert-groups/:id/download
```

**请求头**：
- `Accept: application/zip`

**响应**：
- `Content-Type: application/zip`
- `Content-Disposition: attachment; filename="{id}-{version}.zip"`
- `Content-Length: {downloadSize}`
- 响应体：插件 zip 文件流

**插件包内部结构（服务端打包规范）**：

```
{expertGroupId}/
├── .claude-plugin/
│   └── plugin.json            # 插件清单（含 expertGroup 字段）
├── expert-groups/
│   └── {expertGroupId}.json   # 专家团配置清单（含 mainRole.prompt）
├── agents/                    # SubAgent 定义（如有）
│   └── {agentName}.md
├── skills/                    # Skills（如有）
│   └── {skillName}/
│       └── SKILL.md
└── .mcp.json                  # MCP 配置（如有）
```

**说明**：服务端打包的 zip 结构与 `installUserPluginZip()` 期望的格式一致，可直接复用现有安装逻辑。

### 11.2 额外注意事项

| 关注点 | 说明 |
|---|---|
| **版本更新检测** | `mergeExpertGroups()` 中已处理，本地版本 < 服务端版本时标记 `remote_update_available`；用户可选择"更新"重新下载 |
| **下载失败重试** | 标记 `remote_download_failed` 后，用户可点击"重新获取"再次下载；不做自动重试避免消耗流量 |
| **断点续传** | 初版不做，下载失败的 zip 直接删除重新下载；大文件场景后续可加 |
| **并发下载** | 允许同时下载多个专家团；每个独立跟踪进度 |
| **离线场景** | 无网络时使用磁盘缓存的服务端列表；`remote_not_downloaded` 的专家团不可召唤，但本地已安装的可正常使用 |
| **下载中应用退出** | 退出时取消所有进行中的下载，未完成的临时文件在下次启动时清理 |
| **Skills 可用性判定** | `remote_not_downloaded` 的专家团，其 `skills` 字段已从服务端返回，前端可以预先判定"该专家团需要技能 A，但本地未安装"——可在卡片上展示提示，但无需阻止下载（下载时会将 skills 一并安装） |
| **MCP 依赖** | 同上，下载的插件包内含 `.mcp.json`，安装后由现有 MCP 管理流程处理 |
| **与插件市场的关系** | 服务端专家团本质上是一个"官方应用市场"的简化版；可与现有 `plugin-marketplace-service` 复用部分下载/安装逻辑，但保持独立的频道和缓存 |
| **backward compat** | 现有 `listAgentExpertGroups()` 不变，新增 `fetchServerExpertGroups()` IPC；`agentExpertGroupsAtom` 从单一数据源变为合并后的数据源 |
