# 专家团下载进度条与可取消方案

编辑时间：2026年6月22日

***

## 一、背景与问题

Agent 技能「专家」tab 页中，从服务端下载专家团插件包时存在两类体验问题：

1. **整个进程卡死**：专家包较大时，下载完成后的安装阶段会冻结整个应用 UI（无法点击、无法切换）。
2. **无进度反馈、无法取消**：下载耗时较长或专家包较大时，用户看不到任何进度，也无法中途取消。

### 三个根因

| 根因 | 位置 | 说明 |
| --- | --- | --- |
| **同步解压/拷贝阻塞主进程** | [`installUserPluginZip`](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/plugin-registry-service.ts#L746) → [`extractPluginZipSafely`](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/plugin-registry-service.ts#L659) | 逐条 `entry.getData()` + `writeFileSync` 同步写盘，再 `cpSync` 递归拷贝。大包长时间占用事件循环 → 应用冻结。**这是真正卡死的元凶**，下载本身是 `fetch` 流式非阻塞。 |
| **渲染端未消费进度** | 全渲染层 | 主进程已通过 `DOWNLOAD_PROGRESS` 广播 [`RemoteDownloadProgress`](file:///d:/AiWorkbench-workmate/packages/shared/src/types/agent.ts)，但**没有任何组件订阅**，用户看不到进度。 |
| **取消为空实现** | [`CANCEL_DOWNLOAD` handler](file:///d:/AiWorkbench-workmate/apps/electron/src/main/ipc.ts#L2511) | 仅打印日志，不真正中断。 |

### 现有可复用基建

- 进度类型 `RemoteDownloadProgress`、IPC 通道 `DOWNLOAD_REMOTE_EXPERT` / `DOWNLOAD_PROGRESS` / `CANCEL_DOWNLOAD` / `ENSURE_LATEST` 已存在。
- preload 已暴露 `downloadRemoteExpert` / `cancelRemoteDownload` / `onExpertDownloadProgress` / `ensureExpertGroupLatest`。
- 状态枚举 `remote_downloading` / `remote_download_failed` 已存在于 `AgentExpertGroupStatus`。
- 统一召唤入口 [`useSummonExpert`](file:///d:/AiWorkbench-workmate/apps/electron/src/renderer/experts/hooks/useSummonExpert.ts)、卡片 [`ExpertCard`](file:///d:/AiWorkbench-workmate/apps/electron/src/renderer/experts/card/ExpertCard.tsx) 已存在。

## 二、目标与非目标

### 目标

1. 安装/解压阶段**不再冻结主进程**，应用全程可交互。
2. 卡片**内联**展示下载进度（百分比 + 进度条），三个触发场景复用同一套 UI。
3. 下载 + 安装**全程可取消**，取消后清理临时文件、回退状态。
4. 下载完成后**自动打开会话的现有行为保持不变**。
5. 召唤时若检测到需要下载（未下载 / 版本升级），同样走进度条与可取消。

### 非目标

- 断点续传。
- 多专家并行下载的全局下载管理器。
- worker_threads / child_process 解压（见方案选型，本期用异步分片解压即可）。

## 三、设计选型

「不卡死」的关键是让解压/拷贝不阻塞事件循环。三种方案对比：

| 方案 | 做法 | 取舍 |
| --- | --- | --- |
| **A 异步分片解压（采用）** | `extractPluginZipSafely` 改 `async`，逐条 `fs.promises.writeFile` 并在条间 `await` 让出事件循环；`cpSync` 换 `fs.promises.cp`；条间检查 abort 标志 | 改动小、零打包风险、天然支持安装期取消。专家包多为大量小文件，逐条让出足以保持 UI 响应 |
| B worker_threads | 解压+拷贝整体丢子线程 | 即使超大文件也不阻塞，但需单独打包 worker、桥接 cancel/progress 复杂，对当前包体属过度工程 |
| C child_process unzip | 调系统 unzip | 完全不阻塞，但 Windows 无内置 unzip、丢失路径穿越安全校验，否决 |

**结论：采用方案 A。**

## 四、整体架构

```
主进程                          IPC                      渲染进程
─────────────────────────────────────────────────────────────────────
expert-download-service
  ├ AbortController 注册表    ◄──CANCEL_DOWNLOAD────  ExpertCard 取消按钮
  │  (per groupId)
  ├ fetch(signal) 流式下载    ───DOWNLOAD_PROGRESS──► 全局订阅 hook
  └ installUserPluginZipAsync                          │ 写入
     (异步分片解压 + await cp,                          ▼
      条间检查 abort)                            expertDownloadProgressAtom
                                                  Map<groupId, Progress>
                                                       │ 派生切片
                                                       ▼
                                                 ExpertCard 内联进度条
```

**三个触发场景统一走这套进度 + 取消：**

1. 专家 tab 页点「下载」→ `downloadRemoteExpert`
2. 召唤未下载专家 → `useSummonExpert` 分支一（下载后自动开会话，行为不变）
3. 召唤已下载专家的版本检测升级 → `ensureExpertGroupLatest` 内部复用 `downloadAndInstallRemoteExpert`，进度自动广播

## 五、主进程：去阻塞 + 真正可取消

### 5.1 异步分片解压（消除卡死）

在 [`plugin-registry-service.ts`](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/plugin-registry-service.ts) 中，`extractPluginZipAsync` 和 `installUserPluginZipAsync` 已落地，但内部仍调用同步 `copyPluginAtomically` 和 `assertNoDuplicateExpertGroups`（见 §5.4 / §5.5）。本节描述其异步解压核心逻辑（已实现，无需改动），不改动现有同步 `installUserPluginZip`（仍供本地 zip 上传等场景使用）：

```ts
interface InstallAsyncOptions extends InstallUserPluginZipOptions {
  signal?: AbortSignal               // 安装期可取消
}

// extractPluginZipSafely 的异步版：逐条 await writeFile，条间 yield + 检查 abort
async function extractPluginZipAsync(zipPath, extractDir, signal?): Promise<void> {
  const zip = new AdmZip(zipPath)
  for (const entry of zip.getEntries()) {
    if (signal?.aborted) throw new DownloadCancelledError()
    // ...沿用现有路径穿越安全校验...
    if (entry.isDirectory) { await fsp.mkdir(targetPath, { recursive: true }); continue }
    await fsp.mkdir(dirname(targetPath), { recursive: true })
    await fsp.writeFile(targetPath, entry.getData())   // 条间让出事件循环
  }
}

export async function installUserPluginZipAsync(zipPath, options): Promise<AgentPluginInfo> {
  // 结构同步版完全一致，仅把 extract 换异步、cpSync → await fsp.cp、关键节点检查 signal
}
```

> `entry.getData()` 单条仍是同步 CPU，但专家包是大量小文件（md/json），逐条 `await` 已足够让 UI 保持响应——符合方案 A 的判断。

### 5.2 取消注册表 + 贯穿下载与安装

[`expert-download-service.ts`](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/expert-download-service.ts) 改造：

```ts
// per-groupId 的 AbortController 注册表
const activeDownloads = new Map<string, AbortController>()

export function cancelRemoteDownload(groupId: string): void {
  activeDownloads.get(groupId)?.abort()
}

export async function downloadAndInstallRemoteExpert(groupId, options) {
  const controller = new AbortController()
  activeDownloads.set(groupId, controller)
  try {
    // 下载：fetch 的 signal 用 controller.signal（连接超时仍保留，二者合并）
    await downloadFile(url, tempPath, onProgress, controller.signal)
    broadcastProgress({ status: 'installing', ... })
    // 安装：传 signal，安装期也能取消
    const plugin = await installUserPluginZipAsync(tempPath, { ..., signal: controller.signal })
    broadcastProgress({ status: 'done', ... })
    return plugin
  } catch (err) {
    if (controller.signal.aborted) {
      broadcastProgress({ status: 'cancelled', ... })   // 新增状态
      throw new DownloadCancelledError()
    }
    broadcastProgress({ status: 'error', error, ... })
    throw err
  } finally {
    try { unlinkSync(tempPath) } catch {}          // 取消/失败都清理临时 zip
    activeDownloads.delete(groupId)
  }
}
```

- `downloadFile` 增加 `signal` 参，与现有 15s 连接超时的 controller 合并（`AbortSignal.any` 或共用同一 controller）。
- 取消时已解压的半成品目录：`installUserPluginZipAsync` 内部捕获 abort 后清理临时 `extractDir` 与 `.installing-*` 目录（复用现有 [`copyPluginAtomically`](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/plugin-registry-service.ts#L700) 的失败清理逻辑）。

### 5.3 IPC + 类型

### 5.4 异步原子拷贝（消除 cpSync 阻塞）

虽然 §5.1 的 `extractPluginZipAsync` 已消除解压阻塞，但 [`copyPluginAtomically`](file:///d:/AiWorkbench-workmate/apps/electron/src/main/lib/plugin-registry-service.ts#L805) 仍使用同步 `cpSync` + `rmSync` + `renameSync`，大包拷贝时仍会冻结主进程。新增异步版：

```ts
async function copyPluginAtomicallyAsync(
  sourceDir: string, targetDir: string, overwrite: boolean, signal?: AbortSignal
): Promise<'installed' | 'overwritten'> {
  if (signal?.aborted) throw new DownloadCancelledError()
  const existed = existsSync(targetDir)
  if (existed && !overwrite) throw new Error(`插件已存在: ${basename(targetDir)}`)

  const parent = dirname(targetDir)
  const tmp = join(parent, `.${basename(targetDir)}.installing-${Date.now()}`)
  await fsp.mkdir(parent, { recursive: true })
  await fsp.rm(tmp, { recursive: true, force: true })
  await fsp.cp(sourceDir, tmp, { recursive: true })

  if (signal?.aborted) {
    await fsp.rm(tmp, { recursive: true, force: true })
    throw new DownloadCancelledError()
  }

  try {
    if (existed) await fsp.rm(targetDir, { recursive: true, force: true })
    await fsp.rename(tmp, targetDir)
  } catch (error) {
    await fsp.rm(tmp, { recursive: true, force: true })
    if (!existed) await fsp.rm(targetDir, { recursive: true, force: true })
    throw error
  }

  return existed ? 'overwritten' : 'installed'
}
```

`installUserPluginZipAsync` 内部把 `copyPluginAtomically(...)` 替换为 `await copyPluginAtomicallyAsync(..., options.signal)`。

### 6.3 卡片内联 UI（方案 A：底部状态区）

[`ExpertCard`](file:///d:/AiWorkbench-workmate/apps/electron/src/renderer/experts/card/ExpertCard.tsx) 订阅 `expertDownloadProgressFamily(group.id)`，把进度从右上角 `absolute`（与收藏星争位、视觉层级弱）移到**卡片底部独立状态区**（`border-top` 分隔），与召唤/下载按钮共用同一块底部空间，切换自然：

```
┌──────────────────────────────────┐
│ ★                                │
│ 🤖 产品专家团                      │
│ 主角色：产品经理                    │
│ 覆盖需求分析、PRD 撰写…            │
│ [产品] [需求] [PRD]               │
├──────────────────────────────────┤  ← border-top 分隔
│ ● 正在下载              63%   ✕   │  ← 状态行：状态点 + 文案 + 百分比 + 取消
│ ▓▓▓▓▓▓▓▓▓▓▓░░░░░░░░░░░░░░░░░    │  ← 确定态进度条（按 progress 填充）
└──────────────────────────────────┘
```

**完整状态机：**

| 状态 | 底部区域内容 |
| --- | --- |
| 无下载（默认） | 召唤/下载按钮（hover 显示，现有行为不变） |
| `downloading` | 蓝色状态点 +「正在下载」+ 百分比（`tabular-nums` 防抖动）+ **确定态进度条**（按 `progress` 填充）+ 取消 ✕ |
| `installing` | 紫色状态点 +「正在安装…」+ **不确定态流动条纹**进度条（不再假装静态 100%）+ 取消 ✕ |
| `error` | 红色状态点 +「下载失败」+「重试」按钮（内联，1.5s 后由 bridge 清理回退默认态） |
| `cancelled` | 灰色状态点 +「已取消」+「下载」按钮（内联回退，1.5s 后由 bridge 清理） |

**细节说明：**

- 安装态的不确定进度条用 CSS 动画实现流动条纹（`linear-gradient` + `translateX`），替代原先的静态 100% 误导。
- 进度条仍沿用项目内手写样式（ui/ 下无 Progress 组件），宽度不写死，自适应底部区域。
- 取消按钮用 `✕` 图标，hover 时背景加深，与状态行在同一行右侧。
- `error` / `cancelled` 为终态，由 bridge 的 `setTimeout(removeEntry, 1500)` 自动清理；卡片只需在 `downloadProgress` 存在时渲染对应终态 UI。

## 七、错误处理与边界

| 场景 | 行为 |
| --- | --- |
| 下载期取消 | `fetch` abort → 清理临时 zip → 广播 `cancelled` → 卡片回退按钮 |
| 安装期取消 | 解压循环检测 abort 抛出 → 清理 `extractDir`/`.installing-*` → 广播 `cancelled` |
| 网络失败 / HTTP 非 200 | 广播 `error` → 卡片短暂提示后回退；召唤场景降级本地版（升级路径）或 toast 失败（未下载路径） |
| 召唤未下载时取消 | `downloadRemoteExpert` reject（`DownloadCancelledError`）→ `useSummonExpert` catch 中区分取消与失败：取消不弹 error toast，不开会话 |
| 切 tab 后回来 | 进度 Map 为全局 atom，下载在主进程后台继续，回来照常显示 |
| 同一专家重复点下载 | 主进程 `activeDownloads` 已有该 groupId 时直接复用/忽略，避免并发写 |

## 八、测试要点

- **主进程单测**：`extractPluginZipAsync` 正确解压 + 路径穿越拦截不变；`signal.aborted` 时抛 `DownloadCancelledError` 且清理临时目录；`cancelRemoteDownload` 能 abort 进行中的下载。
- **去阻塞验证**：下载安装大包期间主进程事件循环仍可响应（可用大体积 zip fixture 验证 install 期间定时器仍按时触发）。
- **召唤行为回归**：未下载专家召唤 → 下载完成后自动开会话；版本升级召唤 → 进度展示 + 失败降级本地版不阻断。
