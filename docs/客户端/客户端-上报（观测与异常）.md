## 一、设计思路

客户端内聚一个 `ObservabilityService`，收集事件（用户行为 + 异常），定时批量 POST 到服务端 `POST /workmate/observability/events`，并集成全局异常捕获。

### 架构图

```
┌─────────────────────────────────────────────────────────┐
│                 ObservabilityService                     │
├─────────────────────────────────────────────────────────┤
│  面包屑 (breadcrumbs: Breadcrumb[], 最近 20 条)         │
│       │  addBreadcrumb() 入队                            │
│       ▼  reportErrorEvent() 自动附加                     │
│                                                          │
│  内存队列 (eventQueue: ObservabilityEventItem[])        │
│       │  每事件携带 eventId (UUID) 用于去重              │
│       ▼                                                  │
│  ┌──────────────┐   失败    ┌──────────────────────┐    │
│  │ 定时冲刷      │ ────────→ │ 本地磁盘缓存          │    │
│  │ (5s 间隔)     │           │ ~/.proma/            │    │
│  │ 最多 50 条/批 │ ←──────── │ obs-queue.jsonl      │    │
│  └──────┬───────┘   恢复     └──────────────────────┘    │
│         │ 成功                                            │
│         ▼                                                 │
│  ┌──────────────────────────────────────────────────┐   │
│  │ POST {serverUrl}/workmate/observability/events   │   │
│  │ Headers:                                          │   │
│  │   Content-Type: application/json                  │   │
│  │   Cookie: EIPGW-TOKEN=<jwt>                      │   │
│  │ (EIP 网关自动注入 X-EIPGW-USERID)                  │   │
│  │ Body: { events: [...] }                           │   │
│  └──────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

> 采样/限流层已从架构图中移除（流量低，不需要）。`ObservabilityConfig.sampleRate` / `maxEventsPerMinute` 字段保留。

## 三、数据模型

详见 `apps/electron/src/types/workmate.ts`：

```typescript
type ObservabilityEventType =
  | 'user_login' | 'user_logout' | 'chat_question' | 'agent_question'
  | 'error' | 'upgrade_check'

interface ObservabilityEventItem {
  eventId: string             // UUID 去重
  type: ObservabilityEventType
  userId: string              // 用户工号（服务端以 req.jobId 覆盖）
  timestamp: number           // Date.now()
  question?: string
  questionLength?: number
  modelId?: string
  channelId?: string
  sessionId?: string
  workspaceId?: string
  result?: 'success' | 'failure' | 'pending'
  responseDurationMs?: number
  error?: { type: string; message: string; stack?: string; fingerprint?: string }
  breadcrumbs?: Breadcrumb[]  // error 事件自动附加
  tags?: Record<string, string>
  client: { appVersion: string; platform: string; osVersion: string }
}

interface Breadcrumb {
  type: 'navigation' | 'user_action' | 'system' | 'http'
  category: string
  message: string
  timestamp: number
  data?: Record<string, string | number | boolean | null>
}

interface ObservabilityConfig {
  enabled: boolean
  url: string           // 完整上报地址
  timeoutMs?: number
  maxQueueSize?: number
  flushIntervalMs?: number
  maxBatchSize?: number
  sampleRate?: number             // 暂不启用
  maxEventsPerMinute?: number     // 暂不启用
  enableBreadcrumbs?: boolean
  maxBreadcrumbs?: number
  maxEventBytes?: number
}
```

## 四、观测上报服务

### 4.1 初始化与配置

**文件**: `workmate-init.ts`

```typescript
// 由 main/index.ts bootstrap 阶段调用
export function initWorkmateServices(): void {
  registerGlobalErrorHandlers()

  const serverUrl = resolveApiBase()  // → hteip-client → eipGatewayBase
  const config: ObservabilityConfig = {
    enabled: true,
    url: `${serverUrl}/workmate/observability/events`,
    timeoutMs: 5000,
    maxQueueSize: 200,
    flushIntervalMs: 5000,
    maxBatchSize: 50,
    enableBreadcrumbs: true,
    maxBreadcrumbs: 20,
    maxEventBytes: 256 * 1024,
  }
  initObservability(config)
}
```

域名由 `hteip-client.ts` 的 `resolveApiBase()` 统一解析（→ `settings.eipGatewayBase` → 默认 `https://eip.workmate.com`），与其他 WorkMate API 保持一致。

---

### 4.2 核心服务 API

**文件**: `observability-service.ts`

| 函数 | 说明 |
|------|------|
| `init(config)` | 初始化：恢复磁盘缓存 + 启动定时冲刷 |
| `shutdown()` | 关闭：等待当前冲刷完成 + 最后一次冲刷 |
| `addBreadcrumb(crumb)` | 记录面包屑（最多 20 条，FIFO） |
| `reportEvent(item)` | 通用事件入队（自动生成 eventId、体积保护） |
| `reportChatEvent(params)` | Chat 对话事件 → `type: 'chat_question'` |
| `reportAgentEvent(params)` | Agent 对话事件 → `type: 'agent_question'` |
| `reportLoginEvent(jobId, result, error?)` | 登录事件 → `type: 'user_login'` |
| `reportLogoutEvent(jobId)` | 登出事件 → `type: 'user_logout'` |
| `reportUpgradeCheckEvent(result, error?)` | 升级检测事件 → `type: 'upgrade_check'` |
| `reportErrorEvent(error, context?)` | 异常上报（自动附加面包屑 + 指纹） |
| `flushQueue()` | 冲刷上报（防并发、失败回写队列 + 落盘） |

---

### 4.3 事件入队与体积保护

`reportEvent()` 入队时通过 `observability-event-utils.ts` 做两层保护：

1. **字符串截断** — `normalizeObservabilityEvent()` 对 `question`（8KB）、`error.message`（4KB）、`error.stack`（16KB）、`breadcrumb.data`（1KB）等字段做截断
2. **字节数检查** — `getSerializedEventBytes()` 按 UTF-8 字节数计算，超过 `maxEventBytes`（默认 256KB）拒绝入队

---

### 4.4 冲刷上报

`flushQueue()` 执行流程：

```
flushQueue()
  ├── 已在进行中？→ await 当前 flushPromise（防并发）
  ├── 队列为空？→ return
  ├── 从队列头部取出 maxBatchSize 条
  ├── fetch(config.url, POST)
  │     ├── Header: Cookie: EIPGW-TOKEN=<getToken()>
  │     ├── Body: safeStringify({ events: batch })
  │     ├── Timeout: AbortSignal.timeout(timeoutMs)
  │     └── 校验: HTTP 200 + 业务 code === 0
  ├── 成功 → 事件已发送
  └── 失败 → eventQueue.unshift(batch) 回写队列 + writeToDiskCache(batch) 落盘
```

---

### 4.5 磁盘缓存（失败兜底）

- **写入**: `upsertDiskCacheContent()` — 按 `eventId` 去重合并，以 `writeFileSync` 写回 `obs-queue.jsonl`
- **恢复**: `restoreFromDiskCache()` — 启动时从 jsonl 逐行解析恢复，成功后清空文件
- 路径: `~/.proma/obs-queue.jsonl`

---

### 4.6 辅助模块

| 模块 | 职责 |
|------|------|
| `observability-event-utils.ts` | `normalizeObservabilityEvent()` — 字符串截断；`getSerializedEventBytes()` — UTF-8 字节数；`upsertDiskCacheContent()` — eventId 去重合并 |
| `safe-stringify.ts` | `safeStringify()` — JSON.stringify + 长字符串截断（512KB）+ 循环引用保护 |
| `observability-ipc.ts` | 渲染进程 → `reportRendererErrorViaIpc()` → IPC → 主进程 `reportErrorEvent()` |

## 五、异常捕获

### 5.1 全局异常监听

**文件**: `error-handler.ts`，在 `initWorkmateServices()` 最先注册。

| 监听点 | 触发方式 | 上报 |
|--------|---------|------|
| `process.on('uncaughtException')` | 未捕获异常 | `reportErrorEvent(error, { source: 'uncaughtException' })` |
| `process.on('unhandledRejection')` | Promise 未 catch | `reportErrorEvent(error, { source: 'unhandledRejection' })` |
| `app.on('render-process-gone')` | 渲染进程崩溃 | `reportErrorEvent(error, { source, reason, exitCode, url })` |
| `app.on('child-process-gone')` | 子进程崩溃 | `reportErrorEvent(error, { source, type, reason, exitCode })` |

致命错误（ENOMEM / 堆溢出 / 模块加载失败）上报后延迟 1s 退出进程。

### 5.2 渲染进程 ErrorBoundary

渲染进程通过 `observability-ipc.ts` → IPC `reportRendererError` → 主进程 `ipc.ts` L4462-L4470 → `dynamic import` → `reportErrorEvent()` 上报，`tags.source = 'renderer-error-boundary'`。

### 5.3 防递归保护

`reportErrorEvent()` 内部 `isReportingError` 标志防止上报过程自身抛错导致无限递归。

### 5.4 指纹生成

```
fingerprint = md5(errorType + ':' + errorMessage.slice(0, 100)).slice(0, 16)
```

客户端生成仅供透传，服务端入库时以自有算法统一重算覆盖。

## 六、各事件调用入口

```
登录          auth/ipc-handlers.ts  →  dynamic import → reportLoginEvent()
登出          auth/ipc-handlers.ts  →  dynamic import → reportLogoutEvent()
Chat 对话     chat-service.ts       →  reportChatEvent()
Agent 对话    agent-service.ts      →  reportAgentEvent()
升级检测      auto-updater.ts       →  reportUpgradeCheckEvent()  (仅手动触发)
渲染层错误    observability-ipc.ts  →  IPC → reportErrorEvent()
主进程异常    error-handler.ts      →  reportErrorEvent()
```

所有上报最终汇入 `observability-service.ts` → `flushQueue()` → HTTP POST。

## 七、上报 HTTP 层

当前使用裸 `fetch`，手工管理 Cookie、超时、JSON 解析，未复用 `hteip-client` 的 `httpPost`。原因：观测上报有差异化逻辑（`safeStringify`、失败回写队列 + 去重落盘），与 `httpPost` 的标准请求-响应模式不完全匹配。URL 域名层面已通过 `resolveApiBase()` 收敛。
