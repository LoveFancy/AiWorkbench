## 二、进程架构

### 2.1 当前三进程架构

```
┌──────────────────────────────────────────────────────────┐
│                     Electron App                         │
├──────────────────────────────────────────────────────────┤
│  ┌────────────────────────────────────────────────────┐ │
│  │              主进程 (Main Process)                   │ │
│  │  src/main/index.ts — 应用生命周期 + 窗口管理          │ │
│  │  src/main/ipc.ts   — IPC 通道注册                   │ │
│  │  src/main/lib/     — 服务层（60+ 模块）               │ │
│  └──────────────────────┬─────────────────────────────┘ │
│          IPC (contextBridge)                             │
│  ┌──────────────────────┴─────────────────────────────┐ │
│  │              Preload (preload/index.ts)              │ │
│  └──────────────────────┬─────────────────────────────┘ │
│                         │                                │
│  ┌──────────────────────┴─────────────────────────────┐ │
│  │          渲染进程 (Renderer - React 18)              │ │
│  │  src/renderer/ — 全部 UI 组件、状态管理、样式         │ │
│  └────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────┘
```

### 2.2 核心通信路径

```
shared 类型定义 + IPC 常量
        ↓
main/ipc.ts           →  注册 ipcMain.handle / on
        ↓
preload/index.ts      →  contextBridge.exposeInMainWorld('electronAPI', ...)
        ↓
renderer Jotai atoms  →  调用 window.electronAPI.xxx
        ↓
renderer React 组件   →  消费 Jotai 原子状态
```

```

  └── bootstrap()

      ├── 9. createTray()             ← 系统托盘
```

## 五、已有的 Headless 能力

### 5.1 `runAgentHeadless()` - 已有

`agent-service.ts` 中已实现 `runAgentHeadless()`，用于无渲染进程的 Agent 执行：

- **飞书 Bridge** 使用它处理群消息
- **钉钉 Bridge** 使用它处理群消息
- **本地 API Server** 使用它提供 HTTP API
- **自动化调度器** 使用它执行定时任务

调用方只需要提供 `{ onError, onComplete, onTitleUpdated }` 回调，无需 `webContents`。

### 5.2 `LocalApiServer` - 已有

`local-api-server.ts` 已基于 Node 原生 `http` 模块实现了 REST + SSE API：

- `POST /v1/sessions` — 创建会话
- `POST /v1/sessions/:id/messages` — 发送消息（SSE 流式响应）
- `GET /v1/sessions/:id/runs/:runId` — 查询运行状态
- Token 认证 + 权限模式控制

### 5.3 Bridge 体系 - 已有

飞书、钉钉、微信 Bridge 已通过 WebSocket 长连接实现，启动/停止/自愈逻辑在 `bridge-registry.ts` 中统一管理。

## 六、当前构建配置

### 6.1 构建脚本

```json
{
  "build:main": "esbuild src/main/index.ts --bundle --platform=node --format=cjs --outfile=dist/main.cjs --external:electron --external:@anthropic-ai/claude-agent-sdk",
  "build:preload": "esbuild src/preload/index.ts --bundle --platform=node --format=cjs --outfile=dist/preload.cjs --external:electron",
  "build:renderer": "vite build"
}
```

### 6.2 electron-builder 配置

- 当前仅配置 macOS (`dmg`, `zip`) 和 Windows (`nsis`) 目标
- 无 Linux 目标
- `@anthropic-ai/claude-agent-sdk` 的 optionalDependencies 仅包含 darwin/win32 平台，缺少 linux

### 6.3 关键依赖

| 依赖 | 说明 | 跨平台 |
|------|------|--------|
| `@anthropic-ai/claude-agent-sdk` | Agent 运行时 | 需添加 linux 平台 native binary |
| `@primno/dpapi` | Windows DPAPI 加密 | 仅 Windows，Linux 需替代 |
| `electron` | 桌面框架 | 需移除 |
| `pdfjs-dist` | PDF 解析 | 纯 JS，跨平台 |
| `mammoth` | Word 解析 | 纯 JS，跨平台 |

## 七、关键挑战

### 7.1 必须解决的问题

1. **Electron 运行时依赖**：`src/main/index.ts` 的启动流程深度依赖 `app`、`BrowserWindow` 等 Electron API，需要重构启动入口
2. **IPC 通信层**：`ipc.ts` 中 ~2000+ 行的 IPC 处理器全部依赖 `ipcMain`，需要替换为 HTTP 或本地 socket
3. **DPAPI 加密**：`dpapi.ts` 使用 `@primno/dpapi`（Windows 专有），Linux 上 API Key 加密需替代方案
4. **渲染进程交互**：用户追问（`ask_user`）、权限确认、退出计划模式等需要用户交互的功能，在无 GUI 环境下需要替代交互方式
5. **自动更新**：`electron-updater` 依赖 Electron，需要替换
6. **全局快捷键**：`globalShortcut` 依赖 Electron
7. **系统托盘**：`Tray` 依赖 Electron

### 7.2 可以保留的能力

- Agent 编排核心（`agent-orchestrator.ts`）
- 会话管理（`agent-session-manager.ts`）
- 工作区管理（`agent-workspace-manager.ts`）
- 渠道管理（`channel-manager.ts`）
- 桥接服务（飞书/钉钉/微信）
- 本地 HTTP API Server
- 定时任务调度
- 文件日志
- 设置/存储服务
- 运行时检测
- Chat 服务
