# WorkMate Linux 无头服务抽取方案

> 方案日期：2026-06-26
> 目标：抽取 main 主线程核心服务，打包为无 GUI 的 Linux 独立进程，支持 CentOS、Kylin 等操作系统，覆盖 x86_64 和 ARM64 架构

## 二、技术方案

### 2.1 整体思路

**核心策略：不修改现有 Electron 代码，新建独立的 headless 入口，最大化复用现有业务逻辑模块。**

```
apps/electron/src/
├── main/                          # 现有 Electron 主进程（不变）
│   ├── index.ts
│   ├── ipc.ts
│   └── lib/                       # 纯业务逻辑模块（直接复用）
│       ├── agent-orchestrator.ts
│       ├── agent-session-manager.ts
│       ├── agent-workspace-manager.ts
│       ├── channel-manager.ts
│       ├── feishu-bridge.ts
│       ├── local-api-server.ts
│       └── ...
├── headless/                      # 新增：无头服务入口
│   ├── index.ts                   # 启动入口（替代 Electron 生命周期）
│   ├── bootstrap.ts               # 启动流程（替代 index.ts 的 bootstrap）
│   ├── signal-handler.ts          # 进程信号处理（SIGTERM/SIGINT）
│   └── config.ts                  # 无头服务专属配置
├── renderer/                      # 渲染进程（无头版不构建）
└── preload/                       # Preload（无头版不构建）
```

### 2.2 启动流程重构

对比 Electron 版的 `bootstrap()`，无头版简化为：

```
headless/index.ts
  └── bootstrap()
      ├── 1. 解析命令行参数 / 配置文件
      ├── 2. setPromaVersion()
      ├── 3. 初始化文件日志
      ├── 4. seedDefaultSkills / seedDefaultPlugins / seedDefaultConnectors
      ├── 5. loadCacheFromDisk() + initModelRefresh()
      ├── 6. initializeRuntime()            ← Node/Bun/Git 检测
      ├── 7. startAllBridges()              ← 飞书/钉钉/微信
      ├── 8. startBridgeSelfHealing()       ← Bridge 自愈
      ├── 9. startLocalApiServer()          ← HTTP API
      ├── 10. startScheduler()              ← 定时任务
      ├── 11. 注册 SIGTERM/SIGINT 信号处理
      └── 12. 保持进程运行（不退出）
```

### 2.3 需要移除的 Electron GUI 依赖

| 原有模块/功能 | 无头版处理方式 |
|--------------|---------------|
| `BrowserWindow` 创建 | 移除 |
| `createTray()` 系统托盘 | 移除 |
| `createApplicationMenu()` 原生菜单 | 移除 |
| `dialog.showErrorBox()` 错误弹窗 | 改为 stderr + 日志 |
| `screen` 显示器检测 | 移除 |
| `app.dock` macOS Dock | 移除 |
| `powerMonitor` 电源事件 | 移除 |
| `globalShortcut` 全局快捷键 | 移除 |
| `createQuickTaskWindow()` | 移除 |
| `createVoiceDictationWindow()` | 移除 |
| `sendToMainWindow()` / `sendToMainWindowQuiet()` | 移除 |
| `initAutoUpdater()` (electron-updater) | 改为独立更新机制 |
| `workspace-watcher` 中的 `webContents.send()` | 改为 EventBus 事件 |
| `agent-service.ts` 中 `webContents.send()` | 改为 EventBus 事件 |
| 权限确认 / 用户追问弹窗 | 改为 HTTP API 回调 / 默认策略 |

### 2.4 用户交互处理

原 Electron 版中需要用户实时交互的场景：

| 场景 | Electron 版 | 无头版 |
|------|------------|--------|
| 权限确认 (canUseTool) | 弹窗询问 | 配置默认策略（Bypass/Default/Plan） |
| 用户追问 (ask_user) | 聊天框输入 | 通过 HTTP API 的 `ask_user` 回调 |
| 退出计划模式 | 弹窗确认 | 通过 HTTP API 回调 |
| 模型选择 | 下拉菜单 | 配置文件指定 |
| API Key 输入 | 设置面板 | 配置文件 / 环境变量 |

### 2.5 DPAPI 加密替代方案

Linux 上不能使用 `@primno/dpapi`（Windows 专有），需提供替代方案：

```
方案 A（推荐）：文件权限保护
  - API Key 存储为 JSON 文件，权限设为 600（仅 owner 可读写）
  - 简单有效，Linux 服务端常用做法

方案 B：使用 keytar 替代
  - keytar 通过 libsecret (GNOME) / ksecret (KDE) 访问系统密钥环
  - 需要桌面环境支持，不适合无头服务器

方案 C：使用环境变量
  - 通过 systemd 的 EnvironmentFile 或 Docker secrets 注入
  - 配合 systemd 的 PrivateTmp/ProtectSystem 加固
```

### 2.6 配置管理

无头版通过配置文件 + 环境变量管理，不依赖 GUI 设置面板：

```yaml
# /etc/workmate/config.yml 或 ~/.workmate/config.yml
server:
  host: "0.0.0.0"
  port: 5800
  api_token: "${WORKMATE_API_TOKEN}"  # 支持环境变量

channels:
  - id: "default"
    provider: "anthropic"
    api_key: "${ANTHROPIC_API_KEY}"
    model: "claude-sonnet-4-20250514"

bridges:
  feishu:
    enabled: true
    app_id: "xxx"
    app_secret: "${FEISHU_APP_SECRET}"
  dingtalk:
    enabled: false
  wechat:
    enabled: false

agent:
  permission_mode: "bypass"  # default | accept_edits | bypass | plan
  workspace: "default"

automation:
  enabled: true
  tasks:
    - name: "daily-report"
      cron: "0 9 * * 1-5"
      prompt: "生成今日工作日报"
```

## 三、工程实现步骤

### 阶段一：代码抽取（2-3天）

#### Step 1: 创建 headless 入口

```
新建文件：
  apps/electron/src/headless/index.ts        # 启动入口
  apps/electron/src/headless/bootstrap.ts    # 启动流程
  apps/electron/src/headless/config.ts       # 配置加载
  apps/electron/src/headless/signal-handler.ts # 信号处理
```

`index.ts` 核心逻辑：

```typescript
#!/usr/bin/env node
// 移除所有 Electron 导入，仅使用 Node.js 标准库
import { bootstrap } from './bootstrap'

async function main() {
  try {
    await bootstrap()
    // 保持进程运行
    process.stdin.resume()
  } catch (err) {
    console.error('Failed to start WorkMate headless:', err)
    process.exit(1)
  }
}

main()
```

#### Step 2: 重构 `agent-service.ts`

- 抽取 `AgentService` 类，将 `webContents` 相关逻辑移除
- EventBus 事件直接分发给订阅者，不再经过 IPC 转发
- `runAgentHeadless()` 已经是无 GUI 的，保持不变

#### Step 3: 移除 IPC 依赖

- `ipc.ts` 整体不加载
- `agent-ask-user-service.ts` 改为 HTTP 回调模式
- `agent-permission-service.ts` 的确认逻辑改为配置驱动

#### Step 4: 替换 DPAPI

- 新增 `credential-store.ts`，抽象加密接口
- Linux 实现：文件权限保护（chmod 600）+ 可选 AES 加密
- 保留 Windows 的 DPAPI 实现

### 阶段二：构建配置（1天）

#### Step 5: 新增构建脚本

```json
{
  "scripts": {
    "build:headless": "esbuild src/headless/index.ts --bundle --platform=node --format=cjs --outfile=dist/headless.cjs --external:@anthropic-ai/claude-agent-sdk",
    "dist:headless:linux-x64": "bun run build:headless && <打包脚本>",
    "dist:headless:linux-arm64": "bun run build:headless && <打包脚本>"
  }
}
```

#### Step 6: 添加 Linux 平台的 SDK native binary

在 `package.json` 的 `optionalDependencies` 中添加：

```json
{
  "optionalDependencies": {
    "@anthropic-ai/claude-agent-sdk-linux-x64": "0.3.153",
    "@anthropic-ai/claude-agent-sdk-linux-arm64": "0.3.153"
  }
}
```

### 阶段三：打包方案（1-2天）

#### Step 7: 打包策略

**方案 A：Node.js 单文件打包（推荐）**

使用 `esbuild` 打包为单个 CJS 文件，附带 native binary 和资源文件：

```
dist/headless/
├── workmate-headless.cjs       # 主程序（esbuild bundle）
├── node_modules/
│   └── @anthropic-ai/
│       └── claude-agent-sdk-linux-x64/
│           └── native-binary
├── default-skills/             # 技能模板
├── default-connectors/         # 连接器
└── bundled-plugins/            # 插件
```

打包为 tar.gz：

```bash
tar -czf workmate-headless-linux-x64.tar.gz dist/headless/
```

**方案 B：Docker 镜像**

```dockerfile
FROM node:20-slim
WORKDIR /app
COPY dist/headless/ /app/
RUN chmod +x /app/workmate-headless.cjs
EXPOSE 5800
CMD ["node", "/app/workmate-headless.cjs"]
```

**方案 C：systemd 服务 + deb/rpm 包**

```
/usr/lib/workmate/
├── workmate-headless.cjs
├── node_modules/
├── default-skills/
└── bundled-plugins/
/etc/workmate/config.yml
/usr/lib/systemd/system/workmate.service
```

#### Step 8: CI/CD 矩阵构建

在 `.github/workflows/release.yml` 中增加 Linux 构建矩阵：

```yaml
strategy:
  matrix:
    os: [ubuntu-22.04, ubuntu-22.04-arm]
    include:
      - os: ubuntu-22.04
        arch: x64
      - os: ubuntu-22.04-arm
        arch: arm64
```

### 阶段四：部署与运维（1-2天）

#### Step 9: systemd 服务文件

```ini
[Unit]
Description=WorkMate Headless Agent Service
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=workmate
ExecStart=/usr/bin/node /usr/lib/workmate/workmate-headless.cjs
WorkingDirectory=/var/lib/workmate
Restart=on-failure
RestartSec=10
EnvironmentFile=/etc/workmate/env
PrivateTmp=yes
ProtectSystem=strict
ProtectHome=read-only
ReadWritePaths=/var/lib/workmate

[Install]
WantedBy=multi-user.target
```

#### Step 10: 健康检查与监控

```
GET /health → 200 { "status": "ok", "uptime": 3600 }
GET /health/ready → 200 { "status": "ready", "bridges": { "feishu": "connected" } }
```

## 五、风险与注意事项

### 5.1 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| SDK Linux binary 不兼容旧内核 | 无法在 CentOS 7 等旧系统运行 | 静态链接或提供 glibc 版本要求说明 |
| `@primno/dpapi` 无替代 | API Key 无法安全存储 | 使用文件权限 + 可选 AES 加密 |
| 权限/追问交互丢失 | 某些场景自动化失败 | 配置默认策略 + HTTP API 回调 |
| 第三方依赖的平台兼容性 | 某些 npm 包在 ARM64 上有问题 | CI 多架构测试 |

### 5.2 运维风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 进程崩溃后无自动恢复 | 服务中断 | systemd Restart=on-failure |
| 日志文件无限增长 | 磁盘占满 | logrotate 配置 |
| 多实例竞争 | 数据损坏 | 文件锁（类似 Electron 的 singleInstanceLock） |

## 七、附录：关键代码示意

### 7.1 headless bootstrap 示意

```typescript
// headless/bootstrap.ts
import { installFileLogger } from '../main/lib/file-logger'
import { seedDefaultSkills, seedDefaultPlugins, seedDefaultConnectors } from '../main/lib/config-paths'
import { initializeRuntime } from '../main/lib/runtime-init'
import { startAllBridges, startBridgeSelfHealing } from '../main/lib/bridge-registry'
import { startLocalApiServer } from '../main/lib/local-api-service'
import { startScheduler } from '../main/lib/automation-scheduler'
import { loadConfig } from './config'
import { registerSignalHandlers } from './signal-handler'
import { setPromaVersion } from '@proma/core'

export async function bootstrap(): Promise<void> {
  const config = loadConfig()

  // 日志
  installFileLogger(config.logsDir)

  // 版本
  setPromaVersion(config.version)

  // 默认资源
  seedDefaultSkills()
  seedDefaultPlugins()
  seedDefaultConnectors()

  // 运行时检测
  await initializeRuntime()

  // Bridge
  if (config.bridges) {
    await startAllBridges()
    startBridgeSelfHealing()
  }

  // HTTP API
  await startLocalApiServer()

  // 定时任务
  if (config.automation?.enabled) {
    startScheduler()
  }

  // 信号处理
  registerSignalHandlers()

  console.log('[WorkMate Headless] Started successfully')
}
```

### 7.2 构建脚本示意

```typescript
// scripts/dist-headless.ts
import { execSync } from 'node:child_process'
import { mkdirSync, cpSync, existsSync } from 'node:fs'
import { join } from 'node:path'

const OUT_DIR = 'dist/headless'
const ARCH = process.argv[2] || 'x64'  // x64 | arm64

// 1. esbuild 打包
execSync(
  `esbuild src/headless/index.ts --bundle --platform=node --format=cjs --outfile=${OUT_DIR}/workmate-headless.cjs --external:@anthropic-ai/claude-agent-sdk`,
  { stdio: 'inherit' }
)

// 2. 复制 resources
cpSync('default-skills', join(OUT_DIR, 'default-skills'), { recursive: true })
cpSync('default-connectors', join(OUT_DIR, 'default-connectors'), { recursive: true })
cpSync('bundled-plugins', join(OUT_DIR, 'bundled-plugins'), { recursive: true })

// 3. 复制 native binary
const sdkBinary = `node_modules/@anthropic-ai/claude-agent-sdk-linux-${ARCH}`
if (existsSync(sdkBinary)) {
  cpSync(sdkBinary, join(OUT_DIR, sdkBinary), { recursive: true })
}

// 4. 打包 tar.gz
execSync(`tar -czf workmate-headless-linux-${ARCH}.tar.gz -C ${OUT_DIR} .`, { stdio: 'inherit' })
```
