## 一、需求概述

### 1.1 功能目标

为 WorkMate 客户端提供问题上报能力，支持：

1. **客户端提交问题**：上传问题描述文字 + 多张截图（顺序敏感），通过 HTTP multipart 提交
2. **管理台查看**：管理台支持分页查看所有上报问题，查看详情（含图片展示）
3. **图片顺序保持**：多张图片按提交顺序存储和展示

### 1.2 核心流程

```
┌──────────────────────────────────────────────────────────┐
│                     客户端 (Client)                        │
│  输入问题描述 + 选择多张截图（有序）                           │
│      │                                                    │
│      ▼                                                    │
│  POST /workmate/api/issues                                 │
│  multipart/form-data                                       │
│  { description, files[0], files[1], ... }                  │
└──────────────────────────────────────────────────────────┘
                        │
                        ▼
           ┌───────────────────────┐
           │    IssueController     │
           │    → IssueService      │
           │      → 校验参数         │
           │      → 保存图片到磁盘    │
           │      → 写入DB          │
           └───────────────────────┘
                        │
                        ▼
           ┌───────────────────────┐
           │  文件系统               │

           │    images/             │
           │      {uuid1}.png       │
           │      {uuid2}.jpg       │
           │      ...               │

┌──────────────────────────────────────────────────────────┐
│                   管理台 (Admin Console)                    │
├──────────────────────────────────────────────────────────┤
│  问题列表页：分页列表，支持按状态/时间筛选                      │
│      │                                                    │

└──────────────────────────────────────────────────────────┘
```

## 二、存储设计

### 2.1 存储架构

```
文件系统 (${workmate.issue.storage-path})
└── images/
    ├── a1b2c3d4-xxxx.png   ← 问题 #1 第1张图片
    ├── e5f6g7h8-xxxx.jpg   ← 问题 #1 第2张图片
    ├── i9j0k1l2-xxxx.png   ← 问题 #2 第1张图片
    └── ...

MySQL
├── issue_reports           ← 问题主表（一条问题一行）
└── issue_report_images     ← 图片明细表（一张图片一行，通过 sort_order 保持顺序）
```

> **设计决策**：图片与问题分离存储，一个问题的多张图片作为子表记录，通过 `sort_order` 字段保持客户端提交的顺序。

### 2.2 为什么用子表而非 JSON 数组

| 方案         | 优点                           | 缺点                               |
| ---------- | ---------------------------- | -------------------------------- |
| JSON 数组字段  | 查询简单，一次取出                    | 无法单独索引图片，不方便按图片维度查询；JSON 更新需整体替换 |
| **子表（采用）** | 可单独索引；方便扩展（如图片审核状态）；排序查询天然有序 | 需要 JOIN 或两次查询                    |

## 三、数据库设计

### 3.1 issue\_reports 表

```sql
-- V10__issue_report.sql
CREATE TABLE `issue_reports` (
    `id`            INTEGER      NOT NULL AUTO_INCREMENT COMMENT '主键',
    `user_id`       VARCHAR(128) NOT NULL                COMMENT '上报用户标识（jobId）',
    `description`   TEXT         NOT NULL                COMMENT '问题描述',
    `status`        ENUM('PENDING', 'PROCESSING', 'RESOLVED', 'CLOSED') NOT NULL DEFAULT 'PENDING' COMMENT '处理状态',
    `client_version` VARCHAR(32)  NULL                   COMMENT '客户端版本号',
    `client_platform` VARCHAR(32) NULL                   COMMENT '客户端平台：windows/mac/linux',
    `created_at`    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '上报时间',
    `updated_at`    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (`id`),
    INDEX `idx_user_id` (`user_id`),
    INDEX `idx_status` (`status`),
    INDEX `idx_created_at` (`created_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='问题上报主表';
```

### 3.2 issue\_report\_images 表

```sql
CREATE TABLE `issue_report_images` (
    `id`            INTEGER      NOT NULL AUTO_INCREMENT COMMENT '主键',
    `report_id`     INTEGER      NOT NULL                COMMENT '关联问题ID',
    `file_name`     VARCHAR(255) NOT NULL                COMMENT '服务端存储的文件名（UUID + 扩展名）',
    `original_name` VARCHAR(255) NOT NULL                COMMENT '原始文件名',
    `file_size`     BIGINT       NOT NULL DEFAULT 0      COMMENT '文件大小（字节）',
    `sort_order`    INTEGER      NOT NULL DEFAULT 0      COMMENT '图片顺序（从0开始）',
    `created_at`    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    PRIMARY KEY (`id`),
    INDEX `idx_report_id` (`report_id`),
    INDEX `idx_report_id_sort` (`report_id`, `sort_order`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='问题上报图片明细';
```

### 3.3 查询模式

| 查询场景          | SQL 路径                                                                      | 索引                              |
| ------------- | --------------------------------------------------------------------------- | ------------------------------- |
| 管理台列表（时间倒序分页） | `SELECT * FROM issue_reports ORDER BY created_at DESC LIMIT ?,?`            | `idx_created_at`                |
| 按状态筛选         | `WHERE status = ? ORDER BY created_at DESC`                                 | `idx_status` + `idx_created_at` |
| 按用户筛选         | `WHERE user_id = ? ORDER BY created_at DESC`                                | `idx_user_id`                   |
| 查看问题图片        | `SELECT * FROM issue_report_images WHERE report_id = ? ORDER BY sort_order` | `idx_report_id_sort`            |

## 四、API 接口设计

### 4.1 客户端接口（开放，无需管理台权限）

#### 4.1.1 提交问题上报

```
POST /workmate/api/issues
Content-Type: multipart/form-data
```

**请求参数**：

| 参数               | 类型               | 必填 | 说明                 |
| ---------------- | ---------------- | -- | ------------------ |
| `description`    | String           | 是  | 问题描述文字             |
| `files`          | MultipartFile\[] | 否  | 截图文件（支持多张），顺序即数组顺序 |
| `clientVersion`  | String           | 否  | 客户端版本号             |
| `clientPlatform` | String           | 否  | 客户端平台              |

**校验规则**：

| 检查项            | 规则                            | 说明       |
| -------------- | ----------------------------- | -------- |
| description 非空 | 长度 1-5000 字符                  | 提供有意义的描述 |
| 图片数量上限         | 1-9 张                         | 限制单次上传数量 |
| 图片格式白名单        | png / jpg / jpeg / gif / webp | 常见图片格式   |
| 单张大小上限         | 10MB                          | 避免超大文件   |
| 图片非空           | 文件大小 > 0                      | —        |

**成功响应**（200）：

```json
{
  "code": 0,
  "message": "提交成功",
  "data": {
    "id": 1,
    "status": "PENDING",
    "imageCount": 3,
    "createdAt": "2026-06-15T10:30:00"
  }
}
```

#### 4.1.2 查看图片

```
GET /workmate/api/issues/images/{fileName}
```

直接返回图片二进制流，`Content-Type` 根据文件扩展名设置。

> 图片访问无需鉴权，因为是问题截图的静态资源，且文件名是 UUID 不可枚举。

### 5.5 Controller 设计

客户端接口（新增 IssueController）：

```java
@RestController
@RequestMapping("/workmate/api/issues")
@RequiredArgsConstructor
public class IssueController {

    private final IssueService issueService;

    @PostMapping
    public ApiResponse<Map<String, Object>> submit(
            HttpServletRequest httpRequest,
            @RequestParam String description,
            @RequestParam(required = false) String clientVersion,
            @RequestParam(required = false) String clientPlatform,
            @RequestParam(required = false) List<MultipartFile> files) {
        // 从请求属性获取 userId
        String userId = (String) httpRequest.getAttribute("jobId");
        IssueReport report = issueService.submit(userId, description,
                clientVersion, clientPlatform, files);
        return ApiResponse.ok(Map.of(
            "id", report.getId(),
            "status", report.getStatus(),
            "imageCount", files != null ? files.size() : 0,
            "createdAt", report.getCreatedAt()
        ), "提交成功");
    }
}
```

### 5.7 安全设计

| 关注点     | 措施                                                            |
| ------- | ------------------------------------------------------------- |
| 图片文件名校验 | 服务端生成 UUID 文件名，丢弃原始文件名                                        |
| 图片格式校验  | Magic Number 校验，不能仅信任扩展名                                      |
| 路径穿越防御  | 文件名不含 `/`、`\`、`..`                                            |
| 图片大小限制  | 单张 ≤ 10MB，由 `spring.servlet.multipart.max-file-size` 和应用层双重兜底 |
| 图片数量限制  | 单次 ≤ 9 张                                                      |
| 用户身份    | 客户端接口从 `UserIdInterceptor` 注入的 `jobId` 获取用户标识                 |
| 管理台鉴权   | 复用 `AdminWhitelistInterceptor`，仅白名单用户可访问管理接口                  |

### 7.2 调用时机

### 7.3 调用频率限制

通过 `RateLimitFilter` 对 `/workmate/api/issues` 路径做频率限制（与观测上报共用限流策略），每人每分钟最多提交 2 次问题。
