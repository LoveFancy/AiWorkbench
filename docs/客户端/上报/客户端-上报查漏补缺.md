## 一、事件类型 → 实现状态

> **全局前置拦截**：下表所有事件在入队前都会先经过「EIP 未登录不上报」的统一拦截（详见[第六节](#六全局拦截未登录不上报新增)）。未登录（Token 缺失或已过期）时，任何事件都不会上报。

| 事件类型 | 触发场景 | 实际实现 | 状态 |
|----------|---------|---------|------|
| `user_login` | 登录成功/失败 | `auth/ipc-handlers.ts` → `reportLoginEvent()` | ✅ |
| `user_logout` | 用户登出 | `auth/ipc-handlers.ts` → `reportLogoutEvent()` | ✅ |
| `chat_question` | Chat 正常完成 | `chat-service.ts` L450 → `reportChatEvent({ result: 'success' })` | ✅ |

| `chat_question` | **用户中止 / 超时** | `chat-service.ts` L497 → `reportChatEvent({ result: 'failure', error: new Error('用户中止') })` | ✅ 已修复 |

## 二、新增

### 2.1 app_startup — App 启动事件上报

**位置**: 

| 文件 | 变更 |
|------|------|
| `types/workmate.ts` | 新增 `'app_startup'` 事件类型和 `startupDurationMs` 字段 |
| `observability-service.ts` | 新增 `reportStartupEvent(startupDurationMs)` 方法 |
| `workmate-init.ts` | `initWorkmateServices()` 接收 `startupDurationMs` 参数，观测初始化后立即上报 |
| `index.ts` | `bootstrap()` 中计算 `Date.now() - STARTUP_TIME` 传入 `initWorkmateServices()` |

**上报时机**: 观测上报服务初始化完成后立即上报，位于 `initializeRuntime()` 之后、窗口创建之前。启动耗时从 `index.ts` 模块级 `STARTUP_TIME`（进程加载首行）起算。

**上报格式**:

```ts
reportStartupEvent(Date.now() - STARTUP_TIME)
// →
reportEvent({
  type: 'app_startup',
  userId: getJobId() ?? 'unknown',
  timestamp: Date.now(),
  result: 'success',
  startupDurationMs: 1234,         // 启动耗时（毫秒）
  client: {
    appVersion: app.getVersion(),  // 复用 getClientInfo()
    platform: platform(),          // 复用 getClientInfo()
    osVersion: release(),          // 复用 getClientInfo()
  },
})
```

> `client` 信息复用 `observability-service.ts` 中已有的 `getClientInfo()` 函数，与其他事件类型保持一致。

## 三、已修复

### 3.1 Chat 中止/超时 — 已补上报

**位置**: `chat-service.ts` L497

```ts
// 上报用户中止事件（仅上报用户问题，不带上报模型回复，防止日志过大）
try {
  reportChatEvent({
    userId: getJobId() ?? 'unknown',
    question: userMessage,
    modelId,
    result: 'failure',

```

## 四、刻意不上报

### 4.1 Agent 手动停止 不上报

**原因**: Agent 的回复不应上报，防止日志过大。用户手动停止视为正常交互行为，不需要记录。

### 4.2 升级检测自动轮询 不上报

**原因**: 后台定时轮询频率高、数据量大，不上报避免噪音。仅用户主动"检查更新"时上报。

### 4.3 面包屑 不上报

**原因**: 面包屑 `addBreadcrumb()` 已实现但未接入。当前上报场景有限，面包屑收益/成本比低，暂不接入。

## 五、实现偏离设计之处

| 设计预期 | 实际代码 | 评估 |
|---------|---------|------|
| Chat 失败由监听 `STREAM_ERROR` 事件触发，不修改 `chat-service` | `chat-service.ts` 直接 `import { reportChatEvent }` 调用 | 直接调用更简单可靠，保留 |
| Agent 失败由 `AgentEventBus` 事件触发 | `agent-service.ts` 在 `onError`/`onComplete` 回调中直接调用 | 同上，保留 |
| `result: 'pending'` 为中/超时场景保留 | 从未使用 | 搁置 |
| 磁盘缓存用 `appendFileSync` | 实际用 `upsertDiskCacheContent` 按 eventId 去重 → `writeFileSync` | 实际优于设计 |

## 六、全局拦截：未登录不上报（新增）

编辑时间：2026年6月22日

### 6.1 需求

EIP 未登录时不应产生任何上报：未登录意味着没有有效的用户身份（无 Token / Token 已过期），此时上报既无法被服务端正确归属，也属于无效噪音，应一律丢弃。

### 6.2 方案（单点守卫）

所有事件——无论 `user_login` / `chat_question` / `agent_question` / `error` / `upgrade_check` / `app_startup`——最终都经由 `observability-service.ts` 的 `reportEvent()` 入队。因此只在该单一漏斗处加一道前置守卫即可覆盖全部事件类型，无遗漏、无重复。

**位置**：`apps/electron/src/main/lib/observability-service.ts` → `reportEvent()`

```ts
export function reportEvent(item: Omit<ObservabilityEventItem, 'eventId'>): void {
  if (!config?.enabled) return
  // 未登录（含 Token 缺失/已过期）时不上报：EIP 未登录则无有效身份，事件一律丢弃
  if (!isLoggedIn()) return
  // ... 其余入队逻辑不变
}
```

`isLoggedIn` 从 `../../auth` 导入（该模块原已导入 `getToken` / `getJobId`，不引入新依赖、无循环引用）。

### 6.3 登录判断依据

复用 `auth/auth-service.ts` 的 `isLoggedIn()`：

- `getToken()` 存在（Token 未缺失）；
- 且 `Date.now() <= auth.json.expiresAt`（**以文件内的过期时间为准，当前为 30 天**）；
- 且能从 JWT 解析出工号（`parseJobId` 非空）。

任一不满足 → 视为未登录 → 拦截。

### 6.4 对「边界事件」的影响（确认为：全部拦截）

| 事件 | 上报时的登录状态 | 结果 |
|------|----------------|------|
| 登录成功 / `CHECK_SESSION` 恢复 | Token 已写入且有效 | ✅ 正常上报 |
| 登录失败 `reportLoginEvent(jobId, 'failure')` | 未写入 Token | 🚫 拦截 |
| 登出 `reportLogoutEvent()`（在 `logout()` 清空文件之后调用） | 已无 Token | 🚫 拦截 |
| chat / agent / error / upgrade_check / app_startup | 取决于当时登录态 | 未登录则 🚫 拦截 |

> 经与需求方确认：未登录时**全部拦截**，包含登录失败与登出这两类边界事件，不做例外。

### 6.5 测试

`apps/electron/src/main/lib/observability-service.test.ts` 新增 `describe('reportEvent 登录拦截')`：

- 未登录（`isLoggedIn() === false`）→ `reportEvent()` 后 `flushQueue()` 不触发 `fetch`（事件未入队）；
- 已登录（`isLoggedIn() === true`）→ `reportEvent()` 后 `flushQueue()` 触发一次 `fetch`（事件正常入队上报）。

通过 `mock.module('../../auth')` 控制 `isLoggedIn` 返回值，`mock.module('electron')` / `mock.module('./config-paths')` 隔离运行环境。
