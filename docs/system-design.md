## 1. 产品概述

WorkMate 是一个本地优先的 AI 桌面应用，将通用 Agent、多模型对话、Skills/MCP 扩展和记忆能力整合在统一的 Electron 桌面客户端中。

简单问题用 Chat，复杂任务交给 Agent，数据和配置尽量留在本地。

### 核心能力

| 能力 | 说明 |
|------|------|
| **Agent 任务执行** | 基于 Claude Agent SDK，支持文件操作、命令执行、多步骤任务、工作区隔离 |
| **多模型 Chat** | 支持 Anthropic、OpenAI、DeepSeek、Google 等 13 种 AI 供应商 |
| **Skills & MCP** | 工作区级别的 Skill 管理和 MCP Server 配置 |
| **Agent 插件** | 从插件市场安装完整的功能扩展包（Skill + MCP + Agent 组合） |
| **本地优先** | 所有数据存储在用户目录，JSON/JSONL 纯文件组织，无本地数据库 |

### 技术栈

| 层级 | 技术 |
|------|------|
| 桌面框架 | Electron |
| 前端 | React + TypeScript + Tailwind CSS + Radix UI |
| 状态管理 | Jotai |
| Agent 引擎 | Claude Agent SDK |
| 运行时 | Bun |

## 2. 总体架构

### 2.1 进程模型

WorkMate 采用 Electron 标准多进程架构：

- **主进程**：负责文件系统访问、加密存储、Agent SDK 编排、系统集成等底层能力
- **渲染进程**：负责 React UI 渲染，通过类型安全的桥接层访问主进程能力
- **Preload 桥接层**：在主进程和渲染进程之间建立类型安全的 IPC 通信桥梁

渲染进程无法直接访问文件系统或系统 API，所有底层操作都通过 IPC 通道委托给主进程执行。

### 2.2 通信模式

IPC 通道按功能模块组织（渠道、对话、Agent、记忆等），采用两种通信模式：

- **请求-响应模式**：适用于 CRUD 操作和配置读取，渲染进程发起请求，主进程返回结果
- **推送通知模式**：适用于 Agent 流式事件和状态变更广播，主进程主动向渲染进程推送增量数据

新增功能模块需要在四个层面同步：类型定义 → 主进程处理器 → 预加载桥接 → 渲染进程调用。

### 2.3 主进程服务层

主进程服务按职责划分为以下几个领域：

**Agent 核心**：编排调度、会话管理、提示词构建、权限检查、用户交互处理、工作区管理

**Chat 服务**：多模型流式对话编排、对话管理

**扩展体系**：插件注册与管理、插件市场、SkillHub 企业市场

**基础设施**：渠道与 API Key 管理、文件监听、运行时初始化、自动更新

## 3. Agent 引擎集成

### 3.1 核心流程

用户输入消息后，编排层依次完成：并发守卫检查 → 渠道查找与 API Key 解密 → 构建子进程运行环境 → 构建系统提示词 → 汇总 MCP 和插件配置 → 注入工作区上下文 → 调用 SDK 发起查询。

SDK 作为外部原生进程运行，执行 Agent 循环（读取文件、搜索代码、执行命令、调用 MCP 工具、触发 Skill），通过消息流持续返回执行状态，WorkMate 将这些消息转换为统一的 UI 事件驱动界面更新。

### 3.2 消息类型

SDK 返回的消息涵盖 AI 响应、工具调用、子任务状态、权限请求、用户问答、计划审批、上下文压缩、Token 用量统计等。WorkMate 将其转换为统一的 UI 事件，包括流式文本、工具调用卡片、子任务面板、权限确认弹窗等。

### 3.3 子任务与 SubAgent

复杂任务会被自动拆分为子任务执行，WorkMate 内置了三种专用 SubAgent：

| SubAgent | 用途 |
|---------|------|
| 代码审查 | 审查代码质量、发现潜在问题 |
| 代码探索 | 快速搜索代码库、理解项目结构 |
| 技术调研 | 技术调研、方案对比、架构分析 |

### 3.4 用户交互

Agent 执行过程中可能触发三类需要用户介入的交互：工具权限确认、问答信息收集、计划审批。这些交互通过按会话隔离的排队机制处理，保证多会话并行时不会混乱。

### 3.5 错误处理

SDK 调用失败时，系统将错误映射为 30 多种分类错误码，每种错误配有标题、描述和恢复建议。支持自动重试，并在 UI 中展示可操作的恢复按钮。

## 4. 多 Agent SDK 适配器

### 4.1 设计动机

当前 Agent 能力基于 Claude Agent SDK 实现。为支撑未来多引擎共存（如接入其他 Agent SDK），在主进程和具体 SDK 之间引入了适配器抽象层。

### 4.2 架构

编排层通过抽象接口依赖注入，不直接依赖任何具体 SDK。每个 SDK 提供各自的适配器实现，对外暴露统一的查询、中止、资源释放等行为契约。

### 4.3 设计原则

- **接口从简，实现从厚**：接口只定义行为契约，配置细节由各适配器内部封装
- **消息统一透传**：所有适配器返回统一的消息流，UI 层完全与底层 SDK 解耦
- **长生命周期通道**：适配器内部封装多轮对话机制，对外暴露一致的消息注入接口
- **进程全生命周期管理**：适配器自行管理 SDK 子进程的启动、软中断、强中止和退出清理

接入新 SDK 时，无需修改任何 UI 代码，只需实现适配器接口并扩展消息转换逻辑。

## 5. AI 多模型接入

### 5.1 适配器架构

Chat 模式通过统一的适配器接口接入多家 AI 供应商。核心组件包括：

- **Provider 注册表**：按标识符查找适配器，统一管理所有供应商
- **适配器实现**：将各供应商的 API 协议适配到统一接口
- **SSE 流读取器**：通用的流式响应解析器，各适配器复用

### 5.2 兼容性

Agent 模式底层使用 Anthropic 兼容协议，因此仅支持该协议的供应商可用于 Agent。Chat 模式则支持所有供应商，包括 OpenAI 协议和 Gemini 协议。

### 5.3 渠道管理

每个供应商连接称为一个"渠道"，支持创建、编辑、删除、连接测试和模型拉取。API Key 通过操作系统级安全存储加密，UI 层永远不接触明文密钥。

## 6. 工作区

### 6.1 概念

工作区是 Agent 的组织容器，用于将相关会话、Skill、MCP Server 和文件资源分组在一个独立的项目环境中。每个工作区通过唯一的标识符（slug）进行目录隔离，名称可随时修改但 slug 在创建后不可变更。

系统在首次启动时自动创建一个默认工作区，用户可按需创建更多工作区以区分不同的项目或任务场景。

### 6.2 隔离逻辑

工作区通过目录级别分离实现隔离：

- **Skill 隔离**：每个工作区维护自己启用和禁用的 Skill 副本。内置默认 Skill 在创建工作区时自动注入，后续升级时按版本号比较提示更新
- **MCP 隔离**：每个工作区拥有独立的 MCP Server 配置
- **文件隔离**：工作区提供共享的文件目录，该工作区内的所有会话均可访问
- **附加资源隔离**：工作区级别的附加目录和文件仅对该工作区内的会话生效

启用/禁用 Skill 通过在不同目录间移动实现，Agent 运行时仅扫描启用目录。工作区间若要共享 Skill，支持跨工作区导入并追踪来源版本信息。

### 6.3 工作区与会话的关系

Agent 会话绑定到特定工作区，每个会话在工作区目录下拥有独立的子目录作为其运行时的工作目录。Agent 在执行任务时，所有文件操作（读、写、执行命令）均以该子目录为当前工作目录。同一工作区的多个会话互不干扰，因为各自拥有独立的文件空间。

会话可在工作区之间迁移，迁移后其关联的 Skill 和 MCP 配置会切换到目标工作区的上下文。

每次发送消息时，系统提示词中会注入工作区名称、可用的 MCP Server 列表和 Skill 提示，使 Agent 清楚自身所在的工作区环境。

### 6.4 附加目录与文件

Agent 运行时可配置附加目录，使外部资源可供 Agent 访问而无需拷贝到会话工作目录中。附加目录和文件的配置分为四个层级：

1. **实时附加**：单次消息携带的临时目录，仅当轮有效
2. **会话级附加目录**：持久绑定到特定会话，该会话的所有消息均可访问
3. **会话级附加文件**：附加文件所在的父目录自动纳入可访问范围
4. **工作区级附加资源**：共享给该工作区内所有会话的目录和文件

所有附加资源在 Agent 启动时汇总传递给 SDK，使 Agent 拥有完整的外部文件访问视图，同时保持文件操作限制在受管理的目录范围内。

## 7. 扩展体系

### 6.1 双轨模型

扩展体系采用双轨并行设计：

| 维度 | 单体 Skill | 完整 Plugin |
|------|-----------|------------|
| **安装范围** | 工作区级别 | 全局 |
| **内容** | 单个 Skill 定义 | Skill + 命令 + Agent + MCP 配置 |
| **启用方式** | 按工作区启用/禁用 | 全局启用/禁用 |
| **来源** | 内置模板、企业市场、跨工作区导入 | 内置插件、插件市场安装 |

### 6.2 Skill 体系

Skill 以 Markdown 文件定义，包含元数据和指令内容。运行时 Agent 会自动识别并调用当前工作区启用的 Skill。Skill 支持版本管理，内置 Skill 更新时可提示用户升级。

### 6.3 SkillHub 企业市场

SkillHub 是基于 HTTP 的标准目录服务，提供 Skill 清单浏览和文件下载能力。用户可在应用内浏览市场清单、安装所需 Skill 到工作区。

### 6.4 插件市场

插件市场支持 GitHub、Gitee、GitLab、原始 URL 和本地目录五种来源。插件是包含 Skill、命令、Agent 定义和 MCP 配置的完整扩展包。用户可添加市场、浏览插件、一键安装。

### 6.5 运行时能力汇总

Agent 启动时从工作区 Skill、全局插件 Skill、内置模板和用户引用的 Skill 中汇总完整的能力列表，注入到 SDK 的配置中。

## 8. 权限与安全

### 8.1 API Key 保护

用户输入的明文 Key 经操作系统级安全存储加密后保存为密文，使用时仅在内存中短暂解密，不写入日志或磁盘。UI 层只能获取"是否已配置"的布尔状态，永不可接触明文。

### 8.2 Agent 权限模式

| 模式 | 行为 |
|------|------|
| **完全自动** | 所有工具自动允许，无需确认 |
| **自动审批** | 危险操作需用户确认，可记住选择 |
| **计划模式** | 仅规划不执行，用户审批后才执行 |

权限模式可按会话动态切换。

### 8.3 数据隔离

- 工作区按独立目录隔离
- 会话数据存储在独立文件中
- 渲染进程无法直接访问文件系统
- 动态注入的配置仅单次会话有效

## 9. 数据存储

### 9.1 存储方案

所有数据存储在用户目录下，纯文件组织，无需任何数据库：

- **JSON 文件**：存储索引和配置（渠道、对话索引、设置、用户档案等），采用全量覆写
- **JSONL 文件**：存储消息日志，采用追加写入，流式友好

### 9.2 设计优势

- 纯文件存储，可备份、迁移、版本控制
- 人类可读格式，方便排查问题
- 原子写入保证数据一致性
- 无需额外数据库依赖，降低部署复杂度
