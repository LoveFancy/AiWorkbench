## 技术选型原因

| 选型 | 原因 |
|------|------|
| **Docker** | 标准化运行环境，消除环境差异；便于 CI/CD 集成和快速扩缩容 |
| **PM2** | Node.js 生态成熟的进程管理器，支持 cluster 模式零停机重载、日志管理、崩溃自动重启 |
| **Nginx** | 高性能反向代理，处理静态资源缓存、请求转发、SSL 终结；在企业内部部署中为标准实践 |
| **MySQL 8.0** | 关系型数据库，支持复杂查询；企业中运维成熟度高 |
| **TypeScript** | 类型安全，减少运行时错误；复杂的数据模型（如多表关联的升级策略）受益于静态类型检查 |

## 一、本地启动

本章介绍在本地开发机器上启动服务端和前端管理台的完整流程。

### 1.1 前置条件

在开始前，请确保本地已安装以下软件：

| 软件 | 最低版本 | 验证命令 |
|------|----------|----------|
| Node.js | 18.x（推荐 20.x） | `node -v` |
| 包管理器 | npm / bun | `npm -v` 或 `bun -v` |
| MySQL | 8.0 | `mysql --version` |

### 1.2 首次启动步骤

#### 1）安装服务端依赖

```bash
cd workmate-server
npm install            # 或 bun install
```

#### 2）创建数据库

```bash
# 使用 MySQL 客户端创建数据库
mysql -u root -p <<EOF
CREATE DATABASE workmate_server CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
EOF
```

#### 3）配置环境变量

```bash
# 从模板创建 .env 文件（如没有 .env.example，直接创建）
cat > .env <<'EOF'
PORT=3000
HOST=0.0.0.0
DATABASE_URL="mysql://root:password@127.0.0.1:3306/workmate_server"
USER_ID_ENCRYPTION_KEY=0123456789abcdef0123456789abcdef
REQUIRE_USER_ID=false
DEFAULT_USER_ID=test_user
MODEL_PLATFORM_API_URL=http://model-platform.htsc.com/api/v1
MODEL_PLATFORM_TIMEOUT_MS=10000
OBSERVABILITY_SAMPLE_RATE=1.0
OBSERVABILITY_MAX_EVENTS_PER_MINUTE=60
LOG_LEVEL=info
LOG_DIR=./logs
EOF
```

> 编辑 `.env` 文件，将 `DATABASE_URL` 中的 `password` 改为你的 MySQL 密码。`REQUIRE_USER_ID=false` 在本地开发时跳过 EIP 身份校验。

#### 4）初始化数据库表结构

```bash
npx prisma migrate dev
```

#### 5）写入种子数据

```bash
npx tsx prisma/seed.ts
```

> 种子数据会创建一个初始管理员白名单，工号为配置中指定的用户。

#### 6）启动服务端

```bash
npm run dev     # tsx watch 监听模式下启动，代码修改后自动重启
```

服务端启动成功后终端输出：

```
WorkMate Server 已启动
  host: 0.0.0.0
  port: 3000
  env: development
  endpoints:
    client:  http://0.0.0.0:3000/workmate
    console: http://0.0.0.0:3000/workmate/console
    health:  http://0.0.0.0:3000/workmate/health
```

#### 7）启动管理台前端（另一个终端）

```bash
cd admin
npm install            # 或 bun install
npm run dev            # Vite dev server
```

前端启动成功后的终端输出：

```
VITE v6.4.3  ready in 454 ms

  ➜  Local:   http://localhost:5173/
```

### 1.3 启动后验证

| 验证项 | 操作 | 预期结果 |
|--------|------|----------|
| 后端健康检查 | `curl http://localhost:3000/workmate/health` | `{"status":"ok"}` |
| 模型接口（关闭身份校验时） | `curl http://localhost:3000/workmate/models` | 返回模型 JSON 或空列表 |
| 升级检测 | `curl http://localhost:3000/workmate/upgrade/check` | 返回 `{"code":0}` |
| 管理台前端 | 浏览器打开 `http://localhost:5173` | 可看到管理台仪表盘 |
| 管理台前端（生产构建） | `cd admin && npm run build && cd ..` 后浏览器打开 `http://localhost:3000/workmate/admin` | Express 托管的管理台页面 |

### 1.4 本地开发 Tips

| 场景 | 操作 |
|------|------|
| 代码修改后自动重启 | 服务端使用 `tsx watch`（`npm run dev`），前端 Vite 自带 HMR |
| 重新初始化数据库 | `npx prisma migrate reset`（会清空数据） |
| 运行单元测试 | `npm test`（执行 85 个测试用例） |
| 数据库可视化浏览 | `npx prisma studio`（浏览器打开 `http://localhost:5555`） |
| 查看 Prisma 操作日志 | 在 `.env` 设置 `LOG_LEVEL=debug` |
| `prisma migrate dev` 失败 | 确保 MySQL 正在运行：`systemctl status mysqli` 或 `brew services list | grep mysql` |

### 1.5 常见启动问题

| 问题 | 原因 | 解决 |
|------|------|------|
| `Can't reach database server` | MySQL 未启动或连接信息错误 | `systemctl start mysqli` 并检查 `.env` 中 `DATABASE_URL` |
| `Unknown database 'workmate_server'` | 未创建数据库 | 执行步骤 2 |
| `P1001: Can't reach database` | IP/端口不匹配 | 尝试 `DATABASE_URL="mysql://root:password@localhost:3306/workmate_server"` |
| `USER_ID_ENCRYPTION_KEY must be 16 bytes` | 密钥长度不正确 | 确保 `.env` 中密钥为 32 位 Hex 字符串（即 16 字节） |
| 端口 3000 被占用 | 其他服务占用了 3000 端口 | `.env` 中修改 `PORT=3001` |
| 前端报"缺少用户身份信息" | 未设置 `REQUIRE_USER_ID=false` | 修改 `.env` 后重启服务端 |
| 管理台页面空白 | 未初始化种子数据或白名单缺失 | 执行 `npx tsx prisma/seed.ts` |

## 三、Docker 部署

Docker 将应用及其依赖打包为镜像，在任何安装了 Docker 的环境中一致运行。

### 3.1 安装 Docker

```bash
# ===== Windows / macOS =====
# 下载 Docker Desktop: https://www.docker.com/products/docker-desktop
# 安装后启动 Docker Desktop 即可在终端使用 docker 命令

# ===== Rocky Linux 8/9 =====
sudo dnf install -y dnf-plugins-core
sudo dnf config-manager --add-repo https://download.docker.com/linux/rhel/docker-ce.repo
sudo dnf install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin
sudo systemctl enable docker
sudo systemctl start docker

# ===== CentOS 7 =====
sudo yum install -y yum-utils
sudo yum-config-manager --add-repo https://download.docker.com/linux/centos/docker-ce.repo
sudo yum install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin
sudo systemctl enable docker
sudo systemctl start docker

# ===== 验证安装 =====
docker --version
docker compose version

# ===== 允许非 root 用户执行 docker（可选）=====
sudo usermod -aG docker $USER
# 退出重新登录后生效
```

> Docker Desktop（Windows/macOS）安装后 Docker Compose 已内置。Linux 下 V2 版本用 `docker compose`（空格），V1 用 `docker-compose`（连字符）。

### 3.2 所需文件

| 文件 | 作用 |
|------|------|
| `.dockerignore` | 排除不需要打进镜像的文件，减小镜像体积、加快构建 |
| `Dockerfile` | 镜像构建脚本，定义如何从源码构建出可运行镜像 |
| `docker-compose.yml` | 编排 MySQL + App 两个容器一并启动 |

### 3.3 .dockerignore

```dockerignore
node_modules
dist
admin/dist
.env
.env.test
.git
logs
*.md
!README.md
```

> 注意：`admin/dist` 不忽略——管理台前端构建产物需要打进镜像。如果前端在镜像外构建，则忽略它并在 Dockerfile 中构建。

### 3.4 Dockerfile

```dockerfile
# ===== 阶段 1: 构建阶段 =====
# 使用多阶段构建：先在一个临时容器里编译 TypeScript，再把产物复制到运行容器
FROM node:20-alpine AS builder

WORKDIR /app

# 先复制 package 文件，利用 Docker 缓存层——package 不变时跳过 npm ci
COPY package.json package-lock.json ./
RUN npm ci --production=false

# 复制源码和配置
COPY tsconfig.json ./
COPY prisma/ ./prisma/
COPY src/ ./src/

# 编译 TypeScript → dist/，生成 Prisma Client
RUN npm run build
RUN npx prisma generate

# ===== 阶段 2: 运行阶段 =====
FROM node:20-alpine

WORKDIR /app

# 从构建阶段复制产物
COPY --from=builder /app/dist/ ./dist/
COPY --from=builder /app/node_modules/ ./node_modules/
COPY --from=builder /app/prisma/ ./prisma/
COPY --from=builder /app/package.json ./

# 管理台前端静态文件（需在宿主机先执行 cd admin && npm run build）
COPY admin/dist/ ./public/admin/

EXPOSE 3000

# 启动时先执行数据库迁移，再启动服务
CMD ["sh", "-c", "npx prisma migrate deploy && node dist/index.js"]
```

### 3.5 构建与运行

#### 步骤 1：构建管理台前端

```bash
cd admin
npm install
npm run build     # 产物在 admin/dist/
cd ..
```

#### 步骤 2：构建 Docker 镜像

```bash
# 在 workmate-server 根目录执行
docker build -t workmate-server:latest .

# 查看构建的镜像
docker images | grep workmate-server
```

构建过程说明：
1. 拉取 `node:20-alpine` 基础镜像（~50MB 精简 Linux）
2. 在 builder 容器内 `npm ci` 安装依赖
3. `tsc` 编译 TypeScript → `dist/`，`prisma generate` 生成客户端代码
4. 将 `dist/`、`node_modules/`、`prisma/`、`public/admin/` 复制到最终镜像
5. 最终镜像大小约 200-300MB

#### 步骤 3：运行容器

```bash
# 仅启动应用容器（需已有 MySQL）
docker run -d \
  --name workmate-server \
  -p 3000:3000 \
  -e DATABASE_URL="mysql://root:password@host.docker.internal:3306/workmate_server" \
  -e USER_ID_ENCRYPTION_KEY="0123456789abcdef0123456789abcdef" \
  -e MODEL_PLATFORM_API_URL="http://model-platform.htsc.com/api/v1" \
  -e REQUIRE_USER_ID="true" \
  workmate-server:latest

# 查看日志
docker logs -f workmate-server

# 验证
curl http://localhost:3000/workmate/health
```

> `host.docker.internal` 是 Docker Desktop 提供的特殊 DNS，指向宿主机。Linux 下需用 `--add-host=host.docker.internal:host-gateway` 或直接用宿主机 IP。

### 3.6 Docker Compose（MySQL + App 一键编排）

```yaml
# docker-compose.yml
version: '3.8'

services:
  # MySQL 数据库
  mysql:
    image: mysql:8.0
    container_name: workmate-mysql
    restart: unless-stopped
    environment:
      MYSQL_ROOT_PASSWORD: root_password
      MYSQL_DATABASE: workmate_server
    ports:
      - "3306:3306"
    volumes:
      - mysql_data:/var/lib/mysql        # 数据持久化
    healthcheck:                          # 健康检查，确保 MySQL 就绪后再启 App
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 5

  # WorkMate 后端应用
  app:
    build: .
    container_name: workmate-server
    restart: unless-stopped
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: "mysql://root:root_password@mysql:3306/workmate_server"
      USER_ID_ENCRYPTION_KEY: "0123456789abcdef0123456789abcdef"
      MODEL_PLATFORM_API_URL: "http://model-platform.htsc.com/api/v1"
      REQUIRE_USER_ID: "true"
    depends_on:
      mysql:
        condition: service_healthy       # 等 MySQL 健康检查通过后才启动
    volumes:
      - ./logs:/var/log/workmate-server  # 日志挂载到宿主机

volumes:
  mysql_data:
```

一键启动：

```bash
# 构建镜像 + 启动所有服务（后台运行）
docker compose up -d --build

# 查看运行状态
docker compose ps

# 查看日志
docker compose logs -f app

# 停止
docker compose down

# 停止并删除数据卷（⚠ 会清空数据库）
docker compose down -v
```

### 3.7 常用 Docker 命令速查

```bash
# 查看运行容器
docker ps

# 进入容器调试
docker exec -it workmate-server sh

# 查看实时日志
docker logs -f workmate-server

# 重启容器
docker restart workmate-server

# 删除容器
docker rm -f workmate-server

# 删除镜像
docker rmi workmate-server:latest

# 清理无用镜像和容器
docker system prune -a
```

## 四、PM2 部署

### 4.1 什么是 PM2

PM2 是一个 Node.js 的生产级进程管理器，负责：

- **进程守护** — 应用崩溃后自动重启，保证服务高可用
- **Cluster 模式** — 利用多核 CPU，开启多个进程实例负载均衡（如 `instances: 2` = 两个进程并行处理请求）
- **零停机重载** — `pm2 reload` 逐个重启进程，实现滚动更新，用户无感知
- **日志管理** — 自动收集 stdout/stderr，支持日志轮转，省去手动管理日志文件
- **开机自启** — 配合 `pm2 startup` 生成 systemd 脚本，服务器重启后自动拉起应用

简单说：`node dist/index.js` 只能跑一个进程，挂了就停了；PM2 帮你跑多个进程，挂了自动重启，还能零停机更新。

### 4.2 安装

```bash
npm install -g pm2
```

### 4.3 配置文件

```javascript
// ecosystem.config.cjs
module.exports = {
  apps: [{
    name: 'workmate-server',
    script: './dist/index.js',
    env: { NODE_ENV: 'production', PORT: 3000 },
    instances: 2,
    exec_mode: 'cluster',
    max_memory_restart: '500M',
    log_date_format: 'YYYY-MM-DD HH:mm:ss',
  }],
}
```

### 4.4 常用 PM2 命令

```bash
# 启动
pm2 start ecosystem.config.cjs      # 用配置文件启动
pm2 start dist/index.js --name api  # 直接启动

# 查看
pm2 status            # 进程列表（状态、内存、CPU、重启次数）
pm2 list              # 同上
pm2 show workmate-server  # 查看某个应用的详细信息
pm2 monit             # 实时监控面板（CPU、内存、事件循环延迟）

# 日志
pm2 logs                       # 所有应用日志
pm2 logs workmate-server       # 只看 workmate-server
pm2 logs --lines 200           # 最近 200 行
pm2 flush                      # 清空日志

# 重启
pm2 restart workmate-server    # 硬重启（有短暂中断）
pm2 reload workmate-server     # 零停机重载（生产推荐）

# 停止
pm2 stop workmate-server       # 停止
pm2 delete workmate-server     # 停止并从列表移除

# 开机自启
pm2 save                        # 保存当前进程列表
pm2 startup                     # 生成自启脚本（按提示执行）
```

#### 5.3.6 SELinux 配置（如启用）

如果服务器启用了 SELinux（`getenforce` 输出 `Enforcing`），Nginx 反向代理或应用访问文件系统时可能被拦截。两种处理方式：

**方式一：放行 Nginx 代理（推荐）**

```bash
# 允许 Nginx 建立网络连接（反向代理需要）
sudo setsebool -P httpd_can_network_connect on
```

**方式二：关闭 SELinux（仅测试环境）**

```bash
sudo setenforce 0                # 临时关闭
sudo sed -i 's/SELINUX=enforcing/SELINUX=disabled/' /etc/selinux/config  # 永久关闭
```

### 5.8 Nginx 反向代理配置

```nginx
# /etc/nginx/conf.d/workmate-server.conf

upstream workmate_backend {
    server 127.0.0.1:3000;
    keepalive 64;
}

server {
    listen 80;
    server_name workmate-server.example.com;

    access_log /var/log/nginx/workmate-access.log;
    error_log  /var/log/nginx/workmate-error.log;
    client_max_body_size 5m;

    # API 接口代理
    location /workmate {
        proxy_pass http://workmate_backend;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_pass_request_headers on;
        proxy_connect_timeout 30s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;
    }

    location /workmate/console {
        proxy_pass http://workmate_backend;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_pass_request_headers on;
        proxy_connect_timeout 30s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;
    }

    location /workmate/health {
        proxy_pass http://workmate_backend;
        proxy_set_header Host $host;
    }

    # 管理台前端静态资源
    location /workmate/admin {
        alias /opt/workmate-server/admin/dist;
        index index.html;
        try_files $uri $uri/ /workmate/admin/index.html;

        location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2|ttf|eot)$ {
            expires 30d;
            add_header Cache-Control "public, immutable";
        }
    }

    location = / {
        return 301 /workmate/admin;
    }
}
```

## 七、打包部署（ZIP 分发）

当目标机器无法使用 Docker 或 Git 拉取代码时，可在开发机上编译打包为 ZIP，直接拷贝到目标机器运行。

### 7.1 打包命令

项目内置打包脚本，提供**三种打包模式**，覆盖从同架构部署到跨平台自包含部署的所有场景：

| 模式 | Linux/Mac 命令 | Windows 命令 | 说明 |
|------|---------------|-------------|------|
| **完整包** | `npm run pack` | `npm run pack:win` | 含 `node_modules`，适用于**同架构**部署（Linux→Linux / Win→Win） |
| **自包含包** ⭐ | `npm run pack:linux` | `npm run pack:win:linux` | 含 `node_modules` + 预下载 Linux 引擎，**跨平台部署**（Win/Mac→Linux），目标仅需 Node.js，**解压即运行，无需网络** |
| **精简包** | `npm run pack:lite` | `npm run pack:win:lite` | 不含 `node_modules`，适用于**跨 CPU 架构**（x86→ARM/Kylin），在目标机器上自动安装依赖 |

> 也可直接调用脚本指定版本和目标平台：`bash scripts/pack.sh "" bundled debian-openssl-3.0.x` 或 `powershell -File scripts/pack.ps1 1.2.0 bundled debian-openssl-3.0.x`

> **重要**：`node_modules` 中包含平台相关的原生二进制文件（Prisma 查询引擎 `.node` 文件和 Schema 引擎）。完整包仅限同架构；跨平台请使用自包含包；跨 CPU 架构（如 ARM）请使用精简包。

### 7.2 打包原理

Prisma 需要两类平台相关的原生二进制文件才能运行：

| 引擎 | 用途 | Windows | Linux x64 |
|------|------|---------|-----------|
| 查询引擎 | `@prisma/client` 运行时 SQL 查询 | `query_engine-windows.dll.node` | `libquery_engine-debian-openssl-3.0.x.so.node` |
| Schema 引擎 | `prisma migrate deploy` 数据库迁移 | `schema-engine-windows.exe` | `schema-engine-debian-openssl-3.0.x` |

**自包含包**通过以下机制解决跨平台问题：

1. **`prisma/schema.prisma` 配置 `binaryTargets`** — `prisma generate` 同时生成 Windows 和 Linux 查询引擎到 `node_modules/.prisma/client/`，运行时自动选择正确平台
2. **`@prisma/fetch-engine` 预下载 Schema 引擎** — 使用 Prisma 官方的引擎下载机制，将 Linux Schema 引擎下载到 `engines/` 目录
3. **启动脚本自动注入环境变量** — `start.sh` 设置 `PRISMA_SCHEMA_ENGINE_BINARY` 指向预打包的 Linux Schema 引擎，`prisma migrate deploy` 无需网络下载

### 7.3 打包流程

脚本自动执行以下步骤：

| 步骤 | 完整包 | 自包含包 | 精简包 |
|------|--------|---------|--------|
| 1. `prisma generate` | ✅ | ✅ | ✅ |
| 2. `npm run build` | ✅ | ✅ | ✅ |
| 3. 构建管理台前端 | ✅ | ✅ | ✅ |
| 4. 下载 Linux 引擎 | ❌ | ✅ | ❌ |
| 5. 复制 `node_modules` | ✅ | ✅ | ❌ |
| 6. 复制 `engines/` | ❌ | ✅ | ❌ |
| 7. 压缩打包 | ZIP | ZIP | ZIP |

```bash
# 5. 启动服务（自动执行数据库迁移 + 启动应用）
```

#### 启动脚本说明

**自包含包**的 `start.sh` 会依次执行：

1. 检查 `.env` 文件是否存在
2. 设置 `PRISMA_SCHEMA_ENGINE_BINARY` 和 `PRISMA_QUERY_ENGINE_LIBRARY` 环境变量，指向预打包的 Linux 引擎（**无需网络下载**）
3. `npx prisma migrate deploy` — 使用预打包的 Schema 引擎执行数据库迁移
4. `node dist/index.js` — 启动服务（@prisma/client 自动从 `node_modules/.prisma/client/` 加载 Linux 查询引擎）

**完整包**的 `start.sh` 直接运行 `prisma migrate deploy` + `node dist/index.js`（引擎已在 `node_modules` 中）。

**精简包**的 `start.sh` 首次运行时会自动 `npm install --production` + `npx prisma generate`，再执行迁移和启动。

### 7.8 常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| `Cannot find module '@prisma/client'` | 打包时未执行 `prisma generate` 或 binaryTargets 未配置 | 确认 `prisma/schema.prisma` 含 `binaryTargets`，重新 `npm run pack` |
| `prisma migrate deploy` 报错 | 数据库未创建或 `.env` 中 `DATABASE_URL` 错误 | 先创建数据库，检查连接串 |
| 管理台页面 404 | `public/admin/` 目录缺失 | 确认打包时 `admin/dist/` 已复制到 `public/admin/` |
| `EADDRINUSE: address already in use :::3000` | 端口被占用 | 修改 `.env` 中 `PORT=3001` 或停掉占用进程 |
| `node_modules` 体积太大 | 包含了开发依赖 | 目标机器执行 `npm prune --production` 移除 devDependencies |
| ARM 机器上 Prisma 报 `ELF: wrong class` | x86 的 `node_modules` 在 ARM 上不兼容 | 使用精简包（`pack:lite`），让目标机器自行安装依赖 |
| 自包含包部署后 `prisma migrate deploy` 报 `cannot find schema-engine` | `engines/` 目录缺失或平台不匹配 | 确认打包时使用了 `bundled` 模式，且目标平台参数正确 |
| `npm install` 在 ARM 上卡住 | 网络不通或 npm 源慢 | 配置内网 npm 镜像：`npm config set registry https://your-internal-registry/` |

### 7.9 麒麟 V10 ARM 部署

银河麒麟 V10（Kylin V10）是国产 Linux 操作系统，运行在 ARM（aarch64）芯片上。WorkMate Server 完全支持在该平台部署，只需注意 Node.js 安装方式和依赖编译。

#### 7.9.1 环境安装

```bash
# ===== 确认系统架构 =====
uname -m
# 预期输出: aarch64

cat /etc/os-release
# 预期包含: Kylin Linux Advanced Server V10

# ===== 安装 Node.js 20.x (ARM64) =====
# 方式1: 官方二进制包（推荐，无需编译）
curl -fsSL https://nodejs.org/dist/v20.18.0/node-v20.18.0-linux-arm64.tar.xz -o node.tar.xz
sudo tar -xf node.tar.xz -C /usr/local --strip-components=1
node -v  # v20.18.0

# 方式2: 如果有内网 npm 镜像，可通过 nvm 安装
# export NVM_NODEJS_ORG_MIRROR=https://your-mirror/nodejs
# nvm install 20

# ===== 安装 MySQL 8.0 =====
# 麒麟 V10 通常自带或可从仓库安装
sudo yum install -y mysql-server
# 或
sudo dnf install -y mysql-server

sudo systemctl enable mysqld
sudo systemctl start mysqld

# 初始化数据库
mysql -u root -p <<EOF
CREATE DATABASE workmate_server CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'workmate'@'127.0.0.1' IDENTIFIED BY 'your_password';
GRANT ALL PRIVILEGES ON workmate_server.* TO 'workmate'@'127.0.0.1';
FLUSH PRIVILEGES;
EOF
```

#### 7.9.2 打包与部署

在 x86 开发机上使用**精简包**模式打包（不含 `node_modules`）：

```bash
# Windows 开发机
npm run pack:win:lite

# Linux/macOS 开发机
npm run pack:lite
```

产物为 `workmate-server-{版本}-lite.zip`，体积约 5-10MB（不含依赖）。

上传到麒麟机器后：

```bash
# 1. 上传并解压
scp dist-pack/workmate-server-1.0.0-lite.zip user@kylin-host:/opt/
ssh user@kylin-host
cd /opt && unzip workmate-server-1.0.0-lite.zip
cd workmate-server-1.0.0-lite

# 2. 配置环境变量
cp .env.example .env
vi .env
# 修改 DATABASE_URL 等配置

# 3. 启动（精简包的 start.sh 会自动安装依赖）
bash start.sh
```

`start.sh` 首次运行时自动执行：
1. `npm install --production` — 安装生产依赖（ARM 版本）
2. `npx prisma generate` — 生成 ARM 版 Prisma Client（ARM 上的 binaryTargets `native` = ARM64）
3. `npx prisma migrate deploy` — 执行数据库迁移
4. `node dist/index.js` — 启动服务

> 如果目标机器无法访问外网 npm 源，需提前配置内网镜像：
> ```bash
> npm config set registry https://your-internal-npm-registry/
> ```

#### 7.9.3 离线部署（无外网）

如果麒麟机器完全无法访问外网，需要在**有网络的 ARM 机器**上预装依赖后再打包：

```bash
# 在一台有外网的 ARM 机器上
cd workmate-server-1.0.0
npm install --production
npx prisma generate
tar -czf workmate-server-1.0.0-arm64.tar.gz .

# 将 tar.gz 拷贝到目标机器
scp workmate-server-1.0.0-arm64.tar.gz user@offline-kylin:/opt/
```
