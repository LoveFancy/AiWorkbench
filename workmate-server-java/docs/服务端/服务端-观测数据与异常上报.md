## 一、错误指纹（Error Fingerprinting）


**问题**：相同的错误因不同用户、不同时间上报产生大量重复记录，难以统计和聚合。

**优化**：在 `observability_events` 表增加 `error_fingerprint` 字段。

```prisma
model ObservabilityEvent {
  // ... 现有字段 ...
  errorFingerprint String? @map("error_fingerprint") @db.VarChar(64)  // 错误指纹，用于聚合相同错误
  breadcrumbs      String? @db.Text  // JSON: 事件发生前的用户操作路径
  tags             String? @db.Text  // JSON: 自定义标签 key:value 对

  @@index([errorFingerprint])
}
```

**指纹生成算法**（服务端）：

```typescript
import { createHash } from 'node:crypto'

function generateErrorFingerprint(event: { errorType?: string; errorMessage?: string; errorStack?: string }): string {
  const raw = [event.errorType ?? '', extractTopFrame(event.errorStack) ?? '', event.errorMessage ?? ''].join('|')
  return createHash('md5').update(raw).digest('hex').slice(0, 16)
}

function extractTopFrame(stack?: string): string | null {
  if (!stack) return null
  const match = stack.match(/at\s+(.+?)(?:\s+\(|$)/)
  return match ? match[1] : null
}
```

**客户端侧**：在 `error` 事件中增加 `fingerprint` 字段，由客户端根据 `error.type + error.message` 生成，服务端再根据 `errorType + errorStack 第一帧` 补全校验。

## 二、面包屑（Breadcrumbs）

**问题**：仅捕获异常堆栈难以还原用户操作上下文。

**优化**：客户端维护最近 20 条面包屑（用户操作记录），在异常上报时附加。

```typescript
export interface Breadcrumb {
  type: 'navigation' | 'user_action' | 'system' | 'http'
  category: string       // 如 'ui.click', 'navigation', 'api.call'
  message: string        // 人类可读描述
  timestamp: number
  data?: Record<string, unknown>  // 附加数据（URL、按钮ID等）
}
```

客户端 `ObservabilityService` 增加面包屑功能：

```typescript
private breadcrumbs: Breadcrumb[] = []
private static MAX_BREADCRUMBS = 20

addBreadcrumb(crumb: Breadcrumb): void {
  this.breadcrumbs.push(crumb)
  if (this.breadcrumbs.length > ObservabilityService.MAX_BREADCRUMBS) {
    this.breadcrumbs.shift()
  }
}

reportErrorEvent(error: Error, context?: Record<string, unknown>): void {
  this.reportEvent({
    type: 'error',
    error: { type: error.name, message: error.message, stack: error.stack },
    breadcrumbs: [...this.breadcrumbs],   // 附加面包屑
    ...context,
  })
}
```

集成点：
- 路由切换：`addBreadcrumb({ type: 'navigation', category: 'route', message: '进入设置页' })`
- 按钮点击：`addBreadcrumb({ type: 'user_action', category: 'ui.click', message: '点击发送按钮' })`
- API 调用：`addBreadcrumb({ type: 'http', category: 'api.call', message: 'POST /workmate/observability/events' })`

## 三、客户端上报采样与限流

**问题**：高并发场景下客户端可能产生大量事件，冲击服务端。

**优化**：客户端增加采样率和限流机制。

```typescript
export interface ObservabilityConfig {
  enabled: boolean
  url: string              // 完整上报地址，如 https://host/workmate/observability/events
  timeoutMs?: number
  maxQueueSize?: number
  flushIntervalMs?: number
  maxBatchSize?: number
  sampleRate?: number         // 采样率 0.0~1.0，默认 1.0（全量上报）
  maxEventsPerMinute?: number // 每分钟最大事件数，默认 60
  enableBreadcrumbs?: boolean // 是否启用面包屑，默认 true
}
```

**采样逻辑**：

```typescript
private eventsThisMinute = 0
private minuteWindowStart = Date.now()

reportEvent(item: ObservabilityEventItem): void {
  // error 类型始终上报，不受采样率和限流影响
  if (item.type !== 'error') {
    // 限流：每分钟上限
    const now = Date.now()
    if (now - this.minuteWindowStart > 60_000) {
      this.eventsThisMinute = 0
      this.minuteWindowStart = now
    }
    this.eventsThisMinute++
    if (this.eventsThisMinute > (this.config.maxEventsPerMinute ?? 60)) return

    // 采样
    if (Math.random() > (this.config.sampleRate ?? 1.0)) return
  }

  this.eventQueue.push(item)
  if (this.eventQueue.length >= (this.config.maxQueueSize ?? 100)) {
    this.flushQueue()
  }
}
```

**注意**：`error` 类型事件始终全量上报，不受采样率和限流影响（仅受队列大小限制）。

## 四、服务端事件去重

**问题**：客户端可能因网络超时重试导致同一事件重复写入。

**优化**：客户端为每个事件生成 `eventId`，服务端做幂等去重。

```typescript
// 客户端
import { randomUUID } from 'node:crypto'

export interface ObservabilityEventItem {
  eventId: string          // 事件唯一标识，用于服务端去重
  type: ObservabilityEventType
  // ... 其余字段 ...
}

// 服务端（observability.service.ts）
// 注：双表路由的完整 writeEvents 实现见第 5.6 节；
//     此处仅展示原始单表去重逻辑示意，实际落地以双表版本为准。
async function writeEvents(
  events: ObservabilityEventItem[],
  jobId: string,
): Promise<{ received: number; duplicated: number }> {
  const existingIds = await prisma.observabilityEvent.findMany({
    where: { eventId: { in: events.map(e => e.eventId).filter(Boolean) } },
    select: { eventId: true },
  })
  const existingSet = new Set(existingIds.map(e => e.eventId))
  const newEvents = events.filter(e => !existingSet.has(e.eventId))

  if (newEvents.length > 0) {
    await prisma.observabilityEvent.createMany({ data: newEvents.map(e => transformEvent(e, jobId)) })
  }

  return { received: events.length, duplicated: events.length - newEvents.length }
}

/**
 * 客户端事件 → DB 行的转换器（双表版本见下方 transformBusinessEvent / transformErrorEvent）。
 * 关键策略：
 *  1) created_at 使用客户端 timestamp，保留事件真实发生时间（事件可能在磁盘缓存中滞留）
 *  2) error_fingerprint 服务端统一重算并覆盖，避免客户端实现偏差导致聚合失效
 *  3) user_id 用网关 jobId 覆盖，保证身份不可伪造；登录失败等无 jobId 场景退回 'unknown'
 *  4) breadcrumbs / tags 由对象序列化为 JSON 字符串后入库
 */
function transformEvent(e: ObservabilityEventItem, jobId: string) {
  // 服务端重算 fingerprint，统一聚合口径（覆盖客户端传入值）
  const errorFingerprint = e.error
    ? generateErrorFingerprint({
        errorType: e.error.type,
        errorMessage: e.error.message,
        errorStack: e.error.stack,
      })
    : null

  return {
    eventId: e.eventId,
    userId: jobId || e.userId || 'unknown',     // jobId 优先；无 jobId（如登录失败）退回客户端值或 unknown
    eventType: e.type,
    question: e.question ?? null,
    questionLength: e.questionLength ?? null,
    modelId: e.modelId ?? null,
    channelId: e.channelId ?? null,
    sessionId: e.sessionId ?? null,
    workspaceId: e.workspaceId ?? null,
    result: e.result ?? null,
    responseDurationMs: e.responseDurationMs ?? null,
    errorType: e.error?.type ?? null,
    errorMessage: e.error?.message ?? null,
    errorStack: e.error?.stack ?? null,
    errorStatusCode: e.error?.statusCode ?? null,
    errorFingerprint,                            // 服务端重算后的值，覆盖客户端 e.error.fingerprint
    breadcrumbs: e.breadcrumbs ? JSON.stringify(e.breadcrumbs) : null,
    tags: e.tags ? JSON.stringify(e.tags) : null,
    clientVersion: e.client.appVersion,
    clientPlatform: e.client.platform,
    clientOsVersion: e.client.osVersion ?? null,
    createdAt: new Date(e.timestamp),            // 使用客户端事件发生时间，而非 now()
  }
}

// ===== 双表路由的转换器（供第 5.6 节 writeEvents 调用） =====

/** 业务事件 → observability_events 行（无 error/question/stack 字段） */
function transformBusinessEvent(e: ObservabilityEventItem, jobId: string) {
  return {
    eventId: e.eventId,
    userId: jobId || e.userId || 'unknown',
    eventType: e.type,                           // user_login|user_logout|chat_question|agent_question|upgrade_check
    questionLength: e.questionLength ?? null,    // 仅长度，不存内容（隐私合规）
    modelId: e.modelId ?? null,
    channelId: e.channelId ?? null,
    sessionId: e.sessionId ?? null,
    workspaceId: e.workspaceId ?? null,
    result: e.result ?? null,
    responseDurationMs: e.responseDurationMs ?? null,
    clientVersion: e.client.appVersion,
    clientPlatform: e.client.platform,
    clientOsVersion: e.client.osVersion ?? null,
    createdAt: new Date(e.timestamp),
  }
}

/** 异常事件 → observability_errors 行（stack 截断到 1000B；breadcrumbs 已由客户端按 source 差异化存储） */
function transformErrorEvent(e: ObservabilityEventItem, jobId: string) {
  const errorFingerprint = e.error
    ? generateErrorFingerprint({ errorType: e.error.type, errorMessage: e.error.message })
    : null

  return {
    eventId: e.eventId,
    userId: jobId || e.userId || 'unknown',
    sessionId: e.sessionId ?? null,
    workspaceId: e.workspaceId ?? null,
    errorType: e.error?.type ?? null,
    errorMessage: e.error?.message ?? null,
    errorStack: truncateStack(e.error?.stack, 1000),  // 按字节截断到 1000B，保留顶部帧
    errorFingerprint,
    errorStatusCode: e.error?.statusCode ?? null,
    breadcrumbs: e.breadcrumbs ? JSON.stringify(e.breadcrumbs) : null,
    tags: e.tags ? JSON.stringify(e.tags) : null,
    clientVersion: e.client.appVersion,
    clientPlatform: e.client.platform,
    clientOsVersion: e.client.osVersion ?? null,
    createdAt: new Date(e.timestamp),
  }
}

/**
 * 堆栈按字节截断（UTF-8 安全）。
 * 优先保留前 N 字节（最贴近错误抛出位置的调用帧），尾部加 '...[truncated]' 标记。
 * 中文字符按 UTF-8 字节计算，不会截断到字符中间。
 */
function truncateStack(stack: string | undefined | null, maxBytes: number): string | null {
  if (!stack) return null
  const buf = Buffer.from(stack, 'utf-8')
  if (buf.length <= maxBytes) return stack
  // 截断到 maxBytes-15 字节，留出 '...[truncated]' 的空间
  const sliced = buf.subarray(0, maxBytes - 15).toString('utf-8')
  return sliced + '...[truncated]'
}
```

> **注**：`eventId` 字段设为 `UNIQUE` 索引，实际实现中可使用 `INSERT IGNORE` / `ON DUPLICATE KEY UPDATE` 实现高效去重。
>
> **身份覆盖策略**：当 `req.jobId` 存在时强制覆盖 `user_id`（防伪造）；仅当 `jobId` 缺失（如 `user_login` result=failure 时网关可能未注入 `X-EIPGW-USERID`）才退回客户端上报值或 `'unknown'`，避免登录失败事件丢失上下文。
>
> **created_at 策略**：客户端事件可能在磁盘缓存中滞留数小时（断网/重启场景），服务端必须使用 `event.timestamp` 转换写入 `created_at`，否则会丢失真实事件发生时间。
>
> **fingerprint 策略**：服务端在 `transformEvent` 中统一重算 `errorFingerprint` 并覆盖客户端传入值。客户端算法（`md5(type:message)`）与服务端算法（`md5(type|stackTopFrame|message)`）不一致，必须以服务端为准才能正确聚合相同错误。

## 五、数据模型最终版

### 5.1 拆表策略

为优化存储与运维，将事件拆分为**两张表**，均采用 RANGE 分区：

| 表名 | 内容 | 保留期 | 分区策略 | 行格式 |
|------|------|-------|---------|-------|
| `observability_events` | 业务事件：`user_login` / `user_logout` / `chat_question` / `agent_question` / `upgrade_check` | 永久（用于长期统计） | **按年 RANGE 分区** | `COMPRESSED` |
| `observability_errors` | 异常事件：`error` 类型 | **仅保留 6 个月**（定期 DROP PARTITION） | **按月 RANGE 分区** | `COMPRESSED` |

**拆分理由**：
- **生命周期不同**：业务事件用于长期统计与用户行为分析需永久留存；异常事件只用于近期排障，超过 6 个月失去意义
- **行宽差异巨大**：error 单行含 `error_stack`（截断到 1000B）+ `breadcrumbs` 等大字段平均 ~1700 B，业务事件仅 ~350 B，差 5 倍。分开后业务表可保持高密度小行
- **删除高效**：异常表按月分区，`DROP PARTITION` 秒级释放空间；混在一起 `DELETE WHERE` 慢且不释放磁盘
- **索引精简**：业务表不需要 `error_fingerprint` 索引，异常表不需要 `model_id` 索引

**分区粒度选择**：
- **业务事件按年**：单年数据量可控（~13 GB 压缩后），管理台默认按"当前年份"查询，跨年查询时只扫描 1~2 个分区；分区数量少便于维护
- **异常事件按月**：6 个月保留期下分区数 ≤ 8 个，`DROP PARTITION` 粒度细，便于精细化清理

**查询约定**：所有管理台查询接口**强制带年度范围**（默认当年，可切换历史年份），通过分区裁剪将单次查询锁定在 1 个分区内，避免跨分区全表扫描。详见 §6.2。

### 5.2 业务事件表 `observability_events`（按年分区 + 永久保留）

```
┌──────────────────────────────────────┐
│         observability_events         │
├──────────────────────────────────────┤
│ id                  BIGINT           │  ← 与 created_at 组成 PK

│ user_id             VARCHAR(64)      │  ← 服务端 req.jobId 覆盖
│ event_type          VARCHAR(32)      │  ← user_login|user_logout|chat_question|agent_question|upgrade_check
│ question_length     INT              │  ← 仅长度，不存内容（隐私合规）
│ model_id            VARCHAR(128)     │
│ channel_id          VARCHAR(128)     │

└──────────────────────────────────────┘
```

**DDL**（按年 RANGE 分区 + COMPRESSED 行格式）：

```sql
CREATE TABLE observability_events (
  id                    BIGINT NOT NULL AUTO_INCREMENT,
  event_id              VARCHAR(36) NOT NULL,
  user_id               VARCHAR(64) NOT NULL,
  event_type            VARCHAR(32) NOT NULL,
  question_length       INT NULL,
  model_id              VARCHAR(128) NULL,
  channel_id            VARCHAR(128) NULL,
  session_id            VARCHAR(64) NULL,

  result                VARCHAR(16) NULL,
  response_duration_ms  INT NULL,
  client_version        VARCHAR(32) NOT NULL,
  client_platform       VARCHAR(32) NOT NULL,
  client_os_version     VARCHAR(64) NULL,
  created_at            DATETIME NOT NULL,
  -- 分区表约束：分区键必须包含在所有 UNIQUE 与 PRIMARY KEY 中
  PRIMARY KEY (id, created_at),
  UNIQUE KEY uk_event_id (event_id, created_at),
  KEY idx_event_type_created (event_type, created_at),
  KEY idx_user_id_created (user_id, created_at),
  KEY idx_created_at (created_at),
  KEY idx_client_version (client_version)

ENGINE=InnoDB
ROW_FORMAT=COMPRESSED                  -- 压缩存储，节省 50%~70% 空间
KEY_BLOCK_SIZE=8                       -- 8KB 压缩块，权衡 CPU 与压缩率
DEFAULT CHARSET=utf8mb4
PARTITION BY RANGE (YEAR(created_at)) (
  PARTITION p_2026 VALUES LESS THAN (2027),
  PARTITION p_2027 VALUES LESS THAN (2028),
  PARTITION p_2028 VALUES LESS THAN (2029),
  PARTITION p_2029 VALUES LESS THAN (2030),
  PARTITION p_2030 VALUES LESS THAN (2031),
  PARTITION p_max  VALUES LESS THAN MAXVALUE
);
```

> **分区裁剪生效条件**：查询必须带 `WHERE created_at >= 'YYYY-01-01' AND created_at < 'YYYY+1-01-01'` 形式的范围条件，MySQL 才会跳过其他年份分区。管理台所有查询接口已强制约定该形式（详见 §6.2）。

> **error_stack 截断策略**：堆栈对排障非常关键（指出错误抛出的代码位置），但完整堆栈通常 2KB+ 且重复度高。服务端在 `transformErrorEvent` 中**按字节截断到 1000 B**（约保留前 5~8 帧），既能定位关键位置又能控制存储。截断时保留最顶部帧（最贴近错误的调用栈）。

### 5.4 自动化分区维护（定时任务）

#### 业务事件表 `observability_events`（按年）
每年 11 月 1 日执行：**新增下一年分区**（永不删除）。

```sql
-- 添加下一年分区（提前 2 个月预留）
ALTER TABLE observability_events
  REORGANIZE PARTITION p_max INTO (
    PARTITION p_2031 VALUES LESS THAN (2032),
    PARTITION p_max  VALUES LESS THAN MAXVALUE
  );
```

#### 异常事件表 `observability_errors`（按月）
每月 1 日 02:00 执行：**新增下月分区 + 删除 6 个月前的分区**。

```sql
-- 1) 添加下下月分区（提前一个月预留，避免月初无分区可用）
ALTER TABLE observability_errors
  REORGANIZE PARTITION p_max INTO (
    PARTITION p_202608 VALUES LESS THAN (TO_DAYS('2026-09-01')),
    PARTITION p_max    VALUES LESS THAN MAXVALUE
  );

-- 2) 删除 6 个月前的分区（秒级释放空间，远比 DELETE 高效）
ALTER TABLE observability_errors DROP PARTITION p_202512;
```

> **运维建议**：通过 `cron` / `xxl-job` 调度执行，并对分区数监控告警（持续无新增分区会导致写入失败）。
> **关键告警**：业务表 `p_max` 分区不能积累数据（说明定时任务没执行 / 跨年了未及时建分区）；可通过 `SELECT partition_name, table_rows FROM information_schema.partitions WHERE table_name='observability_events' AND partition_name='p_max'` 监控。

### 5.6 写入路由（writeEvents 适配双表）

```typescript
async function writeEvents(
  events: ObservabilityEventItem[],
  jobId: string,
): Promise<{ received: number; duplicated: number }> {
  // 1) 按事件类型分流到不同表
  const errorEvents = events.filter(e => e.type === 'error')
  const businessEvents = events.filter(e => e.type !== 'error')

  // 2) 各自做去重
  const [existingEventIds, existingErrorIds] = await Promise.all([
    businessEvents.length === 0 ? [] : prisma.observabilityEvent.findMany({
      where: { eventId: { in: businessEvents.map(e => e.eventId) } },
      select: { eventId: true },
    }),
    errorEvents.length === 0 ? [] : prisma.observabilityError.findMany({
      where: { eventId: { in: errorEvents.map(e => e.eventId) } },
      select: { eventId: true },
    }),
  ])
  const eventDup = new Set(existingEventIds.map(e => e.eventId))
  const errorDup = new Set(existingErrorIds.map(e => e.eventId))
  const newBusiness = businessEvents.filter(e => !eventDup.has(e.eventId))
  const newErrors   = errorEvents.filter(e => !errorDup.has(e.eventId))

  // 3) 并行批量插入
  await Promise.all([
    newBusiness.length > 0 && prisma.observabilityEvent.createMany({
      data: newBusiness.map(e => transformBusinessEvent(e, jobId)),
    }),
    newErrors.length > 0 && prisma.observabilityError.createMany({
      data: newErrors.map(e => transformErrorEvent(e, jobId)),
    }),
  ])

  const received = events.length
  const duplicated = received - newBusiness.length - newErrors.length
  return { received, duplicated }
}
```

### 5.7 存储估算（启用 COMPRESSED 后）

`ROW_FORMAT=COMPRESSED + KEY_BLOCK_SIZE=8` 对 TEXT/VARCHAR 字段压缩率约 **50%~70%**（实际取决于内容重复度）。

按真实模型（15000 注册 / 30% DAU / 人均 30 提问 / 5% 异常率）：

| 表 | 单行原始 | 压缩后单行 | 日事件数 | 存储 | 分区 / 保留 |
|----|---------|----------|---------|------|------------|
| `observability_events` | ~350 B | **~150 B** | ~13.5 万 | **~7 GB / 年** | 按年分区，永久 |
| `observability_errors` | ~2700 B（含 1000B stack） | **~1000 B** | ~6,750 | **~200 MB / 月** = **~1.2 GB / 6 月** | 按月分区，仅 6 月 |
| **合计（首年）** | | | | **~8.2 GB** | |
| **合计（5 年）** | | | | **~36 GB**（业务 35 GB + 异常 1.2 GB 滚动） | |

5 年累积仅 ~36 GB，MySQL 单实例轻松承载。即使按年扩展，单年分区也远低于 50 GB 警戒线。

> **写入开销**：`ROW_FORMAT=COMPRESSED` 写入时 CPU 开销增加约 10%~20%，但在当前 QPS（~2）下完全可忽略。
> **stack 字段成本**：每条 error 增加 ~400 B 压缩后存储，6 个月异常表多用 ~280 MB —— 换来排障定位能力，性价比极高。

## 七、相关数据模型

完整表定义见**第五节 数据模型最终版**，此处列关键要点：

### observability_events（业务事件表）
- `event_id` VARCHAR(36) — 与 `created_at` 组成 UNIQUE，用于去重
- `user_id` VARCHAR(64) — 服务端从 X-EIPGW-USERID 解密获取，覆盖客户端上报值
- `event_type` 可选值：`user_login` / `user_logout` / `chat_question` / `agent_question` / `upgrade_check`
- `question` 字段不存储（隐私合规，仅存储 `question_length`）
- 行格式：`ROW_FORMAT=COMPRESSED`，**按年 RANGE 分区**，永久保留

### observability_errors（异常事件表）
- `event_id` VARCHAR(36) — 与 `created_at` 组成 UNIQUE，用于去重
- `error_stack` VARCHAR(1000) — 堆栈，**服务端截断到 1000 字节**入库
- `error_fingerprint` VARCHAR(64) — 错误指纹，聚合相同错误
- `breadcrumbs` TEXT — JSON，事件发生前的用户操作路径
- 行格式：`ROW_FORMAT=COMPRESSED`，**按月 RANGE 分区**，保留 6 个月
