推荐采用方案 A：保留当前 `lib/updater` 模块、IPC 通道、preload API、Jotai atom 和现有 UI 入口，内部替换检测接口与下载实现。


```ts

scheduleRandomChecks()
loadDownloadedInstallerManifest()
```

```text

  -> updateStatusAtom

  -> 继续消费 updater 状态
```

## 三、问题与解决方案清单

### 1. 配置字段与方案描述不一致

**问题**

升级方案写的是按 `workmate.json` 的 `workmate.serverUrl` 判断启用 WorkMate 升级检测；当前客户端代码里已有的配置字段是 `settings.workmateServerUrl`，参考 `model-service.ts` 的读取方式。

**影响**

如果实现时按文档新增 `workmate.serverUrl`，会产生两套配置来源，模型服务与升级服务可能使用不同的服务器地址。

**方案是否已有解决**

原方案提到了“按 serverUrl 配置互斥”，但没有和当前客户端实际字段 `workmateServerUrl` 对齐。

**建议解决方案**

统一使用现有 `settings.workmateServerUrl`。升级服务读取逻辑参考 `apps/electron/src/models/model-service.ts`：

```ts
const settingsPath = getSettingsPath()
const settings = JSON.parse(readFileSync(settingsPath, 'utf-8'))
const serverUrl = settings.workmateServerUrl?.trim()
```

**建议落地位置**

- `apps/electron/src/main/lib/updater/auto-updater.ts`
- 或新增 `apps/electron/src/main/lib/updater/workmate-upgrade-service.ts`

### 2. 目录结构与“最小侵入”目标冲突

**问题**

原方案建议新增 `apps/electron/src/upgrade/` 独立目录，并新建 IPC、preload、atoms、弹窗组件。但当前项目已经有完整升级链路：`main/lib/updater`、`updater-ipc`、`preload.updater`、`renderer/atoms/updater.ts`、`UpdateDialog.tsx`。

**影响**

严格按原方案新建目录，会导致两套升级前后端桥接并存，实际改动面更大，不符合“尽可能适配当前逻辑”的目标。

**方案是否已有解决**

原方案在末尾提到若干当前文件“可能修改/不动”，但主体设计仍是全新 `src/upgrade` 模块。

**建议解决方案**

按方案 A 收敛：保留当前 `lib/updater` 模块，只在内部增加 WorkMate Server 分支。新增服务文件时放到：

```text
apps/electron/src/main/lib/updater/workmate-upgrade-service.ts
```

而不是新增完整的 `apps/electron/src/upgrade/` 前后端链路。

**建议落地位置**

- 修改：`apps/electron/src/main/lib/updater/auto-updater.ts`
- 新增：`apps/electron/src/main/lib/updater/workmate-upgrade-service.ts`
- 修改：`apps/electron/src/main/lib/updater/updater-types.ts`

### 3. 当前版本关系校验不足

**问题**

当前实现使用 `electron-updater.checkForUpdates()`，版本比较由 `electron-updater` 内部完成，只支持常规升级，即远端版本大于当前版本。当前业务代码没有显式校验：

- `UPGRADE` 必须 `latestVersion > currentVersion`
- `ROLLBACK` 必须 `latestVersion < currentVersion`
- 安装包真实版本必须等于检测结果中的 `latestVersion`

**影响**

WorkMate 灰度升级支持回退，如果只沿用 `electron-updater` 的判断模型，就无法支持 `ROLLBACK`，也无法防止服务端异常数据或安装包替换。

**方案是否已有解决**

原方案 §5.4 已写端侧二次校验，§4.2 已写安装前版本校验。需要在方案 A 中明确落地到当前 updater 内部。

**建议解决方案**

新增 `validateVersionDirection`，检测返回后立即校验：

```ts
if (releaseType === 'UPGRADE') {
  return latestVersion > currentVersion
}
if (releaseType === 'ROLLBACK') {
  return latestVersion < currentVersion
}
return false
```

安装前再次校验：

```text
1. 本地安装包存在
2. 安装包真实版本 == latestVersion
3. UPGRADE 时安装包版本 > 当前版本
4. ROLLBACK 时安装包版本 < 当前版本
```

任一失败：进入 `error` 状态，不执行安装，并删除异常安装包。

**建议落地位置**

- `apps/electron/src/main/lib/updater/workmate-upgrade-service.ts`
- `apps/electron/src/main/lib/updater/auto-updater.ts`

### 4. 安装包版本解析方案不够可靠

**问题**

原方案代码示例中 `verifyInstallerVersion` 通过文件名 `WorkMate-1.2.0.exe` 提取版本。这个方式简单，但安全性和可靠性不足：文件名可以伪造，且无法证明安装包内部版本一致。

**影响**

如果下载地址内容被替换，文件名仍可能保持正确，客户端会误判通过。

**方案是否已有解决**

原方案已说明“生产环境应使用 rcedit / electron-builder 元数据校验”，但示例代码是简化实现。

**建议解决方案**

分两期处理：

一期最小落地：

- 使用文件名提取版本作为过渡实现。
- 同时要求服务端返回 `sha256`，客户端下载后校验 hash。

二期增强：

- Windows：读取 EXE PE 版本信息。
- macOS：读取安装包内 `.app/Contents/Info.plist` 的 `CFBundleShortVersionString`。
- Linux：按最终包格式补充 AppImage/deb/rpm 版本读取规则。

**建议落地位置**

- 类型扩展：`apps/electron/src/main/lib/updater/updater-types.ts`
- 下载校验：`apps/electron/src/main/lib/updater/workmate-upgrade-service.ts`
- 服务端接口字段：`sha256` 或 `checksum`

### 5. `quitAndInstall` 语义变化较大

**问题**

当前 `quitAndInstall()` 调用的是 `autoUpdater.quitAndInstall(true, true)`，这是 electron-updater 的内置自更新流程。WorkMate 方案改为 `shell.openPath(installerPath)` 启动外部安装器。

**影响**

“立即重启更新”不再准确。对于 Windows NSIS 安装器，如果当前应用仍在运行，安装可能失败或需要用户手动关闭应用。

**方案是否已有解决**

原方案提到 `shell.openPath()`，但没有明确是否需要先退出当前应用。

**建议解决方案**

保留 IPC 名称 `updater:quit-and-install` 以减少改动，但内部改成：

```text
1. 安装前校验
2. shell.openPath(installerPath)
3. 若启动安装器成功：
   - 标记应用正在退出
   - 移除窗口 close 拦截
   - app.quit()
4. 若启动失败：推送 error 状态
```

UI 文案从“立即重启更新”改为“立即安装”。

**建议落地位置**

- `apps/electron/src/main/lib/updater/auto-updater.ts`
- `apps/electron/src/renderer/components/settings/UpdateDialog.tsx`
- `apps/electron/src/renderer/components/settings/AboutSettings.tsx`

### 6. 下载完成后的“下次启动自动提示”缺少持久化设计

**问题**

当前 `currentStatus` 是内存变量，应用退出后会丢失。原方案要求用户点“下次安装”后安装包保留，下次启动自动提示安装。

**影响**

如果只复用当前内存状态，应用重启后不知道本地已有可安装包，无法自动提示。

**方案是否已有解决**

原方案描述了行为，但未明确持久化文件和恢复流程。

**建议解决方案**

在 `userData/upgrade/` 下保存 manifest：

```json
{
  "version": "1.2.0",
  "releaseType": "UPGRADE",
  "forceUpdate": false,
  "installerPath": ".../WorkMate-1.2.0.exe",
  "releaseNotes": "...",
  "downloadedAt": 1717300000000,
  "sha256": "..."
}
```

应用启动时：

```text
1. 读取 manifest
2. 检查 installerPath 是否存在
3. 校验版本/hash
4. 校验方向仍有效
5. 通过则直接 setStatus(downloaded)，渲染端弹窗提示安装
6. 不通过则删除 manifest 和安装包，重新检测
```

**建议落地位置**

- `apps/electron/src/main/lib/updater/workmate-upgrade-service.ts`
- `userData/upgrade/manifest.json`

### 7. 随机检测策略与当前实现不一致

**问题**

当前实现是固定 10 秒首检、固定 4 小时周期检测。原方案要求：

- 首次：10 秒到 60 分钟随机
- 周期：6 小时到 8 小时随机抖动

**影响**

如果保留现状，所有客户端在启动后 10 秒集中请求，无法达到错峰目标。

**方案是否已有解决**

原方案已有随机算法，但需要替换当前 `setTimeout` 和 `setInterval`。

**建议解决方案**

不要使用固定 `setInterval`，改成递归 `setTimeout`：

```text
scheduleNextCheck(delay)
  -> setTimeout(check)
  -> check 完成后重新计算 6h~8h delay
```

这样每次周期都重新随机，而不是启动时只随机一次。

**建议落地位置**

- `apps/electron/src/main/lib/updater/auto-updater.ts`
- 或 `apps/electron/src/main/lib/updater/workmate-upgrade-service.ts`

### 8. 登录成功后触发检测没有落点

**问题**

原方案要求“登录成功 10 秒后立即检查一次”。当前升级模块和认证模块没有连接点。升级检测虽然可以通过 `getToken()` 判断是否登录，但无法感知“刚登录成功”。

**影响**

如果用户启动时未登录，首次随机检测可能跳过；登录后若没有触发检查，可能要等到下一次周期检测。

**方案是否已有解决**

原方案行为矩阵写了登录成功触发，但没有给出当前代码落点。

**建议解决方案**

优先方案：在认证 IPC 登录成功后调用 updater 的轻量触发函数：

```ts
scheduleLoginTriggeredCheck()
```

该函数内部 10 秒后调用 `checkForUpdates()`，但不重置周期计时器。

备选方案：认证模块提供登录成功事件，updater 初始化时订阅。

**建议落地位置**

- `apps/electron/src/auth/ipc-handlers.ts`
- `apps/electron/src/main/lib/updater/auto-updater.ts`

### 9. 下载实现需要处理 backpressure、临时文件和失败清理

**问题**

原方案示例中使用 `response.body.getReader()` 后直接 `fileStream.write(value)`，没有处理 `write()` 返回 `false` 时的 drain，也没有使用 `.tmp` 临时文件。下载中断时可能留下半成品文件。

**影响**

大安装包下载时可能内存压力增大；中断后可能把不完整文件误认为已下载完成。

**方案是否已有解决**

原方案 §4.4 提到中断与重试，但示例代码未完整实现。

**建议解决方案**

一期建议：

```text
1. 下载到 WorkMate-<version>.<ext>.tmp
2. 成功完成后 rename 为正式文件
3. 失败时删除 tmp
4. 正式文件存在时仍需校验 size/hash/version
```

下载进度可用 `ReadableStream` 逐 chunk 统计，但写入需等待 drain。

二期再支持 Range 断点续传。

**建议落地位置**

- `apps/electron/src/main/lib/updater/workmate-upgrade-service.ts`

### 10. 缺少安装包完整性校验字段

**问题**

原接口返回 `downloadUrl`，但没有返回 `sha256`、签名或文件大小字段。仅做版本号校验不足以证明文件未被篡改。

**影响**

如果下载链路或文件存储被替换，客户端可能下载到版本号相同但内容异常的包。

**方案是否已有解决**

原方案仅覆盖版本一致性，没有覆盖 hash/signature。

**建议解决方案**

服务端响应增加：

```json
{
  "sha256": "...",
  "fileSize": 123456789
}
```

客户端下载完成后：

```text
1. 校验 fileSize（如果提供）
2. 校验 sha256
3. 再做版本和方向校验
```

**建议落地位置**

- 服务端升级检测接口响应 DTO
- `apps/electron/src/main/lib/updater/updater-types.ts`
- `apps/electron/src/main/lib/updater/workmate-upgrade-service.ts`

### 12. `releaseNotes` 来源和渲染方式不一致

**问题**

当前 `UpdateDialog.tsx` 和 `AboutSettings.tsx` 会调用 `getReleaseByTag(v${version})` 获取 GitHub Release，并用 `ReleaseNotesViewer` 渲染。WorkMate 接口直接返回 `releaseNotes` 字符串。

**方案是否已有解决**

**建议解决方案**

状态中增加 `source`：

```ts
source: 'github' | 'workmate'
```

```text
github: 保留 getReleaseByTag + ReleaseNotesViewer
workmate: 直接展示 releaseNotes 字符串，可按换行渲染；若约定 Markdown，则复用 Markdown 渲染能力但不调用 GitHub API
```

**建议落地位置**

### 13. 状态类型与当前渲染端不兼容

**问题**

原方案定义的 `UpgradeStatus` 使用 `result: UpgradeCheckResult` 嵌套结构；当前渲染端使用扁平结构：

```ts
status.version
status.releaseNotes
status.progress
```

**影响**

如果直接替换类型，`UpdateDialog.tsx`、`AboutSettings.tsx`、`atoms/updater.ts` 都要大改。

**方案是否已有解决**

原方案没有考虑当前 renderer 的兼容。

**建议解决方案**

方案 A 中保留扁平字段，同时增加可选 `upgrade`：

```ts
export type UpdateStatus =
  | { status: 'idle' }
  | { status: 'checking' }
  | { status: 'not-available' }
  | {
      status: 'downloading'
      version: string
      releaseNotes?: string
      progress: DownloadProgress
      upgrade?: WorkmateUpgradeInfo
    }
  | {
      status: 'downloaded'
      version: string
      releaseNotes?: string
      installerPath?: string
      upgrade?: WorkmateUpgradeInfo
    }
  | { status: 'error'; error: string }
```

这样旧 UI 的 `status.version` 不会失效，新 UI 可读取 `status.upgrade.forceUpdate` 等扩展信息。

**建议落地位置**

- `apps/electron/src/main/lib/updater/updater-types.ts`
- `apps/electron/src/renderer/atoms/updater.ts`
- `apps/electron/src/preload/index.ts` 类型声明

### 15. 下载 URL 的安全边界不明确

**问题**

接口返回 `downloadUrl`，客户端直接下载。当前方案没有限制协议、域名或来源。

**影响**

如果服务端配置错误或被注入异常 URL，客户端可能下载非预期地址的文件。

**方案是否已有解决**

**建议解决方案**

```text
1. 仅允许 https，开发环境可配置允许 http localhost
2. 默认要求 downloadUrl 与 workmateServerUrl 同源，或在配置中明确允许的下载域名
3. 禁止 file://、ftp:// 等协议
```

### 16. 下载进度 total=-1 时 UI 会展示异常

**问题**

原方案 `total=-1` 表示未知大小。当前 `UpdateDialog.tsx` 的 `formatBytes` 会把负数格式化成 `-0 KB` 之类的结果。

**影响**

如果服务端没有 `content-length`，下载弹窗显示不专业。

**方案是否已有解决**

原方案定义了 `-1` 语义，但当前 UI 没有处理未知总量。

**建议解决方案**

UI 格式化调整：

```ts
if (bytes < 0) return '未知大小'
```

进度条：

```text
total > 0: 显示百分比进度
total <= 0: 显示不确定进度或只显示已下载大小/速度
```

**建议落地位置**

- `apps/electron/src/renderer/components/settings/UpdateDialog.tsx`
- `apps/electron/src/renderer/components/settings/AboutSettings.tsx`

### 17. `minVersion` 语义没有客户端处理规则

**问题**

接口返回 `minVersion`，原方案字段说明为“最低允许的当前版本”，但客户端如何使用没有明确规则。

**影响**

当前版本低于 `minVersion` 时，理论上应强制升级；如果客户端不处理，强制策略只能依赖服务端返回 `forceUpdate=true`。

**方案是否已有解决**

原方案定义了字段，但未写端侧规则。

**建议解决方案**

约定：

```text
如果 minVersion 存在且 currentVersion < minVersion：
  客户端将该更新视为 forceUpdate=true
```

但最终是否强制仍建议由服务端直接返回 `forceUpdate=true`，客户端规则作为兜底。

**建议落地位置**

- `apps/electron/src/main/lib/updater/workmate-upgrade-service.ts`

---

### 18. 并发检查和并发下载需要更严格保护

**问题**

当前 `checkForUpdates()` 只跳过 `downloading` 和 `downloaded`，但 WorkMate 流程需要同时避免：

- 多次手动点击造成并发 check
- 周期检测与登录触发重叠
- 同一版本重复下载
- 下载中又触发安装

**影响**

可能出现状态覆盖、重复下载、manifest 被覆盖等问题。

**方案是否已有解决**

原方案未细化并发控制。

**建议解决方案**

维护内部 promise：

```ts
let checkingPromise: Promise<void> | null = null
let downloadingPromise: Promise<string> | null = null
```

相同任务复用已有 promise；不同版本更新需先取消或等待当前下载完成。

**建议落地位置**

- `apps/electron/src/main/lib/updater/workmate-upgrade-service.ts`

---

### 19. `serverUrl` 未配置时的 fallback 行为需明确

**问题**

原方案上半部分写“serverUrl 未配置启用 GitHub Release”，行为矩阵又写“serverUrl 未配置不启动检测”。两处描述有冲突。

**影响**

实现时可能出现分歧：到底是沿用 GitHub 自动更新，还是完全关闭更新。

**方案是否已有解决**

原方案存在不一致。

**建议解决方案**

按方案 A 明确为：

```text
serverUrl 已配置：WorkMate Server 升级检测
serverUrl 未配置：保留当前 electron-updater GitHub Release 流程
```

如果产品要求企业版必须只走 WorkMate Server，可再通过构建配置禁用 GitHub fallback。

**建议落地位置**

- 文档：`客户端-升级检测.md`
- 代码：`apps/electron/src/main/lib/updater/auto-updater.ts`

---

### 20. 平台与安装包后缀规则不完整

**问题**

原方案示例中 Windows 使用 `.exe`，macOS 使用 `.dmg`，Linux 没有明确处理。当前接口只传 `platform=win32/darwin/linux`，没有传架构和包类型。

**影响**

多架构、多安装包格式时，服务端无法准确返回对应安装包；客户端也无法可靠推断本地文件扩展名。

**方案是否已有解决**

原方案仅覆盖基本 platform。

**建议解决方案**

请求参数增加：

```text
platform=win32|darwin|linux
arch=x64|arm64
```

服务端返回增加：

```json
{
  "fileName": "WorkMate-1.2.0-x64.exe",
  "packageType": "nsis"
}
```

客户端优先使用服务端 `fileName`，不要本地硬编码推断扩展名。

**建议落地位置**

- 服务端接口
- `apps/electron/src/main/lib/updater/workmate-upgrade-service.ts`
