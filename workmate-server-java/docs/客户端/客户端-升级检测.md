## 一、设计结论

直接在现有 `apps/electron/src/main/lib/updater/` 模块中实现 WorkMate Server 升级链路，替换原有的 `electron-updater`（GitHub Release）方式。

保留现有 IPC、preload、renderer atom、设置页入口和弹窗入口不变：

```text
main/index.ts
  → initAutoUpdater(mainWindow)
main/ipc.ts
  → 保留现有 updater IPC 注册
preload/index.ts
  → window.electronAPI.updater
renderer/atoms/updater.ts
  → updateStatusAtom
UpdateDialog.tsx / AboutSettings.tsx
  → 继续消费 updater 状态
```

升级链路：

```text
应用启动 / 定时 / 手动触发
  ↓
调用 WorkMate Server 检测接口
  ↓
有更新 → 自动下载安装包（支持进度展示）
  ↓
下载完成 → 校验 sha256 / fileSize / 版本
  ↓
弹窗提示安装 → 用户确认 → 启动安装器 → 退出应用
```

职责划分：

| 层 | 职责 |
|------|------|
| 检测 API | 调用 `/workmate/upgrade/check`，解析响应，做端侧版本方向校验。 |
| 下载器 | 下载到临时文件、进度统计、backpressure 处理、sha256/fileSize 校验。 |
| 安装器 | 安装前校验、版本提取、`shell.openPath` 启动安装器并退出应用。 |
| manifest | 持久化已下载安装包信息，启动时恢复以便重新提示安装。 |
| 主入口 | 统一状态管理、定时检测调度、对外 API（`checkForUpdates`、`quitAndInstall` 等）。 |

## 二、配置设计

### 2.1 配置字段

登录和升级检测共用同一个 EIP Lite 地址。配置在客户端 `settings.json` 中：

```json
{
  "eipGatewayBase": "http://eiplite.htsc.com.cn",
  "allowedUpgradeDownloadHosts": ["eiplite.htsc.com.cn"]
}
```

字段说明：

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `eipGatewayBase` | string | `http://eiplite.htsc.com.cn` | EIP Lite 地址，登录和升级检测共用。开发环境可配置为 `http://localhost:6173`。 |
| `allowedUpgradeDownloadHosts` | string[] | 空 | 可选下载域名白名单。为空时只校验 hostname 与 `eipGatewayBase` 一致，不要求端口一致。 |

### 2.2 配置读取

建议抽一个公共函数 `getEipGatewayBase()`，登录和升级检测统一读取同一个配置字段。

```ts
// 参考 apps/electron/src/models/model-service.ts 的读取方式
const settingsPath = getSettingsPath()
const settings = JSON.parse(readFileSync(settingsPath, 'utf-8'))
const serverUrl = settings.eipGatewayBase?.trim()
```

## 四、状态类型

保留现有扁平字段，增加 WorkMate 扩展信息。

```ts
export interface DownloadProgress {
  transferred: number
  total: number          // -1 表示未知大小
  bytesPerSecond: number
  percent: number
}

export interface UpgradeInfo {
  currentVersion?: string
  latestVersion: string
  releaseType?: 'UPGRADE' | 'ROLLBACK'
  forceUpdate?: boolean
  minVersion?: string | null
  downloadUrl?: string
  releaseNotes?: string | null
  hint?: string | null
  publishedAt?: string
  sha256?: string | null
  fileSize?: number | null
  fileName?: string | null
  packageType?: 'exe' | 'dmg' | 'appimage' | 'deb' | 'rpm' | null
  installerPath?: string
}

export type UpdateStatus =
  | { status: 'idle' }
  | { status: 'checking' }
  | { status: 'not-available' }
  | {
      status: 'available'
      version: string
      releaseNotes?: string
      forceUpdate?: boolean
      releaseType?: 'UPGRADE' | 'ROLLBACK'
      hint?: string | null
      upgradeInfo?: UpgradeInfo
    }
  | {
      status: 'downloading'
      version: string
      releaseNotes?: string
      progress: DownloadProgress
      forceUpdate?: boolean
      releaseType?: 'UPGRADE' | 'ROLLBACK'
      hint?: string | null
      upgradeInfo?: UpgradeInfo
    }
  | {
      status: 'downloaded'
      version: string
      releaseNotes?: string
      installerPath?: string
      forceUpdate?: boolean
      releaseType?: 'UPGRADE' | 'ROLLBACK'
      hint?: string | null
      upgradeInfo?: UpgradeInfo
    }
  | { status: 'error'; error: string }
```

状态流转规则：

- `not-available`：手动检查后无更新时使用；自动静默检查无更新时回到 `idle`。
- `downloading` / `downloaded`：跳过后续自动检查，避免并发下载。
- 各状态中 `version`、`releaseNotes` 等为旧字段兼容；WorkMate 扩展信息放在 `upgradeInfo` 中。

## 五、检测接口

### 5.1 请求

```http
GET /workmate/upgrade/check?currentVersion=X&platform=Y&arch=Z
Cookie: EIPGW-TOKEN=<jwt>
```

请求参数：

| 参数 | 类型 | 说明 |
|------|------|------|
| `currentVersion` | string | 当前客户端版本号，来自 `app.getVersion()`。 |
| `platform` | string | `win32` / `darwin` / `linux`。 |
| `arch` | string | `x64` / `arm64` 等，来自 `process.arch`。 |

### 5.2 响应

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "hasUpdate": true,
    "forceUpdate": false,
    "releaseType": "UPGRADE",
    "latestVersion": "1.2.0",
    "downloadUrl": "http://eiplite.htsc.com.cn/workmate/releases/1.2.0/WorkMate-1.2.0.exe",
    "releaseNotes": "修复了若干已知问题，优化了性能",
    "minVersion": "1.0.0",
    "hint": "有新版本可用，建议更新",
    "publishedAt": "2026-05-20T10:00:00Z",
    "sha256": "...",
    "fileSize": 123456789,
    "fileName": "WorkMate-1.2.0.exe",
    "packageType": "exe"
  },
  "timestamp": 1717300000000
}
```

字段说明：

| 字段 | 必填 | 说明 |
|------|------|------|
| `hasUpdate` | 是 | 是否有可用更新。 |
| `forceUpdate` | 是 | 是否强制升级。 |
| `releaseType` | 有更新时必填 | `UPGRADE` 或 `ROLLBACK`。 |
| `latestVersion` | 有更新时必填 | 目标版本。 |
| `downloadUrl` | 有更新时必填 | 安装包下载地址。 |
| `releaseNotes` | 否 | 发布说明。 |
| `minVersion` | 否 | 最低允许运行版本。 |
| `hint` | 否 | 提示文案，如回退提示。 |
| `publishedAt` | 否 | 发布时间。 |
| `sha256` | 建议必填 | 安装包 SHA-256，用于完整性校验。 |
| `fileSize` | 建议必填 | 安装包大小（byte），用于下载完成校验和 UI 展示。 |
| `fileName` | 建议必填 | 安装包文件名，用于保存和展示。 |
| `packageType` | 建议必填 | 安装包类型。 |

> **注意**：当前服务端 `UpgradeCheckResponse` 未包含 `sha256`、`fileSize`、`fileName`、`packageType`、`arch` 字段，`forceUpdate` 恒为 `false`。需要服务端同步补齐这些字段，并改为从 `UpgradeRelease` 表读取文件元数据。详见 §11。

### 5.3 端侧二次校验

收到 `hasUpdate=true` 后立即校验：

```text
releaseType = UPGRADE:
  latestVersion > currentVersion 才有效，否则忽略更新

releaseType = ROLLBACK:
  latestVersion < currentVersion 才有效，否则忽略更新

minVersion 兜底：
  如果 minVersion 存在且 currentVersion < minVersion：
    客户端将 forceUpdate 视为 true
```

版本比较必须使用 SemVer 解析，不能用字符串大小比较。

## 六、启动与定时检测

### 6.1 初始化流程

```text
auto-updater.init(mainWindow)
  ↓
读取 eipGatewayBase 配置
  ↓
恢复 manifest.json（getConfigDir()/upgrade/manifest.json）
  ↓
如果 manifest 存在且校验通过：
  setStatus(downloaded)，等待渲染端弹窗提示安装
  ↓
启动随机首检定时器
```

### 6.2 随机检测策略

```text
首次检测延迟:
  delay = random(10_000, 3_600_000)    // 10秒到60分钟

周期检测间隔:
  delay = random(6 * 3600_000, 8 * 3600_000)   // 6到8小时
```

使用递归 `setTimeout`，每次检测完成后重新计算下一次间隔。

### 6.3 行为矩阵

| 场景 | 行为 |
|------|------|
| 正常启动 | 10秒~60分钟随机首检 |
| 周期检测 | 每次6~8小时重新随机 |
| 登录成功 | 10秒后触发一次检查，不影响周期计时器 |
| 用户手动检查 | 立即检查，不影响周期计时器 |
| 登出 | 不主动触发检查；已有定时检查发现无 token 时跳过 |
| 状态为 downloading/downloaded | 跳过自动检查，避免并发下载 |

### 6.4 并发保护

`checkForUpdates()` 加锁，同一时间只允许一个检查流程。手动检查时若已处于 `checking` 状态则忽略。

## 七、下载与持久化

### 7.1 下载流程

```text
检测到有效更新
  ↓
校验 downloadUrl 安全边界（见 §7.2）
  ↓
setStatus(downloading) 并推送进度
  ↓
下载到临时文件：<fileName>.tmp
  ↓
下载完成后校验 fileSize 和 sha256（如果服务端提供）
  ↓
rename 为正式安装包文件
  ↓
写入 manifest.json
  ↓
setStatus(downloaded)，渲染端弹窗提示安装
```

下载实现要求：

1. 使用 `.tmp` 临时文件，成功后再 `rename`。
2. 失败时删除 `.tmp`，不保留半成品。
3. 写入文件流时处理 backpressure：`write()` 返回 `false` 时等待 `drain`。
4. 支持通过 `content-length` 计算下载进度；缺失时 `total=-1`，UI 展示"未知大小"。
5. 一期不要求 Range 断点续传。
6. 下载失败后自动重试 1 次（间隔 5 秒）。两次都失败则进入 `error` 状态。

### 7.2 下载 URL 安全校验

下载前校验 URL：

```text
1. 仅允许 https；开发环境允许 http://localhost 或 http://127.0.0.1。
2. 如果 allowedUpgradeDownloadHosts 非空，downloadUrl.hostname 必须在白名单内。
3. 如果 allowedUpgradeDownloadHosts 为空，downloadUrl.hostname 必须与 eipGatewayBase.hostname 一致。
4. 禁止 file://、ftp:// 等非 HTTP(S) 协议。
```

> 注意：hostname 比对不要求端口一致，因为下载走 Nginx（如 18080 端口），检测走 Spring Boot（如 6173 端口）。

### 7.3 manifest 持久化

保存位置：

```text
getConfigDir()/upgrade/manifest.json
```

`getConfigDir()` 为客户端已有配置根目录，manifest 与 `settings.json`、`auth.json` 放在同一目录下。

示例：

```json
{
  "version": "1.2.0",
  "releaseType": "UPGRADE",
  "forceUpdate": false,
  "installerPath": ".../upgrade/WorkMate-1.2.0.exe",
  "releaseNotes": "修复了若干已知问题，优化了性能",
  "downloadedAt": 1717300000000,
  "sha256": "...",
  "fileSize": 123456789,
  "fileName": "WorkMate-1.2.0.exe",
  "packageType": "exe"
}
```

启动恢复流程：

```text
读取 manifest
  ↓
检查 installerPath 是否存在
  ↓
校验 fileSize / sha256（如果 manifest 中有）
  ↓
校验安装包版本 == manifest.version
  ↓
校验版本方向仍符合当前 app.getVersion()
  ↓
通过：setStatus(downloaded)
失败：删除 manifest 和安装包，重新走检测
```

## 八、安装前校验与安装

### 8.1 安装前校验

```text
1. 当前状态必须是 downloaded
2. installerPath 文件必须存在
3. fileSize 与 manifest 一致（如果提供）
4. sha256 与 manifest 一致（如果提供）
5. 安装包版本 == latestVersion
6. UPGRADE 时安装包版本 > 当前 app.getVersion()
7. ROLLBACK 时安装包版本 < 当前 app.getVersion()
```

任一失败：进入 `error` 状态，删除异常安装包和 manifest，不执行安装。

### 8.2 安装包版本读取

| 平台 | 一期方案 | 二期增强 |
|------|----------|----------|
| Windows | 文件名提取版本 + sha256 校验 | 读取 EXE PE VersionInfo |
| macOS | 文件名提取版本 + sha256 校验 | 解析包内 `.app/Contents/Info.plist` |
| Linux | 文件名提取版本 + sha256 校验 | 按 AppImage/deb/rpm 格式读取元数据 |

### 8.3 执行安装

```text
shell.openPath(installerPath)
  ↓
成功：标记应用正在退出，移除 window close 拦截，app.quit()
失败：setStatus(error)
```

保留现有 `quitAndInstall()` 函数和 `updater:quit-and-install` IPC 通道名。UI 文案从"立即重启更新"改为"立即安装"。

## 十一、服务端待配合事项

当前服务端与本文档存在以下差距，需要服务端配合改造：

| 事项 | 当前状态 | 需要改为 |
|------|----------|----------|
| `UpgradeCheckResponse` 缺文件元数据 | 无 `sha256`/`fileSize`/`fileName`/`packageType` | 增加这些字段，从 `UpgradeRelease` 表读取 |
| `UpgradeCheckRequest` 缺 `arch` | 只有 `currentVersion`/`platform` | 增加 `arch` 参数 |
| `forceUpdate` 恒为 `false` | 硬编码 `false` | 支持从策略或配置中读取强制升级标记 |
| 检测接口的 `downloadUrl` 来源 | 取自策略表的冗余字段 | 改为查 `UpgradeRelease` 表，返回完整文件元数据 |
