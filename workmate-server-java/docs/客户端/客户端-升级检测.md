# 客户端：升级检测方案

编辑时间：2026年6月5日

> **关联文档**：对应后端服务设计见 [服务端-workmate升级检测和灰度升级管理.md](../服务端/服务端-workmate升级检测和灰度升级管理.md)。客户端登录方案见 [客户端-登录.md](./客户端-登录.md)。问题复核见 [客户端--升级检测方案问题与解决方案.md](./客户端--升级检测方案问题与解决方案.md)。请求经过 EIP 网关，网关自动注入 `X-EIPGW-USERID` Header；客户端侧需要携带 `EIPGW-TOKEN` Cookie。

## 一、设计结论

本方案采用“**主升级路径 + 策略模式**”改造方式，尽可能复用当前 Proma 客户端已有升级链路。

主升级路径继续使用现有 `apps/electron/src/main/lib/updater/`，保留当前 IPC、preload、renderer atom、设置页入口和弹窗入口。主路径只负责：

1. 读取配置。
2. 选择升级策略。
3. 维护统一状态。
4. 向渲染进程广播状态变化。
5. 对外保持现有 updater API 不变。

具体升级能力由策略实现：

| 策略 | 目录 | 说明 |
|------|------|------|
| WorkMate 升级策略 | `apps/electron/src/main/lib/updater/strategies/workmate/` | 使用 WorkMate Server API 检测、下载、校验和安装，支持灰度、强制升级、回退。 |
| Proma 老升级策略 | `apps/electron/src/main/lib/updater/strategies/proma/` | 封装现有 `electron-updater` 逻辑，用于切换回老的 Proma/GitHub Release 升级方式。 |

配置决定实际执行策略。主路径中通过读取配置判断应该委托给哪个策略执行，而不是在多个入口里分别注册升级逻辑。

---

## 二、核心原则

1. **复用原有升级路径**：保留 `initAutoUpdater`、`checkForUpdates`、`getUpdateStatus`、`quitAndInstall`、`cleanupUpdater` 等对外函数签名。
2. **策略隔离**：WorkMate 与 Proma 使用不同目录实现同一 `UpgradeStrategy` 接口，互不侵入。
3. **配置切换**：通过客户端配置选择 `workmate` 或 `proma` 策略，不再用“是否新建一套 IPC/preload”来区分升级系统。
4. **状态兼容**：保留当前渲染端已使用的 `UpdateStatus` 扁平字段，同时增加 WorkMate 扩展字段。
5. **严格校验**：WorkMate 策略必须校验版本方向、安装包版本、hash、文件大小和下载 URL 安全边界。
6. **随机分散**：WorkMate 策略启动后使用随机首检和随机周期检测，避免集中请求。
7. **可回退**：配置切回 Proma 策略后，继续走老的 `electron-updater` 升级方式。

## 三、配置设计

### 3.1 配置字段

当前客户端登录已使用 `settings.eipGatewayBase` 作为 EIP Lite 地址，升级检测请求同样经过 EIP Lite，因此升级方案统一复用该字段，不再使用示例域名，也不复用 `workmateServerUrl` 作为升级检测地址。

建议抽一个公共配置读取函数，例如 `getEipGatewayBase()`，登录、升级检测等所有经过 EIP Lite 的请求都从同一个函数读取地址。

新增升级策略配置：

```json
{
  "upgradeStrategy": "workmate",
  "eipGatewayBase": "http://eiplite.htsc.com.cn",
  "allowedUpgradeDownloadHosts": ["eiplite.htsc.com.cn"]
}
```

字段说明：

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `upgradeStrategy` | `'workmate' \| 'proma' \| 'auto'` | `workmate` | 升级策略。默认走 WorkMate 升级策略。`workmate` 或未配置时使用新策略，`proma` 显式切回老策略，`auto` 作为兼容选项按 `eipGatewayBase` 是否存在自动选择。 |
| `eipGatewayBase` | string | `http://eiplite.htsc.com.cn` | EIP Lite 地址，登录和升级检测共用。开发环境可配置为 `http://localhost:6173`。 |
| `allowedUpgradeDownloadHosts` | string[] | 空 | 可选下载域名白名单。为空时默认要求 `downloadUrl` 与 `eipGatewayBase` 同源。 |

### 3.2 策略选择规则

```text
读取 settings.json
  ↓
upgradeStrategy = 'workmate' 或未配置
  → 使用 WorkMateUpgradeStrategy
  → 读取 eipGatewayBase；若未配置则使用默认 EIP Lite 地址

upgradeStrategy = 'proma'
  → 使用 PromaUpgradeStrategy
  → 走 electron-updater 老逻辑

upgradeStrategy = 'auto'
  → eipGatewayBase 存在或可使用默认值：使用 WorkMateUpgradeStrategy
  → 明确关闭 WorkMate 或要求老升级：使用 PromaUpgradeStrategy
```

默认值使用 `workmate`。客户端未配置 `upgradeStrategy` 时直接走 WorkMate 升级策略；如需回退老 Proma/GitHub Release 升级方式，必须显式配置 `upgradeStrategy: "proma"`。

`auto` 仅作为兼容选项保留；常规部署不推荐依赖 `auto`。

## 四、目录结构

### 4.1 主升级路径

```text
apps/electron/src/main/lib/updater/
├── auto-updater.ts              # 主入口：读配置、选择策略、统一对外 API
├── updater-ipc.ts               # 保留现有 IPC 注册
├── updater-types.ts             # 统一状态、策略接口、通用升级信息类型
├── updater-config.ts            # 新增：读取升级配置并选择策略类型
├── updater-state.ts             # 新增：统一状态维护和广播
├── strategy-registry.ts         # 新增：唯一策略注册表，新增策略只在此注册一次
└── strategies/
    ├── proma/
    │   └── proma-upgrade-strategy.ts
    └── workmate/
        ├── workmate-upgrade-strategy.ts
        ├── workmate-upgrade-api.ts
        ├── workmate-downloader.ts
        ├── workmate-installer.ts
        ├── workmate-manifest.ts
        └── workmate-version.ts
```

策略扩展规则：

1. 新增策略必须放在 `strategies/<strategy-type>/` 独立目录内。
2. 主路径不能直接 import 具体策略实现，只通过 `strategy-registry.ts` 创建策略。
3. 新增策略时，除新增策略包外，只允许修改 `strategy-registry.ts` 中的注册项和必要的配置类型定义。
4. 渲染端不按具体策略写散落分支；需要展示的字段应优先沉淀为通用 `UpdateStatus` 字段。

### 4.2 渲染端复用路径

```text
apps/electron/src/preload/index.ts
apps/electron/src/renderer/atoms/updater.ts
apps/electron/src/renderer/components/settings/UpdateDialog.tsx
apps/electron/src/renderer/components/settings/AboutSettings.tsx
```

这些文件不新增平行实现，只在现有基础上扩展 WorkMate 字段和 UI 行为。

## 五、策略接口

所有升级策略实现同一个接口，由主路径统一调用。

```ts
export type UpgradeStrategyType = string

export interface UpgradeStrategyContext {
  mainWindow: BrowserWindow
  setStatus: (status: UpdateStatus) => void
  getStatus: () => UpdateStatus
}

export interface UpgradeStrategy {
  readonly type: UpgradeStrategyType
  init(context: UpgradeStrategyContext): Promise<void> | void
  checkForUpdates(options?: { manual?: boolean }): Promise<void>
  quitAndInstall(): Promise<void> | void
  cleanup(): void
  onLoginSuccess?(): void
}
```

策略创建统一经由 `strategy-registry.ts`，避免主入口出现多处 `switch` 或直接 import 具体策略。

```ts
const strategyFactories: Record<string, (config: UpdaterConfig) => UpgradeStrategy> = {
  proma: (config) => new PromaUpgradeStrategy(config),
  workmate: (config) => new WorkmateUpgradeStrategy(config),
}

export function createUpgradeStrategy(config: UpdaterConfig): UpgradeStrategy {
  const factory = strategyFactories[config.resolvedStrategy]
  if (!factory) throw new Error(`Unsupported upgrade strategy: ${config.resolvedStrategy}`)
  return factory(config)
}
```

主入口 `auto-updater.ts` 不关心具体策略内部如何检测、下载和安装，只负责把现有公共 API 委托给当前策略。

```ts
let currentStrategy: UpgradeStrategy | null = null

export function initAutoUpdater(mainWindow: BrowserWindow): void {
  const config = readUpdaterConfig()
  currentStrategy = createUpgradeStrategy(config)
  currentStrategy.init({ mainWindow, setStatus, getStatus })
}

export async function checkForUpdates(): Promise<void> {
  await currentStrategy?.checkForUpdates({ manual: true })
}

export function quitAndInstall(): void {
  void currentStrategy?.quitAndInstall()
}

export function cleanupUpdater(): void {
  currentStrategy?.cleanup()
  currentStrategy = null
}
```

## 六、统一状态类型

保留当前渲染端使用的扁平字段，增加策略来源和 WorkMate 扩展信息。

```ts
export interface DownloadProgress {
  transferred: number
  total: number
  bytesPerSecond: number
  percent: number
}

export interface UpgradeInfo {
  currentVersion?: string
  latestVersion: string
  releaseType?: 'UPGRADE' | 'ROLLBACK'
  forceUpdate?: boolean
  minVersion?: string | null
  downloadUrl?: string
  releaseNotes?: string | null
  hint?: string | null
  publishedAt?: string
  sha256?: string | null
  fileSize?: number | null
  fileName?: string | null
  packageType?: 'exe' | 'dmg' | 'appimage' | 'deb' | 'rpm' | null
  installerPath?: string
}

export type UpdateStatus =
  | { status: 'idle'; source?: string }
  | { status: 'checking'; source?: string }
  | { status: 'not-available'; source?: string }
  | {
      status: 'available'
      source: string
      version: string
      releaseNotes?: string
      forceUpdate?: boolean
      releaseType?: 'UPGRADE' | 'ROLLBACK'
      hint?: string | null
      upgradeInfo?: UpgradeInfo
    }
  | {
      status: 'downloading'
      source: string
      version: string
      releaseNotes?: string
      progress: DownloadProgress
      forceUpdate?: boolean
      releaseType?: 'UPGRADE' | 'ROLLBACK'
      hint?: string | null
      upgradeInfo?: UpgradeInfo
    }
  | {
      status: 'downloaded'
      source: string
      version: string
      releaseNotes?: string
      installerPath?: string
      forceUpdate?: boolean
      releaseType?: 'UPGRADE' | 'ROLLBACK'
      hint?: string | null
      upgradeInfo?: UpgradeInfo
    }
  | { status: 'error'; source?: string; error: string }
```

类型边界规则：

1. `UpdateStatus` 只放渲染端确实需要展示或交互的通用字段，例如 `version`、`releaseNotes`、`progress`、`forceUpdate`、`releaseType`、`hint`、`installerPath`。
2. `upgradeInfo` 只承载升级域通用信息，不以 `workmate` 命名，避免后续新增策略继续扩展 WorkMate 专属结构。
3. 策略私有字段留在策略内部或 manifest 内，不进入通用 renderer 状态。
4. 新增策略如果需要 UI 展示新信息，先评估是否可抽象为通用字段，再修改 `UpdateStatus`。

兼容规则：

1. Proma 策略继续填充 `version`、`releaseNotes`、`progress` 等旧字段。
2. WorkMate 策略同时填充旧字段和通用扩展字段，例如 `forceUpdate`、`releaseType`、`hint`、`upgradeInfo`。
3. 渲染端旧逻辑仍可读取 `status.version`；新逻辑优先读取 `status.forceUpdate`、`status.releaseType` 等通用字段。
4. 保留 `not-available`，用于手动检查后提示“已是最新版本”。自动静默检查无更新时可回到 `idle`。

## 七、WorkMate 策略流程

### 7.1 初始化流程

```text
WorkmateUpgradeStrategy.init
  ↓
读取并校验 eipGatewayBase
  ↓
恢复 getConfigDir()/upgrade/manifest.json
  ↓
如果已有已下载安装包且校验通过：
  setStatus(downloaded)
  等渲染端弹窗提示安装
  ↓
启动随机首检定时器
```

### 7.2 随机检测策略

```text
首次检测延迟:
  delay = random(10_000, 3_600_000)    // 10秒到60分钟

周期检测间隔:
  delay = random(6 * 3600_000, 8 * 3600_000)
```

使用递归 `setTimeout`，不要使用固定 `setInterval`。

```text
scheduleNextCheck(delay)
  -> setTimeout(checkForUpdates)
  -> 检查完成后重新计算下一次 6h~8h delay
```

行为矩阵：

| 场景 | 行为 |
|------|------|
| 正常启动 | 10 秒到 60 分钟随机首检 |
| 周期检测 | 每次 6 小时到 8 小时重新随机 |
| 登录成功 | 10 秒后触发一次检查，不影响周期计时器 |
| 用户手动检查 | 立即检查，不影响周期计时器 |
| 登出 | 不主动触发检查；已有定时检查发现无 token 时直接跳过 |
| 当前状态 downloading/downloaded | 跳过自动检查，避免并发下载 |

### 7.3 检测接口

```http
GET /workmate/upgrade/check?currentVersion=X&platform=Y&arch=Z
Cookie: EIPGW-TOKEN=<jwt>
```

请求参数：

| 参数 | 类型 | 说明 |
|------|------|------|
| `currentVersion` | string | 当前客户端版本号，来自 `app.getVersion()`。 |
| `platform` | string | `win32` / `darwin` / `linux`。 |
| `arch` | string | `x64` / `arm64` 等，来自 `process.arch`。 |

响应建议：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "hasUpdate": true,
    "forceUpdate": false,
    "releaseType": "UPGRADE",
    "latestVersion": "1.2.0",
    "downloadUrl": "http://eiplite.htsc.com.cn/workmate/releases/1.2.0/WorkMate-1.2.0.exe",
    "releaseNotes": "修复了若干已知问题，优化了性能",
    "minVersion": "1.0.0",
    "hint": "有新版本可用，建议更新",
    "publishedAt": "2026-05-20T10:00:00Z",
    "sha256": "...",
    "fileSize": 123456789,
    "fileName": "WorkMate-1.2.0.exe",
    "packageType": "exe"
  },
  "timestamp": 1717300000000
}
```

字段说明：

| 字段 | 必填 | 说明 |
|------|------|------|
| `hasUpdate` | 是 | 是否有可用更新。 |
| `forceUpdate` | 是 | 是否强制升级。 |
| `releaseType` | 有更新时必填 | `UPGRADE` 或 `ROLLBACK`。 |
| `latestVersion` | 有更新时必填 | 目标版本。 |
| `downloadUrl` | 有更新时必填 | 安装包下载地址。 |
| `releaseNotes` | 否 | 发布说明，可按 Markdown 约定渲染。 |
| `minVersion` | 否 | 最低允许运行版本，当前版本低于该值时客户端兜底视为强制升级。 |
| `sha256` | 建议必填 | 安装包 SHA-256，用于完整性校验。 |
| `fileSize` | 建议必填 | 安装包大小，用于下载完成校验和 UI 展示。 |
| `fileName` | 建议必填 | 服务端建议文件名，避免客户端硬编码文件名。 |
| `packageType` | 建议必填 | 安装包类型，避免客户端硬编码后缀。 |

### 7.4 版本方向校验

收到 `hasUpdate=true` 后，客户端必须立即做端侧二次校验：

```text
releaseType = UPGRADE:
  latestVersion > currentVersion 才有效
  latestVersion <= currentVersion 视为异常数据，忽略更新

releaseType = ROLLBACK:
  latestVersion < currentVersion 才有效
  latestVersion >= currentVersion 视为异常数据，忽略更新
```

`minVersion` 兜底规则：

```text
如果 minVersion 存在且 currentVersion < minVersion：
  客户端将 forceUpdate 视为 true
```

版本比较必须使用 SemVer 解析比较，不能直接用字符串大小比较。

## 八、WorkMate 下载与持久化

### 8.1 下载流程

```text
检测到有效更新
  ↓
校验 downloadUrl 安全边界
  ↓
设置 downloading 状态并推送进度
  ↓
下载到临时文件：<fileName>.tmp
  ↓
下载完成后校验 fileSize 和 sha256
  ↓
rename 为正式安装包文件
  ↓
写入 manifest.json
  ↓
设置 downloaded 状态，渲染端弹窗提示安装
```

下载实现要求：

1. 使用 `.tmp` 临时文件，成功后再 `rename`。
2. 失败时删除 `.tmp`，不得把半成品当作正式安装包。
3. 写入文件流时处理 backpressure，`write()` 返回 `false` 时等待 `drain`。
4. `content-length` 缺失时 `total=-1`，UI 展示“未知大小”，不展示负数字节。
5. 一期不要求 Range 断点续传；后续可在服务端支持 Range 后增强。

### 8.2 下载 URL 安全校验

下载前必须校验 URL：

```text
1. 仅允许 https；开发环境可允许 http://localhost 或 http://127.0.0.1。
2. 如果 allowedUpgradeDownloadHosts 非空，则 downloadUrl.host 必须在白名单内。
3. 如果 allowedUpgradeDownloadHosts 为空，则默认要求 downloadUrl 与 eipGatewayBase 同源。
4. 禁止 file://、ftp:// 等非 HTTP(S) 协议。
```

### 8.3 manifest 持久化

保存位置：

```text
getConfigDir()/upgrade/manifest.json
```

说明：`getConfigDir()` 对应当前客户端已有配置根目录，正式版本通常为 `~/.proma`，开发模式通常为 `~/.proma-dev`。manifest 与 `settings.json`、`auth.json` 保持同一配置根目录，便于排查和迁移。

示例：

```json
{
  "source": "workmate",
  "version": "1.2.0",
  "releaseType": "UPGRADE",
  "forceUpdate": false,
  "installerPath": ".../upgrade/WorkMate-1.2.0.exe",
  "releaseNotes": "修复了若干已知问题，优化了性能",
  "downloadedAt": 1717300000000,
  "sha256": "...",
  "fileSize": 123456789,
  "fileName": "WorkMate-1.2.0.exe",
  "packageType": "exe"
}
```

启动恢复流程：

```text
读取 manifest
  ↓
检查 installerPath 是否存在
  ↓
校验 fileSize / sha256
  ↓
校验安装包版本 == manifest.version
  ↓
校验版本方向仍符合当前 app.getVersion()
  ↓
通过：setStatus(downloaded)
失败：删除 manifest 和安装包，重新走检测
```

该设计解决“用户选择下次安装后，下次启动自动提示安装”的问题。

## 九、WorkMate 安装前校验与安装

安装前必须完成以下校验：

```text
1. 当前状态必须是 downloaded。
2. installerPath 文件必须存在。
3. fileSize 与服务端返回值一致（如果提供）。
4. sha256 与服务端返回值一致（如果提供）。
5. 安装包真实版本必须等于 latestVersion。
6. UPGRADE 时安装包版本必须大于当前 app.getVersion()。
7. ROLLBACK 时安装包版本必须小于当前 app.getVersion()。
```

任一失败：

```text
setStatus(error)
删除异常安装包
删除 manifest
不执行安装
```

安装包真实版本读取：

| 平台 | 一期方案 | 二期增强 |
|------|----------|----------|
| Windows | 文件名提取版本 + sha256 校验 | 读取 EXE PE VersionInfo |
| macOS | 文件名提取版本 + sha256 校验 | 挂载/解析包内 `.app/Contents/Info.plist` |
| Linux | 文件名提取版本 + sha256 校验 | 按 AppImage/deb/rpm 格式读取元数据 |

执行安装：

```text
shell.openPath(installerPath)
  ↓
成功：标记应用正在退出，移除 close 拦截，app.quit()
失败：setStatus(error)
```

为了复用现有 IPC，仍保留 `quitAndInstall()` 函数和 `updater:quit-and-install` 通道名；但 WorkMate 策略下 UI 文案改为“立即安装”。

## 十、Proma 老策略

Proma 策略封装当前 `electron-updater` 实现，作为可配置回退策略。

```text
PromaUpgradeStrategy.init
  ↓
配置 autoUpdater.autoDownload = true
  ↓
注册 checking-for-update / update-available / download-progress / update-downloaded / error 等事件
  ↓
按老逻辑启动首检和周期检查
```

Proma 策略保持原有能力边界：

| 能力 | Proma 策略 |
|------|------------|
| 检测源 | GitHub Release / `latest.yml` |
| 版本方向 | 仅支持升级，不支持回退 |
| 下载 | `electron-updater` 内部处理 |
| 安装 | `autoUpdater.quitAndInstall(true, true)` |
| forceUpdate | 不支持 |
| manifest | 不需要 |
| sha256 | 由 `electron-updater` 机制处理，不走 WorkMate manifest |

Proma 策略目录只维护老逻辑，不引入 WorkMate API 字段。

## 十一、IPC、preload 与渲染端

### 11.1 IPC

继续复用现有 IPC 通道，不新增一套 `upgrade:*` 通道。

通道名必须以 `apps/electron/src/main/lib/updater/updater-types.ts` 中的 `UPDATER_IPC_CHANNELS` 为准，当前源码通道为：

```text
updater:check
updater:get-status
updater:quit-and-install
updater:status-changed
```

`updater-ipc.ts` 基本不改，只继续调用主入口导出的函数。`preload/index.ts` 也不要写新的通道名，只继续通过现有 `window.electronAPI.updater` 暴露能力。

### 11.2 preload

`window.electronAPI.updater` 保持不变，仅更新 TypeScript 类型，让状态能包含 `source`、`forceUpdate`、`releaseType`、`hint`、`upgradeInfo` 等通用字段。

### 11.3 renderer atom

继续复用 `updateStatusAtom`、`initializeUpdater`、`checkForUpdates` 等现有名称。

### 11.4 UpdateDialog

WorkMate 策略下弹窗行为：

| 场景 | 行为 |
|------|------|
| downloading | 展示下载进度；`total <= 0` 时展示未知大小。 |
| downloaded + `forceUpdate=false` | 展示“下次安装”和“立即安装”。 |
| downloaded + `forceUpdate=true` | 只展示“立即安装”，不可关闭。 |
| releaseType=`UPGRADE` | 展示“升级”。 |
| releaseType=`ROLLBACK` | 展示“回退”。 |
| source=`workmate` | 直接使用接口返回的 `releaseNotes`，不再调用 GitHub Release API。 |
| source=`proma` | 保留现有 GitHub Release Notes 渲染方式。 |

实现约束：`UpdateDialog.tsx` 中不要散落多个 `source === 'workmate'` 判断。建议在组件顶部或独立 helper 中集中派生展示模型，例如：

```ts
const viewModel = getUpdateDialogViewModel(updateStatus)
```

该派生逻辑负责统一计算标题、说明文案、按钮文案、是否可关闭、是否需要拉取 GitHub Release。组件主体只消费 `viewModel`。

### 11.5 AboutSettings

关于页继续作为手动检查入口。扩展展示：

1. 当前策略来源：WorkMate 或 Proma。
2. 下载中进度。
3. 强制更新标识。
4. WorkMate 的 `hint`、`releaseType` 和 `latestVersion`。
5. 手动检查无更新时保留 `not-available` 提示。

实现约束：关于页同样通过集中派生函数生成展示文案和操作按钮，例如 `getUpdateCardViewModel(status)`。避免在 `AboutSettings.tsx` 中重复实现与 `UpdateDialog.tsx` 不一致的策略判断。

## 十二、并发与错误处理

### 12.1 并发保护

WorkMate 策略维护以下内部状态：

```ts
let checkingPromise: Promise<void> | null = null
let downloadingPromise: Promise<void> | null = null
let loginCheckTimer: ReturnType<typeof setTimeout> | null = null
let periodicTimer: ReturnType<typeof setTimeout> | null = null
```

职责和长度约束：

1. `workmate-upgrade-strategy.ts` 只负责状态机、并发保护、定时器和模块编排，不直接实现 HTTP 解析、文件下载、hash 校验、manifest 读写或安装器启动细节。
2. 单个源文件目标不超过 800 行；单个方法目标不超过 200 行。
3. 如果 `checkForUpdates`、`downloadAndPersist`、`restoreManifest` 等方法接近 200 行，必须继续下沉到 API、downloader、manifest、installer 或 version 模块。
4. Proma 策略只封装 `electron-updater` 老逻辑，不依赖 WorkMate API 类型。

规则：

| 状态 | 新检查请求处理 |
|------|----------------|
| checking | 复用当前 `checkingPromise` 或忽略自动检查。 |
| downloading | 忽略自动检查；手动检查提示正在下载。 |
| downloaded | 不再重复下载同版本，直接提示安装。 |
| error | 手动检查可重试，自动检查按周期重试。 |

### 12.2 错误状态

| 错误 | 处理 |
|------|------|
| 无 token | 静默跳过，回到 idle。 |
| serverUrl 缺失但强制 workmate | 设置 error。 |
| 检测接口失败 | 自动检查回到 idle；手动检查设置 error 或 not-available 视产品交互决定。 |
| 服务端返回版本方向异常 | 忽略更新，记录日志，手动检查可提示 error。 |
| 下载失败 | 删除 tmp，设置 error。 |
| hash/fileSize 校验失败 | 删除安装包和 manifest，设置 error。 |
| 安装包版本异常 | 删除安装包和 manifest，设置 error。 |
| 安装器启动失败 | 设置 error，保留安装包以便重试。 |

## 十三、登录成功触发

WorkMate 策略需要在登录成功后 10 秒触发一次检查。

推荐落点：认证 IPC 登录成功后调用 updater 主入口的轻量函数。当前客户端登录 IPC 位于 `apps/electron/src/auth/ipc-handlers.ts`，应在 `auth:login` 返回 `success=true` 后触发，避免只依赖渲染层 `onLoginSuccess` 而漏掉其他登录入口。

```ts
export function notifyUpdaterLoginSuccess(): void {
  currentStrategy?.onLoginSuccess?.()
}
```

WorkMate 策略实现：

```text
onLoginSuccess()
  ↓
清理旧 loginCheckTimer
  ↓
10 秒后 checkForUpdates({ manual: false })
  ↓
不重置周期检测 timer
```

Proma 策略可以不实现 `onLoginSuccess`。

## 十四、服务端检测逻辑参考

服务端仍按策略判断是否给当前用户返回更新：

```text
GET /workmate/upgrade/check
  ↓
1. 查 ACTIVE 策略，无策略则 hasUpdate=false
  ↓
2. 按已执行阶段规则合并白名单/灰度规则
  ↓
3. 校验当前用户是否命中灰度范围
  ↓
4. 按 releaseType 比较版本
   - UPGRADE: targetVersion > currentVersion
   - ROLLBACK: targetVersion < currentVersion
  ↓
5. 返回检测结果、下载地址、hash、文件大小、发布说明等
```

客户端不能完全信任服务端结果，必须保留端侧二次校验和安装前校验。

## 十七、推荐实施顺序

```text
阶段 1：策略化主路径
  - 抽出 UpgradeStrategy 接口
  - 主入口读取配置并选择策略
  - 把现有 electron-updater 迁移到 PromaUpgradeStrategy

阶段 2：WorkMate 基础闭环
  - 检测接口
  - 版本方向校验
  - 自动下载
  - manifest 持久化
  - 安装前校验
  - shell.openPath 安装

阶段 3：UI 与触发补齐
  - forceUpdate 弹窗
  - 下次安装
  - 登录成功触发检测
  - About 页状态展示

阶段 4：安全增强
  - sha256 / fileSize 强校验
  - downloadUrl 白名单
  - 安装包真实版本读取
  - Range 断点续传
```

首版建议至少完成阶段 1、阶段 2、阶段 3，并将 `sha256` / `fileSize` 校验纳入首版，避免只依赖文件名版本判断。
