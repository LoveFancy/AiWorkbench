## 1. 项目概述

Proma 是一个本地优先的 AI 桌面应用，集成多模型 Chat、通用 Agent、工作区、Skills、MCP、远程机器人和记忆能力。采用 Electron 桌面应用架构，Bun Workspace Monorepo 管理。

核心定位：不只是聊天框，而是一个可以长期沉淀个人工作流的 Agent 工作台——简单问题用 Chat，复杂任务交给 Agent，数据和配置尽量留在本地。

## 2. 开发环境准备

### 2.1 必装工具

#### Git

```powershell
# 下载安装
https://git-scm.com/download/win

# 验证安装
git --version
```

安装时勾选 "Add to PATH"，推荐使用 Git Bash 作为辅助终端。

#### Bun（核心运行时，替代 Node.js/npm/pnpm）

```powershell
# PowerShell 安装
powershell -c "irm bun.sh/install.ps1 | iex"

# 验证安装
bun --version   # 要求 >= 1.2.5
```

Bun 是本项目的核心运行时，用于：
- 安装依赖：`bun install`
- 运行脚本：`bun run <script>`
- 运行测试：`bun test`（内置测试运行器）
- 自动加载 `.env` 文件（无需 dotenv）

> **重要**：项目强制使用 Bun，禁止使用 node/npm/pnpm/vite 命令直接运行。参见 `.cursor/rules/use-bun-instead-of-node-vite-npm-pnpm.mdc`。

#### Python（Skills 脚本依赖）

部分 Skills（pdf、docx、pptx、xlsx、dpmp-assist）的脚本使用 Python 编写：

```powershell
# 下载安装 Python 3.10+
https://www.python.org/downloads/

# 安装时勾选 "Add Python to PATH"

# 验证安装
python --version
```

### 2.2 推荐开发工具

| 工具 | 用途 | 说明 |
|------|------|------|
| VS Code / Cursor | 代码编辑 | 推荐安装 TypeScript、Tailwind CSS、ESLint 扩展 |
| Windows Terminal | 终端 | 支持 PowerShell 和 Git Bash 多标签 |
| PowerShell 5+ | 脚本执行 | Windows 自带，用于构建和运行命令 |

### 2.3 克隆项目与安装依赖

```powershell
# 克隆仓库
git clone https://github.com/ErlichLiu/Proma.git
cd Proma

# 安装所有依赖（Bun Workspace 会自动链接内部包）
bun install
```

### 2.4 Electron 国内镜像配置

`bun install` 和 `electron-builder` 打包时需要下载 Electron 二进制文件，默认从 GitHub 下载，国内网络可能超时。配置方式：

**方式一：项目 `.npmrc`（已配置）**

`apps/electron/.npmrc` 已包含镜像配置：
```
electron_mirror=https://npmmirror.com/mirrors/electron/
electron_builder_binaries_mirror=https://npmmirror.com/mirrors/electron-builder-binaries/
```

**方式二：环境变量（推荐，确保生效）**

打包时在 PowerShell 中设置环境变量：
```powershell
cd apps/electron
$env:ELECTRON_MIRROR="https://npmmirror.com/mirrors/electron/"
$env:ELECTRON_BUILDER_BINARIES_MIRROR="https://npmmirror.com/mirrors/electron-builder-binaries/"
bun run dist:win
```

### 2.5 环境验证

```powershell
# 检查 Bun 版本
bun --version

# 运行类型检查，验证环境是否正常
bun run typecheck

# 启动开发模式
bun run dev
```

如果 `bun run dev` 能成功启动 Electron 窗口并加载界面，说明环境准备完成。

### 2.6 常见 Windows 环境问题

| 问题 | 解决方案 |
|------|----------|
| `bun install` 报错 | 确保 Bun 版本 >= 1.2.5，删除 `bun.lock` 后重试 |
| Electron 下载慢 | 设置镜像：`$env:ELECTRON_MIRROR="https://npmmirror.com/mirrors/electron/"` |
| `bun run dev` 白屏 | 检查 Vite 是否启动成功，查看终端是否有端口冲突 |
| Python Skills 报错 | 确认 Python 已加入 PATH，必要时安装 requirements.txt 中的依赖 |
| PowerShell 执行策略 | 以管理员运行：`Set-ExecutionPolicy RemoteSigned -Scope CurrentUser` |
| 打包时 out 目录被占用 | 关闭所有 WorkMate/Electron 进程后删除 `apps/electron/out` 目录重试 |

## 3. 打包构建

### 3.1 日常开发

```powershell
# 开发模式（推荐 — 自动启动 Vite + Electron + 热重载）
bun run dev

# 手动开发模式（调试时更稳定，需要两个终端）
# 终端 1：
cd apps/electron; bun run dev:vite
# 终端 2：
cd apps/electron; bun run dev:electron
```

### 3.2 构建

```powershell
# 构建所有包
bun run build

# 构建 Electron 应用
bun run electron:build

# 构建并启动
bun run electron:start
```

### 3.3 Electron 应用内部分步构建（`apps/electron/` 目录下）

```powershell
bun run build:main        # esbuild → dist/main.cjs
bun run build:preload     # esbuild → dist/preload.cjs
bun run build:renderer    # Vite → dist/renderer/
bun run build:resources   # 复制 resources/ 到 dist/
```

### 3.4 类型检查

```powershell
# 检查所有包
bun run typecheck

# 检查单个包
cd packages/core; bun run typecheck
cd packages/shared; bun run typecheck
cd packages/ui; bun run typecheck
cd apps/electron; bun run typecheck
```

### 3.5 测试

```powershell
# 运行所有测试
bun test

# 运行单个测试文件
bun test apps/electron/src/main/lib/channel-presets.test.ts
```

### 3.6 分发打包

```powershell
cd apps/electron

# Windows 安装包（国内镜像环境变量）
$env:ELECTRON_MIRROR="https://npmmirror.com/mirrors/electron/"
$env:ELECTRON_BUILDER_BINARIES_MIRROR="https://npmmirror.com/mirrors/electron-builder-binaries/"
bun run dist:win

# macOS 安装包
bun run dist:mac

# Linux 安装包
bun run dist:linux

# 当前架构快速打包
bun run dist:fast
```

### 3.7 完整构建与打包流程

```powershell
# 1. 安装依赖
bun install

# 2. 类型检查
bun run typecheck

# 3. 运行测试

cd apps/electron; bun run build

```

## 5. 核心架构

### 5.1 IPC 通信模式（最重要的架构模式）

Proma 的 Electron 架构遵循严格的 IPC 四层通信模式，添加任何新功能涉及 IPC 调用时，**必须同步修改以下四个位置**：

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. @proma/shared                                                │
│    定义 IPC Channel 名称和 TypeScript 类型                       │
│    文件：packages/shared/src/types/*.ts                         │
├─────────────────────────────────────────────────────────────────┤
│ 2. Main Process Handler                                         │
│    在 ipc.ts 中注册 ipcMain.handle 处理器                       │
│    文件：apps/electron/src/main/ipc.ts                          │
├─────────────────────────────────────────────────────────────────┤
│ 3. Preload Bridge                                               │
│    通过 contextBridge.exposeInMainWorld 暴露 API                │
│    文件：apps/electron/src/preload/index.ts                     │
├─────────────────────────────────────────────────────────────────┤
│ 4. Renderer Call                                                │
│    渲染进程通过 window.electronAPI.* 调用                        │
│    文件：apps/electron/src/renderer/                            │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 IPC Channel 分组

| 分组 | 常量名 | 职责 |
|------|--------|------|
| 基础通道 | `IPC_CHANNELS` | 运行时、Git、环境 |
| 渠道管理 | `CHANNEL_IPC_CHANNELS` | AI 供应商渠道 CRUD |
| Chat 功能 | `CHAT_IPC_CHANNELS` | 对话、消息、流式调用 |
| Agent 功能 | `AGENT_IPC_CHANNELS` | Agent 会话、权限、工作区 |
| 环境检查 | `ENVIRONMENT_IPC_CHANNELS` | 环境检测、依赖检查 |
| 代理设置 | `PROXY_IPC_CHANNELS` | HTTP 代理配置 |
| 系统提示词 | `SYSTEM_PROMPT_IPC_CHANNELS` | 系统提示词管理 |
| 记忆功能 | `MEMORY_IPC_CHANNELS` | 跨会话记忆 |
| Chat 工具 | `CHAT_TOOL_IPC_CHANNELS` | 内置工具管理 |
| 飞书集成 | `FEISHU_IPC_CHANNELS` | 飞书机器人桥接 |
| GitHub | `GITHUB_RELEASE_IPC_CHANNELS` | GitHub 发布检查 |

## 6. 包职责详解

### @proma/shared（v0.1.35）

**定位**：零运行时依赖的共享基础包，为所有其他包提供类型定义和常量。

```
导出模块：
  "."          → src/index.ts        # 统一入口
  "./types"    → src/types/index.ts  # 所有类型定义
  "./config"   → src/config/index.ts # 配置
  "./utils"    → src/utils/index.ts  # 工具函数
```

**关键类型**：
- `AgentMessage` — Agent 消息
- `ChatMessage` — Chat 消息
- `Channel` — AI 供应商渠道
- `PermissionRequest` — 权限请求
- `FeishuConfig` / `DingtalkConfig` / `WechatConfig` — 远程机器人配置

**依赖**：仅 TypeScript（零运行时依赖）

### @proma/core（v0.2.17）

**定位**：AI Provider 适配层和代码高亮服务，是 Chat 功能的核心。

```
导出模块：
  "."           → src/index.ts
  "./types"     → src/types/index.ts
  "./utils"     → src/utils/index.ts
  "./providers" → src/providers/index.ts  # Provider 适配器注册表
  "./highlight" → src/highlight/index.ts  # Shiki 代码高亮
```

**Provider 适配器**：
- `anthropic-adapter.ts` — Anthropic Messages API
- `openai-adapter.ts` — OpenAI Chat Completions API
- `google-adapter.ts` — Google Gemini API
- `sse-reader.ts` — 统一 SSE 流式读取

**依赖**：`@proma/shared`、`shiki`
**Peer 依赖**：`@anthropic-ai/claude-agent-sdk`、`@anthropic-ai/sdk`、`@modelcontextprotocol/sdk`

### @proma/ui（v0.1.9）

**定位**：跨包共享的 React UI 组件库。

**关键组件**：
- `CodeBlock` — 代码块（语法高亮 + 复制按钮）
- `MermaidBlock` — Mermaid 图表渲染
- `useSmoothStream` — 流式文本平滑渲染 Hook

**依赖**：`@proma/core`、`beautiful-mermaid`、`shiki`
**Peer 依赖**：`react@^18.3.0`、`react-dom@^18.3.0`

### @proma/electron（v0.10.24）

**定位**：Electron 桌面应用主体，集成所有包，是最终交付的应用。

**关键依赖**：
- `@anthropic-ai/claude-agent-sdk@0.3.143` — Agent SDK（核心）
- `@larksuiteoapi/node-sdk` — 飞书集成
- `dingtalk-stream-sdk-nodejs` — 钉钉集成
- Radix UI 全家桶 — UI 组件
- TipTap 全家桶 — 富文本编辑
- 文件解析：`pdf-parse`、`officeparser`、`word-extractor`、`mammoth`、`pdfjs-dist`

**内部包引用**（使用 `workspace:*`）：
```json
"@proma/core": "workspace:*",
"@proma/shared": "workspace:*",
"@proma/ui": "workspace:*"
```

## 7. Skills 插件系统

### 7.1 概述

Skills 是 Proma Agent 的可复用能力单元。每个工作区可以独立配置 Skills、MCP Server 和工作区文件。

### 7.2 Skill 目录结构

```
skill-name/
├── SKILL.md               # 必需 — Skill 定义（YAML frontmatter + Markdown 指令）
├── scripts/               # 可选 — 可执行脚本（Python/JS）
├── references/            # 可选 — 参考文档（按需加载到上下文）
└── assets/                # 可选 — 资源文件（模板、图标、字体）
```

### 7.3 SKILL.md 格式

```yaml
---
name: skill-name           # Skill 标识符
description: 描述内容       # 触发条件 + 功能描述（核心触发机制）
version: "1.0.0"           # 版本号
---

# Skill 标题

详细的 Markdown 指令内容...
```

`description` 是 Skill 的主要触发机制，需要同时包含"做什么"和"何时使用"。建议描述稍微"积极"一些，以防止 Agent 欠触发。

### 7.6 三级加载系统

Skills 使用渐进式加载，优化上下文窗口占用：

1. **元数据**（name + description）— 始终在上下文中，约 100 词
2. **SKILL.md 正文** — Skill 触发时加载，理想长度 < 500 行
3. **捆绑资源** — 按需加载，无限制（脚本可无需加载直接执行）

## 8. 编码规范与约定

### 8.1 通用规范

- **运行时**：必须使用 Bun，禁止使用 node/npm/pnpm 命令
- **Bun 原生 API 优先**：`Bun.file` > `node:fs`，`Bun.$\`command\`` > `execa`
- **状态管理**：全部使用 Jotai，不使用 Redux / Zustand / Context
- **本地存储优先**：善用配置文件优于 localStorage，不采用本地数据库方案
- **注释语言**：中文注释为主，保留必要的专业术语
- **依赖安装**：先搜索综合判断版本，不默认采用某个版本

### 8.2 UI 设计规范

- UI 组件推荐 **ShadcnUI** 风格
- 用**卡片和阴影**取代边框
- 使用**符合主题的饱满色彩**
- 设置界面要设置背景，为未来做不同主题留下空间
- 保证充分的**组件化**和人类的**可读性**

### 8.3 开发方法论

- 采用 **BDD 行为驱动开发**
- 每次完成改动后运行代码简化，保持简单直接不过度设计
- 保证充分的组件化

### 8.4 包命名规范

所有内部包使用 `@proma/*` 作用域：
- `@proma/shared` — 共享包
- `@proma/core` — 核心包
- `@proma/ui` — UI 组件包
- `@proma/electron` — Electron 应用包

### 8.5 测试规范

- 使用 Bun 内置测试运行器：`import { test, expect } from "bun:test"`
- 测试文件命名：`*.test.ts`
- 测试文件与源文件同目录放置

### 8.6 安全规范

- API Key 通过 Electron `safeStorage`（AES-256-GCM）加密
- 不暴露或日志记录密钥
- 不将密钥提交到代码仓库

## 9. 调试与排错

### 9.1 开发模式调试

```powershell
# 推荐方式 — 自动启动 Vite + Electron
bun run dev

# 分屏调试 — 更稳定，Vite 和 Electron 分开运行
# 终端 1：Vite Dev Server（渲染进程热重载）
cd apps/electron; bun run dev:vite

# 终端 2：Electron 主进程（需手动重启刷新）
cd apps/electron; bun run dev:electron
```

### 9.2 Electron DevTools

- 渲染进程：`Ctrl + Shift + I` 打开 DevTools
- 主进程：使用 `--inspect` 标志启动，通过 Chrome DevTools 连接

### 9.3 常见问题排查

| 现象 | 排查方向 |
|------|----------|
| 白屏 | 检查 Vite 是否正常启动，查看 DevTools Console 报错 |
| IPC 调用无响应 | 检查 `ipc.ts` 是否注册对应 handler，检查 preload 桥接 |
| Agent 无法启动 | 检查渠道配置（需要 Anthropic 或兼容协议），检查 API Key |
| Skills 不触发 | 检查工作区配置，确认 SKILL.md 的 description 描述是否匹配 |
| 构建失败 | 运行 `bun run typecheck` 检查类型错误 |
| 打包时 out 目录占用 | 关闭所有 WorkMate/Electron 进程，删除 `apps/electron/out` 后重试 |

### 9.4 日志

主进程日志输出到终端，渲染进程日志输出到 DevTools Console。部分服务（如 `file-logger.ts`）支持将日志写入文件。

│   └── {workspace-slug}/
