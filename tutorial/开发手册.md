## 1. 项目概述

WorkMate 是一个本地优先的 AI 桌面应用，集成多模型 Chat、通用 Agent、工作区、Skills、MCP、远程机器人和记忆能力。采用 Electron 桌面应用架构，Bun Workspace Monorepo 管理。

核心定位：不只是聊天框，而是一个可以长期沉淀个人工作流的 Agent 工作台——简单问题用 Chat，复杂任务交给 Agent，数据和配置尽量留在本地。

### 2.1 必装工具

#### Git


#### Bun（核心运行时，替代 Node.js/npm/pnpm）

```powershell
# 方式一：PowerShell 安装
powershell -c "irm bun.sh/install.ps1 | iex"

# 方式二：npm 安装（适合已有 Node.js 环境的用户）
```

Bun 是本项目的核心运行时，用于：

- 安装依赖：`bun install`

- 自动加载 `.env` 文件（无需 dotenv）

> **重要**：项目强制使用 Bun，禁止使用 node/npm/pnpm/vite 命令直接运行。参见 `.cursor/rules/use-bun-instead-of-node-vite-npm-pnpm.mdc`。

## 3. 打包构建

### 3.1 本地启动调试（不打包）

开发时无需打包，直接启动开发模式即可热重载调试：

```powershell
# 一键开发模式（推荐 — 自动启动 Vite DevServer + Electron，支持热重载）
bun run dev
```

**工作原理**：

| 进程             | 工具                     | 说明                                                         |
| -------------- | ---------------------- | ---------------------------------------------------------- |
| Vite DevServer | `bun run dev:vite`     | 启动渲染进程开发服务器（HMR 热重载），监听 `localhost:5173`                   |
| Electron 主进程   | `bun run dev:electron` | 先用 esbuild 编译 main/preload，再用 electronmon 监听文件变化自动重启       |
| Preload 桥接     | esbuild                | 编译 `src/preload/index.ts` → `dist/preload.cjs`，主进程变化时自动重编译 |

> 手动分屏调试更稳定（适合排查主进程问题），需要两个终端：
>
> ```powershell
> # 终端 1：Vite DevServer（渲染进程热重载）
> cd apps/electron; bun run dev:vite
> # 终端 2：Electron 主进程（需手动重启刷新）
> cd apps/electron; bun run dev:electron
> ```

### 3.2 构建（编译源码为可执行产物）

构建是将 TypeScript/React 源码编译为 Electron 可加载的产物，**不生成安装包**，用于本地验证或分发打包的前置步骤。

#### Monorepo 顶层构建

```powershell
# 构建所有 packages（shared → core → ui），生成各自的 dist/
bun run build
```

| 包             | 源码                     | 产物                      | 目的                                                 |
| ------------- | ---------------------- | ----------------------- | -------------------------------------------------- |
| @proma/shared | `packages/shared/src/` | `packages/shared/dist/` | 编译共享类型和工具函数，供 core/ui/electron 通过 `workspace:*` 引用 |
| @proma/core   | `packages/core/src/`   | `packages/core/dist/`   | 编译 AI Provider 适配器和代码高亮服务                          |
| @proma/ui     | `packages/ui/src/`     | `packages/ui/dist/`     | 编译共享 React 组件（CodeBlock、MermaidBlock 等）            |

#### Electron 应用构建（`apps/electron/` 目录下）

```powershell
# 一键构建全部（main + preload + renderer + resources）
bun run build
```

等价于依次执行以下四步：

| 步骤         | 命令                        | 源码                     | 产物                 | 目的                                                    |
| ---------- | ------------------------- | ---------------------- | ------------------ | ----------------------------------------------------- |
| 1. 主进程     | `bun run build:main`      | `src/main/index.ts`    | `dist/main.cjs`    | 将主进程 TypeScript 编译为单文件 CJS 包，供 Electron 运行时加载         |
| 2. Preload | `bun run build:preload`   | `src/preload/index.ts` | `dist/preload.cjs` | 编译 Preload 桥接脚本，通过 `contextBridge` 安全暴露 IPC API 给渲染进程 |
| 3. 渲染进程    | `bun run build:renderer`  | `src/renderer/`        | `dist/renderer/`   | Vite 打包 React 应用（HTML/JS/CSS），生成浏览器可加载的静态资源           |
| 4. 资源文件    | `bun run build:resources` | `resources/`           | `dist/resources/`  | 复制应用图标、权限声明等资源文件到 dist 目录                             |

> **构建后直接运行**（不打包安装包）：
>
> ```powershell
> cd apps/electron
> bun run build          # 先构建
> bun run start          # 启动 electron . 加载 dist/ 产物
> ```

### 3.5 分发打包（生成安装包）

分发打包是将构建产物 + Electron 运行时 + 资源文件打包为可分发的安装程序（exe/dmg/deb），**需要先完成构建**。

```powershell
cd apps/electron

# Windows 安装包（国内镜像环境变量）
$env:ELECTRON_MIRROR="https://npmmirror.com/mirrors/electron/"
$env:ELECTRON_BUILDER_BINARIES_MIRROR="https://npmmirror.com/mirrors/electron-builder-binaries/"
bun run dist:win

# macOS 安装包
bun run dist:mac

# Linux 安装包
bun run dist:linux

# 当前架构快速打包（开发测试用）
bun run dist:fast
```

**打包流程说明**：

| 步骤                 | 做了什么                                             | 产物                                    |
| ------------------ | ------------------------------------------------ | ------------------------------------- |
| 1. 下载 Electron 二进制 | 从镜像下载对应平台的 Electron 运行时（约 137MB）                 | 缓存到 `~/.cache/electron/`              |
| 2. 打包 ASAR         | 将 `dist/` + `node_modules/` 等打包为 `app.asar` 归档文件 | `out/win-unpacked/resources/app.asar` |
| 3. 组装应用目录          | 将 Electron 运行时 + app.asar + 额外资源组装为完整应用          | `out/win-unpacked/`                   |
| 4. 生成安装包           | 使用 NSIS 制作 Windows 安装程序                          | `out/WorkMate Setup x.x.x.exe`        |

> 如果打包失败提示 `out` 目录被占用，关闭所有 WorkMate/Electron 进程后删除 `apps/electron/out` 重试。

## 5. 核心架构

### 5.1 IPC 通信模式（最重要的架构模式）

#### 什么是 IPC 通信模式？

IPC（Inter-Process Communication）通信模式是 Electron 架构的**核心规范和约定**，不是某个组件或库。它规定了渲染进程（React UI）如何安全地调用主进程（Node.js）的能力。

**为什么需要它？** Electron 出于安全考虑，渲染进程和主进程运行在完全隔离的环境中：

- **主进程**拥有完整 Node.js 能力（文件系统、网络、系统 API）
- **渲染进程**是浏览器环境，受沙箱限制，无法直接访问系统资源
- 两者之间只能通过 Electron 提供的 `ipcMain` / `ipcRenderer` 通道通信

WorkMate 将这种底层通信封装为**四层规范**，确保每次新增功能时类型安全、接口一致。

#### 四层通信规范

添加任何新功能涉及 IPC 调用时，**必须同步修改以下四个位置**：

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. @proma/shared — 定义协议（类型 + Channel 名称）              │
│                                                                 │
│    在 packages/shared/src/types/*.ts 中：                       │
│    - 定义 Channel 名称常量（如 'channel:list'）                  │
│    - 定义请求/响应的 TypeScript 类型                              │
│    - 这是最底层的"契约"，所有层共用                               │
├─────────────────────────────────────────────────────────────────┤
│ 2. Main Process Handler — 实现业务逻辑                          │
│                                                                 │
│    在 apps/electron/src/main/ipc.ts 中：                        │
│    - ipcMain.handle('channel:list', async () => { ... })        │
│    - 注册处理器，接收参数、执行业务、返回结果                      │
│    - 这是最核心的业务实现层（57KB 的大文件）                       │
├─────────────────────────────────────────────────────────────────┤
│ 3. Preload Bridge — 安全桥接                                    │
│                                                                 │
│    在 apps/electron/src/preload/index.ts 中：                   │
│    - ipcRenderer.invoke('channel:list') → 调用主进程             │
│    - contextBridge.exposeInMainWorld('electronAPI', { ... })    │
│    - 将底层 IPC 调用包装为语义化的 API 函数                       │
│    - 这一层是安全边界，决定了渲染进程能调用什么                     │
├─────────────────────────────────────────────────────────────────┤
│ 4. Renderer Call — UI 层调用                                    │
│                                                                 │
│    在 apps/electron/src/renderer/ 中：                          │
│    - window.electronAPI.listChannels()                          │
│    - React 组件 / Jotai atom 中直接调用，像普通 async 函数一样    │
│    - 完全不感知 IPC 的存在                                       │
└─────────────────────────────────────────────────────────────────┘
```

#### 一次完整的 IPC 调用流程（以"获取渠道列表"为例）

```
React 组件                         Preload                          主进程
──────────                         ───────                          ──────
window.electronAPI.listChannels()
    │
    ├──→ ipcRenderer.invoke('channel:list')
    │         │
    │         ├──→ Electron IPC 通道 ───→ ipcMain.handle('channel:list')
    │         │                                    │
    │         │                                    ├── 读取 channels.json
    │         │                                    ├── 解密 API Key
    │         │                                    └── return Channel[]
    │         │
    │         ├──← Promise<Channel[]> ←────────────┘
    │
    ├──← Channel[]
```

#### 四层修改示例（新增"删除渠道"功能）

```typescript
// 1. @proma/shared — 定义 Channel 名称和类型
// packages/shared/src/types/channel.ts
export const CHANNEL_IPC_CHANNELS = {
  // ...已有的 LIST, CREATE 等
  DELETE: 'channel:delete',  // ← 新增
}
export interface ChannelDeleteInput { id: string }  // ← 新增类型

// 2. Main Process Handler — 实现业务逻辑
// apps/electron/src/main/ipc.ts
ipcMain.handle(CHANNEL_IPC_CHANNELS.DELETE,
  async (_, id: string): Promise<void> => {
    // 执行删除逻辑
  }
)

// 3. Preload Bridge — 暴露 API
// apps/electron/src/preload/index.ts
deleteChannel: (id: string) => ipcRenderer.invoke(CHANNEL_IPC_CHANNELS.DELETE, id),

// 4. Renderer — UI 调用
// apps/electron/src/renderer/xxx.tsx
await window.electronAPI.deleteChannel(channelId)
```

## 6. 包职责详解

### @proma/shared（v0.1.35）

**定位**：零运行时依赖的共享基础包，为所有其他包提供类型定义和常量。

```
导出模块：
  "."          → src/index.ts        # 统一入口
  "./types"    → src/types/index.ts  # 所有类型定义
  "./config"   → src/config/index.ts # 配置
  "./utils"    → src/utils/index.ts  # 工具函数
```

**关键类型**：

- `AgentMessage` — Agent 消息
- `ChatMessage` — Chat 消息
- `Channel` — AI 供应商渠道
- `PermissionRequest` — 权限请求
- `FeishuConfig` / `DingtalkConfig` / `WechatConfig` — 远程机器人配置

**依赖**：仅 TypeScript（零运行时依赖）

### @proma/core（v0.2.17）

**定位**：AI Provider 适配层和代码高亮服务，是 Chat 功能的核心。

```
导出模块：
  "."           → src/index.ts
  "./types"     → src/types/index.ts
  "./utils"     → src/utils/index.ts
  "./providers" → src/providers/index.ts  # Provider 适配器注册表
  "./highlight" → src/highlight/index.ts  # Shiki 代码高亮
```

**Provider 适配器**：

- `anthropic-adapter.ts` — Anthropic Messages API
- `openai-adapter.ts` — OpenAI Chat Completions API
- `google-adapter.ts` — Google Gemini API
- `sse-reader.ts` — 统一 SSE 流式读取

**依赖**：`@proma/shared`、`shiki`
**Peer 依赖**：`@anthropic-ai/claude-agent-sdk`、`@anthropic-ai/sdk`、`@modelcontextprotocol/sdk`

### @proma/ui（v0.1.9）

**定位**：跨包共享的 React UI 组件库。

**关键组件**：

- `CodeBlock` — 代码块（语法高亮 + 复制按钮）
- `MermaidBlock` — Mermaid 图表渲染
- `useSmoothStream` — 流式文本平滑渲染 Hook

**依赖**：`@proma/core`、`beautiful-mermaid`、`shiki`
**Peer 依赖**：`react@^18.3.0`、`react-dom@^18.3.0`

### @proma/electron（v0.10.24）

**定位**：Electron 桌面应用主体，集成所有包，是最终交付的应用。

**关键依赖**：

## 7. Skills 插件系统

### 7.1 概述

Skills 是 WorkMate Agent 的可复用能力单元。每个工作区可以独立配置 Skills、MCP Server 和工作区文件。

### 7.2 Skill 目录结构

```
skill-name/
├── SKILL.md               # 必需 — Skill 定义（YAML frontmatter + Markdown 指令）
├── scripts/               # 可选 — 可执行脚本（Python/JS）
├── references/            # 可选 — 参考文档（按需加载到上下文）
└── assets/                # 可选 — 资源文件（模板、图标、字体）
```

### 7.3 SKILL.md 格式

```yaml
---
name: skill-name           # Skill 标识符
description: 描述内容       # 触发条件 + 功能描述（核心触发机制）
version: "1.0.0"           # 版本号
---

# Skill 标题

详细的 Markdown 指令内容...
```

`description` 是 Skill 的主要触发机制，需要同时包含"做什么"和"何时使用"。建议描述稍微"积极"一些，以防止 Agent 欠触发。

### 7.4 内置 Default Skills

Skills 使用渐进式加载，优化上下文窗口占用：

2. **SKILL.md 正文** — Skill 触发时加载，理想长度 < 500 行
3. **捆绑资源** — 按需加载，无限制（脚本可无需加载直接执行）

## 8. 编码规范与约定

### 8.1 通用规范

- **运行时**：必须使用 Bun，禁止使用 node/npm/pnpm 命令
- **Bun 原生 API 优先**：`Bun.file` > `node:fs`，`Bun.$\`command\``>`execa\`
- **状态管理**：全部使用 Jotai，不使用 Redux / Zustand / Context
- **本地存储优先**：善用配置文件优于 localStorage，不采用本地数据库方案
- **注释语言**：中文注释为主，保留必要的专业术语
- **依赖安装**：先搜索综合判断版本，不默认采用某个版本

### 8.2 UI 设计规范

- UI 组件推荐 **ShadcnUI** 风格
- 用**卡片和阴影**取代边框
- 使用**符合主题的饱满色彩**
- 设置界面要设置背景，为未来做不同主题留下空间
- 保证充分的**组件化**和人类的**可读性**

### 8.3 开发方法论

- 采用 **BDD 行为驱动开发**
- 每次完成改动后运行代码简化，保持简单直接不过度设计
- 保证充分的组件化

### 8.4 包命名规范

所有内部包使用 `@proma/*` 作用域：

- `@proma/shared` — 共享包
- `@proma/core` — 核心包
- `@proma/ui` — UI 组件包
- `@proma/electron` — Electron 应用包

### 8.5 测试规范

- 使用 Bun 内置测试运行器：`import { test, expect } from "bun:test"`
- 测试文件命名：`*.test.ts`
- 测试文件与源文件同目录放置

### 8.6 安全规范

- API Key 通过 Electron `safeStorage`（AES-256-GCM）加密
- 不暴露或日志记录密钥
- 不将密钥提交到代码仓库

## 9. 调试与排错

### 9.1 开发模式调试

```powershell
# 推荐方式 — 自动启动 Vite + Electron
bun run dev

# 分屏调试 — 更稳定，Vite 和 Electron 分开运行
# 终端 1：Vite Dev Server（渲染进程热重载）
cd apps/electron; bun run dev:vite

# 终端 2：Electron 主进程（需手动重启刷新）
cd apps/electron; bun run dev:electron
```

### 9.2 Electron DevTools

- 渲染进程：`Ctrl + Shift + I` 打开 DevTools
- 主进程：使用 `--inspect` 标志启动，通过 Chrome DevTools 连接

### 9.3 常见问题排查

| 现象           | 排查方向                                                 |
| ------------ | ---------------------------------------------------- |
| 白屏           | 检查 Vite 是否正常启动，查看 DevTools Console 报错                |
| IPC 调用无响应    | 检查 `ipc.ts` 是否注册对应 handler，检查 preload 桥接             |
| Agent 无法启动   | 检查渠道配置（需要 Anthropic 或兼容协议），检查 API Key                |
| Skills 不触发   | 检查工作区配置，确认 SKILL.md 的 description 描述是否匹配             |
| 构建失败         | 运行 `bun run typecheck` 检查类型错误                        |
| 打包时 out 目录占用 | 关闭所有 WorkMate/Electron 进程，删除 `apps/electron/out` 后重试 |

### 9.4 日志

主进程日志输出到终端，渲染进程日志输出到 DevTools Console。部分服务（如 `file-logger.ts`）支持将日志写入文件。

### 本地数据目录（运行时）

WorkMate 运行时将数据存储在 `~/.proma/` 目录下：

```
~/.proma/
├── channels.json              # 渠道配置（API Key 加密存储）
├── conversations.json         # 对话列表
├── conversations/             # 对话消息（JSONL 追加写入）
│   └── {conversation-id}.jsonl
├── agent-sessions.json        # Agent 会话列表
├── agent-sessions/            # Agent 会话消息
│   └── {session-id}.jsonl
├── agent-workspaces/          # Agent 工作区
│   └── {workspace-slug}/
│       ├── workspace-files/   #   工作区文件
│       ├── mcp.json           #   MCP Server 配置
│       └── skills/            #   工作区专属 Skills
├── attachments/               # 附件文件
├── user-profile.json          # 用户资料
├── settings.json              # 全局设置
└── sdk-config/                # SDK 配置
```

> API Key 通过 Electron `safeStorage`（AES-256-GCM）加密后写入 `channels.json`。项目不使用本地数据库，核心数据以 JSON 配置 + JSONL 追加日志为主。
