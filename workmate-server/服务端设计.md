# WorkMate Server

WorkMate 伴行后端服务，为 AiWorkbench 客户端提供模型权限查询、升级检测、观测数据接收和管理后台。

## 项目概述

本项目是 WorkMate 伴行（Proma）的配套后端服务，独立于 Electron 桌面客户端，采用 Node.js + MySQL 架构，包含管理台前端页面。

### 核心功能

| 序号 | 功能 | 说明 |
|------|------|------|
| 1 | 模型列表查询 | 代理查询大模型平台，返回用户可用的模型列表 |
| 2 | 升级检测 | 客户端查询是否有新版本可升级，支持白名单灰度发布 |
| 3 | 观测数据接收 | 接收客户端上报的用户提问观测数据和异常数据 |
| 4 | 管理后台 | Web 管理台，配置升级白名单、查看观测数据 |

### 关键设计决策

| 决策 | 原因 |
|------|------|
| **不存储用户表** | 用户身份通过 EIP 网关注入的 `X-EIPGW-USERID` Header（AES-128-GCM 加密）确定 |
| **不存储模型表** | 模型列表由大模型平台统一管理，本服务仅做代理转发，不做本地缓存 |
| **白名单用规则匹配** | 支持逗号分隔列表、范围区间、前缀/后缀通配符，灵活覆盖组织架构场景 |
| **管理台访问控制** | 管理台不设独立登录，通过 EIP 网关注入的工号 + `admin_whitelist` 表规则匹配控制 |

### 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 运行时 | Node.js 20+ | 服务端运行时 |
| 语言 | TypeScript | 类型安全 |
| Web 框架 | Express.js | HTTP 服务 + RESTful API |
| 数据库 | MySQL 8.0 | 关系型数据库 |
| ORM | Prisma | 类型安全的数据库访问 |
| 前端框架 | React 18 + Vite | 管理台 SPA |

| 状态管理 | Jotai | 与客户端一致的状态管理 |
| 构建 | esbuild | 快速打包 |
| 部署 | Docker + PM2 | 容器化 + 进程管理 |

## 二、身份传递约定

### 2.1 整体流程

客户端请求经过 EIP 网关后，网关自动在 Header 中注入 AES-128-GCM 加密的工号：

```
客户端 ──→ EIP 网关（校验 Cookie + 网关鉴权）──→ WorkMate Server
                                                  │
                                   Header: X-EIPGW-USERID: <AES-128-GCM 密文 (Base64)>
```

服务端解密该 Header 即可获取工号，客户端无需自行处理加密逻辑。

### 2.2 加解密方案

| 参数 | 值 | 说明 |
|------|-----|------|
| 算法 | AES-128-GCM | 128 位密钥（16 字节） |
| IV 长度 | 12 字节 | |
| 密钥 | 16 字节 | EIP 网关与后端共享，通过环境变量 `USER_ID_ENCRYPTION_KEY` 配置 |
| 编码 | Base64 | 密文（含 IV + AuthTag）经 Base64 编码放入 Header |

#### 数据流

```
EIP 网关侧：
  1. 生成 12 字节随机 IV
  2. cipher = AES-128-GCM.encrypt(plainJobId, key, iv)
  3. combined = iv(12B) + cipher(N bytes) + authTag(16B)
  4. Header: X-EIPGW-USERID: <base64(combined)>

WorkMate Server 侧（extract-user-id 中间件）：
  1. combined = Buffer.from(header, 'base64')
  2. iv = combined.subarray(0, 12)
  3. authTag = combined.subarray(combined.length - 16)
  4. ciphertext = combined.subarray(12, combined.length - 16)
  5. plainJobId = AES-128-GCM.decrypt(ciphertext, key, iv, authTag)
  6. req.jobId = plainJobId
```

### 2.3 extract-user-id 中间件

```typescript
// middleware/extract-user-id.ts
//
// 从请求 Header 中解密 EIP 网关注入的 X-EIPGW-USERID，获取用户工号。
// 如果 Header 缺失或解密失败，返回 403。

import { Request, Response, NextFunction } from 'express'
import { createDecipheriv } from 'node:crypto'

declare global {
  namespace Express {
    interface Request {
      jobId?: string
    }
  }
}

const KEY_HEX = process.env.USER_ID_ENCRYPTION_KEY ?? ''
const KEY = Buffer.from(KEY_HEX, 'hex')

function decryptJobId(encryptedBase64: string): string {
  const combined = Buffer.from(encryptedBase64, 'base64')
  const iv = combined.subarray(0, 12)
  const authTag = combined.subarray(combined.length - 16)
  const ciphertext = combined.subarray(12, combined.length - 16)

  const decipher = createDecipheriv('aes-128-gcm', KEY, iv, { authTagLength: 16 })
  decipher.setAuthTag(authTag)
  const decrypted = Buffer.concat([decipher.update(ciphertext), decipher.final()])
  return decrypted.toString('utf-8')
}

export function extractUserId(req: Request, res: Response, next: NextFunction): void {
  const encrypted = req.headers['x-eipgw-userid'] as string | undefined

  if (!encrypted) {
    res.status(403).json({ code: 403, message: '缺少用户身份信息', timestamp: Date.now() })
    return
  }

  try {
    req.jobId = decryptJobId(encrypted)
    next()
  } catch (error) {
    res.status(403).json({ code: 403, message: '用户身份验证失败', timestamp: Date.now() })
  }
}
```

### 3.2 Prisma Schema

```prisma
// prisma/schema.prisma

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "mysql"
  url      = env("DATABASE_URL")
}

// ===== 升级发布表 =====

model UpgradeRelease {
  id           Int         @id @default(autoincrement())
  version      String      @db.VarChar(32)
  releaseType  ReleaseType @default(UPGRADE) @map("release_type")
  releaseNotes String      @map("release_notes") @db.Text
  downloadUrl  String      @map("download_url") @db.VarChar(512)
  platform     String      @db.VarChar(32)   // win32 / darwin / linux
  minVersion   String?     @map("min_version") @db.VarChar(32) // 升级场景的最低版本（回退忽略）
  isActive     Boolean     @default(true) @map("is_active")
  publishedAt  DateTime    @default(now()) @map("published_at")

  @@map("upgrade_releases")
}

enum ReleaseType {
  UPGRADE   // 升级（目标版本 > 当前版本）
  ROLLBACK  // 回退（目标版本 < 当前版本）
}

// ===== 升级白名单表 =====
//
// 通过规则匹配用户工号，决定是否允许升级到指定版本。
// 例如：
//   rule_type='list',   rule_value='022480' → 逗号分隔工号列表 022480
//   rule_type='range',   rule_value='022480-023480' → 工号在区间内
//   rule_type='prefix',  rule_value='022*' → 工号以 022 开头
//   rule_type='suffix',  rule_value='*022' → 工号以 022 结尾

model UpgradeWhitelist {
  id               Int      @id @default(autoincrement())
  sourceStrategyId Int?     @map("source_strategy_id")
  ruleType         String   @map("rule_type") @db.VarChar(16)
  ruleValue        String   @map("rule_value") @db.VarChar(256)
  targetVersion    String?  @map("target_version") @db.VarChar(32)
  platform         String?  @db.VarChar(32)
  isActive         Boolean  @default(true) @map("is_active")
  createdAt        DateTime @default(now()) @map("created_at")

  strategy         UpgradeStrategy? @relation(fields: [sourceStrategyId], references: [id], onDelete: SetNull)

  @@index([isActive])
  @@index([sourceStrategyId])
  @@map("upgrade_whitelist")
}

// ===== 升级策略表 =====
//
// 支持多阶段灰度升级。每个策略包含多个阶段，每个阶段有不同的白名单规则。
// 当前活跃阶段的白名单规则会自动同步到 upgrade_whitelist 表。
// 同步时：targetVersion = 策略的 targetVersion，platform = 策略的 platform。
//
// 状态流转: draft → active → paused → finished
//   - draft: 草稿，可编辑
//   - active: 已启动，current_stage > 0
//   - paused: 暂停，暂不推进
//   - finished: 已完成（所有阶段已执行完毕）

model UpgradeStrategy {
  id            Int           @id @default(autoincrement())
  name          String        @db.VarChar(128)
  targetVersion String        @map("target_version") @db.VarChar(32)
  downloadUrl   String        @map("download_url") @db.VarChar(512)
  releaseNotes  String?       @map("release_notes") @db.Text
  platform      String        @db.VarChar(32)   // win32 / darwin / linux
  minVersion    String?       @map("min_version") @db.VarChar(32)
  totalStages        Int           @map("total_stages")
  currentStage       Int           @default(0) @map("current_stage")   // 0=未启动
  soakTimeMinutes    Int?          @map("soak_time_minutes")  // 阶段间最小浸泡时间（分钟），0或不填=不限
  autoPauseErrorRate Decimal?      @map("auto_pause_error_rate") @db.Decimal(5, 4)  // 自动暂停错误率阈值
  autoPauseEnabled   Boolean       @default(false) @map("auto_pause_enabled")
  status             StrategyStatus @default(DRAFT)
  createdAt     DateTime      @default(now()) @map("created_at")
  updatedAt     DateTime      @updatedAt @map("updated_at")

  stages        UpgradeStrategyStage[]
  whitelistRules UpgradeWhitelist[]

  @@map("upgrade_strategies")
}

enum StrategyStatus {
  DRAFT     // 草稿
  ACTIVE    // 进行中
  PAUSED    // 已暂停
  FINISHED  // 已完成
}

// ===== 升级策略阶段表 =====

model UpgradeStrategyStage {
  id           Int      @id @default(autoincrement())
  strategyId   Int      @map("strategy_id")
  stageOrder   Int      @map("stage_order")   // 1, 2, 3...
  name         String   @db.VarChar(64)       // "内部测试" / "部门推广" / "全量发布"
  releaseNotes String?  @map("release_notes") @db.Text
  advancedAt   DateTime? @map("advanced_at")   // 该阶段激活时间，用于浸泡时间计算
  createdAt    DateTime @default(now()) @map("created_at")

  strategy     UpgradeStrategy             @relation(fields: [strategyId], references: [id], onDelete: Cascade)
  rules        UpgradeStrategyStageRule[]

  @@unique([strategyId, stageOrder])
  @@map("upgrade_strategy_stages")
}

// ===== 策略阶段规则表 =====

model UpgradeStrategyStageRule {
  id        Int    @id @default(autoincrement())
  stageId   Int    @map("stage_id")
```

## 四、白名单匹配引擎

### 4.1 规则类型

| 类型 | rule_value 示例 | 匹配逻辑 | 匹配的工号示例 |
|------|----------------|----------|---------------|
| `list` | `022480,021220,012333` | `ruleValue.split(',').includes(jobId)` | `022480,021220,012333` |
| `range` | `022480-023480` | `+jobId >= rangeStart && +jobId <= rangeEnd` | `022480`, `022500`, `023480` |
| `prefix` | `022*` | `jobId.startsWith(prefix)` | `022123`, `022999`, `022000` |
| `suffix` | `*022` | `jobId.endsWith(suffix)` | `123022`, `456022` |

### 4.2 核心匹配算法

```typescript
// utils/whitelist-matcher.ts

export type WhitelistRuleType = 'list' | 'range' | 'prefix' | 'suffix'

export interface WhitelistRule {
  id: number
  ruleType: WhitelistRuleType
  ruleValue: string
}

export function matchJobId(jobId: string, rules: WhitelistRule[]): boolean {
  for (const rule of rules) {
    if (matchSingleRule(jobId, rule)) {
      return true
    }
  }
  return false
}

function matchSingleRule(jobId: string, rule: WhitelistRule): boolean {
  switch (rule.ruleType) {
    case 'list':
      return rule.ruleValue.split(',').includes(jobId)

    case 'range': {
      const [start, end] = rule.ruleValue.split('-')
      if (!start || !end) return false
      const jobIdNum = parseInt(jobId, 10)
      const startNum = parseInt(start, 10)
      const endNum = parseInt(end, 10)
      return jobIdNum >= startNum && jobIdNum <= endNum
    }

    case 'prefix': {
      const prefix = rule.ruleValue.slice(0, -1)
      return jobId.startsWith(prefix)
    }

    case 'suffix': {
      const suffix = rule.ruleValue.slice(1)
      return jobId.endsWith(suffix)
    }

    default:
      return false
  }
}
```

### 4.3 whitelist.service.ts 服务封装

```typescript
// services/whitelist.service.ts

import { PrismaClient } from '@prisma/client'
import { matchJobId, type WhitelistRule } from '../utils/whitelist-matcher'

const prisma = new PrismaClient()

export async function isInUpgradeWhitelist(
  jobId: string,
  targetVersion?: string,
  platform?: string,
): Promise<boolean> {
  const rules = await prisma.upgradeWhitelist.findMany({
    where: {
      isActive: true,
      ...(targetVersion ? { targetVersion } : {}),
      ...(platform ? { platform } : {}),
    },
    select: { id: true, ruleType: true, ruleValue: true },
  }) as WhitelistRule[]

}

export async function isInAdminWhitelist(jobId: string): Promise<boolean> {
  const rules = await prisma.adminWhitelist.findMany({
    where: { isActive: true },
    select: { id: true, ruleType: true, ruleValue: true },

  return matchJobId(jobId, rules)
}
```

**升级检测核心流程**：

```
1. 客户端请求 GET /upgrade/check?currentVersion=X&platform=Y
   (经过 EIP 网关，自动注入 X-EIPGW-USERID)
2. extract-user-id 中间件解密 Header → req.jobId
3. 首先检查是否有活跃的升级策略（upgrade_strategies 中 status=ACTIVE）：
   a. 如果有活跃策略：
      - targetVersion = 策略的 targetVersion
      - releaseType = 'upgrade'
      - downloadUrl / releaseNotes = 策略中的值
      - 白名单：从 upgrade_whitelist 中查找 targetVersion=策略.targetVersion 的规则
        （这些规则由策略推进时自动同步到 upgrade_whitelist）
      - 策略的信息优先级高于 upgrade_releases 表
   b. 如果没有活跃策略：
      - 查询 upgrade_releases 表：WHERE platform=Y AND isActive=true LIMIT 1
      - 使用表中的 releaseType / targetVersion / downloadUrl 等
4. 如果无 active 发布且无 active 策略 → hasUpgrade=false
5. 获取 targetVersion 和 releaseType：
   a. 如果 targetVersion === currentVersion → hasUpgrade=false（已是最新/已在目标版本）
   b. 如果 releaseType='upgrade' 且 targetVersion < currentVersion
      → hasUpgrade=false（当前版本已高于升级目标）
   c. 如果 releaseType='rollback' 且 targetVersion > currentVersion
      → hasUpgrade=false（当前版本已低于回退目标）
6. 需要执行：
   a. releaseType='upgrade'：
      - 检查 minVersion：currentVersion < minVersion → upgradeAllowed=false
      - 否则继续检查白名单
   b. releaseType='rollback'：
      - 不检查 minVersion（回退场景不适用）
      - 直接检查白名单
   c. 白名单检查：
      - 查询 upgrade_whitelist WHERE isActive=true AND (targetVersion IS NULL OR targetVersion=targetVersion)
        AND (platform IS NULL OR platform=Y)
      - 白名单为空 → 全员可用，upgradeAllowed=true
      - 白名单非空 → 用 req.jobId 匹配规则
        - 匹配成功 → upgradeAllowed=true
        - 匹配失败 → upgradeAllowed=false
7. 返回结果
```

**策略与发布表的关系**：

| 场景 | 数据来源 | upgrade_whitelist 来源 |
|------|----------|------------------------|
| 有活跃策略 | `upgrade_strategies` 表 | 策略推进时自动同步 |
| 无活跃策略 + 有活跃发布 | `upgrade_releases` 表 | 手动在管理台配置 |
| 无活跃策略 + 无活跃发布 | — | hasUpgrade=false |

**版本比较优先级**：

| 当前版本 | 目标版本 | releaseType | 结果 |
|----------|----------|-------------|------|
| 0.10.24 | 0.11.0 | upgrade | hasUpgrade=true → 引导升级 |
| 0.11.0 | 0.10.24 | rollback | hasUpgrade=true → 引导回退 |
| 0.11.0 | 0.10.24 | upgrade | hasUpgrade=false → 当前已更新 |
| 0.10.24 | 0.10.24 | 任意 | hasUpgrade=false → 已是目标版本 |
| 0.10.24 | 0.9.0 | rollback | hasUpgrade=true → 引导回退 |
| 0.9.0 | 0.10.24 | rollback | hasUpgrade=false → 无需回退 |

#### 升级策略管理接口详解

升级策略支持**多阶段灰度发布**：管理员提前配置若干阶段的策略（每个阶段的白名单规则不同），然后使用管理台控制逐步推进。

**阶段白名单规则 = 当前阶段及之前所有阶段的规则累加**。

**策略创建示例**：


**阶段规则生效逻辑**：

```
策略启动后，阶段推进时的白名单累积：

第 0 阶段（未启动）→ upgrade_whitelist 为空，无人可升级
第 1 阶段（内部测试）→ 白名单 = Stage 1 的规则（022480, 021220, 012333）
第 2 阶段（部门推广）→ 白名单 = Stage 1 + Stage 2 的规则
                            （list: 022480,021220,012333 + range: 022000-029999）
第 3 阶段（全量放开）→ 白名单 = Stage 1 + Stage 2 + Stage 3 的规则
                            （list + range + prefix: 02* + suffix: *01）
```

### admin-whitelist 中间件

```typescript
// middleware/admin-whitelist.ts

import { Request, Response, NextFunction } from 'express'
import { isInAdminWhitelist } from '../services/whitelist.service'

export async function adminWhitelistGuard(
  req: Request, res: Response, next: NextFunction,
): Promise<void> {
  const jobId = req.jobId
  if (!jobId) {
    res.status(403).json({ code: 403, message: '身份信息缺失', timestamp: Date.now() })
    return
  }

  const hasAccess = await isInAdminWhitelist(jobId)
  if (!hasAccess) {
    res.status(403).json({ code: 403, message: '无管理台访问权限', timestamp: Date.now() })
    return
  }

  next()
}
```

## 十、灰度升级策略优化

> 参考业界 Canary Release、Ring-based Deployment、Feature Flag 等成熟实践，对现有多阶段灰度升级方案进行增强。

### 10.1 浸泡时间（Soak Time）

**问题**：当前设计允许管理员在极短时间内连续推进多个阶段，缺乏对每阶段稳定性的观察窗口。

**优化**：在 `upgrade_strategies` 表增加 `soak_time_minutes` 字段，在 `upgrade_strategy_stages` 表增加 `advanced_at` 字段。

```prisma
model UpgradeStrategy {
  // ... 现有字段 ...
  soakTimeMinutes  Int?     @map("soak_time_minutes")  // 阶段间最小浸泡时间（分钟），0或不填=不限
}

model UpgradeStrategyStage {
  // ... 现有字段 ...
  advancedAt       DateTime? @map("advanced_at")       // 该阶段激活时间，用于浸泡时间计算
}
```

**推进前校验**：`next-stage` 接口先检查 `now() - currentStage.advancedAt >= strategy.soakTimeMinutes * 60s`，未达浸泡时间则拒绝推进并返回剩余等待时间。

**管理台前端**：阶段推进确认弹窗显示"浸泡进度"（已过 X 分钟 / 需 Y 分钟），未达标时按钮置灰。

### 10.2 健康度检查与自动暂停

**问题**：当前阶段推进完全依赖人工判断，无自动化指标支撑。

**优化**：增加升级健康度检查机制。

#### 10.2.1 健康度指标定义

| 指标 | 来源 | 说明 |
|------|------|------|
| 已升级用户错误率 | `observability_events` 表 | `SELECT COUNT(*) WHERE event_type='error' AND client_version=targetVersion` / 同版本总事件数 |
| 已升级用户活跃率 | `observability_events` 表 | 有上报事件的已升级用户比例 |
| 平均响应时长 | `observability_events` 表 | 新版本用户的平均 `response_duration_ms` |

#### 10.2.2 策略表增加阈值字段

```prisma
model UpgradeStrategy {
  // ... 现有字段 ...
  soakTimeMinutes        Int?     @map("soak_time_minutes")
  autoPauseErrorRate     Decimal? @map("auto_pause_error_rate") @db.Decimal(5, 4)  // 自动暂停错误率阈值，如 0.05=5%
  autoPauseEnabled       Boolean  @default(false) @map("auto_pause_enabled")
}
```

#### 10.2.3 健康度检查 API

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/workmate-console/upgrade/strategies/:id/health` | 获取策略当前健康度 |

```typescript
// Response
{
  "code": 0,
  "data": {
    "strategyId": 1,
    "currentStage": 2,
    "targetVersion": "0.11.0",
    "metrics": {
      "upgradedUsers": 120,
      "errorRate": 0.03,
      "avgResponseMs": 2800,
      "activeRate": 0.85
    },
    "threshold": { "autoPauseErrorRate": 0.05 },
    "healthStatus": "healthy",       // healthy | warning | critical
    "soakProgress": {
      "elapsedMinutes": 180,
      "requiredMinutes": 240,
      "isReady": false
    }
  }
}
```

#### 10.2.4 自动暂停流程

定时任务（每 10 分钟）检查所有 `status=ACTIVE` 的策略：
1. 查询已升级到 targetVersion 的用户错误率
2. 如果错误率 >= `autoPauseErrorRate` → 自动将策略 status 设为 `PAUSED`，记录日志
3. 管理台推送通知（WebSocket 或轮询）告警

### 10.3 阶段推进前置条件

推进到下一阶段前，系统自动检查：

| 检查项 | 条件 | 不通过时 |
|--------|------|----------|
| 浸泡时间 | `now() - advancedAt >= soakTimeMinutes * 60s` | 返回剩余时间 |
| 错误率 | 当前阶段错误率 < `autoPauseErrorRate` | 返回警告，允许强制推进 |
| 上一阶段用户数 | 当前阶段已有 >= N 名用户成功升级 | 返回已升级人数，允许强制推进 |

**API 扩展**：`next-stage` 接口支持 `force: boolean` 参数，默认 `false`。非强制模式拒绝不满足前置条件的推进；强制模式跳过检查但仍记录警告日志。

## 十一、观测数据与异常上报优化

> 参考 Sentry、OpenTelemetry、ELK 等业界实践，对观测数据模型和上报策略进行增强。

### 11.1 错误指纹（Error Fingerprinting）

**问题**：相同的错误因不同用户、不同时间上报产生大量重复记录，难以统计和聚合。

**优化**：在 `observability_events` 表增加 `error_fingerprint` 字段。

```prisma
model ObservabilityEvent {
  // ... 现有字段 ...
  errorFingerprint String? @map("error_fingerprint") @db.VarChar(64)  // 错误指纹，用于聚合相同错误
  breadcrumbs      String? @db.Text  // JSON: 事件发生前的用户操作路径
  tags             String? @db.Text  // JSON: 自定义标签 key:value 对

  @@index([errorFingerprint])
}
```

**指纹生成算法**（服务端）：

```typescript
import { createHash } from 'node:crypto'

function generateErrorFingerprint(event: { errorType?: string; errorMessage?: string; errorStack?: string }): string {
  const raw = [event.errorType ?? '', extractTopFrame(event.errorStack) ?? '', event.errorMessage ?? ''].join('|')
  return createHash('md5').update(raw).digest('hex').slice(0, 16)
}

function extractTopFrame(stack?: string): string | null {
  if (!stack) return null
  const match = stack.match(/at\s+(.+?)(?:\s+\(|$)/)
  return match ? match[1] : null
}
```

**客户端侧**：在 `error` 事件中增加 `fingerprint` 字段，由客户端根据 `error.type + error.message` 生成，服务端再根据 `errorType + errorStack 第一帧` 补全校验。

### 11.2 面包屑（Breadcrumbs）

**问题**：仅捕获异常堆栈难以还原用户操作上下文。

**优化**：客户端维护最近 20 条面包屑（用户操作记录），在异常上报时附加。

```typescript
export interface Breadcrumb {
  type: 'navigation' | 'user_action' | 'system' | 'http'
  category: string       // 如 'ui.click', 'navigation', 'api.call'
  message: string        // 人类可读描述
  timestamp: number
  data?: Record<string, unknown>  // 附加数据（URL、按钮ID等）
}
```

客户端 `ObservabilityService` 增加面包屑功能：

```typescript
private breadcrumbs: Breadcrumb[] = []
private static MAX_BREADCRUMBS = 20

addBreadcrumb(crumb: Breadcrumb): void {
  this.breadcrumbs.push(crumb)
  if (this.breadcrumbs.length > ObservabilityService.MAX_BREADCRUMBS) {
    this.breadcrumbs.shift()
  }
}

reportErrorEvent(error: Error, context?: Record<string, unknown>): void {
  this.reportEvent({
    type: 'error',
    error: { type: error.name, message: error.message, stack: error.stack },
    breadcrumbs: [...this.breadcrumbs],   // 附加面包屑
    ...context,
  })
}
```

集成点：
- 路由切换：`addBreadcrumb({ type: 'navigation', category: 'route', message: '进入设置页' })`
- 按钮点击：`addBreadcrumb({ type: 'user_action', category: 'ui.click', message: '点击发送按钮' })`
- API 调用：`addBreadcrumb({ type: 'http', category: 'api.call', message: 'POST /workmate/observability/events' })`

### 11.3 客户端上报采样与限流

**问题**：高并发场景下客户端可能产生大量事件，冲击服务端。

**优化**：客户端增加采样率和限流机制。

```typescript
export interface ObservabilityConfig {
  enabled: boolean
  url: string              // 完整上报地址，如 https://host/workmate/observability/events
  timeoutMs?: number
  maxQueueSize?: number
  flushIntervalMs?: number
  maxBatchSize?: number
  sampleRate?: number         // 采样率 0.0~1.0，默认 1.0（全量上报）
  maxEventsPerMinute?: number // 每分钟最大事件数，默认 60
  enableBreadcrumbs?: boolean // 是否启用面包屑，默认 true
}
```

**采样逻辑**：

```typescript
private eventsThisMinute = 0
private minuteWindowStart = Date.now()

reportEvent(item: ObservabilityEventItem): void {
  // error 类型始终上报，不受采样率和限流影响
  if (item.type !== 'error') {
    // 限流：每分钟上限
    const now = Date.now()
    if (now - this.minuteWindowStart > 60_000) {
      this.eventsThisMinute = 0
      this.minuteWindowStart = now
    }
    this.eventsThisMinute++
    if (this.eventsThisMinute > (this.config.maxEventsPerMinute ?? 60)) return

    // 采样
    if (Math.random() > (this.config.sampleRate ?? 1.0)) return
  }

  this.eventQueue.push(item)
  if (this.eventQueue.length >= (this.config.maxQueueSize ?? 100)) {
    this.flushQueue()
  }
}
```

**注意**：`error` 类型事件始终全量上报，不受采样率和限流影响（仅受队列大小限制）。

### 11.4 服务端事件去重

**问题**：客户端可能因网络超时重试导致同一事件重复写入。

**优化**：客户端为每个事件生成 `eventId`，服务端做幂等去重。

```typescript
// 客户端
import { randomUUID } from 'node:crypto'

export interface ObservabilityEventItem {
  eventId: string          // 事件唯一标识，用于服务端去重
  type: ObservabilityEventType
  // ... 其余字段 ...
}

// 服务端（observability.service.ts）
async function writeEvents(events: ObservabilityEventItem[]): Promise<{ received: number; duplicated: number }> {
  const existingIds = await prisma.observabilityEvent.findMany({
    where: { eventId: { in: events.map(e => e.eventId).filter(Boolean) } },
    select: { eventId: true },
  })
  const existingSet = new Set(existingIds.map(e => e.eventId))
  const newEvents = events.filter(e => !existingSet.has(e.eventId))

  if (newEvents.length > 0) {
    await prisma.observabilityEvent.createMany({ data: newEvents.map(transformEvent) })
  }

  return { received: events.length, duplicated: events.length - newEvents.length }
}
```

> **注**：`eventId` 字段设为 `UNIQUE` 索引，实际实现中可使用 `INSERT IGNORE` / `ON DUPLICATE KEY UPDATE` 实现高效去重。
