## 二、目标行为

- 登录后调用 `GET /workmate/models` → 拿到 `apiKey` + `models[]`
- 写入本地 `~/.proma/models-cache.json`（apiKey 用 `safeStorage` 加密）
- 应用启动时优先读取缓存；缓存命中即直接返回
- 按 TTL（默认 1 小时）后台静默刷新
- 未登录 / 未配置 serverUrl：返回空结果，**不抛异常**
- 登出时清除缓存

## 三、存储设计

### 3.1 落盘路径

- 路径：`~/.proma/models-cache.json`
- 命名空间：与 `auth.json`、`workmate.json` 同级，不侵入 Proma 上游

### 3.2 文件结构

```typescript
interface ModelsCache {
  /** safeStorage 加密后的 API Key（base64） */
  encryptedApiKey?: string
  /** safeStorage 不可用时的明文 Key（不推荐，仅 fallback） */
  apiKey?: string
  /** 可用模型列表（仅保留 enabled=true） */
  models: ModelInfo[]
  /** 模型总数 */
  total: number
  /** 缓存写入时间（Unix 毫秒），用于 TTL 判断 */
  cachedAt: number
  /** 缓存关联的工号，账号切换时缓存失效 */
  jobId?: string
}
```

### 3.3 关键设计点

1. **API Key 必须加密落盘**：与 `auth.json` 中 `encryptedToken` 同样的 `safeStorage` 模式
2. **缓存与 jobId 绑定**：账号切换后旧缓存自动失效，避免 Key 串号
3. **模型列表明文落盘**：模型元数据（id/name/description）无敏感性，可明文存储
4. **失败降级**：网络/服务端失败时返回旧缓存，确保离线场景下仍能用已缓存的 Key 调用大模型 API

## 四、缓存策略

### 4.1 TTL 判定

```
if (缓存不存在)                → 未命中
else if (Date.now() - cachedAt > TTL) → 过期失效
else if (缓存.jobId !== 当前 jobId)   → 账号不一致失效
else                                  → 命中
```

### 4.2 TTL 默认值

- 默认 1 小时（3_600_000 ms）
- 可在 `~/.proma/workmate.json` 中通过 `modelsCacheTtlMs` 自定义

### 4.3 三级缓存

```
┌─────────────────────────────────────────────┐
│  L1 内存缓存（进程内变量，启动时从 L2 加载）  │
├─────────────────────────────────────────────┤
│  L2 磁盘缓存（~/.proma/models-cache.json）  │
├─────────────────────────────────────────────┤
│  L3 服务端（GET /workmate/models）          │
└─────────────────────────────────────────────┘
```

读取顺序：L1 → L2 → L3
写入顺序：L3 命中 → 同时写 L1 + L2

## 五、API 设计（不实现，仅签名）

### 5.1 核心函数

```typescript
// 拉取并落盘（带缓存）
export async function fetchUserModels(forceRefresh?: boolean): Promise<ModelListResponse>

// 同步读缓存中的 API Key（不发起网络）
export async function getApiKey(): Promise<string | null>

// 同步读缓存中的模型列表（不发起网络）
export function getModels(): ModelInfo[]

// 清除缓存（登出时调用）
export function clearCache(): void

// 启动定期刷新（app ready 后调用一次）
export function initModelRefresh(intervalMs?: number): void

// 停止定期刷新
export function shutdownModelRefresh(): void
```

### 5.2 `fetchUserModels` 行为矩阵

| 状态 | 行为 |
|------|------|
| 缓存命中 + 账号一致 | 直接返回缓存（不发请求） |
| 缓存过期 | 发起网络请求，失败时降级为旧缓存 |
| `workmate.serverUrl` 未配置 | 返回 `{ apiKey: '', models: [], total: 0 }` |
| 未登录（无 token） | 返回 `{ apiKey: '', models: [], total: 0 }` |
| 网络失败 / 服务端错误 | 降级为旧缓存，无旧缓存则返回空 |
| 强制刷新（`forceRefresh=true`） | 跳过缓存检查直接请求网络 |

**关键约束**：所有失败路径都不抛异常。

## 六、定期刷新机制

### 6.1 启动入口

- 位置：`workmate-init.ts`（独立模块，详见审查报告策略五）
- 调用时机：`app.on('ready')` 之后
- 调用方式：`initModelRefresh(intervalMs)` 一次

### 6.2 刷新行为

- 静默后台刷新，**不阻塞主流程**
- 刷新失败时仅 console.warn，不弹窗
- 应用退出前调用 `shutdownModelRefresh()` 清理定时器

### 6.3 触发场景

| 场景 | 触发 |
|------|------|
| 定期刷新 | TTL 到期自动触发 |
| 登录成功 | 立即触发一次刷新 |
| 登出 | 清除缓存，**不**触发刷新 |
| 启动时 | 启动定时器，按 TTL 触发 |

## 十、未处理项说明

### 问题 2：服务端模型平台接口缺 `apiKey` 字段

**当前现状**：
- `workmate-server/src/services/model-platform.service.ts` 请求 `/users/{userId}/models`
- 返回结构仅含 `models` 数组，无 `apiKey`
- 客户端 `model-service.ts` 期望服务端响应形如 `{ code, message, data: { apiKey, models, total } }`

**未修复原因**：用户指示"暂时不修复"。

**对客户端的影响**：
- `json.data.apiKey` 将为 `undefined`
- `apiKey` 落盘为空，Chat 时调用大模型 API 会失败
- 模型列表仍能正常拉取

**后续修复路径**（不在本次范围）：
- 服务端 `model-platform.service.ts` 新增 `getUserCredentials(userId)` 请求 `/users/{userId}/credentials`
- 控制器返回 `apiKey + models + total`
- 客户端无需改动
