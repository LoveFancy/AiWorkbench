## 一、客户端-登录.md

### 1.1 逻辑问题

| # | 问题                                | 严重程度 | 说明                                                                                                                                                                                                                        |
| - | --------------------------------- | ---- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1 | `safeStorage` 调用时机风险              | 高    | `saveToken()` 使用了 `safeStorage.encryptString()`，但 `safeStorage` 必须在 app `ready` 之后才能使用。文档代码片段没有体现这个约束。实际项目中 `index.ts` 已特殊处理了 `safeStorage` 的 `userData` 路径问题（沿用 `HtAiWorkBench` 目录名），说明这块很敏感。实现时必须确保登录流程在 app ready 后执行。 |
| 2 | `login()` 对 HTTP 302 重定向处理不当      | 中    | EIP 网关登录可能返回 302 重定向。文档代码用了 `redirect: 'manual'`，但只判断了 `status !== 200`，302 的情况被当成了失败。应该同时处理 302 + Set-Cookie 的情况。                                                                                                        |
| 3 | `parseJobId()` 从 JWT 解析 mid 的容错不足 | 高    | 如果短期 Token 和长期 Token 的 payload 结构不同（比如长期 Token 可能不包含 `mid` 字段），则 `isLoggedIn()` 调用 `parseJobId(getToken())` 在长期 Token 场景下可能误判为未登录。                                                                                        |
| 4 | `logout()` 实现不够清晰                 | 低    | `logout()` 执行 `writeFileSync(getAuthFilePath(), JSON.stringify({}))`，`getToken()` 读取空对象时依赖 `Date.now() > undefined` 返回 false 的隐式行为。逻辑上能工作但不清晰，建议直接删除文件或写入 `{ loggedOut: true }`。                                          |
| 5 | 缺少 Token 过期后的提醒/续期机制              | 中    | 长期 Token（365天）过期后，用户只能手动重新登录。文档没有设计任何过期提醒或自动续期逻辑。                                                                                                                                                                         |

## 三、客户端-上报（观测与异常）.md

### 3.1 逻辑问题

| # | 问题                                      | 严重程度 | 说明                                                                                                                                                                                 |
| - | --------------------------------------- | ---- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1 | 采样/限流逻辑缺失实现                             | 高    | 文档定义了 `sampleRate` 和 `maxEventsPerMinute` 配置，架构图也画了"采样 & 限流层"，但 `reportEvent()` 代码中没有体现——直接 `eventQueue.push(event)`，**关键逻辑缺失**。                                                   |
| 2 | `flushQueue()` 失败时数据丢失                  | 高    | `const batch = eventQueue.splice(0, ...)` 在上报前就移除了事件，如果上报失败这批事件只打了 `console.warn` 就丢了。文档架构图说"失败时写回队列或本地磁盘缓存"，但代码没有回写逻辑。                                                            |
| 3 | `reportErrorEvent` 的 `context` 参数与类型不兼容 | 中    | 代码 `reportEvent({...context, ...})` 使用 spread 传入 `context: Record<string, unknown>`，但 `ObservabilityEventItem` 字段是固定的，spread 会导致类型不安全的字段混入。应该把 context 放到专门的 `tags` 或 `extra` 字段中。 |
| 4 | `getClientInfo()` 未定义                   | 低    | 代码中多处调用 `getClientInfo()` 但没有给出实现。                                                                                                                                                 |
| 5 | 主进程全局异常捕获可能与 Electron 冲突                | 高    | Electron 官方建议 `uncaughtException` 不要仅做日志/上报后继续运行，因为可能导致应用状态不一致。需要加入"是否为致命错误 → 决定是否退出进程"的判断。                                                                                        |
| 6 | 渲染进程 ErrorBoundary 已存在                  | 中    | 项目中已有 `TabErrorBoundary.tsx`，文档要新建 `ErrorBoundary.tsx`。应该扩展现有的而非新建。                                                                                                                |
| 7 | 面包屑对上报 payload 大小的影响                    | 低    | 目前限制 20 条问题不大，但如果未来面包屑携带 `data` 字段（如 HTTP 请求体），单个事件可能很大。建议加总大小限制。                                                                                                                  |

### 3.2 解决方案

```
问题 1 → 补充采样/限流实现：
  let eventCountThisMinute = 0
  let minuteStart = Date.now()

  export function reportEvent(item): void {
    if (!config?.enabled) return

    // error 事件直通
    if (item.type !== 'error') {
      // 采样率检查
      if (Math.random() > (config.sampleRate ?? 1.0)) return
      // 每分钟限流
      const now = Date.now()
      if (now - minuteStart > 60_000) {
        eventCountThisMinute = 0
        minuteStart = now
      }
      if (eventCountThisMinute >= (config.maxEventsPerMinute ?? 60)) return
      eventCountThisMinute++
    }

    eventQueue.push({ ...item, eventId: randomUUID() })
    // ...
  }

问题 2 → flushQueue 失败时回写队列：
  export async function flushQueue(): Promise<void> {
    if (eventQueue.length === 0) return
    const batch = eventQueue.splice(0, config?.maxBatchSize ?? 50)

    try {
      // ... fetch 逻辑
    } catch (error) {
      console.warn('[观测上报] 上报失败:', error)
      // 回写到队列头部

问题 3 → context 放入专门的 tags 字段：

      type: 'error',
      // ... 其他字段

    })
  }

问题 4 → 补充 getClientInfo 实现：
  import { app } from 'electron'
  import { platform, release } from 'node:os'

    return {
      appVersion: app.getVersion(),

  }

问题 5 → uncaughtException 中加入致命错误判断：
  process.on('uncaughtException', (error) => {
    console.error('[Global Error] 未捕获异常:', error)

  function isFatalError(error: Error): boolean {
    // 根据错误类型判断，如 FATAL_ERROR_TYPES 列表

  }

问题 6 → 扩展现有 TabErrorBoundary：
  // 在 TabErrorBoundary.componentDidCatch 中添加上报逻辑
  // 而非新建 ErrorBoundary.tsx
  override componentDidCatch(error: unknown, info: React.ErrorInfo): void {
    console.error('[TabErrorBoundary] 渲染异常:', error, info.componentStack)
    window.electronAPI?.reportError?.({
      name: error instanceof Error ? error.name : 'UnknownError',

      stack: error instanceof Error ? error.stack : undefined,
```

## 四、客户端-升级检测.md

### 4.1 逻辑问题

| # | 问题                         | 严重程度   | 说明                                                                                                                                                                                                                                        |
| - | -------------------------- | ------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1 | 与现有 `auto-updater.ts` 功能冲突 | **严重** | 项目中已有 `updater/auto-updater.ts`，使用 `electron-updater` 做 GitHub Release 自动更新（4小时检测一次，自动下载）。文档设计的 `upgrade-check-service.ts` 是通过服务端 API 检测升级。**两个升级系统会同时运行，弹出两个升级通知**。                                                                      |
| 2 | 服务端响应字段与客户端类型不匹配           | **严重** | 服务端 `UpgradeCheckResponse` 字段是 `hasUpdate`、`latestVersion`、`forceUpdate`；文档客户端 `UpgradeCheckResult` 字段是 `hasUpgrade`、`targetVersion`、`upgradeAllowed`、`releaseType`。命名完全不一致，客户端解析 `json.data.hasUpgrade` 但服务端返回 `hasUpdate`，**升级检测永远失败**。 |
| 3 | URL 拼接问题                   | 中      | `settings.serverUrl` 末尾是否带 `/` 或是否已含 `/workmate` 前缀，文档没有规范，容易拼接出错误 URL。                                                                                                                                                                   |
| 4 | 缺少定时检测的完整实现                | 中      | 文档提到"每 N 小时重复检测"和 `scheduleNextCheck()`，但没有给出实现代码和调用时机。                                                                                                                                                                                   |

### 5.2 推荐的降低合并风险策略

#### 策略一：WorkMate 配置独立存储

用 `~/.proma/workmate.json` 代替在 `AppSettings` 里加字段：

```typescript
// 新建 apps/electron/src/main/lib/workmate-config-service.ts
import { readFileSync, writeFileSync, existsSync } from 'node:fs'
import { join } from 'node:path'
import { getConfigDir } from './config-paths'

interface WorkmateConfig {
  serverUrl?: string
  observabilityEnabled?: boolean
  upgradeCheckIntervalHours?: number
  lastUpgradeCheckAt?: number
}

const CONFIG_FILE = 'workmate.json'

function getConfigPath(): string {
  return join(getConfigDir(), CONFIG_FILE)
}

export function getWorkmateConfig(): WorkmateConfig {
  const filePath = getConfigPath()
  if (!existsSync(filePath)) return {}
  try {
    return JSON.parse(readFileSync(filePath, 'utf-8'))
  } catch { return {} }
}

export function updateWorkmateConfig(updates: Partial<WorkmateConfig>): WorkmateConfig {
  const current = getWorkmateConfig()
  const updated = { ...current, ...updates }
  writeFileSync(getConfigPath(), JSON.stringify(updated, null, 2), 'utf-8')
  return updated
}
```

**好处**：不碰 `settings.ts`，不碰 `AppSettings` 类型，Proma 上游随便改都不冲突。

#### 策略四：不修改 Proma 核心文件，用事件监听接入观测上报

```typescript
// 在 workmate-init.ts 中
import { ipcMain } from 'electron'
import { reportErrorEvent } from './observability-service'

export function initWorkmateServices(mainWindow: BrowserWindow): void {
  // 1. 全局错误捕获
  process.on('uncaughtException', (error) => {
    reportErrorEvent(error, { context: 'uncaughtException' })
  })
  process.on('unhandledRejection', (reason) => {
    const error = reason instanceof Error ? reason : new Error(String(reason))
    reportErrorEvent(error, { context: 'unhandledRejection' })
  })

  // 2. 监听 IPC 错误事件（不修改 chat-service / agent-orchestrator 源码）
  ipcMain.on('workmate:report-error', (_event, errorData) => {
    const error = new Error(errorData.message)
    error.name = errorData.name
    error.stack = errorData.stack
    reportErrorEvent(error, errorData.context)
  })

  // 3. 升级检测
  initUpgradeCheckService(mainWindow)

  // 4. 观测上报初始化
  initObservabilityService()
}
```

在 `TabErrorBoundary.tsx` 中（只需改一个方法）：

```typescript
override componentDidCatch(error: unknown, info: React.ErrorInfo): void {
```

## 六、服务端字段对齐问题（跨文档）

| 客户端字段                               | 服务端字段                               | 状态              | 建议                                                                         |
| ----------------------------------- | ----------------------------------- | --------------- | -------------------------------------------------------------------------- |
| `hasUpgrade`                        | `hasUpdate`                         | **不一致**         | 以服务端为准，客户端改为 `hasUpdate`                                                   |
| `targetVersion`                     | `latestVersion`                     | **不一致**         | 以服务端为准，客户端改为 `latestVersion`                                               |
| `upgradeAllowed`                    | `forceUpdate`（语义不同）                 | **不一致**         | `upgradeAllowed` = 白名单内允许升级，`forceUpdate` = 是否强制升级。两者语义不同，需要客户端类型同时包含这两个字段 |
